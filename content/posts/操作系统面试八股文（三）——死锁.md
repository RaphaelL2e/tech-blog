## 死锁的基本概念

### [](#死锁的定义)死锁的定义

**死锁（Deadlock）** 是指两个或两个以上的进程在执行过程中，因争夺资源而造成的一种**互相等待**的僵局，若无外力作用，它们都将无法继续推进。

> **面试高频问**：什么是死锁？举个生活中的例子？
> 例子：过独木桥——两个人从两端同时上桥，各占一半桥面，互不让步，谁也过不去。

### [](#死锁与饥饿的区别)死锁与饥饿的区别

| 维度 | 死锁 | 饥饿 |
|------|------|------|
| 本质 | 进程互相等待，都无法推进 | 进程长期得不到资源，无法执行 |
| 进程状态 | 阻塞态 | 就绪态（可能长期在就绪队列） |
| 资源 | 占有资源并等待其他资源 | 一直得不到所需资源 |
| 解决 | 需要外部干预 | 可能自行恢复（如优先级动态调整） |

> **注意**：死锁是"大家都动不了"，饥饿是"有人一直得不到机会"。饥饿不是死锁。

### [](#死锁与死循环的区别)死锁与死循环的区别

| 维度 | 死锁 | 死循环 |
|------|------|------|
| 产生原因 | 资源竞争与等待 | 代码逻辑错误 |
| 进程状态 | 阻塞态（等待资源） | 运行态（一直占用CPU） |
| 解决方式 | 外部干预 | 修复代码 |

---

## 死锁产生的四个必要条件（⭐⭐⭐⭐⭐）

死锁的发生必须**同时满足**以下四个条件，只要有一个不满足，就不会发生死锁：

| 条件 | 说明 | 举例 |
|------|------|------|
| **互斥条件** | 资源一次只能被一个进程使用 | 打印机同一时间只能被一个进程使用 |
| **不可剥夺条件** | 进程已获得的资源不能被强行剥夺，只能主动释放 | 进程拿到锁后，其他人不能抢 |
| **请求并保持条件** | 进程已持有资源，又请求新资源而被阻塞 | 拿着左筷子，又去拿右筷子 |
| **循环等待条件** | 存在进程的循环等待链 | P1等P2的资源，P2等P3的资源，P3等P1的资源 |

> **面试必考**：死锁的四个必要条件是什么？破坏其中任何一个就能预防死锁。

---

## 死锁的处理策略

处理死锁有四种策略，按"从严到宽"排列：

| 策略 | 思想 | 优点 | 缺点 |
|------|------|------|------|
| **预防** | 破坏四个必要条件之一 | 简单、死锁不会发生 | 资源利用率低、系统吞吐量下降 |
| **避免** | 动态检测资源分配是否安全 | 资源利用率较高 | 需要事先知道进程的资源需求 |
| **检测+解除** | 允许死锁发生，检测后解除 | 资源利用率最高 | 检测和解除的开销大 |
| **鸵鸟策略** | 忽略死锁 | 零开销 | 死锁发生时系统可能崩溃 |

> 实际操作系统中，大多数采用**鸵鸟策略**（如Linux、Windows），因为死锁概率低，处理开销大。

## 死锁预防

死锁预防是**静态**策略，在死锁发生之前，通过破坏四个必要条件之一来防止死锁。

### [](#破坏互斥条件)破坏互斥条件

将独占资源改为共享资源（如打印机改为Spooling技术）。

**问题**：很多资源天生就是互斥的（如打印机、临界区），无法改为共享。**此方法不实用。**

### [](#破坏不可剥夺条件)破坏不可剥夺条件

当进程请求新资源失败时：
- 方案1：释放已持有的所有资源，以后再重新申请
- 方案2：由系统强制剥夺（优先级高的进程可以抢占低优先级进程的资源）

**问题**：实现复杂；可能导致进程之前的工作白做；容易造成饥饿。

### [](#破坏请求并保持条件)破坏请求并保持条件

要求进程在运行前**一次性申请所有需要的资源**。

**问题**：
- 资源利用率低（有些资源可能很晚才用，但一开始就占了）
- 可能导致饥饿（长时间申请不到所有资源的进程一直等待）
- 编程困难（程序需要提前知道所有需要的资源）

### [](#破坏循环等待条件)破坏循环等待条件（最常用）

给所有资源**编号排序**，规定每个进程必须按编号递增的顺序请求资源。

即：如果进程已持有编号为i的资源，则只能请求编号大于i的资源。

**为什么有效**：按递增顺序请求，不可能形成循环等待链（因为循环意味着某个进程持有小编号资源却请求更小编号的资源，违反了递增规则）。

**问题**：
- 资源编号需要全局统一，增加系统设计复杂度
- 实际使用顺序可能和编号顺序不一致，导致先申请暂时不需要的资源

> **面试真题**：哪种预防方法最实用？
> 破坏循环等待条件最实用，因为它只限制了资源申请的顺序，不需要修改进程的运行逻辑。但要注意，它仍然可能导致资源浪费。

## 死锁避免

死锁避免是**动态**策略，在资源分配时判断是否可能导致死锁，只在不导致死锁时才分配。

### [](#安全状态与不安全状态)安全状态与不安全状态

- **安全状态**：系统能按某种顺序为每个进程分配资源，使每个进程都能顺利完成
- **不安全状态**：不存在这样的安全序列
- **重要**：**不安全状态不一定是死锁**，但死锁一定处于不安全状态

> 安全状态 → 一定不会死锁
> 不安全状态 → 可能死锁（也可能不死锁）
> 死锁 → 一定是不安全状态

**安全序列**：如果系统存在一个进程序列 {P1, P2, ..., Pn}，使得每个进程Pi还需要的资源可以通过当前可用资源 + 前面所有进程释放的资源来满足，则该序列为安全序列。

### [](#银行家算法)银行家算法（⭐⭐⭐⭐⭐）

银行家算法是死锁避免的**经典算法**，由Dijkstra提出。

#### 数据结构

| 数据结构 | 含义 |
|---------|------|
| Available[m] | 每类资源的可用数量（m为资源种类数） |
| Max[n][m] | 每个进程对每类资源的最大需求 |
| Allocation[n][m] | 每个进程已分配的每类资源数量 |
| Need[n][m] | 每个进程还需要的每类资源数量（Need = Max − Allocation） |

#### 安全性算法

1. 初始化：Work = Available，Finish[i] = false（对所有i）
2. 查找：找一个满足 Finish[i] = false 且 Need[i] ≤ Work 的进程Pi
3. 执行：Work = Work + Allocation[i]，Finish[i] = true，回到步骤2
4. 判断：如果所有 Finish[i] = true，则系统处于安全状态；否则不安全

#### 银行家算法流程

当进程Pi请求资源Request[i]时：

1. 若 Request[i] > Need[i]，出错（请求超过最大需求）
2. 若 Request[i] > Available，Pi等待
3. 试探性分配：
   - Available = Available − Request[i]
   - Allocation[i] = Allocation[i] + Request[i]
   - Need[i] = Need[i] − Request[i]
4. 执行安全性算法检查
5. 若安全 → 正式分配；若不安全 → 撤回试探性分配，Pi等待

#### 银行家算法推演实例

设系统有3类资源A、B、C，数量分别为(10, 5, 7)，有5个进程P0~P4：

| 进程 | Max | Allocation | Need |
|------|-----|-----------|------|
| P0 | (7,5,3) | (0,1,0) | (7,4,3) |
| P1 | (3,2,2) | (2,0,0) | (1,2,2) |
| P2 | (9,0,2) | (3,0,2) | (6,0,0) |
| P3 | (2,2,2) | (2,1,1) | (0,1,1) |
| P4 | (4,3,3) | (0,0,2) | (4,3,1) |

Available = (10,5,7) − (0+2+3+2+0, 1+0+0+1+0, 0+0+2+1+2) = **(3,3,2)**

**求安全序列**：

1. Work = (3,3,2)，查找 Need ≤ Work 的进程：
   - P1 Need=(1,2,2) ≤ (3,3,2) ✅ → Work = (3,3,2)+(2,0,0) = (5,3,2)，Finish[P1]=true
2. Work = (5,3,2)：
   - P3 Need=(0,1,1) ≤ (5,3,2) ✅ → Work = (5,3,2)+(2,1,1) = (7,4,3)，Finish[P3]=true
3. Work = (7,4,3)：
   - P0 Need=(7,4,3) ≤ (7,4,3) ✅ → Work = (7,4,3)+(0,1,0) = (7,5,3)，Finish[P0]=true
4. Work = (7,5,3)：
   - P2 Need=(6,0,0) ≤ (7,5,3) ✅ → Work = (7,5,3)+(3,0,2) = (10,5,5)，Finish[P2]=true
5. Work = (10,5,5)：
   - P4 Need=(4,3,1) ≤ (10,5,5) ✅ → Work = (10,5,5)+(0,0,2) = (10,5,7)，Finish[P4]=true

**安全序列：P1 → P3 → P0 → P2 → P4**，系统安全。

> **面试必考**：银行家算法的核心就是——每次分配资源前，先用安全性算法检查分配后系统是否仍处于安全状态。如果安全则分配，否则拒绝。

## 死锁检测与解除

### [](#死锁检测)死锁检测

允许死锁发生，系统定期运行死锁检测算法。

**资源分配图法**：

- 用**圆圈**表示进程，**方框**表示资源
- 进程到资源的**请求边**：进程正在等待该资源
- 资源到进程的**分配边**：资源已分配给该进程

**死锁判定规则**：
1. 如果资源分配图中**没有环** → 一定没有死锁
2. 如果图中有环，且**每类资源只有一个实例** → 一定有死锁
3. 如果图中有环，但**每类资源有多个实例** → 可能有死锁（需要进一步分析）

**简化步骤**：
1. 找一个既不阻塞又非孤立的进程节点Pi，删除Pi的所有请求边和分配边
2. 重复步骤1
3. 如果能删除所有边 → 无死锁；否则剩余的进程就是死锁进程

### [](#死锁解除)死锁解除

一旦检测到死锁，采用以下方法解除：

| 方法 | 说明 | 适用场景 |
|------|------|---------|
| **资源剥夺法** | 挂起某些死锁进程，抢占其资源给其他进程 | 进程优先级差异大时 |
| **撤销进程法** | 强制撤销部分或全部死锁进程 | 紧急情况 |
| **进程回退法** | 让进程回退到足以避免死锁的检查点 | 有检查点机制的系统 |

**选择牺牲进程的考虑因素**：
- 进程优先级
- 已执行时间（执行少的优先牺牲）
- 已使用资源量（使用少的优先牺牲）
- 还需要多少资源
- 进程是交互式还是批处理式

> **注意**：被剥夺资源的进程可能导致"饥饿"（反复被选中作为牺牲者），需要确保每个进程不会永远被剥夺。

## 死锁策略对比总结

| 策略 | 时机 | 方法 | 开销 | 资源利用率 |
|------|------|------|------|-----------|
| 预防 | 死锁前 | 破坏必要条件 | 低 | 低 |
| 避免 | 资源分配时 | 银行家算法 | 中 | 中 |
| 检测+解除 | 死锁后 | 检测算法+撤销进程 | 高 | 高 |
| 鸵鸟策略 | — | 忽略 | 无 | 最高 |

## 经典面试题

### ⭐⭐⭐⭐⭐ 死锁的四个必要条件

互斥、不可剥夺、请求并保持、循环等待。缺一不可，破坏任何一个就能预防死锁。

### ⭐⭐⭐⭐⭐ 银行家算法的核心思想

在每次资源分配前，先进行**试探性分配**，然后运行**安全性算法**检查系统是否仍处于安全状态。如果安全则正式分配，否则拒绝分配、让进程等待。

### ⭐⭐⭐⭐ 如何预防死锁？实际中最常用哪种方法？

- 破坏四个必要条件之一
- 实际中最常用**破坏循环等待条件**（资源有序分配法），因为实现简单，对进程逻辑影响最小
- 编程中常用**破坏请求并保持条件**（一次性申请所有锁，如lock_guard）

### ⭐⭐⭐⭐ 安全状态和不安全状态的关系

- 安全状态 → 一定不会死锁
- 不安全状态 → **可能**死锁（不是必然死锁）
- 死锁 → 一定处于不安全状态
- 不安全状态 ≠ 死锁（可能碰巧没有死锁）

### ⭐⭐⭐ 代码中如何避免死锁？

1. **锁排序**：始终按固定顺序获取锁（破坏循环等待）
2. **一次获取所有锁**：使用tryLock获取所有锁，失败则全部释放重试（破坏请求并保持）
3. **设置超时**：使用带超时的锁获取操作
4. **减少锁粒度**：减少锁的持有时间和范围
