## 一、引言

微服务架构带来了灵活性，也带来了部署复杂性。从单体应用的一键部署，到微服务的几十个服务独立部署、滚动更新、蓝绿发布，部署策略变得至关重要。

本文将从微服务部署的挑战出发，深度解析容器化技术（Docker、Kubernetes）和CI/CD流水线，探讨生产环境中的部署最佳实践和面试高频问题。

### 单体 vs 微服务部署对比

```
单体应用部署
┌─────────────────────────────────┐
│        monolith.war             │
│  （所有功能打包在一个包）        │
└──────────────┬──────────────────┘
               │ 部署
               ▼
┌─────────────────────────────────┐
│     Tomcat/Jetty               │
│  （一个容器运行）               │
└─────────────────────────────────┘
优势：部署简单（一次打包，到处运行）
问题：修改一行代码，需要重新部署整个应用

微服务部署
┌────┐  ┌────┐  ┌────┐  ┌────┐
│Service A│  │Service B│  │Service C│  │Service D│
│ 独立部署  │  │ 独立部署  │  │ 独立部署  │  │ 独立部署  │
└────┘  └────┘  └────┘  └────┘
优势：独立部署、独立扩展、技术栈灵活
问题：部署复杂性指数级上升（几十个服务如何管理？）
```

## 二、Docker 深度解析

### 2.1 Docker 核心概念

| 概念 | 说明 | 类比 |
|------|------|------|
| **镜像（Image）** | 只读模板 | 类（Class） |
| **容器（Container）** | 运行中的镜像 | 对象（Object） |
| **仓库（Registry）** | 镜像存储中心 | Maven仓库 |
| **Dockerfile** | 镜像构建脚本 | Makefile |
| **Volume** | 数据持久化 | 硬盘 |
| **Network** | 容器网络 | 交换机 |

### 2.2 Dockerfile 深度解析

```dockerfile
# 多层构建（减小镜像体积）
# 第一阶段：构建
FROM maven:3.8-openjdk-17 AS builder
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

# 第二阶段：运行
FROM openjdk:17-jre-slim
WORKDIR /app

# 从构建阶段复制jar包
COPY --from=builder /app/target/*.jar app.jar

# 添加非root用户（安全最佳实践）
RUN addgroup --system appgroup && adduser --system --group appuser
USER appuser

# 健康检查
HEALTHCHECK --interval=30s --timeout=3s --start-period=60s \
  CMD curl -f http://localhost:8080/actuator/health || exit 1

# JVM参数优化
ENV JAVA_OPTS="-Xms256m -Xmx512m -XX:+UseG1GC"

EXPOSE 8080
ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]
```

**关键优化点**：

1. **多层构建**：构建环境和运行环境分离，减小镜像体积
2. **分层缓存**：先COPY pom.xml，再COPY src，利用Maven依赖缓存
3. **非root用户**：安全最佳实践
4. **健康检查**：Kubernetes存活探针依赖这个
5. **JVM参数**：通过环境变量注入，灵活调整

```
└───────────────────────────────────────────────┘

四种网络模式：
1. Bridge（桥接模式）：默认，容器通过docker0网桥通信
2. Host（主机模式）：容器直接使用主机网络（性能最好）
3. None（无网络）：容器没有网络（安全场景）
4. Container（容器模式）：多个容器共享网络命名空间
```

### 3.2 Pod 深度解析

```yaml
# Pod 定义
apiVersion: v1
kind: Pod
metadata:
  name: user-service-pod
  labels:
    app: user-service
    version: v1
spec:
  # 1. 容器定义
  containers:
  - name: user-service
    image: user-service:1.0.0
    imagePullPolicy: IfNotPresent
    
    # 端口
    ports:
    - containerPort: 8080
      name: http
    
    # 环境变量
    env:
    - name: SPRING_PROFILES_ACTIVE
      value: "prod"
    - name: MYSQL_HOST
      valueFrom:
        configMapKeyRef:
          name: app-config
          key: mysql.host
    
    # 资源限制（关键！）
    resources:
      requests:
        memory: "256Mi"
        cpu: "250m"
      limits:
        memory: "512Mi"
        cpu: "500m"
    
    # 健康检查（存活探针）
    livenessProbe:
      httpGet:
        path: /actuator/health
        port: 8080
      initialDelaySeconds: 60
      periodSeconds: 10
      timeoutSeconds: 5
      failureThreshold: 3
    
    # 健康检查（就绪探针）
    readinessProbe:
      httpGet:
        path: /actuator/health
        port: 8080
      initialDelaySeconds: 30
      periodSeconds: 5
      timeoutSeconds: 3
      failureThreshold: 3
    
    # 启动探针（保护慢启动容器）
    startupProbe:
      httpGet:
        path: /actuator/health
        port: 8080
      initialDelaySeconds: 10
      periodSeconds: 5
      timeoutSeconds: 3
      failureThreshold: 12
    
    # 挂载卷
    volumeMounts:
    - name: logs
      mountPath: /app/logs
    - name: config
      mountPath: /app/config
  
  # 2. 初始化容器（在主容器启动前执行）
  initContainers:
  - name: init-mysql
    image: mysql:8.0
    command: ["sh", "-c", "until mysqladmin ping -h mysql; do echo waiting for mysql; sleep 2; done"]
  
  # 3. 卷定义
  volumes:
  - name: logs
    emptyDir: {}
  - name: config
    configMap:
      name: app-config
  
  # 4. 服务账户
  serviceAccountName: user-service-account
  
  # 5. 重启策略
  restartPolicy: Always
  
  # 6. 调度约束
  nodeSelector:
    zone: prod-a
  affinity:
    podAntiAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - user-service
        topologyKey: kubernetes.io/hostname
```

### 3.3 Deployment 深度解析

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service
  labels:
    app: user-service
spec:
  # 1. 副本数
  replicas: 3
  
  # 2. 选择器
  selector:
    matchLabels:
      app: user-service
  
  # 3. 滚动更新策略
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 最多多1个Pod
      maxUnavailable: 0   # 最少0个Pod不可用（保证服务不中断）
  
  # 4. 修订历史保留数
  revisionHistoryLimit: 10
  
  # 5.  Pod模板
  template:
    metadata:
      labels:
        app: user-service
        version: v1
    spec:
      containers:
      - name: user-service
        image: user-service:1.0.0
        ports:
        - containerPort: 8080
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"
        livenessProbe:
          httpGet:
            path: /actuator/health
            port: 8080
        readinessProbe:
          httpGet:
            path: /actuator/health
            port: 8080
```

### 3.4 Service 深度解析

```yaml
# Service（服务发现）
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  # 1. 服务类型
  type: ClusterIP  # 可选：ClusterIP、NodePort、LoadBalancer、ExternalName
  
  # 2. 选择器（选择哪些Pod）
  selector:
    app: user-service
  
  # 3. 端口映射
  ports:
  - name: http
    port: 80          # Service端口
    targetPort: 8080   # Pod端口
    protocol: TCP
  
  # 4. Session亲和性（可选）
  sessionAffinity: ClientIP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 10800
```

**Service 类型对比**：

| 类型 | 说明 | 适用场景 |
|------|------|----------|
| **ClusterIP** | 集群内部访问 | 微服务间调用 |
| **NodePort** | 暴露到节点端口 | 测试环境、 legacy 系统 |
| **LoadBalancer** | 云平台负载均衡器 | 生产环境对外服务 |
| **ExternalName** | 映射到外部域名 | 访问外部服务 |

### 4.2 Jenkins Pipeline 深度解析

```groovy
// Jenkinsfile（声明式Pipeline）
pipeline {
    // 1. 代理（运行环境）
    agent {
        kubernetes {
            yaml '''
            apiVersion: v1
            kind: Pod
            spec:
              containers:
              - name: maven
                image: maven:3.8-openjdk-17
                command: ['cat']
                tty: true
              - name: docker
                image: docker:20-dind
                command: ['cat']
                tty: true
                securityContext:
                  privileged: true
            '''
        }
    }
    
    // 2. 环境变量
    environment {
        DOCKER_REGISTRY = 'registry.example.com'
        IMAGE_NAME = 'user-service'
        KUBECONFIG = credentials('kubeconfig')
    }
    
    // 3. 流水线阶段
    stages {
        // 阶段1：拉取代码
        stage('Checkout') {
            steps {
                checkout scm
            }
        }
        
        // 阶段2：编译构建
        stage('Build') {
            steps {
                container('maven') {
                    sh 'mvn clean package -DskipTests'
                }
            }
        }
        
        // 阶段3：单元测试
        stage('Test') {
            steps {
                container('maven') {
                    sh 'mvn test'
                }
            }
            post {
                always {
                    junit 'target/surefire-reports/*.xml'
                }
            }
        }
        
        // 阶段4：代码质量检查
        stage('SonarQube Analysis') {
            steps {
                container('maven') {
                    withSonarQubeEnv('SonarQube') {
                        sh 'mvn sonar:sonar'
                    }
                }
            }
        }
        
        // 阶段5：构建Docker镜像
        stage('Build Docker Image') {
            steps {
                container('docker') {
                    script {
                        def imageTag = "${BUILD_NUMBER}"
                        sh "docker build -t ${DOCKER_REGISTRY}/${IMAGE_NAME}:${imageTag} ."
                        sh "docker push ${DOCKER_REGISTRY}/${IMAGE_NAME}:${imageTag}"
                    }
                }
            }
        }
        
        // 阶段6：部署到测试环境
        stage('Deploy to Test') {
            steps {
                script {
                    sh "kubectl set image deployment/user-service user-service=${DOCKER_REGISTRY}/${IMAGE_NAME}:${BUILD_NUMBER} --namespace=test"
                    sh "kubectl rollout status deployment/user-service --namespace=test --timeout=300s"
                }
            }
        }
        
        // 阶段7：集成测试
        stage('Integration Test') {
            steps {
                container('maven') {
                    sh 'mvn verify -Pintegration-test'
                }
            }
        }
        
        // 阶段8：人工审批（部署生产环境前）
        stage('Approval for Production') {
            steps {
                input message: 'Deploy to Production?', ok: 'Yes'
            }
        }
        
        // 阶段9：部署到生产环境
        stage('Deploy to Production') {
            steps {
                script {
                    sh "kubectl set image deployment/user-service user-service=${DOCKER_REGISTRY}/${IMAGE_NAME}:${BUILD_NUMBER} --namespace=prod"
                    sh "kubectl rollout status deployment/user-service --namespace=prod --timeout=300s"
                }
            }
        }
    }
    
    // 4. 构建后操作
    post {
        success {
            slackSend(color: 'good', message: "构建成功: ${BUILD_TAG}")
        }
        failure {
            slackSend(color: 'danger', message: "构建失败: ${BUILD_TAG}")
            mail to: 'team@example.com', subject: "构建失败", body: "请检查: ${BUILD_URL}"
        }
    }
}
```

```yaml
name: Deploy to Kubernetes

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

env:
  DOCKER_REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # Job 1：构建和测试
  build:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v3
    
    - name: Set up JDK 17
      uses: actions/setup-java@v3
      with:
        java-version: '17'
        distribution: 'temurin'
        cache: maven
    
    - name: Build with Maven
      run: mvn clean package -DskipTests
    
    - name: Run tests
      run: mvn test
    
    - name: Upload test results
      if: always()
      uses: actions/upload-artifact@v3
      with:
        name: test-results
        path: target/surefire-reports/
  
  # Job 2：构建Docker镜像
  docker:
    needs: build
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
    - uses: actions/checkout@v3
    
    - name: Log in to GitHub Container Registry
      uses: docker/login-action@v2
      with:
        registry: ${{ env.DOCKER_REGISTRY }}
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}
    
    - name: Build and push Docker image
      uses: docker/build-push-action@v4
      with:
        context: .
        push: true
        tags: |
          ${{ env.DOCKER_REGISTRY }}/${{ env.IMAGE_NAME }}:latest
          ${{ env.DOCKER_REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
  
  # Job 3：部署到Kubernetes
  deploy:
    needs: docker
    runs-on: ubuntu-latest
    environment: production
    steps:
    - name: Checkout
      uses: actions/checkout@v3
    
    - name: Set up kubectl
      uses: azure/setup-kubectl@v3
      with:
        version: 'v1.28.0'
    
    - name: Set KUBECONFIG
      uses: azure/k8s-set-context@v3
      with:
        method: kubeconfig
        kubeconfig: ${{ secrets.KUBECONFIG }}
    
    - name: Deploy to Kubernetes
      run: |
        kubectl set image deployment/user-service user-service=${{ env.DOCKER_REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }} --namespace=prod
        kubectl rollout status deployment/user-service --namespace=prod --timeout=300s
    
    - name: Verify deployment
      run: |
        kubectl get pods -n prod -l app=user-service
```

## 五、部署策略深度解析

### 5.1 滚动更新（Rolling Update）

```
滚动更新策略
初始状态（3个Pod，版本v1）
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Pod v1  │  │ Pod v1  │  │ Pod v1  │
└─────────┘  └─────────┘  └─────────┘

更新过程（逐个替换）
Step 1: 创建1个v2，删除1个v1
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Pod v2  │  │ Pod v1  │  │ Pod v1  │
└─────────┘  └─────────┘  └─────────┘

Step 2: 创建1个v2，删除1个v1
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Pod v2  │  │ Pod v2  │  │ Pod v1  │
└─────────┘  └─────────┘  └─────────┘

Step 3: 创建1个v2，删除1个v1
┌─────────┐  ┌─────────┐  ┌─────────┐
│ Pod v2  │  │ Pod v2  │  │ Pod v2  │
└─────────┘  └─────────┘  └─────────┘

优势：零停机、渐进式更新
问题：更新过程慢、两个版本共存可能不兼容
```

### 5.2 蓝绿部署（Blue-Green Deployment）

```
蓝绿部署策略
┌─────────┐
│  Client  │
└────┬─────┘
     │
     ▼
┌──────────────┐
│  Load       │
│  Balancer   │
└────┬─────────┘
     │
     ├────────────► 蓝环境（v1，当前生产）
     │             ┌─────────┐  ┌─────────┐
     │             │ Pod v1  │  │ Pod v1  │
     │             └─────────┘  └─────────┘
     │
     └────────────► 绿环境（v2，新版本）
                   ┌─────────┐  ┌─────────┐
                   │ Pod v2  │  │ Pod v2  │
                   └─────────┘  └─────────┘

部署过程：
1. 在绿环境部署v2
2. 测试绿环境（内部测试）
3. 切换Load Balancer到绿环境（瞬间切换）
4. 蓝环境保留（快速回滚）

优势：瞬间切换、快速回滚
问题：需要双倍资源
```

### 5.3 金丝雀发布（Canary Release）

```
金丝雀发布策略
┌─────────┐
│  Client  │
└────┬─────┘
     │
     ▼
┌──────────────┐
│  Ingress     │
│  Controller  │
└────┬─────────┘
     │
     ├────────────► 生产环境（v1，95%流量）
     │             ┌─────────┐  ┌─────────┐  ┌─────────┐
     │             │ Pod v1  │  │ Pod v1  │  │ Pod v1  │
     │             └─────────┘  └─────────┘  └─────────┘
     │
     └────────────► 金丝雀环境（v2，5%流量）
                   ┌─────────┐
                   │ Pod v2  │
                   └─────────┘

部署过程：
1. 部署v2到金丝雀环境
2. 将5%流量路由到v2
3. 监控v2的性能和错误率
4. 如果OK，逐步增加流量（10% → 50% → 100%）
5. 如果失败，快速回滚（0%流量）

优势：渐进式发布、风险可控
问题：需要流量分割能力（Istio/Linkerd）
```

### 5.4 部署策略对比

| 策略 | 零停机 | 快速回滚 | 资源开销 | 风险 | 适用场景 |
|------|--------|----------|----------|------|----------|
| **滚动更新** | ✅ | ❌（慢） | 低 | 中 | 大多数场景 |
| **蓝绿部署** | ✅ | ✅ | 高（双倍） | 低 | 关键业务 |
| **金丝雀发布** | ✅ | ✅ | 中 | 极低 | 高风险变更 |
| **重建部署** | ❌ | ❌ | 低 | 高 | 开发/测试环境 |

## 六、面试高频问题

### Q1：Docker和虚拟机的区别？

**答**：

| 维度 | Docker | 虚拟机 |
|------|--------|--------|
| **隔离级别** | 进程级（共享内核） | 系统级（独立内核） |
| **启动速度** | 秒级 | 分钟级 |
| **资源开销** | 低（MB级） | 高（GB级） |
| **性能** | 接近原生 | 有损耗 |
| **镜像大小** | MB级 | GB级 |

**核心区别**：Docker是容器化技术（进程隔离），虚拟机是虚拟化技术（硬件虚拟化）。

### Q2：Kubernetes的Service和Ingress的区别？

**答**：
- **Service**：L4层（TCP/UDP）服务发现，提供稳定的虚拟IP和DNS
- **Ingress**：L7层（HTTP/HTTPS）路由，基于域名和URL路径路由

关系：Ingress → Service → Pod

### Q3：Kubernetes的滚动更新和蓝绿部署的区别？

**答**：
- **滚动更新**：逐步替换Pod，零停机但更新慢，两个版本可能共存
- **蓝绿部署**：瞬间切换流量，快速回滚但需要双倍资源

**选择**：
- 大多数场景用滚动更新
- 关键业务用蓝绿部署

### Q4：如何实现Kubernetes的健康检查？

**答**：三种探针：
1. **livenessProbe**（存活探针）：检测容器是否存活，失败则重启容器
2. **readinessProbe**（就绪探针）：检测容器是否就绪，失败则从Service endpoints移除
3. **startupProbe**（启动探针）：检测慢启动容器，在启动阶段禁用其他探针

### Q5：CI/CD的核心价值是什么？

**答**：
1. **自动化**：减少人工操作，降低错误率
2. **快速反馈**：提交代码后立即得到构建和测试反馈
3. **频繁发布**：小步快跑，降低发布风险
4. **可追溯**：每次构建都有记录，方便回滚和审计

### Q6：Docker镜像分层存储的原理？

**答**：Docker镜像采用分层存储（UnionFS）：
- 每一层只读
- 容器层可写（Copy-on-Write）
- 共享相同层（节省存储和传输）

**优势**：
- 构建缓存（相同层复用）
- 增量传输（只传输变化的层）
- 节省存储（相同层只存一份）

### Q7：Kubernetes的资源限制（requests vs limits）？

**答**：
- **requests**：调度依据（保证最小资源）
- **limits**：硬限制（最多使用多少资源）

**QoS等级**（根据requests和limits）：
1. **Guaranteed**：requests = limits → 最高优先级
2. **Burstable**：requests < limits → 中优先级
3. **BestEffort**：无requests/limits → 最低优先级

**驱逐顺序**：BestEffort → Burstable → Guaranteed

### Q8：如何实现微服务的配置管理（ConfigMap/Secret）？

**答**：
- **ConfigMap**：存储非敏感配置（配置文件、环境变量）
- **Secret**：存储敏感配置（密码、Token、密钥）

```yaml
# ConfigMap
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  mysql.host: "mysql-service"
  mysql.port: "3306"

---
# Secret
apiVersion: v1
kind: Secret
metadata:
  name: app-secret
type: Opaque
data:
  mysql.password: cm9vdDEyMw==  # base64编码
```

使用方式：
1. 环境变量
2. 文件挂载（Volume）

## 七、总结

### 核心要点

1. **Docker核心价值**：标准化交付、环境一致性、资源隔离
2. **Kubernetes核心能力**：服务编排、自动恢复、弹性伸缩
3. **CI/CD核心价值**：自动化、快速反馈、频繁发布
4. **部署策略选择**：滚动更新（大多数场景）、蓝绿部署（关键业务）、金丝雀发布（高风险变更）

### 技术选型

```
选型决策树

需要容器化？
    │
    ├─ 否 → 直接部署到物理机/虚拟机
    │
    └─ 是 ─→ 需要编排？
              │
              ├─ 否 → Docker Compose（单机）
              │
              └─ 是 ──→ 规模？
                        │
                        ├─ 小（<10节点）→ Docker Swarm（简单）
                        │
                        └─ 大（>10节点）→ Kubernetes（标准）
```
