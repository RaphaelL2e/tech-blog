## 一、设计模式与系统设计的关系

### 1.1 从建模到设计模式的自然过渡

```
系统分析与设计全流程：

  需求分析 ──► 用例建模 ──► 静态建模（类图）──► 动态建模（时序/状态/活动图）
                                                      │
                                                      ▼
                                              系统设计模式实战
                                              （本文主题）
                                                      │
                                                      ▼
                                              架构设计 ──► 详细设计 ──► 实现
```

**设计模式在系统设计中的角色：**

| 维度 | 作用 | 示例 |
|------|------|------|
| 抽象层 | 将多变的需求固化为稳定的接口 | 策略模式定义支付接口 |
| 解耦层 | 降低模块间耦合，提升可替换性 | 工厂模式解耦创建与使用 |
| 协作层 | 定义对象间的通信与协作机制 | 观察者模式实现事件驱动 |
| 复用层 | 将已验证的设计方案复用到新场景 | 模板方法模式复用算法骨架 |

### 1.2 设计模式选型决策树

```
需要灵活创建对象？
  ├── 需要控制创建逻辑 ──► 工厂模式（Factory）
  ├── 需要创建复杂对象 ──► 建造者模式（Builder）
  └── 需要控制实例数量 ──► 单例模式（Singleton）

需要动态切换行为？
  ├── 运行时选择算法 ──► 策略模式（Strategy）
  ├── 状态驱动行为变化 ──► 状态模式（State）
  └── 权限/职责链处理 ──► 责任链模式（Chain of Responsibility）

需要对象间通信？
  ├── 一对多依赖通知 ──► 观察者模式（Observer）
  ├── 中介协调多对象 ──► 中介者模式（Mediator）
  └── 命令请求分离 ──► 命令模式（Command）
```

## 二、工厂模式：从简单到抽象的演进

### 2.1 三种工厂模式对比

| 模式 | 定义 | 适用场景 | 复杂度 |
|------|------|----------|--------|
| 简单工厂 | 一个工厂类根据参数创建不同产品 | 产品类型少，创建逻辑简单 | ★ |
| 工厂方法 | 定义创建接口，子类决定具体产品 | 产品有层级结构，需要扩展 | ★★ |
| 抽象工厂 | 创建相关/依赖的产品族 | 多产品族，需要保证兼容性 | ★★★ |

## 三、策略模式：算法族的管理与切换

### 3.1 核心思想

```
策略模式 = 定义一系列算法，把它们封装起来，并使它们可以相互替换

            ┌──────────────┐
            │   Context    │  ← 持有策略引用
            │              │
            │ +setStrategy()│
            │ +execute()   │
            └──────┬───────┘
                   │
                   ▼
            ┌──────────────┐
            │ <<interface>> │
            │   Strategy    │
            │ +algorithm()  │
            └──────┬───────┘
                   │
         ┌─────────┼─────────┐
         │         │         │
         ▼         ▼         ▼
    ┌────────┐ ┌────────┐ ┌────────┐
    │Concrete│ │Concrete│ │Concrete│
    │Strategy│ │Strategy│ │Strategy│
    │   A    │ │   B    │ │   C    │
    └────────┘ └────────┘ └────────┘
```

### 3.2 实战案例：优惠策略系统

```java
// ========== 策略接口 ==========
public interface DiscountStrategy {
    /**
     * 计算折扣后金额
     * @param originalPrice 原价
     * @return 折后价
     */
    BigDecimal calculate(BigDecimal originalPrice);
    
    /** 策略名称，用于前端展示 */
    String getStrategyName();
}

// ========== 具体策略1：满减 ==========
@Component
public class FullReductionStrategy implements DiscountStrategy {
    private final BigDecimal threshold = new BigDecimal("200");
    private final BigDecimal reduction = new BigDecimal("50");
    
    @Override
    public BigDecimal calculate(BigDecimal originalPrice) {
        if (originalPrice.compareTo(threshold) >= 0) {
            return originalPrice.subtract(reduction);
        }
        return originalPrice;
    }
    
    @Override
    public String getStrategyName() {
        return "满200减50";
    }
}

// ========== 具体策略2：打折 ==========
@Component
public class PercentageStrategy implements DiscountStrategy {
    private final BigDecimal discountRate = new BigDecimal("0.85");
    
    @Override
    public BigDecimal calculate(BigDecimal originalPrice) {
        return originalPrice.multiply(discountRate)
                .setScale(2, RoundingMode.HALF_UP);
    }
    
    @Override
    public String getStrategyName() {
        return "全场85折";
    }
}

// ========== 具体策略3：新人优惠 ==========
@Component
public class NewUserStrategy implements DiscountStrategy {
    @Override
    public BigDecimal calculate(BigDecimal originalPrice) {
        // 新人首单立减30，最低0.01元
        BigDecimal after = originalPrice.subtract(new BigDecimal("30"));
        return after.compareTo(new BigDecimal("0.01")) < 0 
            ? new BigDecimal("0.01") : after;
    }
    
    @Override
    public String getStrategyName() {
        return "新人首单立减30";
    }
}

// ========== 策略上下文 ==========
@Service
public class DiscountService {
    // Spring自动注入所有DiscountStrategy实现
    @Autowired
    private List<DiscountStrategy> strategies;
    
    /**
     * 自动选择最优策略
     */
    public DiscountResult calculateBest(BigDecimal price, UserInfo user) {
        return strategies.stream()
            .map(s -> new DiscountResult(
                s.getStrategyName(),
                s.calculate(price),
                price.subtract(s.calculate(price))))
            .min(Comparator.comparing(DiscountResult::getFinalPrice))
            .orElse(new DiscountResult("无优惠", price, BigDecimal.ZERO));
    }
    
    /**
     * 指定策略计算
     */
    public DiscountResult calculateByType(String type, BigDecimal price) {
        DiscountStrategy strategy = strategies.stream()
            .filter(s -> s.getClass().getSimpleName().toLowerCase().contains(type))
            .findFirst()
            .orElseThrow(() -> new IllegalArgumentException("未知策略: " + type));
        return new DiscountResult(
            strategy.getStrategyName(),
            strategy.calculate(price),
            price.subtract(strategy.calculate(price)));
    }
}
```

**策略模式的价值：**

| 对比项 | 无策略模式（if-else） | 使用策略模式 |
|--------|---------------------|-------------|
| 新增策略 | 修改核心逻辑类 | 新增一个策略类 |
| 单元测试 | 需要Mock大量分支 | 每个策略独立测试 |
| 代码可读性 | if-else嵌套难以维护 | 策略职责清晰 |
| 运行时切换 | 需要改代码 | 动态注入即可 |

### 3.3 if-else vs 策略模式重构

```java
// ❌ 反模式：大量if-else
public BigDecimal calculatePrice(String type, BigDecimal price) {
    if ("full_reduction".equals(type)) {
        return price.compareTo(new BigDecimal("200")) >= 0 
            ? price.subtract(new BigDecimal("50")) : price;
    } else if ("percentage".equals(type)) {
        return price.multiply(new BigDecimal("0.85"));
    } else if ("new_user".equals(type)) {
        return price.subtract(new BigDecimal("30"));
    } else if ("vip".equals(type)) {
        return price.multiply(new BigDecimal("0.8"));
    }
    // 每加一种优惠就要改这里 → 违反开闭原则
    return price;
}

// ✅ 策略模式：Map + 枚举
@Configuration
public class DiscountConfig {
    @Bean
    public Map<String, DiscountStrategy> discountMap(
            List<DiscountStrategy> strategies) {
        Map<String, DiscountStrategy> map = new HashMap<>();
        for (DiscountStrategy s : strategies) {
            // 用注解或约定命名来注册
            String key = s.getClass()
                .getAnnotation(StrategyKey.class).value();
            map.put(key, s);
        }
        return map;
    }
}
```

## 四、观察者模式：事件驱动架构的基石

### 4.1 核心思想

```
观察者模式 = 一对多的依赖关系，当一个对象状态改变时，所有依赖它的对象都得到通知并自动更新

  主题（Subject）                  观察者（Observer）
  ┌──────────────┐               ┌──────────────┐
  │ +attach()    │◄──────注册────│ +update()     │
  │ +detach()    │               └──────────────┘
  │ +notify()    │───通知───────►┌──────────────┐
  └──────────────┘               │ +update()     │
                                 └──────────────┘
                                 ┌──────────────┐
                                 │ +update()     │
                                 └──────────────┘
```

### 4.2 实战案例：订单状态变更通知系统

```java
// ========== 事件定义 ==========
@Data
@AllArgsConstructor
public class OrderEvent {
    private String orderId;
    private OrderStatus fromStatus;
    private OrderStatus toStatus;
    private LocalDateTime eventTime;
    private Map<String, Object> context;
}

// ========== 观察者接口 ==========
public interface OrderObserver {
    /** 关心的状态变更 */
    List<OrderStatus> interestedStatuses();
    
    /** 收到通知后的处理 */
    void onOrderStatusChanged(OrderEvent event);
}

// ========== 具体观察者1：短信通知 ==========
@Component
public class SmsNotifier implements OrderObserver {
    @Override
    public List<OrderStatus> interestedStatuses() {
        return Arrays.asList(
            OrderStatus.PAID,       // 支付成功
            OrderStatus.SHIPPED,    // 已发货
            OrderStatus.DELIVERED   // 已签收
        );
    }
    
    @Override
    public void onOrderStatusChanged(OrderEvent event) {
        String template = switch (event.getToStatus()) {
            case PAID -> "您的订单%s已支付成功，我们将尽快发货";
            case SHIPPED -> "您的订单%s已发货，快递单号：%s";
            case DELIVERED -> "您的订单%s已签收，感谢您的购买";
            default -> null;
        };
        if (template != null) {
            smsService.send(event.getOrderId(), 
                String.format(template, event.getOrderId(), 
                    event.getContext().get("trackingNo")));
        }
    }
}

// ========== 具体观察者2：库存扣减 ==========
@Component
public class InventoryDeductObserver implements OrderObserver {
    @Override
    public List<OrderStatus> interestedStatuses() {
        return List.of(OrderStatus.PAID);
    }
    
    @Override
    public void onOrderStatusChanged(OrderEvent event) {
        // 支付成功 → 真实扣减库存
        inventoryService.deductRealStock(event.getOrderId());
    }
}

// ========== 具体观察者3：积分发放 ==========
@Component
public class PointsAwardObserver implements OrderObserver {
    @Override
    public List<OrderStatus> interestedStatuses() {
        return List.of(OrderStatus.DELIVERED);
    }
    
    @Override
    public void onOrderStatusChanged(OrderEvent event) {
        // 签收后发放积分
        pointsService.awardPoints(event.getOrderId(), 
            event.getContext().get("totalAmount"));
    }
}

// ========== 主题：订单服务（被观察者） ==========
@Service
public class OrderService {
    private final Map<OrderStatus, List<OrderObserver>> observerMap = new HashMap<>();
    
    @Autowired
    public OrderService(List<OrderObserver> observers) {
        // 注册所有观察者
        for (OrderObserver observer : observers) {
            for (OrderStatus status : observer.interestedStatuses()) {
                observerMap.computeIfAbsent(status, k -> new ArrayList<>())
                           .add(observer);
            }
        }
    }
    
    @Transactional
    public void updateOrderStatus(String orderId, OrderStatus newStatus) {
        Order order = orderRepo.findByOrderId(orderId);
        OrderStatus oldStatus = order.getStatus();
        order.setStatus(newStatus);
        orderRepo.save(order);
        
        // 发布事件（异步通知）
        OrderEvent event = new OrderEvent(orderId, oldStatus, newStatus, 
            LocalDateTime.now(), buildContext(order));
        
        // 通知所有关心的观察者
        List<OrderObserver> interested = observerMap.getOrDefault(newStatus, List.of());
        for (OrderObserver observer : interested) {
            // 异步执行，避免阻塞主流程
            eventExecutor.execute(() -> observer.onOrderStatusChanged(event));
        }
    }
}
```

## 五、设计模式组合实战：电商下单流程

### 5.1 需求分析

```
下单流程的复杂性：

  用户下单
     │
     ▼
  校验库存 ──► 计算价格 ──► 应用优惠 ──► 创建订单
     │            │            │            │
     │            │            │            ▼
     │            │            │        支付(多通道)
     │            │            │            │
     │            │            │            ▼
     │            │            │        支付成功
     │            │            │            │
     │            │            │      ┌─────┼─────┐
     │            │            │      │     │     │
     │            │            │   短信  库存  积分

涉及模式：
  工厂模式 → 支付渠道创建
  策略模式 → 优惠策略选择
  观察者模式 → 支付成功通知
```

### 5.2 综合实现

```java
// ========== 下单服务（组合三种模式） ==========
@Service
public class OrderCreateService {
    
    @Autowired
    private Map<String, DiscountStrategy> discountMap;  // 策略模式
    
    @Autowired
    private ApplicationEventPublisher eventPublisher;   // 观察者模式
    
    @Autowired
    private PaymentFactoryRegistry factoryRegistry;     // 工厂模式
    
    /**
     * 创建订单
     */
    @Transactional
    public OrderResult createOrder(CreateOrderRequest request) {
        
        // 1. 策略模式：选择最优优惠
        DiscountResult discount = selectBestDiscount(request);
        
        // 2. 计算最终金额
        BigDecimal finalAmount = request.getTotalAmount()
            .subtract(discount.getDiscountAmount());
        
        // 3. 创建订单
        Order order = new Order();
        order.setOrderId(generateOrderId());
        order.setAmount(finalAmount);
        order.setDiscountStrategy(discount.getStrategyName());
        order.setPaymentChannel(request.getPaymentChannel());
        orderRepo.save(order);
        
        // 4. 工厂模式：获取支付渠道，发起支付
        PaymentFactory paymentFactory = factoryRegistry
            .getFactory(request.getPaymentChannel());
        PayChannel payChannel = paymentFactory.createPayChannel();
        PayResult payResult = payChannel.pay(new PayRequest(order));
        
        if (payResult.isSuccess()) {
            // 5. 观察者模式：发布支付成功事件
            eventPublisher.publishEvent(
                new OrderPaidEvent(this, order.getOrderId(), finalAmount));
        }
        
        return new OrderResult(order.getOrderId(), payResult);
    }
}
```

### 5.3 架构价值总结

| 模式 | 解决的问题 | 系统收益 |
|------|-----------|---------|
| 工厂模式 | 支付渠道扩展 | 新增渠道只需新增工厂类，不改核心逻辑 |
| 策略模式 | 优惠规则多变 | 运营随时新增优惠策略，互不影响 |
| 观察者模式 | 后续流程解耦 | 新增通知渠道（推送/邮件）无需改下单代码 |

## 六、设计模式误用与反模式

### 6.1 过度设计的信号

```
🚨 设计模式过度使用的信号：

1. 为了实现"优雅"，为一个简单的if-else引入了3个类
   → 评价标准：代码行数是否反而增加了？

2. 每个类都套了设计模式，但从来没切换过实现
   → 评价标准：真的有第二个实现了吗？

3. 为了"面向接口编程"，每个类都抽了一个接口
   → 评价标准：这个接口有几个实现？

4. 照搬GoF，不管业务场景是否匹配
   → 评价标准：不用模式会不会更简单？
```

### 6.2 KISS原则优先

```java
// ❌ 过度设计：简单需求用了复杂模式
public interface StringProcessor {
    String process(String input);
}

public class TrimProcessor implements StringProcessor {
    public String process(String input) { return input.trim(); }
}

public class LowerCaseProcessor implements StringProcessor {
    public String process(String input) { return input.toLowerCase(); }
}

// 就为了 trim().toLowerCase() 创建了3个文件...

// ✅ KISS：保持简单
String result = input.trim().toLowerCase();
```

### 6.3 设计模式使用检查清单

```
使用设计模式前，自问三个问题：

☐ 1. 这个"变化点"真的会变吗？
     → 如果未来2个版本都不需要切换实现，暂不引入

☐ 2. 引入模式后，代码是更易读还是更难读？
     → 如果新同事一眼看不懂，可能就是过度设计

☐ 3. 付出的复杂度代价，值得吗？
     → 评估：新增类数量 vs 未来扩展可能性

经验法则：
  2次实现 → 暂不考虑模式
  3次实现 → 可以考虑抽取接口
  4次及以上 → 必须用模式管理复杂度
```

## 七、总结

| 设计模式 | 核心思想 | 一句话 | 典型场景 |
|----------|---------|--------|---------|
| 工厂模式 | 创建与使用分离 | "你要什么我给你建，别管怎么建的" | 支付渠道、数据库驱动 |
| 策略模式 | 算法族可替换 | "算法任你选，接口不变" | 优惠计算、排序算法 |
| 观察者模式 | 状态变更通知 | "状态变了，告诉所有关心的人" | 消息推送、事件驱动 |

**系统设计模式三大心法：**
1. **找到变化点**：识别系统中可能变化的部分，用模式封装变化
2. **面向接口编程**：依赖抽象而非具体实现，为扩展留下空间
3. **先简单后优化**：不要过早引入设计模式，等真正需要时再重构
