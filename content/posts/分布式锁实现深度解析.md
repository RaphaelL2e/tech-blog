## 一、引言

在分布式系统中，多个节点可能同时访问共享资源（如数据库、文件、缓存），需要通过分布式锁来保证互斥访问。分布式锁是面试中的必考题，也是实际开发中最常用的分布式组件之一。

**本文要点**：
1. 分布式锁的核心要求
2. Redis 分布式锁（SET NX + Lua）
3. Redisson 分布式锁（看门狗机制）
4. ZooKeeper 分布式锁（临时顺序节点）
5. etcd 分布式锁（Lease + Revision）
6. Redlock 算法与争议
7. 三种方案对比与选型
8. 面试高频问题

## 二、分布式锁的核心要求

### 2.1 四个基本要求

| 要求 | 描述 |
|------|------|
| **互斥性** | 同一时刻只有一个客户端持有锁 |
| **防死锁** | 持有锁的客户端宕机，锁必须自动释放 |
| **解铃还须系铃人** | 加锁和解锁必须是同一个客户端 |
| **高可用** | 锁服务本身不能成为单点故障 |

### 2.2 额外要求（生产环境）

| 要求 | 描述 |
|------|------|
| **可重入** | 同一个客户端可以多次获取同一把锁 |
| **阻塞/非阻塞** | 获取锁失败时，可以阻塞等待或立即返回 |
| **公平性** | 按请求顺序获取锁（FIFO） |
| **性能** | 加锁/解锁延迟低，吞吐高 |

### 2.3 分布式锁的常见陷阱

```
陷阱一：锁过期但业务未完成
  → 客户端 A 获取锁（TTL=10s）
  → 业务执行了 15s
  → 锁在 10s 时过期，客户端 B 获取了锁
  → 客户端 A 在 15s 时完成，删除了客户端 B 的锁！

陷阱二：误解锁
  → 客户端 A 获取锁，value = "A"
  → 客户端 A 的锁过期
  → 客户端 B 获取锁，value = "B"
  → 客户端 A 尝试解锁，删除了客户端 B 的锁！

陷阱三：GC 停顿导致锁失效
  → 客户端 A 获取锁（TTL=10s）
  → GC 停顿 12s（STW）
  → 锁过期，客户端 B 获取锁
  → GC 恢复后，两个客户端同时持有锁！
```

## 三、Redis 分布式锁

### 3.1 基础实现（SET NX PX）

**加锁**：

```bash
# SET key value NX PX timeout
SET lock:order:123 "uuid-client-A" NX PX 10000
# NX：只在 key 不存在时设置（互斥性）
# PX：设置过期时间（毫秒），防死锁
# value：唯一标识（解铃还须系铃人）
```

**解锁（Lua 脚本）**：

```lua
-- 先比较 value，再删除
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
```

**为什么解锁要用 Lua 脚本**：

```
如果不使用 Lua 脚本：
1. GET lock → 返回 "uuid-A"  ← 客户端 A 的锁
2. （此时锁刚好过期，客户端 B 获取了锁）
3. DEL lock  ← 删除了客户端 B 的锁！

使用 Lua 脚本：
GET + DEL 是原子操作，不会被其他命令穿插
```

**Java 实现**：

```java
public class RedisDistributedLock {
    
    private Jedis jedis;
    
    // 加锁脚本
    private static final String LOCK_SCRIPT =
        "if redis.call('SET', KEYS[1], ARGV[1], 'NX', 'PX', ARGV[2]) then " +
        "    return 1 " +
        "else " +
        "    return 0 " +
        "end";
    
    // 解锁脚本
    private static final String UNLOCK_SCRIPT =
        "if redis.call('GET', KEYS[1]) == ARGV[1] then " +
        "    return redis.call('DEL', KEYS[1]) " +
        "else " +
        "    return 0 " +
        "end";
    
    /**
     * 加锁
     * @param lockKey 锁名
     * @param requestId 唯一标识
     * @param expireMs 过期时间（毫秒）
     */
    public boolean lock(String lockKey, String requestId, long expireMs) {
        Object result = jedis.eval(
            LOCK_SCRIPT,
            Collections.singletonList(lockKey),
            Arrays.asList(requestId, String.valueOf(expireMs))
        );
        return "1".equals(result.toString());
    }
    
    /**
     * 解锁
     */
    public boolean unlock(String lockKey, String requestId) {
        Object result = jedis.eval(
            UNLOCK_SCRIPT,
            Collections.singletonList(lockKey),
            Collections.singletonList(requestId)
        );
        return "1".equals(result.toString());
    }
}
```

### 3.2 基础实现的问题

**问题一：锁续期（看门狗）**

```
客户端 A 获取锁（TTL=10s）
业务执行超过 10s → 锁过期 → 其他客户端获取锁 → 并发冲突！

解决方案：看门狗机制（Watchdog）
→ 后台线程定期续期锁的 TTL
→ Redisson 已实现
```

**问题二：不可重入**

```
客户端 A 获取锁 lock:order:123
方法 A 调用方法 B，方法 B 也需要 lock:order:123
→ 获取失败！死锁！

解决方案：使用 Hash 记录加锁次数
→ field = requestId, value = count
→ 加锁时 count + 1，解锁时 count - 1
→ count = 0 时删除锁
```

**问题三：单点故障**

```
Redis Master 持有锁
→ Master 宕机
→ Slave 未同步到锁数据
→ Slave 升级为新 Master
→ 其他客户端可以获取同一把锁！

解决方案：Redlock 算法（见后文）
```

### 3.3 Redisson 分布式锁

**Redisson** 是 Redis 的 Java 客户端，提供了完善的分布式锁实现。

**基本使用**：

```java
Config config = new Config();
config.useSingleServer().setAddress("redis://127.0.0.1:6379");
RedissonClient redisson = Redisson.create(config);

// 获取锁对象
RLock lock = redisson.getLock("lock:order:123");

try {
    // 加锁（默认看门狗续期）
    lock.lock();
    
    // 或指定等待时间和过期时间
    // lock.lock(10, TimeUnit.SECONDS);  // 10 秒后自动释放
    // lock.tryLock(5, 30, TimeUnit.SECONDS);  // 等待 5 秒，30 秒后自动释放
    
    // 执行业务逻辑
    doBusiness();
    
} finally {
    // 解锁（只有锁持有者才能解锁）
    lock.unlock();
}
```

**Redisson 的加锁 Lua 脚本**：

```lua
-- KEYS[1] = 锁名
-- ARGV[1] = 看门狗续期时间（默认 30 秒 = 30000 毫秒）
-- ARGV[2] = 唯一标识（clientId:threadId）

-- 加锁逻辑
if (redis.call('exists', KEYS[1]) == 0) then
    -- 锁不存在，加锁
    redis.call('hset', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;

if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then
    -- 锁存在且是自己的，重入 +1
    redis.call('hincrby', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;

-- 锁存在且不是自己的，返回剩余 TTL
return redis.call('pttl', KEYS[1]);
```

**Redisson 的看门狗机制**：

```
1. 加锁成功后，启动看门狗定时任务
2. 每隔 lockWatchdogTimeout / 3（默认 10 秒）检查一次
3. 如果锁仍然持有，自动续期 lockWatchdogTimeout（默认 30 秒）
4. 如果客户端宕机，看门狗不再续期，锁自动过期

优点：业务未完成时锁不会过期
优点：客户端宕机时锁自动释放（防死锁）
```

**Redisson 的解锁 Lua 脚本**：

```lua
-- KEYS[1] = 锁名
-- ARGV[1] = 唯一标识
-- ARGV[2] = 看门狗续期时间

-- 解锁逻辑
if (redis.call('hexists', KEYS[1], ARGV[2]) == 0) then
    -- 不是自己的锁
    return nil;
end;

-- 重入计数 -1
local counter = redis.call('hincrby', KEYS[1], ARGV[2], -1);

if (counter > 0) then
    -- 还有重入，续期
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return 0;
else
    -- 重入归零，删除锁
    redis.call('del', KEYS[1]);
    -- 发布解锁消息（通知等待的客户端）
    redis.call('publish', KEYS[2] + ':unlock', ARGV[1]);
    return 1;
end;
```

**Redisson 支持的锁类型**：

| 锁类型 | 类名 | 特点 |
|--------|------|------|
| 可重入锁 | `RedissonLock` | 支持重入、看门狗 |
| 公平锁 | `RedissonFairLock` | 按请求顺序获取锁 |
| 读写锁 | `RedissonReadWriteLock` | 读读共享，写写互斥 |
| 联锁 | `RedissonMultiLock` | 同时加多把锁 |
| 红锁 | `RedissonRedLock` | Redlock 算法实现 |
| 信号量 | `RedissonSemaphore` | 限流 |

## 四、ZooKeeper 分布式锁

### 4.1 实现原理

**基于临时顺序节点**：

```
1. 客户端在 /lock 下创建临时顺序节点
   → /lock/lock-0000000001

2. 获取 /lock 下所有子节点

3. 判断自己是否序号最小的：
   - 是 → 获得锁
   - 否 → Watch 前一个节点

4. 前一个节点删除时，收到通知
   → 重新检查自己是否最小
   → 是 → 获得锁
```

**为什么用临时节点**：
- 客户端宕机 → Session 过期 → 临时节点自动删除
- 不需要手动释放锁（防死锁）

**为什么用顺序节点**：
- 避免羊群效应（只有前一个节点的 Watch 被触发）
- 保证公平性（按创建顺序获取锁）

### 4.2 Curator 实现

```java
CuratorFramework client = CuratorFrameworkFactory.newClient(
    "localhost:2181",
    new ExponentialBackoffRetry(1000, 3)
);
client.start();

// InterProcessMutex：可重入的分布式锁
InterProcessMutex lock = new InterProcessMutex(client, "/lock");

try {
    // 加锁（阻塞等待，超时 30 秒）
    if (lock.acquire(30, TimeUnit.SECONDS)) {
        try {
            doBusiness();
        } finally {
            lock.release();
        }
    }
} catch (Exception e) {
    e.printStackTrace();
}
```

### 4.3 ZooKeeper 锁的优缺点

**优点**：
- 天然防死锁（临时节点 + Session 过期）
- 公平锁（顺序节点保证 FIFO）
- 强一致性（ZAB 协议保证）
- 无看门狗问题（Session 机制自动管理）

**缺点**：
- 性能较低（每次加锁/解锁需要集群同步）
- 不适合高并发场景（ZooKeeper 写性能有限）
- Session 超时可能导致误释放（网络抖动）

## 五、etcd 分布式锁

### 5.1 etcd 简介

**etcd** 是 CoreOS 开发的分布式 KV 存储，使用 Raft 协议保证强一致性。

**核心特性**：
- 强一致性（线性一致性读）
- Lease 机制（TTL 自动删除）
- Revision 机制（全局递增版本号）
- Watch 机制（监听 Key 变化）
- 事务（Compare-And-Set）

### 5.2 etcd 分布式锁实现

**基于 Lease + Revision + Watch**：

```
1. 创建 Lease（TTL = 10 秒）
2. 创建 Key（附带 Lease），value = 客户端唯一标识
3. 获取所有竞争 Key，按 Revision 排序
4. 判断自己是否 Revision 最小的：
   - 是 → 获得锁
   - 否 → Watch 前一个 Key 的删除事件
5. 前一个 Key 删除 → 收到通知 → 获得锁
6. Lease 过期 → Key 自动删除 → 锁释放
```

**并发控制（Concurrency Control）**：

```go
// etcd v3 客户端
cli, _ := clientv3.New(clientv3.Config{Endpoints: []string{"localhost:2379"}})
defer cli.Close()

// 创建 Session（包含 Lease）
session, _ := concurrency.NewSession(cli)
defer session.Close()

// 创建 Mutex
mutex := concurrency.NewMutex(session, "/lock/order/123")

// 加锁
mutex.Lock(context.TODO())
defer mutex.Unlock(context.TODO())

// 执行业务逻辑
doBusiness()
```

### 5.3 etcd 锁的优缺点

**优点**：
- 强一致性（Raft 协议保证）
- Lease 机制天然防死锁
- Revision 天然有序（公平锁）
- 读写性能优于 ZooKeeper

**缺点**：
- 部署复杂度高于 Redis
- 社区和生态不如 Redis 和 ZooKeeper

## 六、Redlock 算法

### 6.1 Redlock 算法步骤

**背景**：单节点 Redis 存在主从切换导致锁丢失的问题。Redlock 通过多节点解决。

**算法步骤**：

```
假设有 5 个独立的 Redis 节点（无主从关系）

1. 获取当前时间戳 T1（毫秒）

2. 依次向 5 个节点发送加锁请求：
   SET lockKey requestId NX PX TTL
   每个请求设置超时时间（远小于 TTL）

3. 计算加锁耗时：T2 = 当前时间 - T1

4. 判断加锁是否成功：
   - 成功在 >= 3 个节点（N/2 + 1）加锁
   - 且 T2 < TTL（锁还有效时间）

5. 如果成功：
   - 锁的有效时间 = TTL - T2 - 时钟漂移
   - 执行业务逻辑

6. 如果失败：
   - 向所有节点发送解锁请求（无论是否加锁成功）
```

### 6.2 Redlock 的争议

**Martin Kleppmann 的批评**（2016）：

1. **时钟依赖**：
   - Redlock 依赖系统时钟
   - 如果某个节点时钟跳变，可能导致锁提前过期

2. **GC 停顿**：
   - GC STW 可能导致客户端长时间停顿
   - 停顿恢复后，锁可能已过期且被其他客户端持有

3. **进程暂停**：
   - 类似 GC 停顿的问题
   - 进程暂停后，锁的有效期计算不再准确

**Martin 的建议**：
- 使用 fencing token（递增令牌）
- 每次加锁返回一个递增的 token
- 资源服务只接受比上次更大的 token

**Antirez（Redis 作者）的回应**：
- GC 停顿在所有分布式锁方案中都存在
- Redlock 的时钟依赖可以接受（时钟漂移通常很小）
- fencing token 需要资源服务配合，实现复杂

### 6.3 Redlock 的实际使用建议

```
1. 如果对正确性要求极高 → 使用 ZooKeeper 或 etcd
2. 如果对性能要求高且能容忍极小概率的不一致 → 使用 Redis 单节点 + Redisson
3. 如果想兼顾两者 → 使用 Redlock，但需要注意时钟同步
4. 无论使用哪种方案，都建议实现 fencing token 机制
```

## 七、三种方案对比

### 7.1 全面对比

| 对比项 | Redis | ZooKeeper | etcd |
|--------|-------|-----------|------|
| **一致性** | 最终一致 | 强一致（ZAB） | 强一致（Raft） |
| **性能** | 高（10万+ QPS） | 低（万级 QPS） | 中（几万 QPS） |
| **防死锁** | 需要设置 TTL | 临时节点自动删除 | Lease 自动过期 |
| **可重入** | Redisson 支持 | Curator 支持 | concurrency 支持 |
| **公平锁** | 需要额外实现 | 顺序节点天然支持 | Revision 天然支持 |
| **看门狗** | Redisson 内置 | Session 机制 | Lease KeepAlive |
| **部署复杂度** | 低 | 中 | 中 |
| **主从切换风险** | 有（锁可能丢失） | 无 | 无 |
| **适用场景** | 高并发，容忍短暂不一致 | 低并发，强一致性 | 中等并发，强一致性 |

### 7.2 选型建议

```
场景一：电商秒杀（高并发，短暂不一致可接受）
→ Redis + Redisson
→ 性能高，Redisson 看门狗解决续期问题

场景二：金融转账（强一致性，不能有并发冲突）
→ ZooKeeper 或 etcd
→ 强一致性保证

场景三：分布式任务调度（中等并发，公平性要求高）
→ ZooKeeper
→ 顺序节点保证公平

场景四：Kubernetes 环境
→ etcd
→ K8s 原生支持，部署简单

场景五：微服务架构
→ Redis + Redisson（默认）
→ ZooKeeper 或 etcd（强一致性场景）
```

## 八、分布式锁最佳实践

### 8.1 锁粒度

```
原则：锁的粒度越细越好

❌ 锁整个订单服务
   lock("order-service")

✅ 锁单个订单
   lock("order:123")

✅✅ 锁订单的特定操作
   lock("order:123:pay")
```

### 8.2 锁超时时间

```
原则：TTL 应该大于业务最长执行时间，但不能太长

推荐公式：
TTL = 业务平均执行时间 × 3 + 安全余量

示例：
业务平均执行时间 = 2 秒
TTL = 2 × 3 + 4 = 10 秒

配合看门狗：
Redisson 默认 lockWatchdogTimeout = 30 秒
续期间隔 = 30 / 3 = 10 秒
```

### 8.3 加锁失败策略

```java
// 策略一：快速失败（非阻塞）
RLock lock = redisson.getLock("lock:order:123");
boolean acquired = lock.tryLock(0, 30, TimeUnit.SECONDS);
if (!acquired) {
    throw new BusinessException("系统繁忙，请稍后重试");
}

// 策略二：阻塞等待
lock.lock(30, TimeUnit.SECONDS);

// 策略三：有限等待 + 重试
int maxRetries = 3;
for (int i = 0; i < maxRetries; i++) {
    if (lock.tryLock(5, 30, TimeUnit.SECONDS)) {
        try {
            doBusiness();
        } finally {
            lock.unlock();
        }
        break;
    }
    Thread.sleep(1000);  // 退避
}
```

### 8.4 防止误解锁

```java
// 始终使用 try-finally 保证解锁
RLock lock = redisson.getLock("lock:order:123");
try {
    lock.lock();
    doBusiness();
} finally {
    // 只有持有锁才解锁
    if (lock.isHeldByCurrentThread()) {
        lock.unlock();
    }
}
```

## 九、面试高频问题

### Q1: Redis 和 ZooKeeper 分布式锁的区别？

**答案要点**：
- Redis：性能高，最终一致，主从切换可能丢锁
- ZooKeeper：强一致，性能低，临时节点防死锁
- Redis 适合高并发，ZooKeeper 适合强一致

### Q2: Redisson 的看门狗是什么？

**答案要点**：
- 后台线程定期续期锁的 TTL
- 默认 30 秒，每 10 秒检查一次
- 客户端宕机则不再续期，锁自动过期
- 解决了业务执行时间超过锁 TTL 的问题

### Q3: Redlock 算法的步骤？

**答案要点**：
1. 获取时间戳
2. 向 N 个独立节点加锁
3. 判断是否在多数节点加锁成功
4. 判断加锁耗时是否小于 TTL
5. 失败则向所有节点解锁

### Q4: 如何保证解铃还须系铃人？

**答案要点**：
- Redis：value 存储唯一标识，解锁时 Lua 脚本先比较再删除
- ZooKeeper：临时节点与 Session 绑定，只有创建者能删除
- etcd：Lease 与客户端绑定

### Q5: 分布式锁的 fencing token 是什么？

**答案要点**：
- 每次加锁返回递增的 token
- 资源服务只接受比上次更大的 token
- 解决 GC 停顿或时钟跳变导致的锁失效
- Martin Kleppmann 提出的方案

## 十、总结

本文系统对比了三种分布式锁实现方案：

1. **Redis 分布式锁**：高性能，Redisson 提供完善的看门狗和可重入支持
2. **ZooKeeper 分布式锁**：强一致，临时顺序节点天然防死锁和公平
3. **etcd 分布式锁**：强一致，Raft 协议保证，K8s 生态

**核心记忆**：
- Redis 加锁：SET NX PX + Lua 解锁
- Redisson 看门狗：每 10 秒续期，30 秒 TTL
- ZooKeeper 锁：临时顺序节点，避免羊群效应
- Redlock：多节点过半确认，但有争议
- 选型：高并发用 Redis，强一致用 ZooKeeper/etcd
