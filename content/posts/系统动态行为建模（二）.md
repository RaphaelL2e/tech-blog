# 系统动态行为建模（二）

> 🎯 **本文目标**：承接《系统静态分析建模（一）》的类图建模，转向面向对象分析的动态建模。从UML时序图、状态图、活动图三大动态模型出发，结合实际案例讲解如何描述系统的动态行为与交互。

---

## 一、动态建模概述

### 1.1 静态与动态的关系

```
面向对象建模的两个维度:

静态建模 (结构):                     动态建模 (行为):
  ┌─────────────────┐                ┌─────────────────┐
  │     用例图       │                │     状态图       │
  │   (Use Case)     │                │  (State Chart)   │
  └─────────────────┘                └─────────────────┘
  ┌─────────────────┐                ┌─────────────────┐
  │     类图         │    ───────►    │     时序图       │
  │   (Class)       │                │ (Sequence)      │
  └─────────────────┘                └─────────────────┘
  ┌─────────────────┐                ┌─────────────────┐
  │    对象图        │                │     活动图       │
  │   (Object)      │                │  (Activity)     │
  └─────────────────┘                └─────────────────┘

  回答: "系统由什么组成?"             回答: "系统如何工作?"
```

**动态建模三要素：**

| 要素 | 问题 | UML图 |
|------|------|-------|
| 交互 | 对象之间如何协作完成功能？ | 时序图、协作图 |
| 状态 | 对象在其生命周期中如何变化？ | 状态图 |
| 活动 | 业务流程如何流转？ | 活动图 |

## 二、时序图（Sequence Diagram）

### 2.1 核心概念

```
时序图: 描述对象之间的交互顺序，强调消息的时间顺序

基本元素:
  ┌─────────┐     参与者 (Actor/Object)
  │  User   │
  └────┬────┘
       │          生命线 (Lifeline) ──── 虚线表示对象存在周期
       │
       │─────────►│  同步消息 (Sync)    实心箭头，等待返回
       │          │
       │- - - - -►│  异步消息 (Async)   空心箭头，不等待返回
       │          │
       │◄- - - - -│  返回消息 (Return)  虚线箭头
       │          │
       │  ╔═════╗ │  激活框 (Activation) 表示对象正在执行
       │  ║     ║ │
       │  ╚═════╝ │
       │          │
       X          X  销毁标记 (Destruction)
```

```

└──┬───┘          └────┬─────┘          └─────┬────┘          └────┬─────┘
   │                   │                      │                    │
   │ enterLogin()      │                      │                    │
   │──────────────────►│                      │                    │
   │                   │ validate(username,pwd)                   │
   │                   │─────────────────────►│                    │
   │                   │                      │ findByUsername()   │
   │                   │                      │───────────────────►│
   │                   │                      │                    │
   │                   │                      │◄───────────────────│
   │                   │                      │    User对象         │
   │                   │                      │                    │
   │                   │                      │ checkPassword()   │
   │                   │                      │───────────────────►│
   │                   │                      │◄───────────────────│
   │                   │                      │      true/false    │
   │                   │◄─────────────────────│                    │

   │◄──────────────────│                      │                    │
   │  显示登录结果      │                      │                    │

### 2.3 时序图绘制要点

**Q1: 时序图绘制时常见错误有哪些？**

```
错误1: 消息顺序混乱
  消息必须按时间从上到下排列，不能交叉

错误2: 缺少返回消息
  虽然UML允许省略返回，但清晰起见应标注重要返回值

错误3: 激活框范围不当
  激活框应覆盖对象正在执行操作的时间段

错误4: 抽象层次不一致
  同一时序图应保持一致细节层次，避免部分过细、部分过粗
```

**循环与条件片段：**

```
┌─────────────────────────────────────────────┐
│ loop [订单条目数]                             │  ← 循环片段
│   │─────────────────────────────────────────│
│   │  处理单个订单条目                          │
│   │─────────────────────────────────────────│
└─────────────────────────────────────────────┘

┌─────────────────────────────────────────────┐
│ alt [支付方式 == "信用卡"]                    │  ← 条件分支
│   │─────────────────────────────────────────│
│   │  调用信用卡支付                            │
│   │─────────────────────────────────────────│
│ [支付方式 == "支付宝"]                        │
│   │─────────────────────────────────────────│
│   │  调用支付宝支付                            │
│   │─────────────────────────────────────────│
└─────────────────────────────────────────────┘
```

## 三、状态图（State Chart Diagram）

### 3.1 核心概念

```
状态图: 描述对象在其生命周期中的状态变化，以及触发状态变化的事件

基本元素:

  ○──────────►(状态1)─────────────►(状态2)
  初始状态         事件[条件]/动作         终止状态(可选)
                   状态
```

**状态机三要素：**

| 要素 | 说明 | 示例 |
|------|------|------|
| 状态 | 对象在某一时刻的条件 | 待支付、已发货 |
| 事件 | 触发状态变化的动作 | 支付成功、发货 |
| 转换 | 状态之间的变迁 | 待支付 → 已支付 |

### 3.3 状态图绘制要点

**Q2: 哪些对象需要画状态图？**

```
需要状态图的对象特征:
  1. 有明确的生命周期
  2. 状态数量有限且重要
  3. 状态变化由事件触发
  4. 不同状态下行为不同

适合画状态图的例子:
  ✓ 订单 (待支付/已支付/已发货/已完成...)
  ✓ 文档 (草稿/审核中/已发布/已归档)
  ✓ 工单 (待处理/处理中/已解决/已关闭)
  ✓ 用户 (未激活/正常/冻结/注销)

不适合画状态图的例子:
  ✗ 简单的DTO对象 (纯数据容器)
  ✗ 工具类 (无状态)
```

**状态内部结构：**

```
┌─────────────────────────────────────┐
│          状态名称                    │
├─────────────────────────────────────┤
│  entry / 进入动作 (entry动作)         │
│  do   / 内部活动 (持续执行)            │
│  exit / 退出动作 (exit动作)            │
├─────────────────────────────────────┤
│  内部转换:                           │
│    event [条件] / 动作               │
│    (不触发entry/exit)                │
└─────────────────────────────────────┘

示例:
┌─────────────────────────┐
│        已发货            │
├─────────────────────────┤
│  entry / 发送短信通知     │
│  do   / 物流跟踪         │
│  exit / 更新完成时间      │
└─────────────────────────┘
```

## 四、活动图（Activity Diagram）

### 4.1 核心概念

```
活动图: 描述业务流程或算法流程，强调活动之间的顺序与并发

基本元素:

  ●──────────►(活动1)──────────►(活动2)──────────►●
  开始节点      活动节点      决策节点        结束节点

  ┌─────────────┐                ┌─────────────┐
  │  分叉条      │  (并发开始)    │   汇合条     │  (并发结束)
  │  ─────      │                │  ─────      │
  └─────────────┘                └─────────────┘
```

### 4.3 活动图绘制要点

**活动图vs流程图：**

| 对比维度 | 活动图 | 流程图 |
|---------|-------|-------|
| 抽象层次 | 面向对象，可关联对象 | 面向过程 |
| 并发支持 | 支持分叉/汇合 | 不支持 |
| 泳道 | 支持划分责任区域 | 无 |
| 复杂度 | 更适合复杂业务流程 | 简单流程即可 |

**泳道（Swimlane）示例：**

```

## 五、三种图的关系与选择

### 5.1 三种图对比

| 图类型 | 关注点 | 适用场景 | 时间维度 |
|--------|-------|---------|---------|
| 时序图 | 对象交互顺序 | 分析用例实现、接口设计 | 单次交互流程 |
| 状态图 | 对象生命周期 | 分析核心实体的状态变化 | 对象生命全周期 |
| 活动图 | 业务流程流转 | 分析业务规则、工作流 | 完整业务流程 |

**Q3: 什么时候用哪种图？**

```
场景决策:

需要分析"用户下单"用例如何实现:
  → 时序图 (展示各对象协作顺序)

需要分析"订单"对象的状态变化:
  → 状态图 (订单从创建到完成/取消的全生命周期)

需要分析"订单处理"完整业务流程:
  → 活动图 (包含库存、支付、物流等完整流程)

综合分析复杂系统:
  → 结合使用，互为补充
```

### 5.2 从用例到动态模型

```
用例驱动建模流程:

  用例图
     │
     ▼
  用例规约 (前置条件、后置条件、基本流程、扩展流程)
     │
     ▼
  ┌───────────────────────────────────────┐
  │  动态建模 (三种图)                       │
  │                                       │
  │   基本流程 → 活动图 (业务流程)          │
  │   交互过程 → 时序图 (对象协作)          │
  │   状态变化 → 状态图 (实体生命周期)      │
  └───────────────────────────────────────┘
     │
     ▼
  静态建模 (类图优化)
```

## 七、总结

| 动态模型 | 核心要素 | 绘制要点 |
|---------|---------|---------|
| 时序图 | 对象、生命线、消息、激活框 | 保持时间顺序、标注关键返回值 |
| 状态图 | 状态、事件、转换、动作 | 识别核心实体、覆盖完整生命周期 |
| 活动图 | 活动、决策、分叉/汇合、泳道 | 划分责任边界、处理并发与异常 |

**动态建模三原则：**
1. **以用例为起点**：每个用例至少对应一张时序图
2. **关注核心实体**：只为有复杂状态的对象画状态图
3. **保持一致性**：三种图与类图互相验证
