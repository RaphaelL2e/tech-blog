## 一、微服务可观测性全景

### 1.1 可观测性三大支柱在微服务场景下的挑战

> 📌 **基础知识提示**：Prometheus指标采集与Grafana仪表盘、ELK/EFK日志平台架构、OpenTelemetry标准等通用可观测性知识已在《分布式系统面试八股文（八）——分布式监控与告警系统实战》中深度覆盖（PromQL查询、Grafana仪表盘设计、ELK索引优化、Alertmanager告警路由等），本文不再重复。本文聚焦微服务视角下特有的可观测性实践。

**Q: 微服务架构下的可观测性相比单体应用有哪些根本性变化？**

```
单体应用 vs 微服务应用的可观测性差异：

┌─────────────────────────────────────────────────────────────────┐
│                     单体应用                                     │
│  ┌───────────────────────────────────────────┐                  │
│  │            monolith.jar                    │                  │
│  │  ┌─────────┐  ┌──────┐  ┌─────────────┐  │                  │
│  │  │日志文件  │  │Metrics│  │   Stack Trace│  │                  │
│  │  │app.log  │  │本地JMX │  │  直接定位    │  │                  │
│  │  └─────────┘  └──────┘  └─────────────┘  │                  │
│  └───────────────────────────────────────────┘                  │
│  问题：一望到底，日志/指标/调用栈都在一个进程内                   │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                     微服务应用                                    │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐                    │
│  │ 网关      │──▶│ 订单服务  │──▶│ 库存服务  │                    │
│  │ app.log  │   │ app.log  │   │ app.log  │                    │
│  │ metrics  │   │ metrics  │   │ metrics  │                    │
│  └──────────┘   └──────────┘   └──────────┘                    │
│                       │                                          │
│                       ▼                                          │
│         问题：一个请求跨越N个服务，日志散落在N台机器上             │
│         需要：统一的traceId串联 + 集中的采集和查询体系             │
└─────────────────────────────────────────────────────────────────┘
```

**核心挑战：**

| 挑战 | 单体应用 | 微服务 |
|------|---------|--------|
| 调用链追踪 | 栈帧即可，不需要额外工具 | 必须注入traceId，跨进程传递 |
| 日志关联 | 一个文件，grep即可 | 散落在N个服务，需要集中聚合 |
| 指标聚合 | 本地JMX足够 | 需要Push/Pull模型 + 聚合查询 |
| 故障定界 | 一眼看出是哪个方法报错 | 需要先定位是哪个服务，再查那个服务的日志 |
| 性能瓶颈 | Profiler直接分析 | 需要端到端追踪 + 逐段分析 |

### 1.2 可观测性成熟度模型

```
Level 1: 事后响应（Reactive）
┌─────────────────────────────────────────────┐
│  用户投诉 → 查日志 → 定位问题 → 修复        │
│  问题：用户比你先发现故障，体验差            │
└─────────────────────────────────────────────┘

Level 2: 主动监控（Monitoring）
┌─────────────────────────────────────────────┐
│  指标告警 → 收到通知 → 排查修复              │
│  问题：告警有阈值窗口，可能漏报或滞后        │
└─────────────────────────────────────────────┘

Level 3: 可观测性（Observability）
┌─────────────────────────────────────────────┐
│  任意维度钻取 → 未预定义的查询 → 快速归因    │
│  特点：不需要事先想到问题是什么就能查        │
└─────────────────────────────────────────────┘

Level 4: 自愈（Self-Healing）
┌─────────────────────────────────────────────┐
│  检测异常 → 自动熔断/降级/扩缩容            │
│  只有在此阶段可观测性才算真正闭环            │
└─────────────────────────────────────────────┘
```

## 二、分布式链路追踪：从 Sleuth 到 Micrometer Tracing

### 2.1 Sleuth 的日落与 Micrometer Tracing 的崛起

**Q: Sleuth 和 Micrometer Tracing 是什么关系？为什么 Spring Boot 3.x 不再使用 Sleuth？**

这是微服务面试中的高频问题，反映了 Spring 生态的演进方向。

```java
// ===== Spring Boot 2.x 时代：Sleuth =====
// pom.xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-sleuth</artifactId>
</dependency>

// 使用方式：自动注入 TraceContext，零代码侵入
@RestController
public class OrderController {
    @Autowired
    private Tracer tracer;  // Sleuth 自己的 Tracer
    
    @GetMapping("/order/{id}")
    public Order getOrder(@PathVariable Long id) {
        // Sleuth 自动在日志中注入 [app-name,traceId,spanId]
        // 2026-07-07 10:00:01.123 [order-service,abc123,def456] INFO ...
        log.info("查询订单: {}", id);
        return orderService.findById(id);
    }
}

// ===== Spring Boot 3.x 时代：Micrometer Tracing =====
// pom.xml
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>
    <!-- 或者使用 OpenTelemetry 桥接：
    <artifactId>micrometer-tracing-bridge-otel</artifactId> -->
</dependency>

// 使用方式：ObservedAware + Micrometer API
@RestController
public class OrderController {
    @Autowired
    private ObservationRegistry registry;
    
    @GetMapping("/order/{id}")
    public Order getOrder(@PathVariable Long id) {
        return Observation.createNotStarted("order.findById", registry)
                .lowCardinalityKeyValue("order.id", id.toString())
                .observe(() -> {
                    log.info("查询订单: {}", id);
                    return orderService.findById(id);
                });
    }
}
```

**演进对比：**

```
Sleuth 时代的架构：
┌──────────────────────────────────────────────┐
│  Sleuth Tracer                                │
│  ├── Brave (Zipkin 兼容)                      │
│  ├── traceId/spanId 注入日志                  │
│  └── 与 Zipkin 集成                           │
└──────────────────────────────────────────────┘
    缺点：Tracer API 与 Micrometer 不一致
          功能单一，与 Metrics 体系割裂

Micrometer Tracing 时代的架构：
┌──────────────────────────────────────────────┐
│  Micrometer Tracing (统一 API)                │
│  ├── Bridge: Brave 或 OpenTelemetry           │
│  ├── 与 Micrometer Metrics 统一               │
│  ├── traceId/spanId 注入日志                  │
│  ├── 支持 Baggage（跨服务透传业务参数）        │
│  └── 原生集成观察 API (Observation)           │
└──────────────────────────────────────────────┘
    优点：API 统一，Tracing + Metrics 一体
          桥接器解耦：随时切换 Brave/OTel
```

### 2.2 traceId 与 spanId 的生命周期

**Q: traceId 和 spanId 是怎么生成和传递的？跨线程/跨MQ怎么处理？**

```java
// ===== traceId/spanId 生成机制 =====
// traceId: 64位或128位，在请求边界入口生成，整个调用链不变
// spanId: 64位，每个新操作生成新的spanId

// Brave（默认实现）生成算法：
// traceId: SecureRandom 生成 128位随机数
// spanId: SecureRandom 生成 64位随机数

// ===== 跨服务传递：通过 HTTP Header =====
// 请求方自动注入：
GET /stock/deduct HTTP/1.1
Host: stock-service:8080
X-B3-TraceId: 463ac35c9f6413ad48485a3953bb6124
X-B3-SpanId: 48485a3953bb6124
X-B3-ParentSpanId: 463ac35c9f6413ad
X-B3-Sampled: 1

// 如果用 W3C Trace Context 标准（推荐）：
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
tracestate: vendor=opaqueValue

// ===== 跨线程传递 =====
CompletableFuture.supplyAsync(() -> {
    // 问题：新线程没有原线程的 traceId
    // 解决：使用 Hooks 自动传播
    Hooks.enableAutomaticContextPropagation(); // Reactor
}).subscribe();

// 手动包装（如果不自动传播）：
Runnable wrapped = Context.current().wrap(runnable);
new Thread(wrapped).start();

// ===== 跨MQ传递 =====
// 发送端：把traceId放入消息头
@Autowired
private RabbitTemplate rabbitTemplate;

public void sendOrderMessage(Order order) {
    // Brave 自动注入 traceId 到消息头
    // Header: b3: 463ac35c9f6413ad-48485a3953bb6124-1
    rabbitTemplate.convertAndSend("order.exchange", "order.created", order);
}

// 消费端：自动从消息头提取traceId，继续追踪
@RabbitListener(queues = "order.queue")
public void handleOrder(Order order) {
    // 这里的日志会自动带上发送端的 traceId
    log.info("收到订单消息: {}", order.getId());
}
```

### 2.3 Baggage：跨服务透传业务参数

**Q: 什么场景需要用 Baggage？和 traceId 有什么区别？**

```java
// traceId 只能传递技术信息，Baggage 可以传递业务参数
// 典型场景：透传租户ID、用户ID、测试标识等

// ===== 设置 Baggage =====
@GetMapping("/order/create")
public Order createOrder(
        @RequestHeader("X-Tenant-Id") String tenantId) {
    
    // 方式1：代码设置
    BaggageField.TENANT_ID.updateValue(tenantId);
    
    // 方式2：通过 Observation 设置
    Observation.createNotStarted("order.create", registry)
            .contextualName("创建订单")
            .highCardinalityKeyValue("tenant.id", tenantId) // 高基数
            .lowCardinalityKeyValue("order.type", "normal")  // 低基数
            .observe(() -> orderService.createOrder());
    
    return order;
}

// ===== 在日志中使用 Baggage =====
// logback-spring.xml 中配置：
// %property{tenantId} 或者直接映射到 MDC
// Micrometer Tracing 会自动把 Baggage 值放到 MDC

// logback-spring.xml 配置示例：
<configuration>
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>
                %d{HH:mm:ss.SSS} [%thread] %-5level [%mdc{traceId:-},%mdc{spanId:-}]
                [tenant=%mdc{tenantId:-}] %logger{36} - %msg%n
            </pattern>
        </encoder>
    </appender>
</configuration>

// 日志输出示例：
// 10:00:01.123 [http-nio-8080-1] INFO  [abc123,def456] [tenant=T001] OrderController - 查询订单: 1001
```

## 三、分布式日志关联与聚合

### 3.1 微服务日志管理的核心痛点

```
痛点全景图：

请求: 用户下单 (userId=100)
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ 网关     │────▶│订单服务  │────▶│库存服务  │────▶│支付服务  │
│ :8080   │     │ :8081   │     │ :8082   │     │ :8083   │
│         │     │         │     │         │     │         │
│ IP:     │     │ IP:     │     │ IP:     │     │ IP:     │
│ 10.0.1.1│     │ 10.0.2.3│     │ 10.0.3.5│     │ 10.0.4.2│
└─────────┘     └─────────┘     └─────────┘     └─────────┘
    │                │                │                │
    ▼                ▼                ▼                ▼
 gateway.log    order.log       stock.log        pay.log
    │                │                │                │
    └────────────────┴────────────────┴────────────────┘
                 没有 traceId 关联时：
                 → 需要登录4台机器，分别grep
                 → 时间线对不齐，根本不知道先后顺序
```

**解决方案：traceId + 日志文件采集 + 集中检索平台**

> 📌 关于 ELK/EFK 日志平台的搭建与优化（Filebeat采集、Logstash管道、Elasticsearch索引策略、Kibana查询），详见《分布式系统面试八股文（八）——分布式监控与告警系统实战》第四节。本文聚焦微服务侧的日志配置与 traceId 串联。

### 3.2 微服务日志最佳实践

```java
// ===== 结构化日志 + traceId 注入 =====
// 使用 logstash-logback-encoder 输出 JSON 格式

// pom.xml
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>

// logback-spring.xml
<configuration>
    <springProperty scope="context" name="appName" source="spring.application.name"/>
    
    <appender name="CONSOLE_JSON" class="ch.qos.logback.core.ConsoleAppender">
        <encoder class="net.logstash.logback.encoder.LogstashEncoder">
            <customFields>{"application":"${appName}"}</customFields>
            <!-- Micrometer Tracing 自动将 traceId/spanId 注入 JSON -->
            <includeMdcKeyName>traceId</includeMdcKeyName>
            <includeMdcKeyName>spanId</includeMdcKeyName>
            <includeMdcKeyName>tenantId</includeMdcKeyName>
        </encoder>
    </appender>
    
    <root level="INFO">
        <appender-ref ref="CONSOLE_JSON"/>
    </root>
</configuration>

// 输出的 JSON 日志：
{
  "@timestamp": "2026-07-07T02:00:01.123Z",
  "level": "INFO",
  "application": "order-service",
  "traceId": "4bf92f3577b34da6a3ce929d0e0e4736",
  "spanId": "00f067aa0ba902b7",
  "tenantId": "T001",
  "logger": "com.example.order.OrderService",
  "message": "订单创建成功: orderId=1001",
  "thread": "http-nio-8080-exec-3"
}
```

### 3.3 Log4j2 的异步日志与性能优化

**Q: 微服务的高并发场景下，日志怎么写才不影响性能？**

```java
// ===== 方案对比 =====
// Logback 同步：每条日志都等磁盘IO完成 → 高并发下成为瓶颈
// Log4j2 异步：日志写入 RingBuffer → 独立线程批量刷盘

// pom.xml — 排除默认 Logback，引入 Log4j2
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
    <exclusions>
        <exclusion>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-logging</artifactId>
        </exclusion>
    </exclusions>
</dependency>
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-log4j2</artifactId>
</dependency>

// log4j2-spring.xml
<Configuration status="WARN">
    <Appenders>
        <!-- 使用 RollingRandomAccessFile：比 RollingFileAppender 快 20-500% -->
        <RollingRandomAccessFile name="RollingFile"
            fileName="logs/app.log"
            filePattern="logs/app-%d{yyyy-MM-dd}-%i.log.gz">
            <PatternLayout>
                <pattern>%d{ISO8601} [%t] %-5level [%X{traceId},%X{spanId}] %logger - %msg%n</pattern>
            </PatternLayout>
            <Policies>
                <TimeBasedTriggeringPolicy/>
                <SizeBasedTriggeringPolicy size="256 MB"/>
            </Policies>
        </RollingRandomAccessFile>
    </Appenders>
    <Loggers>
        <!-- 使用 AsyncLogger：基于 LMAX Disruptor 的高性能异步日志 -->
        <AsyncLogger name="com.example" level="DEBUG" additivity="false">
            <AppenderRef ref="RollingFile"/>
        </AsyncLogger>
        <Root level="INFO">
            <AppenderRef ref="RollingFile"/>
        </Root>
    </Loggers>
</Configuration>

// 性能对比（压测数据，仅供参考）：
// Logback 同步:  ~50,000条/秒
// Log4j2 异步:   ~500,000条/秒 (10倍提升)
// 内存占用:      ~128MB (RingBuffer)
// 风险: 使用完 RingBuffer 会丢弃日志(DiscardPolicy)，生产建议配置 FailoverAppender
```

## 四、故障排查实战方法论

### 4.1 微服务故障排查三板斧

```
三板斧 = 链路追踪定界 + 日志分析定位 + Metrics监控定因

┌─────────────────────────────────────────────────────────────────┐
│                     第一板斧：链路追踪定界                         │
│                                                                 │
│  用户反馈："下单失败"                                            │
│       ↓                                                         │
│  SkyWalking/Zipkin 输入 orderId → 找到这条 Trace                 │
│       ↓                                                         │
│  可视化调用链：                                                  │
│  Gateway(200ms) → OrderService(50ms) → StockService(3000ms!!!)  │
│       ↓                                                         │
│  结论：问题出在 StockService，下面重点排查库存服务                │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                     第二板斧：日志分析定位                         │
│                                                                 │
│  拿到 traceId: 4bf92f3577b34da6a3ce929d0e0e4736                 │
│       ↓                                                         │
│  Kibana/ELK 中搜索：traceId:"4bf92f..."                         │
│       ↓                                                         │
│  按时间排序，还原完整调用过程：                                   │
│  10:00:01.100 [stock-service] 收到扣库存请求: skuId=5001,qty=1  │
│  10:00:01.150 [stock-service] 开始 SQL: SELECT ... FOR UPDATE   │
│  10:00:04.100 [stock-service] SQL 执行完成 (耗时2950ms!!!)       │
│  10:00:04.150 [stock-service] 库存扣减失败: 死锁！！              │
│       ↓                                                         │
│  结论：库存表死锁，导致请求超时3秒                                │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│                     第三板斧：Metrics监控定因                     │
│                                                                 │
│  在 Prometheus + Grafana 中：                                   │
│  - 「库存服务 - DB连接数」峰值达到100（最大连接数）               │
│  - 「死锁次数」每分钟 50 次                                      │
│  - 「行锁等待时间」P99 = 3000ms                                 │
│       ↓                                                         │
│  根因：扣库存的 SQL 锁粒度太大，多个请求争抢同一行 → 死锁        │
│  方案：改用乐观锁（version号）或 Redis 预减库存 + 异步同步DB    │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 常见故障模式与排查路径

```java
// ===== 故障1：服务超时 =====
// 现象：上游调用超时，但不清楚是网络还是下游慢
// 排查路径：

// 1. 查链路追踪 → 确认是哪个服务耗时高
// 2. 查该服务的 P99 耗时指标
rate(http_server_requests_seconds_sum{service="stock-service"}[5m]) 
/ rate(http_server_requests_seconds_count{service="stock-service"}[5m])

// 3. 查数据库慢查询（Arthas 或 数据库慢查询日志）
// 4. 查 JVM GC 情况（Young GC 频繁导致 STW）
// 5. 查线程池使用情况

// ===== 故障2：内存泄漏 → OOM =====
// 排查路径：
// 1. 查 Grafana：堆内存使用率持续上升不回降 → 怀疑内存泄漏
// 2. 通过 Arthas 查看对象分布：
//    dashboard → 查看堆内存
//    heapdump → 下载堆转储文件
// 3. MAT/JProfiler 分析 heapdump

// ===== 故障3：线程池打满 =====
// 排查路径：
// 1. Grafana 查看 tomcat_threads_busy 指标 = max → 线程池满
// 2. 查链路追踪：哪些接口耗时高（雪崩）
// 3. 临时措施：扩大线程池 + 配置熔断
// 4. 根治：优化慢接口

// ===== 故障4：Caffeine/Redis 缓存失效 =====
// 排查路径：
// 1. 检查缓存命中率指标 cache_gets_total / cache_misses
```


### 5.2 自定义 Metrics 与 Tracing 增强

```java
@Component
public class OrderMetricsService {
    
    private final MeterRegistry meterRegistry;
    private final ObservationRegistry observationRegistry;
    
    // ===== 自定义 Counter =====
    private final Counter ordersCreatedTotal;
    // ===== 自定义 Timer =====
    private final Timer orderProcessDuration;
    // ===== 自定义 Gauge =====
    private final AtomicInteger pendingOrders;
    
    public OrderMetricsService(MeterRegistry meterRegistry, 
                                ObservationRegistry observationRegistry) {
        this.meterRegistry = meterRegistry;
        this.observationRegistry = observationRegistry;
        
        this.ordersCreatedTotal = Counter.builder("orders.created.total")
                .description("订单创建总数")
                .tag("service", "order-service")
                .register(meterRegistry);
                
        this.orderProcessDuration = Timer.builder("orders.process.duration")
                .description("订单处理耗时")
                .publishPercentiles(0.5, 0.95, 0.99)
                .register(meterRegistry);
                
        this.pendingOrders = meterRegistry.gauge("orders.pending",
                new AtomicInteger(0));
    }
    
    // 使用 Observation 同时记录 Metrics 和 Tracing
    @Observed(name = "order.create",
              contextualName = "创建订单",
              lowCardinalityKeys = {"order.type"})
    public Order createOrder(CreateOrderRequest request) {
        Timer.Sample sample = Timer.start(meterRegistry);
        
        try {
            Order order = // ... 业务逻辑
            ordersCreatedTotal.increment();
            pendingOrders.incrementAndGet();
            return order;
        } finally {
            sample.stop(orderProcessDuration);
        }
    }
}
```

**Q2: 采样策略有哪些？怎么选择？**

| 策略 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| 全量采样 | 100% 采集 | 数据完整 | 存储成本极高 | 开发环境 |
| 固定比例 | N% 随机采样 | 简单 | 低流量请求可能漏 | 通用场景 |
| 自适应采样 | 根据QPS动态调整 | 存储可控 | 实现复杂 | 高流量生产 |
| 错误优先 | 正常低采样，错误全采集 | 兼顾成本和排错 | 需要后处理 | 推荐方案 |

```java
// 实现自定义采样器
@Bean
public SamplerFunction<HttpRequest> serverRequestSampler() {
    return request -> {
        // 错误请求 100% 采样
        if (request instanceof HttpServerRequest serverRequest) {
            String path = serverRequest.path();
            if (path.startsWith("/error") || path.startsWith("/health")) {
                return false;  // 不采样健康检查
            }
        }
        // 默认 10% 采样
        return Math.random() < 0.1;
    };
}
```

**Q3: traceId 丢失的场景及解决方案？**

```java
// ===== 场景1：异步线程 =====
// 问题：new Thread() 会丢失父线程的 traceId
// 解决：使用包装器
@Async
public CompletableFuture<Order> processAsync() {
    // Spring @Async 配合 Micrometer Tracing 自动传播
    log.info("异步处理"); // traceId 正常传递
    return CompletableFuture.completedFuture(order);
}

// ===== 场景2：@Scheduled 定时任务 =====
// 问题：定时任务没有外部请求，如何生成 traceId？
@Scheduled(fixedRate = 60000)
public void scheduledTask() {
    Observation observation = Observation
            .start("scheduled.task.process", registry);
    try (var scope = observation.openScope()) {
        // 这里有了 traceId 和 spanId
        log.info("定时任务执行");
    } catch (Exception e) {
        observation.error(e);
        throw e;
    } finally {
        observation.stop();
    }
}

// ===== 场景3：自定义线程池 =====
@Bean
public Executor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);
    executor.setMaxPoolSize(20);
    // 关键：包装 Runnable，传播上下文
    executor.setTaskDecorator(task -> {
        Map<String, String> context = MDC.getCopyOfContextMap();
        return () -> {
            if (context != null) MDC.setContextMap(context);
            try {
                task.run();
            } finally {
                MDC.clear();
            }
        };
    });
    return executor;
}
```

### 6.2 核心对比表

| 对比维度 | Sleuth | Micrometer Tracing | SkyWalking Java Agent |
|---------|--------|-------------------|----------------------|
| 代码侵入性 | 零（自动注入） | 零（自动注入） | 零（Java Agent字节码） |
| 与 Metrics 统一 | ❌ 独立API | ✅ 统一 Observation API | ❌ 独立体系 |
| 跟踪协议 | Brave/Zipkin | Brave 或 OTel | 自有协议 |
| Spring Boot 3 支持 | ❌ 不维护 | ✅ 原生支持 | ✅ Agent方式，与版本无关 |
| 性能开销 | 低 | 低 | 中等（Agent拦截） |
| 采样策略 | 固定比例 | 灵活自定义 | 自适应 |
| 适用场景 | 旧项目 | 新项目推荐 | 需要无侵入且Spring Boot 2/3通用 |
