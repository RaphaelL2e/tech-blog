## 一、架构演进——从单体到微服务

### 1.1 单体架构的困境

**Q: 为什么要从单体转向微服务？**

一个典型的电商单体应用：

```
┌──────────────────────────────────────────┐
│              Monolithic E-Commerce        │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │  用户模块  │ │  订单模块  │ │  商品模块  │ │
│  └──────────┘ └──────────┘ └──────────┘ │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │  支付模块  │ │  库存模块  │ │  物流模块  │ │
│  └──────────┘ └──────────┘ └──────────┘ │
│          共享同一个数据库 + War包部署       │
└──────────────────────────────────────────┘
```

**单体架构的痛点：**

| 问题 | 表现 | 后果 |
|------|------|------|
| 🐢 开发效率低 | 10+人在一个代码库改代码 | 频繁冲突、合并地狱 |
| 🐌 部署周期长 | 改一行代码要全量部署 | 月更→季更，不敢频繁发布 |
| 📈 扩展性差 | 只有订单模块压力大 | 只能整体扩容，资源浪费 |
| 💥 故障爆炸半径大 | 非核心模块OOM | 整站不可用 |
| 🔧 技术栈锁定 | 全部用Spring | 无法用Go写高性能服务 |

### 1.2 微服务架构的定义

> 微服务是一种将单个应用程序开发为一组小型服务的方法，每个服务运行在自己的进程中，通过轻量级通信机制（通常是HTTP API）进行通信。这些服务围绕业务能力构建，可以独立部署、独立扩展。—— Martin Fowler

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户服务  │  │ 订单服务  │  │ 商品服务  │  │ 支付服务  │
│ port:8081│  │ port:8082│  │ port:8083│  │ port:8084│
│ 独立DB   │  │ 独立DB   │  │ 独立DB   │  │ 独立DB   │
└────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
     └──────────────┴───────┬─────┴──────────────┘
                            │
                    ┌───────┴───────┐
                    │   API Gateway  │
                    └───────┬───────┘
                            │
                    ┌───────┴───────┐
                    │     客户端     │
                    └───────────────┘
```

### 1.3 微服务 vs 单体 vs SOA

| 对比维度 | 单体架构 | SOA | 微服务 |
|---------|----------|-----|--------|
| 服务粒度 | 全在一個包 | 粗粒度（企业级） | 细粒度（业务级） |
| 通信协议 | 方法调用 | ESB（企业服务总线） | 轻量级HTTP/gRPC/消息 |
| 数据管理 | 共享数据库 | 共享数据库 | **每个服务独立数据库** |
| 部署单元 | 整个应用 | 独立部署 | **独立部署+CI/CD** |
| 技术异构 | 统一技术栈 | 可以异构 | **鼓励异构**（多语言） |
| 团队组织 | 按技术分层 | 按功能 | **按业务能力** |

## 二、分布式理论基石——CAP与BASE

### 2.1 CAP定理

**Q: CAP定理是什么？（面试必考⭐⭐⭐⭐⭐）**

> CAP定理：一个分布式系统最多只能同时满足一致性（Consistency）、可用性（Availability）、分区容错性（Partition Tolerance）中的两项。

```
         C (一致性)
        /\
       /  \
      /    \
     /  CP  \    ← ZooKeeper, Etcd (优先一致性)
    /        \
   /──────────\
  /            \
 /   CA         \  ← 单机系统（无网络分区风险）
/                \
A ──────────────── P
(可用性)           (分区容错性)
        \        /
         \  AP  /  ← Eureka, Cassandra (优先可用性)
          \    /
           \  /
            \/
```

**P（分区容错性）必须满足**：因为网络分区在分布式系统中不可避免，所以实际选择是CP或AP。

**Q: 为什么P必须满足？**

```
场景：服务A和B部署在不同机器上，之间通过网络通信

正常情况：                    网络分区（交换机故障）：
 A ←────→ B                  A ←──[断]──→ B
                               ↓              ↓
                           用户查询A      用户查询B
                           能返回数据    也能返回数据
                           但数据可能不一致
```

在网络分区发生时，你必须在C和A之间抉择：

| 选择 | 行为 | 典型场景 |
|------|------|---------|
| **CP（放弃A）** | 拒绝请求，等数据同步后恢复 | 银行转账、金融交易 |
| **AP（放弃C）** | 返回可能过时的数据，最终一致 | 社交动态、商品浏览 |

### 2.2 BASE理论

**Q: BASE理论是什么？它和ACID的关系？**

```
ACID（传统数据库）            BASE（分布式系统）
─────────────────            ─────────────────
Atomicity   原子性     VS     Basically Available   基本可用
Consistency 强一致性   VS     Soft State            软状态
Isolation   隔离性     VS     Eventually Consistent 最终一致性
Durability  持久性
```

**BASE理论三要素：**

1. **基本可用（Basically Available）**：系统出现故障时，允许损失部分可用性，但保证核心功能

```java
// 秒杀场景：流量过大时，降级非核心功能
@HystrixCommand(fallbackMethod = "recommendFallback")
public List<Product> getRecommendations(Long userId) {
    return recommendationService.get(userId);
}

// 降级方法：返回默认推荐或缓存数据
public List<Product> recommendFallback(Long userId) {
    return getCachedHotProducts(); // 损失个性化推荐，但核心功能（下单）不受影响
}
```

2. **软状态（Soft State）**：允许系统中的数据存在中间状态，且该状态不影响系统整体可用性

```
例子：订单状态流转
  已支付 → [支付确认中...] → 已确认
          ↑ 中间状态允许存在
```

3. **最终一致性（Eventually Consistent）**：不保证实时一致，但保证最终一致

```
场景：用户下单后库存扣减
  T0: 用户下单 → 订单服务(order)写入订单(状态=待支付)
  T1: 订单服务发送消息 → "库存扣减"事件
  T2: 库存服务(inventory)消费消息 → 扣减库存
  T0~T2期间：订单和库存数据不一致（但用户感知不到）
  T2之后：最终一致
```

## 三、服务拆分——DDD与限界上下文

### 3.1 拆分原则

**Q: 微服务怎么拆分？有哪些原则？**

**第一原则：高内聚 + 低耦合**

```
高内聚（High Cohesion）：相关的逻辑放在一个服务里
低耦合（Low Coupling）：服务之间依赖性最小

       好的拆分                         坏的拆分
   ┌──────────┐                    ┌──────────┐
   │  订单服务  │                    │  订单服务  │
   │ ┌──────┐ │  ──轻量调用──→     │ ┌──────┐ │
   │ │创建订单│ │                    │ │创建订单│ │
   │ │取消订单│ │                    │ │扣减库存│ │ ← 库存逻辑不应在订单服务
   │ │查询订单│ │                    │ │查询商品│ │ ← 商品逻辑不应在订单服务
   │ └──────┘ │                    │ └──────┘ │
   └──────────┘                    └──────────┘
```

**六大拆分原则：**

| 原则 | 说明 | 反例 |
|------|------|------|
| **单一职责** | 一个服务只做一件事 | 用户服务里写物流逻辑 |
| **闭包原则** | 微服务是独立闭环 | 下单功能横跨4个服务 |
| **数据自治** | 每个服务有自己的数据库 | 多个服务共享数据库 |
| **可独立部署** | 改一个服务不影响其他 | 一改动所有服务要重启 |
| **轻量通信** | 不要过度依赖同步调用 | 循环调用链A→B→C→A |
| **团队对齐** | 服务边界=团队边界 | 一个服务多个团队维护 |

### 3.2 DDD限界上下文拆分法

**Q: 如何用DDD来指导微服务拆分？**

DDD（Domain-Driven Design，领域驱动设计）的**限界上下文**（Bounded Context）是微服务拆分的最佳指导：

```
电商领域的限界上下文拆分：

┌─────────────────────────────────────────────────┐
│                   电商核心域                      │
│                                                 │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│  │ 商品上下文│    │ 订单上下文│    │ 支付上下文│  │
│  │          │    │          │    │          │  │
│  │ Product  │    │  Order   │    │ Payment  │  │
│  │ 名称     │    │ 下单     │    │ 支付     │  │
│  │ SKU      │    │ 状态流转 │    │ 退款     │  │
│  │ 类目     │    │ 金额计算 │    │ 对账     │  │
│  └──────────┘    └──────────┘    └──────────┘  │
│                                                 │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐  │
│  │ 库存上下文│    │ 用户上下文│    │ 营销上下文│  │
│  │          │    │          │    │          │  │
│  │Inventory │    │   User   │    │Marketing │  │
│  │ 入库/出库│    │ 注册     │    │ 优惠券   │  │
│  │ 库存预警 │    │ 登录     │    │ 满减活动 │  │
│  │ 库存盘点 │    │ 会员     │    │ 秒杀     │  │
│  └──────────┘    └──────────┘    └──────────┘  │
└─────────────────────────────────────────────────┘
```

**Q: 不同上下文中的"商品"含义一样吗？**

```java
// 商品上下文的Product：关注基本信息
class Product {
    String id;
    String name;
    String description;
    List<SKU> skus;
    Category category;
}

// 订单上下文的Product：关注下单时快照
class OrderProduct {
    String productId;
    String nameSnapshot;  // ← 下单时商品名称快照（防止改名后订单信息变化）
    BigDecimal priceSnapshot; // ← 下单时价格快照
    int quantity;
}

// 库存上下文的Product：关注库存数量
class InventoryProduct {
    String productId;
    int totalStock;
    int reservedStock;
    int availableStock; // = totalStock - reservedStock
    int safetyStock;    // 安全库存阈值
}
```

**同一個"商品"在不同的限界上下文中具有完全不同的模型和关注点**，这就是为什么要拆分成独立的微服务。

### 3.3 拆分中的反模式

**❌ 反模式一：按技术层拆分**

```
不好的拆分：
  Controller服务 → Service服务 → DAO服务
  （调用链极长，一个简单查询要经过3个服务）
```

**❌ 反模式二：过度拆分（纳米服务）**

```
过度拆分：UserService → UserReadService + UserWriteService + UserAuthService
（服务太多，运维成本指数级上升）
```

**❌ 反模式三：数据耦合**

```
订单服务                          商品服务
┌──────────┐                    ┌──────────┐
│ Orders   │                    │ Products │
│   表     │←──── 外键 ──────→│   表     │
└──────────┘                    └──────────┘
     └──────── 共用数据库实例 ────────┘
（表面上拆了服务，实际上通过数据库强耦合）
```

## 四、服务通信——同步与异步

### 4.1 通信模型对比

| 维度 | 同步通信（REST/gRPC） | 异步通信（消息队列） |
|------|---------------------|---------------------|
| 耦合度 | 强耦合（调用方等结果） | 松耦合（发送即遗忘） |
| 实时性 | 实时响应 | 最终处理 |
| 可靠性 | 调用失败直接影响 | 消息持久化，可重试 |
| 适用场景 | 查询、即时操作 | 通知、数据同步、削峰 |
| 典型技术 | Feign, Dubbo, gRPC | Kafka, RabbitMQ, RocketMQ |

### 4.2 RESTful API vs gRPC

**RESTful（基于HTTP/JSON）**：

```java
// Spring Cloud Feign 声明式调用
@FeignClient(name = "order-service")
public interface OrderClient {
    
    @GetMapping("/orders/{id}")
    Order getOrder(@PathVariable("id") Long id);
    
    @PostMapping("/orders")
    Order createOrder(@RequestBody CreateOrderRequest request);
}
```

优点：简单、可读性好、跨语言、工具链成熟
缺点：JSON序列化慢、无类型约束、HTTP/1.1队头阻塞

**gRPC（基于HTTP/2 + Protobuf）**：

```protobuf
// order.proto
service OrderService {
  rpc GetOrder (GetOrderRequest) returns (Order);
  rpc CreateOrder (CreateOrderRequest) returns (Order);
  rpc ListOrders (ListOrdersRequest) returns (stream Order); // 服务端流
}
```

优点：二进制序列化（比JSON快5-10倍）、强类型、HTTP/2多路复用、支持流式
缺点：可读性差（二进制）、调试不便、浏览器直调有限制

### 4.3 消息队列——异步解耦

```
同步调用的问题：                          异步消息的方案：
                                         
用户下单                                 用户下单
  ↓                                       ↓
订单服务 ──调用──→ 库存服务                订单服务 ──发送消息──→ Kafka
  ↓                 ↓                      ↓                    ↓
  └──调用──→ 积分服务                     返回"已受理"     库存服务消费
  ↓                 ↓                                        ↓
  └──调用──→ 短信服务                                    积分服务消费
  ↓                 ↓                                        ↓
  等...500ms...1s...2s...               短信服务消费
  用户干等着！😡                              
                                        消息发送后立刻返回，不阻塞用户
```

## 五、三大基础设施

### 5.1 注册中心——服务发现

**Q: 为什么需要注册中心？**

```
没有注册中心：                          有注册中心：
                                        
A服务                  B服务           A服务                注册中心
┌──────┐             ┌──────┐        ┌──────┐    ②获取B地址  ┌──────┐
│      │──调用B的IP?→│      │        │      │←─────────────│      │
│      │  (需要写死IP) │      │        │      │              │ B:    │
└──────┘             └──────┘        │      │──③调用B────→│ 10.0.1.5:8080
                                     └──────┘              └──────┘
                                                                 ↑
                                        ①注册(B:10.0.1.5:8080)   │
                                                              ┌──────┐
                                                              │B服务  │
                                                              └──────┘
```

**主流注册中心对比：**

| 注册中心 | CAP | 一致性协议 | 健康检查 | 特点 |
|---------|-----|-----------|---------|------|
| **Nacos** | CP+AP（可切换） | Raft/Distro | TCP/HTTP/MySQL | 阿里开源，同时支持配置中心 |
| **Eureka** | AP | - | 心跳 | Netflix出品，自我保护机制 |
| **Consul** | CP | Raft | 多种检查 | HashiCorp出品，功能完善 |
| **ZooKeeper** | CP | ZAB | 临时节点+心跳 | 老牌，但作为注册中心有缺陷 |
| **Etcd** | CP | Raft | 租约 | Kubernetes基石 |

**Eureka的自我保护机制（独特面试点）：**

```
正常情况：
  服务提供者 → 每30秒发送心跳 → Eureka Server
  
自我保护触发条件：
  Eureka Server在15分钟内，心跳失败比例超过85%
  
自我保护行为：
  ❌ 不剔除任何服务实例（宁可保留不可用节点）
  ✅ 保护注册信息不丢失
  ⚠️  设计哲学：好死不如赖活着（AP优先）
  
实际意义：
  防止因网络抖动导致大量服务被误下线，引发雪崩
```

### 5.2 配置中心——配置管理

**Q: 为什么要有配置中心？**

```
传统方式（application.yml硬编码）：        配置中心方式：
                                        
每个服务都有自己的配置文件                     ┌──────────┐
  ┌──────────┐                              │ Nacos    │
  │ app.yml  │  database.url=jdbc:...       │ Config   │
  │          │  redis.host=192.168.1.1      │          │
  └──────────┘                              │ db.url=  │
                                            │ redis=   │
                          ┌──────────┐      │ timeout= │
问题：                      │ app.yml  │      └────┬─────┘
  改配置需要：              │          │           │
  1. 改yml文件              └──────────┘     ┌─────┼─────┐
  2. 提交代码                                ↓     ↓     ↓
  3. 重新部署                ┌──────────┐  用户  订单  商品
  4. 等服务重启              │ app.yml   │  服务  服务  服务
  → 耗时10分钟+              └──────────┘
                                        优点：
配置分散在N个服务的N个文件                    - 改一处，所有服务实时生效
  → 管理噩梦                                - 配置版本管理、灰度发布
                                            - 敏感配置加密存储
```

**Nacos配置中心动态刷新：**

```java
@RestController
@RefreshScope  // ← 配置变更时自动刷新Bean
public class ConfigController {
    
    @Value("${app.timeout:3000}")
    private int timeout;
    
    @GetMapping("/timeout")
    public int getTimeout() {
        return timeout; // 修改Nacos配置后，这里的值实时变化
    }
}
```

### 5.3 API网关——统一入口

**Q: API网关的作用是什么？**

```
没有网关：                           有网关：
                                    
客户端 ──→ 用户服务:8081              客户端 ──→ API Gateway :80
客户端 ──→ 订单服务:8082                           ↓ 根据路径路由
客户端 ──→ 商品服务:8083                   /user/**  → 用户服务:8081
客户端 ──→ 支付服务:8084                   /order/** → 订单服务:8082
                                           /product/**→ 商品服务:8083
问题：                                      /pay/**   → 支付服务:8084
  1. 跨域问题（每个服务都要配置）         
  2. 认证要在每个服务做一遍              网关统一处理：
  3. 前端要知道所有服务地址                 - 路由转发
  4. 限流、日志难以统一                    - 认证鉴权
                                          - 限流熔断
                                          - 日志收集
                                          - 跨域处理
```

**Spring Cloud Gateway路由配置：**

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
```

## 六、面试高频真题

### Q1: 微服务有哪些优缺点？

**优点：**
- ✅ 独立部署：改一个服务不用全量发布
- ✅ 技术异构：可以用不同语言写不同服务
- ✅ 独立扩展：只对瓶颈服务扩容
- ✅ 故障隔离：一个服务挂了不影响其他
- ✅ 团队自治：小团队聚焦单一业务

**缺点（微服务的代价）：**
- ❌ 分布式复杂性：网络延迟、分布式事务、数据一致性
- ❌ 运维成本高：需要CI/CD、监控、日志聚合、链路追踪
- ❌ 调试困难：一次请求可能经过多个服务
- ❌ 数据一致性：从ACID到BASE的转变
- ❌ 接口兼容性：服务间API版本管理

### Q2: 什么时候不适合用微服务？

- 项目初期，业务不明确（先单体快速验证）
- 团队规模小（<5人，微服务增加协调成本）
- 对性能有极致要求（避免网络开销）
- 业务逻辑简单（CRUD应用）

### Q3: 服务间调用超时了怎么办？

```java
// 方案1：设置合理的超时时间
@FeignClient(name = "order-service", configuration = FeignConfig.class)
public interface OrderClient {
    // 配置类中设置 connectTimeout=2000, readTimeout=5000
}

// 方案2：重试机制
@Retryable(value = {TimeoutException.class}, maxAttempts = 3, backoff = @Backoff(delay = 1000))
public Order getOrder(Long id) { ... }

// 方案3：熔断降级（最重要）
@HystrixCommand(fallbackMethod = "getOrderFallback",
    commandProperties = {
        @HystrixProperty(name = "execution.isolation.thread.timeoutInMilliseconds", value = "3000"),
        @HystrixProperty(name = "circuitBreaker.requestVolumeThreshold", value = "10"),
        @HystrixProperty(name = "circuitBreaker.errorThresholdPercentage", value = "50")
    })
public Order getOrder(Long id) { ... }

public Order getOrderFallback(Long id) {
    return getOrderFromCache(id); // 降级：返回缓存数据
}
```

### Q4: 分布式事务怎么处理？

| 方案 | 原理 | 适用场景 | 代表技术 |
|------|------|---------|---------|
| **2PC/XA** | 两阶段提交 | 强一致性要求 | JTA、Atomikos |
| **TCC** | Try-Confirm-Cancel | 资金交易 | Seata TCC模式 |
| **SAGA** | 正向补偿+逆向补偿 | 长事务 | Seata SAGA模式 |
| **最终一致性** | 本地消息表+MQ | 高并发场景 | RocketMQ事务消息 |
| **AT模式** | 自动回滚（undo log） | 最常见的业务场景 | Seata AT模式 |
