## 一、微服务测试金字塔

> 📌 **基础知识提示**：测试金字塔的概念、单元测试JUnit/Mockito最佳实践、全链路压测方法论等通用测试知识已在《系统分析与设计（七）——系统测试策略与可靠性工程》中深度覆盖，本文不再重复。本文聚焦微服务独有的测试挑战与解决方案。

**Q: 微服务的测试金字塔和单体应用有什么不同？**

```
单体应用测试金字塔（传统）：
       ┌──────┐
       │ E2E  │  少量：Selenium/Cypress 全功能验证
       ├──────┤
       │ 集成  │  中等：Spring Test + 真实数据库
       ├──────┤
       │ 单元  │  大量：JUnit/Mockito 单类测试
       └──────┘

微服务测试金字塔（加宽腰部）：
       ┌──────────┐
       │   E2E    │  极少：跨服务的关键业务流程
       ├──────────┤
       │ 契约测试  │  ← 微服务独有层级！Consumer-Driven Contract
       ├──────────┤
       │ 集成测试  │  服务级：Testcontainers + 真实依赖
       ├──────────┤
       │ 单元测试  │  类级：JUnit/Mockito
       └──────────┘

微服务测试的核心变化：
1. "腰部"变宽：集成测试比重显著上升（服务间通信是主要风险来源）
2. 新增"契约测试"层：验证服务间接口约定（微服务最大的坑）
3. E2E 比例进一步压缩：跨服务 E2E 成本极高、稳定性差
```

**测试策略决策树：**

```
                有外部依赖？
               /          \
             否              是
             ↓               ↓
         单元测试          跨服务调用？
         (Mock所有)       /          \
                        是            否
                        ↓             ↓
                   契约测试      集成测试
                   (CDC验证)    (Testcontainers)
                        ↓
                  API 格式正确？
                 /            \
               是              否
               ↓               ↓
          通过！         服务端修Bug
```

## 二、契约测试（Consumer-Driven Contract）

### 2.1 为什么微服务必须做契约测试？

**Q: 已经有了单元测试和集成测试，为什么还需要契约测试？**

这是微服务测试中的核心问题。

```
没有契约测试的经典事故：

第一周：订单服务（Consumer）依赖库存服务（Provider）的 /stock/check 接口
  请求：POST /stock/check  body: { "skuId": 1001, "quantity": 1 }
  响应：200 { "available": true, "stock": 100 }

第二周：库存团队重构代码，把字段名改成了 "inStock"（为了语义更清晰）
  响应：200 { "inStock": true, "stock": 100 }

第三周：上线后 → 订单服务解析 available 字段为 null → 大量订单误判为库存不足！
        损失：3小时的订单全部失败

┌─────────────────────────────────────────────────────────────────┐
│  问题根源：Provider 改 API 时，不知道 Consumer 依赖了哪些字段    │
│  契约测试的价值：让 Consumer 定义期望 → Provider 验证符合期望    │
│  核心思想："你承诺什么，我就测试什么"                            │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 Spring Cloud Contract 实战

**Consumer-Driven Contract (CDC) 工作流：**

```
┌──────────────────────────────────────────────────────────────────┐
│                    CDC 工作流                                     │
│                                                                  │
│  ① Consumer 定义契约                                             │
│     ┌─────────────────────┐                                     │
│     │  order-service/     │  需求："调用 POST /stock/check，     │
│     │  contracts/         │        期望返回 available 字段"      │
│     │  shouldCheckStock.groovy                                   │
│     └──────┬──────────────┘                                     │
│            │ ② PR 提交到 Provider                                  │
│            ▼                                                     │
│  ② Provider 验证契约                                             │
│     ┌─────────────────────┐                                     │
│     │  stock-service/     │  Provider 接收 Consumer 的契约,       │
│     │  运行契约测试       │  验证自己的 API 是否符合所有 Consumer │
│     │  → 全部通过 ✅      │  的期望                              │
│     └──────┬──────────────┘                                     │
│            │ ③ Provider 通过后发布 Stub JAR                        │
│            ▼                                                     │
│  ③ Consumer 使用 Stub 自测                                       │
│     ┌─────────────────────┐                                     │
│     │  order-service/     │  Consumer 不再需要启动真实的          │
│     │  运行 Stub Runner   │  库存服务，用 Stub 替代              │
│     │  → 集成测试通过 ✅   │                                      │
│     └─────────────────────┘                                     │
│                                                                  │
│  关键：Provider 不知道 Consumer 是谁，但 Consumer 的契约          │
│        成为了 Provider 的"测试用例"，Provider 改了 API            │
│        会第一时间被契约测试发现                                   │
└──────────────────────────────────────────────────────────────────┘
```

**实战配置：**

```groovy
// ===== ① Consumer 端：定义契约（Groovy DSL）=====
// order-service/src/test/resources/contracts/stock/
// shouldCheckStock.groovy

Contract.make {
    description "查询库存是否足够"
    request {
        method POST()
        url '/api/stock/check'
        headers {
            contentType applicationJson()
        }
        body([
            skuId: 1001,
            quantity: 1
        ])
    }
    response {
        status 200
        headers {
            contentType applicationJson()
        }
        body([
            available: true,
            stock: 100,
            skuId: 1001
        ])
    }
}

// ===== ② Provider 端：自动生成测试 =====
// stock-service/pom.xml
<plugin>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-contract-maven-plugin</artifactId>
    <version>4.1.0</version>
    <extensions>true</extensions>
    <configuration>
        <!-- 基础类，Provider 需要实现 BaseTestClass -->
        <baseClassForTests>com.example.stock.BaseContractTest</baseClassForTests>
    </configuration>
</plugin>

// stock-service/src/test/java/.../BaseContractTest.java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.MOCK)
@AutoConfigureMockMvc
public abstract class BaseContractTest {
    
    @Autowired
    private MockMvc mockMvc;
    
    @BeforeEach
    public void setup() {
        // 关键：把 MockMvc 传给契约验证器
        RestAssuredMockMvc.mockMvc(mockMvc);
        // 打桩：模拟真实 Controller 的行为
        RestAssuredMockMvc.standaloneSetup(new StockController(stockService));
    }
}

// 运行 mvn clean install → 自动验证契约 → 生成 Stub JAR
// → stock-service-1.0.0-stubs.jar 上传到 Maven 仓库

// ===== ③ Consumer 端：使用 Stub 测试 =====
// order-service/pom.xml
<dependency>
    <groupId>com.example</groupId>
    <artifactId>stock-service</artifactId>
    <version>1.0.0</version>
    <classifier>stubs</classifier>
    <scope>test</scope>
</dependency>

// 集成测试中使用 Stub
@SpringBootTest
@AutoConfigureStubRunner(
    ids = "com.example:stock-service:+:stubs:8082",
    stubsMode = StubRunnerProperties.StubsMode.LOCAL
)
class OrderServiceIntegrationTest {
    
    @Autowired
    private OrderService orderService;
    
    @Test
    void shouldCreateOrderWhenStockAvailable() {
        // Stub Runner 自动启动了模拟的库存服务在 8082 端口
        // 当 OrderService 调用库存服务时，Stub 返回契约中定义的响应
        Order order = orderService.createOrder(
            new CreateOrderRequest(1001L, 1));
        
        assertNotNull(order.getId());
        assertEquals(OrderStatus.CREATED, order.getStatus());
    }
}
```

### 2.3 Pact 框架：另一种契约测试方案

```
Spring Cloud Contract vs Pact 对比：

┌───────────────────────┬────────────────────────┬────────────────────────┐
│         维度           │   Spring Cloud Contract │   Pact                  │
├───────────────────────┼────────────────────────┼────────────────────────┤
│ 契约定义方式           │ Groovy DSL / YAML       │ 代码内 DSL (Java/JS等) │
│ Stub 生成              │ ✅ 自动生成 Stub JAR     │ ❌ 需自己搭建           │
│ 契约存储               │ Maven 仓库 (Stub JAR)   │ Pact Broker (独立服务)  │
│ 语言支持               │ JVM 为核心              │ 多语言 (Java/JS/Python) │
│ 学习曲线               │ 中等                    │ 较低                   │
│ 适用场景               │ Spring 生态优先         │ 多语言/异构系统        │
│ Provider 验证          │ Maven 插件自动化        │ 需要写测试代码         │
└───────────────────────┴────────────────────────┴────────────────────────┘

选择建议：
- 纯 Spring 技术栈 → Spring Cloud Contract（与 Maven 深度集成）
- 多语言/异构系统   → Pact（Pact Broker 统一管理契约）
- 多团队大公司      → Pact（Pact Broker 的契约矩阵非常直观）
```

```java
// Pact Consumer 端示例（订单服务 - Java）
@ExtendWith(PactConsumerTestExt.class)
class OrderServicePactTest {
    
    // 定义期望
    @Pact(consumer = "order-service", provider = "stock-service")
    V4Pact checkStockPact(PactDslWithProvider builder) {
        // 定义如果请求是 {skuId: 1001, quantity: 1}，
        // 期望返回 {available: true, stock: 100}
        return builder
            .given("SKU 1001 has stock 100")
            .uponReceiving("check stock availability")
                .method("POST")
                .path("/api/stock/check")
                .body("{\"skuId\": 1001, \"quantity\": 1}")
            .willRespondWith()
                .status(200)
                .body("{\"available\": true, \"stock\": 100, \"skuId\": 1001}")
            .toPact();
    }
    
    @Test
    @PactTestFor(pactMethod = "checkStockPact")
    void shouldGetStockAvailability(MockServer mockServer) {
        // mockServer 模拟了库存服务
        StockClient client = new StockClient(mockServer.getUrl());
        StockCheckResult result = client.check(1001L, 1);
        assertTrue(result.isAvailable());
    }
}
```

## 三、Testcontainers：真实依赖的集成测试

### 3.1 为什么不能用 H2/Embedded DB 替代

**Q: 为什么集成测试要用 Testcontainers 而不是 H2 嵌入式数据库？**

```java
// ===== H2 的问题 =====
// H2 和 MySQL 在以下场景行为不一致：

// 1. SQL 方言不同
// MySQL: INSERT INTO orders SET status = 'CREATED';   ✅ OK
// H2:    INSERT INTO orders SET status = 'CREATED';   ❌ 不支持 MySQL 语法

// 2. 函数不同
// MySQL: SELECT DATE_FORMAT(create_time, '%Y-%m-%d') FROM orders;   ✅
// H2:    SELECT DATE_FORMAT(create_time, '%Y-%m-%d') FROM orders;   ❌ 没有此函数

// 3. 非标准 SQL
// MySQL: SELECT * FROM orders LIMIT 10 OFFSET 0;     ✅
// H2:    SELECT * FROM orders LIMIT 10 OFFSET 0;     ✅ 但分页逻辑可能不同

// 4. 事务行为差异（最隐蔽！）
// MySQL 默认 REPEATABLE READ，H2 默认 READ COMMITTED
// → 测试通过，上线后才暴露事务 bug

// ===== Testcontainers 的优势 =====
// 直接用真实 MySQL Docker 容器测试
```

### 3.4 Testcontainers vs Embedded Database 面试应答

**Q: 什么情况下可以用 Embedded DB，什么情况下必须用 Testcontainers？**

```
使用 Embedded DB (H2/HSQLDB) 的场景：
✅ 纯 CRUD 的简单 Repository 测试
✅ 使用了 JPA/Hibernate 的标准映射，没有原生 SQL
✅ 对数据库特有功能没有依赖

必须使用 Testcontainers 的场景：
❌ 使用了数据库特有能力（MySQL JSON类型、PostGIS空间索引等）
❌ 写了原生 SQL / 存储过程
❌ 验证数据库迁移脚本（Flyway/Liquibase）
❌ 测试特定的索引策略或查询优化
❌ 验证事务隔离级别行为
❌ 多数据源场景

面试回答模板：
"我们团队的做法是能跑 Testcontainers 就跑 Testcontainers。
原因很简单：用 H2 测试通过但生产 MySQL 报错的教训太多了。
如果 CI 环境不方便用 Docker（比如一些企业的 CI 环境），
那就用 Testcontainers 的 Ryuk 容器自动管理生命周期，
或者降级用 Embedded DB 但要写专项的数据库兼容性测试。"
```

## 四、服务虚拟化：WireMock 与 Hoverfly

### 4.1 WireMock 实战

**Q: 下游服务不稳定或未开发完成时，如何做集成测试？**

```java
// ===== WireMock 基础用法 =====
@SpringBootTest
@WireMockTest(httpPort = 8089) // 自动启动 WireMock 在 8089
class OrderServiceTest {
    
    @Autowired
    private OrderService orderService;
    
    @Test
    void shouldCallStockService() {
        // 模拟库存服务的行为
        stubFor(post(urlEqualTo("/api/stock/check"))
                .withRequestBody(equalToJson("{\"skuId\":1001,\"quantity\":1}"))
                .willReturn(aResponse()
                        .withStatus(200)
                        .withHeader("Content-Type", "application/json")
                        .withBody("{\"available\":true,\"stock\":100}")));
        
        boolean available = orderService.checkStock(1001L, 1);
        assertTrue(available);
        
        // 验证是否真的调用了库存服务
        verify(postRequestedFor(urlEqualTo("/api/stock/check"))
                .withRequestBody(containing("1001")));
    }
}

// ===== WireMock 故障模拟 =====
@Test
void shouldHandleStockServiceTimeout() {
    // 模拟库存服务超时
    stubFor(post(urlEqualTo("/api/stock/check"))
            .willReturn(aResponse()
                    .withFixedDelay(5000)));  // 5000ms 后才响应
    
    // 验证熔断机制是否生效
    assertThrows(TimeoutException.class, () -> 
        orderService.checkStock(1001L, 1));
}

@Test
void shouldHandleStockServiceDown() {
    // 模拟库存服务 500
    stubFor(post(urlEqualTo("/api/stock/check"))
            .willReturn(serverError()
                    .withBody("Internal Server Error")));
    
    // 测试降级逻辑
    StockCheckResult result = orderService.checkStockWithFallback(1001L, 1);
    assertFalse(result.isAvailable()); // 降级后保守处理：不可用
}

// ===== WireMock 状态机：模拟复杂场景 =====
@Test
void shouldRetryAndSucceed() {
    // 前两次返回 503，第三次返回 200（模拟重试成功）
    stubFor(post(urlEqualTo("/api/stock/check"))
            .inScenario("retry scenario")
            .whenScenarioStateIs(Scenario.STARTED)
            .willReturn(aResponse().withStatus(503))
            .willSetStateTo("first-failure"));
    
    stubFor(post(urlEqualTo("/api/stock/check"))
            .inScenario("retry scenario")
            .whenScenarioStateIs("first-failure")
            .willReturn(aResponse().withStatus(503))
            .willSetStateTo("second-failure"));
    
    stubFor(post(urlEqualTo("/api/stock/check"))
            .inScenario("retry scenario")
            .whenScenarioStateIs("second-failure")
            .willReturn(aResponse().withStatus(200)
                    .withBody("{\"available\":true,\"stock\":100}")));
    
    // 验证重试机制（配置了 maxAttempts=3）
    StockCheckResult result = orderService.checkStock(1001L, 1);
    assertTrue(result.isAvailable());
}
```

### 4.2 WireMock vs Hoverfly 对比

| 维度 | WireMock | Hoverfly |
|------|----------|----------|
| 使用方式 | 代码内 DSL / 独立服务 | 独立服务（HTTP代理或Web服务器） |
| 录制-回放 | 支持但非核心功能 | 核心功能 |
| 故障注入 | 支持（延迟、错误码、状态机） | 更丰富（延迟、频率限制、数据篡改） |
| 性能测试 | 中等 | 强（支持中间件模式） |
| Spring 集成 | @WireMockTest 注解 | 需要手动管理 |
| 学习曲线 | 低 | 中等 |

**JaCoCo 覆盖率指标的面试要点：**

### 5.3 Mutation Testing：覆盖率的试金石

**Q: 如果团队发现代码覆盖率数据很好但 bug 不断，你会怎么诊断？**

```java
// 这就是 Mutation Testing 要解决的问题！
// 原理：自动修改你的代码（引入变异），看测试是否能发现

// 原始代码：
public boolean isEligibleForDiscount(User user) {
    return user.getAge() > 18 && user.isVip();
}

// 变异1：把 > 改成 >=
// return user.getAge() >= 18 && user.isVip();
// → 如果测试还通过 → 说明你的测试没有覆盖 age=18 的边界情况

// 变异2：把 && 改成 ||
// return user.getAge() > 18 || user.isVip();
// → 如果测试还通过 → 你的测试根本就没验证逻辑的"与"关系

// ===== 使用 PIT (Pitest) =====
// pom.xml
<plugin>
    <groupId>org.pitest</groupId>
    <artifactId>pitest-maven</artifactId>
    <version>1.15.0</version>
    <dependencies>
        <dependency>
            <groupId>org.pitest</groupId>
            <artifactId>pitest-junit5-plugin</artifactId>
            <version>1.2.0</version>
        </dependency>
    </dependencies>
    <configuration>
        <targetClasses>
            <param>com.example.order.service.*</param>
        </targetClasses>
        <targetTests>
            <param>com.example.order.service.*</param>
        </targetTests>
        <mutators>
            <mutator>STRONGER</mutator>  <!-- 更强的变异算子 -->
        </mutators>
    </configuration>
</plugin>

// mvn pitest:mutationCoverage
// 输出报告：Mutation Coverage 85%（行覆盖率可能是 95%，但变异覆盖率只有 85%）
```

## 七、面试高频问答

**Q1: 微服务测试中最大的挑战是什么？**

```
答：最大的挑战是"服务间依赖的不确定性"。

具体来说：
1. 下游 API 悄悄修改 → 契约测试解决
2. 下游服务不可用 → 熔断 + 降级 + WireMock 模拟
3. 数据状态不一致 → Testcontainers 用真实 DB 验证
4. 异步消息丢失 → 集成测试验证 Kafka 消息发送和消费

一句话总结：不要相信任何外部依赖，把它们都测试到。
```

**Q2: 代码覆盖率应该设多少？100% 有必要吗？**

```
答：没有必要追求 100%，这是一个常见的误区。

我的建议：
- 行覆盖率：70% 是基准，80% 优秀
- 分支覆盖率：50% 是基准，更重要
- 增量代码：≥ 80%

100% 覆盖率的问题：
1. getter/setter/toString 也要写测试？浪费
2. 容易"刷"覆盖率：只关注执行不关注断言质量
3. 边际成本急剧上升：从 80% 到 100% 的劳动量可能翻倍

更好的做法：关注"关键路径覆盖率"
- 支付流程：必须 100% 覆盖（含异常分支）
- 日志配置：0% 也可以
- 用户推荐算法：核心逻辑 90%+
```

**Q3: 契约测试和集成测试有什么区别？**

```
契约测试 = 验证 Provider 的 API 是否符合 Consumer 的期望
集成测试 = 验证 Consumer 能否正确地与真实的（或接近真实的）Provider 交互

类比：
- 契约测试像"接口文档的自动校验"：你说 GET /user/1 返回 name 字段，
  那就自动验证真的返回了 name 字段
- 集成测试像"真实对话"：启动真实服务，真的发请求，验证端到端流程

关系：契约测试通过后，集成测试才有意义。
```

**Q4: Testcontainers 和 WireMock 的使用场景如何区分？**

| 场景 | 推荐工具 | 原因 |
|------|---------|------|
| 测试自己的数据库操作 | Testcontainers | 需要真实 SQL 行为 |
| 测试第三方 API 调用 | WireMock | 第三方服务不可控 |
| 测试内部微服务集成 | 优先 Spring Cloud Contract Stub<br>备选 WireMock | Stub 更贴近真实行为 |
| 测试 Kafka/Redis 交互 | Testcontainers | 需要真实中间件行为 |
| 模拟故障场景 | WireMock | 灵活的状态机和延迟控制 |
| CI 流水线（无 Docker） | WireMock | CI 环境可能无法运行容器 |
