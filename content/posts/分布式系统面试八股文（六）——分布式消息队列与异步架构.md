### 选型面试应答

**Q: 电商订单系统用哪个MQ？为什么？**

> **RocketMQ**。原因：(1) 事务消息保障「下单+减库存+发优惠券」的分布式事务一致性 (2) 支持延时消息（下单30分钟未支付自动取消） (3) 支持消息回溯（运维定位问题） (4) 亿级消息堆积能力（双11削峰） (5) 阿里双11验证过的稳定性。Kafka虽然吞吐更高但缺事务/延时消息，RabbitMQ延迟低但堆积能力和事务消息不如RocketMQ。

## 三、消息可靠性保障 —— 不丢消息

### 面试高频问题

| # | 问题 | 重要度 | 核心要点 |
|---|------|--------|---------|
| 1 | 消息丢失的三个环节？ | ⭐⭐⭐⭐⭐ | 生产→Broker→消费，各环节都需要ACK |
| 2 | 生产端如何保证不丢？ | ⭐⭐⭐⭐ | 同步发送+确认回调+重试+本地消息表兜底 |
| 3 | Broker端如何保证不丢？ | ⭐⭐⭐⭐⭐ | 同步刷盘+多副本(ISR)+Leader确认 |
| 4 | 消费端如何保证不丢？ | ⭐⭐⭐⭐⭐ | 先消费后ACK，处理好重试 |
| 5 | 各类MQ的不丢配置？ | ⭐⭐⭐⭐ | 不同MQ的参数组合 |

### 消息丢失全链路分析

```
Producer                    Broker                      Consumer
   │                          │                            │
   │──(1) send(msg)────→     │                            │
   │                          │                            │
   │  风险1: 网络丢包/异步刷盘  │                            │
   │  方案: 同步发送+重试      │──(2) persist──→           │
   │←── sendResult ──────     │                            │
   │                          │  风险2: 磁盘故障+单点      │
   │                          │  方案: 多副本+ISR+同步刷盘  │
   │                          │                            │
   │                          │──(3) push/consumer pull──→│
   │                          │                            │
   │                          │  风险3: 消费后未处理就ACK   │
   │                          │  方案: 业务处理完再ACK     │
   │                          │                            │
   │                          │                    ← ─ ACK─
```

### 各MQ不丢配置

```java
// Kafka Producer — 不丢消息配置
Properties props = new Properties();
props.put("acks", "all");                     // 等待所有ISR确认
props.put("retries", Integer.MAX_VALUE);       // 无限重试
props.put("max.in.flight.requests.per.connection", 1); // 保证顺序
props.put("enable.idempotence", true);         // 幂等生产者

// Kafka Broker
// min.insync.replicas = 2  (至少2个ISR副本确认)
// unclean.leader.election.enable = false  (不允许落后副本当选Leader)

// Kafka Consumer — 不丢消息配置
props.put("enable.auto.commit", "false");     // 手动提交
// 消费处理完后: consumer.commitSync();

// RocketMQ Producer
producer.setSendMsgTimeout(3000);
producer.setRetryTimesWhenSendFailed(3);
// Broker: flushDiskType=SYNC_FLUSH, 同步双写(syncMaster=true)

// RocketMQ Consumer
consumer.registerMessageListener(new MessageListenerConcurrently() {
    @Override
    public ConsumeConcurrentlyStatus consumeMessage(List<MessageExt> msgs, ...) {
        try {
            // 业务处理(幂等)
            processBusiness(msgs);
            return ConsumeConcurrentlyStatus.CONSUME_SUCCESS; // 成功才ACK
        } catch (Exception e) {
            return ConsumeConcurrentlyStatus.RECONSUME_LATER; // 失败重试
        }
    }
});
```

### 面试追问应答

**Q: Kafka acks=all 和 min.insync.replicas=2 配合使用才能保证不丢？**

> `acks=all` 要求所有ISR副本确认后才返回成功，但如果ISR只有一个（Leader），那本质上还是单副本。配置 `min.insync.replicas=2` 强制要求至少2个ISR副本，当ISR不足2个时拒绝写入抛出 `NotEnoughReplicasException`，从源头上防止数据丢失。两者配合：acks=all定义了写入成功的标准，min.insync.replicas保证了写入时至少有多少副本。

## 四、幂等性保障

### 面试高频问题

| # | 问题 | 重要度 | 核心要点 |
|---|------|--------|---------|
| 1 | 消息为什么会重复？ | ⭐⭐⭐⭐ | 生成端重试/网络超时/Broker故障转移 |
| 2 | 幂等性如何实现？ | ⭐⭐⭐⭐⭐ | 数据库唯一索引 + 业务状态机 + Redis去重 |
| 3 | Kafka的幂等生产者原理？ | ⭐⭐⭐⭐ | PID + Sequence Number 去重 |
| 4 | 支付场景幂等怎么设计？ | ⭐⭐⭐⭐⭐ | 预生成全局唯一支付号 + 状态机 |

### 幂等方案对比

| 方案 | 实现方式 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| 数据库唯一索引 | `INSERT ... ON DUPLICATE KEY` | 简单可靠 | 插入性能损耗 | 订单/流水 |
| Redis去重 | `SETNX msgId:xxx 1 EX 3600` | 高性能 | Redis故障存风险 | 高频去重 |
| 状态机 | IF status='待支付' THEN UPDATE | 天然幂等 | 状态管理复杂 | 订单/流程 |
| Kafka幂等 | `enable.idempotence=true` | Producer端自动 | 仅单分区+单会话 | Kafka专用 |

### 支付场景幂等实战

```java
@Service
public class PaymentService {
    
    // 方案一：数据库唯一索引 + 状态机
    @Transactional
    public void processPayment(PaymentMessage msg) {
        // 1. 根据业务唯一键查重
        PaymentRecord exist = paymentMapper.selectByBizNo(msg.getBizOrderNo());
        if (exist != null) {
            log.info("重复消息, 当前状态: {}", exist.getStatus());
            return; // 已处理, 幂等返回
        }
        
        // 2. 插入支付记录（唯一索引保证并发安全）
        PaymentRecord record = new PaymentRecord();
        record.setBizOrderNo(msg.getBizOrderNo());  // biz_order_no 唯一索引
        record.setStatus("PENDING");
        try {
            paymentMapper.insert(record);
        } catch (DuplicateKeyException e) {
            return; // 并发插入 → 幂等
        }
        
        // 3. 执行业务
        executePayment(msg);
        record.setStatus("SUCCESS");
        paymentMapper.updateById(record);
    }

    // 方案二：Redis去重 + DB兜底
    public void processWithRedis(PaymentMessage msg) {
        String key = "payment:dedup:" + msg.getBizOrderNo();
        // SETNX 原子去重, 设置2倍最大消费时间
        Boolean set = redisTemplate.opsForValue()
            .setIfAbsent(key, "1", Duration.ofMinutes(30));
        if (Boolean.FALSE.equals(set)) {
            log.info("Redis去重命中, 跳过");
            return;
        }
        try {
            processPayment(msg); // 使用方案一的DB去重做兜底
        } catch (Exception e) {
            redisTemplate.delete(key); // 失败删除Key, 让重试再次处理
            throw e;
        }
    }
}
```

### 面试追问应答

**Q: 唯一索引去重在分库分表场景下怎么办？**

> 如果订单号不能路由到同一分片，有两种方案：(1) **集中式幂等表** — 单独一张不分库的幂等记录表，统一管理去重key，由定时任务清理过期记录 (2) **路由键改造** — 在订单号中嵌入分片键，使业务唯一键能够路由到固定分片。方案(1)简单但有单点风险，方案(2)优雅但对订单号生成有约束。

### 面试高频问题

| # | 问题 | 重要度 | 核心要点 |
|---|------|--------|---------|
| 1 | 为什么要保证消息顺序？ | ⭐⭐⭐⭐ | 创建→更新→删除必须有序 |

| 3 | Kafka如何实现顺序？ | ⭐⭐⭐⭐ | 单分区 + 单消费者 + max.in.flight=1 |
| 4 | RocketMQ顺序消息原理？ | ⭐⭐⭐⭐ | MessageQueueSelector + 锁机制 |
| 5 | 顺序消息的代价？ | ⭐⭐⭐ | 牺牲吞吐量+可用性 |

### 局部顺序实现方案

```
                      ┌──────────────┐
                      │   Topic      │
                      └──────┬───────┘
                ┌────────────┼────────────┐
                ↓            ↓            ↓
          Partition 0   Partition 1   Partition 2
          (orderId%3=0) (orderId%3=1) (orderId%3=2)
                │            │            │
                ↓            ↓            ↓
           Consumer 1   Consumer 2   Consumer 3
              串行消费      串行消费      串行消费

核心原则: 同一订单的所有操作 → 同一分区 → 同一消费者 → 串行处理
```

### RocketMQ顺序消息实现

```java
// Producer — 顺序发送
SendResult sendResult = producer.send(msg, new MessageQueueSelector() {
    @Override
    public MessageQueue select(List<MessageQueue> mqs, Message msg, Object arg) {
        Long orderId = (Long) arg;
        int index = (int) (orderId % mqs.size());
        return mqs.get(index); // 同一订单进入同一Queue
    }
}, orderId);

// Consumer — 顺序消费 (MessageListenerOrderly)
consumer.registerMessageListener(new MessageListenerOrderly() {
    @Override
    public ConsumeOrderlyStatus consumeMessage(List<MessageExt> msgs, 
                                                ConsumeOrderlyContext ctx) {
        // 加锁 → 保证同一Queue同一时刻只有一个线程消费
        for (MessageExt msg : msgs) {
            // 串行处理
            processMessage(msg);
        }
        return ConsumeOrderlyStatus.SUCCESS;
    }
});
```

### 面试追问应答

**Q: 为什么全局顺序几乎不可行？代价是什么？**

> 全局顺序要求：(1) 只有一个分区 (2) 只有一个消费者 (3) 生产者不能并发发送。这完全丧失了MQ的**水平扩展能力**和**高吞吐量**优势。一个分区打满单机磁盘IO，一个消费者打满单机CPU。实际业务只需要**局部有序**——如同一订单的操作有序，不同订单间可以并行。这恰好符合分区模型。

            .collect(Collectors.toList());

        orderMapper.batchInsert(orders);
        
        // 手动提交 offset (确保入库成功再提交)
        // ack.acknowledge();
    }
    
    // 2. 并行消费 — 提升吞吐
    @KafkaListener(topics = "order-topic", concurrency = "6")
    public void parallelConsume(ConsumerRecord<String, String> record) {
        // concurrency=6 → 每个消费者6个线程并行
        processOrder(record.value());
    }
    
    // 3. 异步化下游 — 避免阻塞

        }, executor);
    }

### 面试追问应答

**Q: 增加Consumer节点一定能解决消息堆积吗？**

> **不一定**。Kafka的分区数是固定的——一个分区同一时刻只能被Consumer Group中的一个消费者消费。如果Topic只有3个Partition，你增加到10个Consumer实例，也只有3个在处理，其他7个空闲。**消费并行度上限 = min(Partition数, Consumer数)**。所以消息堆积的根因如果是分区不足，需要先增加分区，再扩容消费者。

**Q: RocketMQ的重试机制和死信队列是什么关系？**

> RocketMQ消费失败后，消息进入重试队列(延迟级别递进：1s→10s→30s→...→2h)。重试16次后仍未成功，消息转入死信队列(%DLQ%+ConsumerGroup)。死信队列需要人工介入或配置定时补偿脚本处理，避免占用重试资源。

### 同步调用 vs 异步解耦

```
同步调用 (RPC):                      异步解耦 (MQ):
                                    
    下单服务                              下单服务
    │  │  │                                │
    ↓  ↓  ↓                                ↓
  库存 支付 通知                            MQ ──→ 库存服务
    (串行等待)                              ├──→ 支付服务
  总耗时 = Σ(Ti)                           └──→ 通知服务
  (依赖强耦合)                              (异步并行, 解耦独立)
  失败 = 全链路异常                         失败 = 各自重试, 局部影响
```

### CDC (Change Data Capture) 事件驱动

```
业务数据库 (MySQL)
    │
    │ Binlog 监听
    ↓
Canal/Debezium
    │
    │ 解析变更事件
    ↓
Kafka/RocketMQ
    │
    ├──→ 搜索服务 (ES索引同步)
    ├──→ 缓存服务 (Redis更新)
    ├──→ 数据仓库 (离线分析)
    └──→ 业务通知 (推送/邮件)
```

### 面试追问应答

**Q: 用MQ解耦后，如何保证发消息和本地事务的一致性？**

> 三种方案：(1) **事务消息**（RocketMQ）— 先发半消息→执行本地事务→根据结果Commit或Rollback，Broker通过回查兜底 (2) **本地消息表** — 在业务数据库中建 `outbox` 表，本地事务中写入待发送消息，定时任务扫描发送+确认 (3) **CDC监听Binlog** — Canal监听MySQL Binlog，自动捕获变更事件发送到MQ，零侵入但依赖数据库。

**Q: 事件驱动的最大挑战是什么？**

> (1) **数据一致性** — 事件投递和本地事务的原子性难保证 (2) **事件顺序** — 并发下事件可能乱序到达 (3) **可观测性** — 事件链路追踪比RPC调用链复杂 (4) **整体性理解** — 系统行为分散在多个事件处理者中，新人难以理解完整业务流程。最佳实践：事务消息 + 全链路追踪 + 事件Schema治理。

## 九、面试加分弹药库

### 经典面试连环问

**面试官：为什么用RocketMQ而不用Kafka？**

> 1. 我们业务场景是电商交易，对事务消息有强需求（下单+减库存需原子保障），RocketMQ原生支持，Kafka的事务是流式场景的**恰好一次**语义，不完全对等
> 2. 延时消息（下单30分钟未支付取消）是刚需，Kafka没有现成方案
> 3. 消息回溯按时间维度在线上故障排查中高频使用
> 4. RocketMQ的多语言客户端成熟度虽不如Kafka，但Java生态更适配我们全栈Java的技术栈
> 5. Kafka更适合日志/流计算/大数据场景，吞吐极高但业务消息的功能支持不如RocketMQ

**面试官：Kafka高吞吐量的技术原理，说透？**

> 1. **顺序读写** — Kafka将消息追加到日志末尾，无随机IO，充分利用磁盘顺序写(~600MB/s)远快于随机写(~100KB/s)
> 2. **零拷贝(Zero-Copy)** — sendfile()系统调用直接将数据从PageCache传输到Socket Buffer，不经过用户态（省2次拷贝+2次上下文切换）
> 3. **PageCache** — 消息先写入OS的PageCache（内存），由OS异步刷盘，读取也优先命中PageCache
> 4. **批量+压缩** — Producer端按大小/时间窗口批量发送，支持GZip/Snappy/LZ4压缩，Broker直接原封转发
> 5. **分区并行** — Topic分为多个Partition，每个Partition独立读写，线性扩展

**面试官：怎么设计一个能支撑双11的MQ架构？**

> (1) **削峰填谷** — 前端请求不直接访问后端，通过MQ暂存，后端按自身能力匀速消费 (2) **分区规划** — 提前评估TPS，设计足够分区数保证并行度 (3) **容量评估** — 按照峰值QPS×削峰时间窗口估算堆积量 (4) **弹性扩容** — Consumer支持HPA自动伸缩，Broker集群提前扩容 (5) **降级熔断** — 消息堆积超阈值时触发降级（跳过非核心消息） (6) **监控告警** — 消费延迟、堆积量、消费失败率三个核心监控指标 (7) **流量染色** — 全链路压测使用影子Topic/消息标记，隔离真实流量

## 十、总结

| 主题 | 核心要点 | 一句话总结 |
|------|---------|-----------|
| 选型 | Kafka大数据/RocketMQ业务/RabbitMQ低延迟 | 业务消息首选RocketMQ |
| 不丢消息 | 生产端同步+确认、Broker多副本+同步刷盘、消费端手动ACK | 三条链路各司其职 |
| 幂等 | 唯一索引/Redis去重/状态机 | 宁可重复处理，不能重复扣款 |
| 顺序 | 同一Key→同一分区→同一Consumer→串行 | 全局顺序不可行，局部顺序就够了 |
| 堆积治理 | 扩容+批量+降级+监控 | 分钟级发现，小时级清零 |
| 异步解耦 | 事务消息/本地消息表/CDC | 发消息和业务操作必须原子 |

> 🎯 **消息队列的本质**：在分布式系统中充当**异步中枢**，将同步的、耦合的调用链变为异步的、解耦的事件流。理解MQ不仅仅是会用API——理解它的可靠性机制、性能原理和适用边界，才能在设计系统时做出正确的技术选型。
