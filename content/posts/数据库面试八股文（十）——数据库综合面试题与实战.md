## 一、SQL优化实战

### 1.1 慢查询定位与分析

**Q：如何定位和优化慢查询？**

定位手段：
```sql
-- 开启慢查询日志
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过1秒记录
SET GLOBAL log_queries_not_using_indexes = ON;

-- 查看慢查询日志位置
SHOW VARIABLES LIKE 'slow_query_log_file';

-- 使用mysqldumpslow分析
mysqldumpslow -s t -t 10 /var/lib/mysql/slow.log
```

EXPLAIN分析关键指标：

| 字段 | 关注点 | 优化方向 |
|------|--------|---------|
| type | ALL全表扫描最差 | 至少达到range级别 |
| key | 是否使用索引 | 添加合适索引 |
| rows | 预估扫描行数 | 减少扫描范围 |
| Extra | Using filesort/temporary | 消除额外排序和临时表 |
| filtered | 过滤比例 | 提高索引选择性 |

### 1.2 索引优化黄金法则

**Q：如何设计高效的索引？**

```
索引设计原则：
1. 选择性高的列优先（区分度 > 0.1）
2. 遵循最左前缀匹配
3. 覆盖索引避免回表
4. 避免在索引列上做运算/函数
5. 联合索引遵循ESR原则（Equality → Sort → Range）
6. 字符串索引考虑前缀索引
7. 频繁更新的列慎建索引
```

实战案例——避免索引失效：
```sql
-- ❌ 索引失效
SELECT * FROM orders WHERE YEAR(create_time) = 2026;
SELECT * FROM users WHERE name LIKE '%张';
SELECT * FROM orders WHERE id + 1 = 100;
SELECT * FROM users WHERE name = '张三' OR age = 25;  -- age无索引

-- ✅ 索引有效
SELECT * FROM orders WHERE create_time >= '2026-01-01' AND create_time < '2027-01-01';
SELECT * FROM users WHERE name LIKE '张%';
SELECT * FROM orders WHERE id = 99;
SELECT * FROM users WHERE name = '张三' UNION SELECT * FROM users WHERE age = 25;
```

### 1.3 复杂查询优化

**Q：如何优化JOIN查询和子查询？**

```sql
-- 小表驱动大表原则
-- ✅ 先过滤小表再JOIN
SELECT o.* FROM orders o
INNER JOIN (
    SELECT id FROM users WHERE status = 1  -- 先过滤，减少JOIN行数
) u ON o.user_id = u.id;

-- 子查询 → JOIN优化
-- ❌ 依赖子查询，每行执行一次
SELECT * FROM orders o
WHERE EXISTS (SELECT 1 FROM users u WHERE u.id = o.user_id AND u.vip = 1);

-- ✅ 改为JOIN
SELECT o.* FROM orders o
INNER JOIN users u ON o.user_id = u.id AND u.vip = 1;
```

## 二、分库分表实战

### 2.1 为什么需要分库分表

**Q：什么时候该分库分表？有哪些方案？**

```
触发条件：
- 单表数据量 > 5000万行（MySQL B+树3层上限）
- 单表数据文件 > 20GB
- QPS超过单机承载能力（通常5000-8000 QPS）
- 慢查询频繁且优化无效

方案对比：
┌──────────┬────────────────┬──────────────────┐
│  方案     │  适用场景       │  限制             │
├──────────┼────────────────┼──────────────────┤
│ 读写分离   │ 读多写少       │ 主从延迟          │
│ 垂直分库   │ 业务耦合大     │ 跨库JOIN/事务     │
│ 垂直分表   │ 大字段多       │ 主键冗余          │
│ 水平分库   │ 数据量极大     │ 路由/扩容复杂     │
│ 水平分表   │ 单表数据量大   │ 路由计算          │
└──────────┴────────────────┴──────────────────┘
```

### 2.2 分片策略选择

**Q：分片键如何选择？有哪些路由算法？**

分片键选择原则：
- **高基数**：取值范围大，分布均匀
- **查询必带**：大多数查询包含该字段
- **不可变**：值不会更新
- **业务相关**：优先按租户/用户维度

路由算法对比：

| 算法 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| 取模 | hash(key) % N | 数据均匀 | 扩容需迁移 |
| 范围 | key在[start,end) | 扩容方便 | 热点问题 |
| 一致性哈希 | 虚拟节点环 | 扩容迁移少 | 实现复杂 |
| 基因法 | 嵌入分片基因 | 避免二次查询 | 设计复杂 |

### 2.3 分库分表后的核心问题

```python
# 一、分布式ID生成
# 方案：Snowflake → 64bit: 1符号 + 41时间 + 10机器 + 12序列
```

## 三、数据迁移方案

### 3.1 全量+增量迁移

**Q：如何实现不停机的数据库迁移？**

```
不停机迁移核心流程：

1. 全量迁移（历史数据）
   └── mysqldump / DataX / Canal

2. 增量同步（变更数据）
   └── Canal监听binlog → 写入新库

3. 数据校验
   └── 对比新旧库数据一致性

4. 灰度切流
   └── 10% → 50% → 100% 流量切换

5. 观察期
   └── 旧库保留7天，新库无问题后下线旧库
```

### 3.2 双写一致性

**Q：数据迁移过程中如何保证双写一致？**

```java
// 双写模式 - 同步写旧库 + 异步写新库
@Transactional
public void saveOrder(Order order) {
    // 1. 写旧库（主流程）
    oldOrderMapper.insert(order);
    
    // 2. 异步写新库（保证最终一致）
    mqTemplate.send("order-sync-topic", order);
}

// 消费者：写新库 + 对账
@RocketMQMessageListener(topic = "order-sync-topic")
public class OrderSyncConsumer implements RocketMQListener<Order> {
    public void onMessage(Order order) {
        newOrderMapper.insert(order);
        // 定时对账：补偿丢失数据
    }
}
```

## 四、容量规划与高可用

### 4.1 容量评估

**Q：如何进行数据库容量规划？**

```
容量规划公式：

数据量评估：
  总数据量 = 日增数据 × 保留天数 × 副本数 × 冗余系数(1.5)
  
  示例：日增100万行 × 365天 × 3副本 × 1.5 = 16.4亿行
  每行约500B → 总量约820GB → 需分4个分片

QPS评估：
  峰值QPS = 日均QPS × 峰值系数(3~5)
  
  示例：日均5000 QPS × 5 = 峰值25000 QPS
  单机承载5000 → 需要5个分片（含1个冗余）

存储规划：
  SSD: IOPS = 容量GB × 30（通用SSD）
  需要IOPS = 峰值QPS × 10（读写比1:10）
```

### 4.2 高可用架构

**Q：数据库高可用有哪些方案？**

```
MySQL高可用演进：

1. 主从复制 + 手动切换
   └── 简单但需要人工干预

2. MHA（Master High Availability）
   └── 自动故障转移，30秒内切换
   └── 局限：只支持一主多从

3. MGR（MySQL Group Replication）
   └── Paxos协议，自动选主
   └── 单主/多主模式

4. InnoDB Cluster
   └── MGR + MySQL Router + MySQL Shell
   └── Oracle官方推荐方案

5. 云数据库高可用
   └── 跨可用区部署 + 自动故障切换
   └── RDS/PolardB/TiDB
```

## 五、综合面试真题

### 5.1 场景题

**Q1：线上MySQL突然变慢，如何排查？**

```
排查步骤：
1. 查看监控：CPU、内存、磁盘IO、连接数
2. SHOW PROCESSLIST：是否有锁等待、长事务
3. 慢查询日志：最近新增的慢SQL
4. InnoDB状态：SHOW ENGINE INNODB STATUS
5. 锁信息：SELECT * FROM information_schema.INNODB_LOCKS

常见原因：
- 突发大量查询 → 缓存击穿
- 锁等待 → 大事务未提交
- 磁盘IO高 → checkpoint刷脏页
- 连接数满 → 连接泄漏
```

**Q2：如何设计一个秒杀系统的数据库层？**

```
秒杀数据库设计要点：

1. 库存扣减
   ├── 方案1：Redis预扣 + 异步落DB
   ├── 方案2：UPDATE stock SET count=count-1 WHERE id=? AND count>0
   └── 避免超卖：乐观锁/Redis原子操作

2. 订单创建
   ├── 先写Redis，异步MQ写DB
   └── 本地消息表保证最终一致

3. 限流保护
   ├── DB前加Redis令牌桶
   └── 连接池限制最大连接数

4. 扩容
   ├── 热点商品分片
   └── 读写分离 + 缓存
```

**Q3：100亿数据如何高效查询？**

```
100亿数据查询方案：

1. 分库分表
   ├── 按 userID 分256个表
   └── 每表约4000万行

2. 分层存储
   ├── 热数据（近3月）→ MySQL
   ├── 温数据（3-12月）→ ClickHouse
   └── 冷数据（>1年）→ HDFS/Hive

3. 搜索能力
   └── Elasticsearch 全文检索 + 聚合

4. 缓存策略
   ├── 热点数据 → Redis
   └── 布隆过滤器防穿透
```

**Q4：设计一个多租户SaaS的数据库架构？**

```
多租户架构选择：

┌──────────┬──────────────┬──────────────┬──────────────┐
```
