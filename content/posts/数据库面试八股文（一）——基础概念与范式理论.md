## [](#概述)概述

数据库是软件开发中最核心的基础设施之一，也是技术面试的必考内容。本系列将系统地讲解数据库面试的核心知识点，从基础概念到MySQL进阶，再到Redis实战，帮助你构建扎实的数据库知识体系。

本文作为系列开篇，重点讲解数据库的基础概念和范式理论——这是理解关系型数据库的基石。

## [](#数据库基本概念)数据库基本概念

### [](#什么是数据库)什么是数据库

**数据库（Database）**是按照数据结构来组织、存储和管理数据的仓库。

**数据库管理系统（DBMS）** 是操作和管理数据库的软件，常见的有：
- **关系型数据库**：MySQL、PostgreSQL、Oracle、SQL Server
- **非关系型数据库**：Redis（KV）、MongoDB（文档）、Cassandra（列族）、Neo4j（图）

### [](#数据库vs文件系统)数据库vs文件系统

| 对比项 | 文件系统 | 数据库 |
|--------|----------|--------|
| 数据组织 | 文件 | 表（行+列） |
| 查询效率 | 低（全扫描） | 高（索引） |
| 数据冗余 | 高 | 低（范式） |
| 并发控制 | 无 | 有（锁、事务） |
| 数据安全 | 低 | 高（权限、备份） |
| 数据一致性 | 难保证 | 易保证（ACID） |

### [](#关系型数据库的核心概念)关系型数据库的核心概念

```
数据库（Database）
    ├── 表（Table）= 实体（Entity）
    ├── 行（Row）= 记录（Record）= 元组（Tuple）
    ├── 列（Column）= 字段（Field）= 属性（Attribute）
    ├── 主键（Primary Key）= 唯一标识
    ├── 外键（Foreign Key）= 表间关联
    └── 索引（Index）= 加速查询
```

**关系（Relation）**的本质：数学上的集合（笛卡尔积的子集）

## [](#SQL语言分类)SQL语言分类

SQL（Structured Query Language）分为四大类：

### [](#DDL-数据定义语言)DDL（数据定义语言）

用于定义数据库对象（数据库、表、列）：

```sql
-- 创建数据库
CREATE DATABASE mydb CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

-- 创建表
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) NOT NULL UNIQUE,
    password VARCHAR(100) NOT NULL,
    email VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 修改表
ALTER TABLE users ADD COLUMN age INT DEFAULT 0;
ALTER TABLE users MODIFY COLUMN username VARCHAR(100);
ALTER TABLE users DROP COLUMN age;

-- 删除表
DROP TABLE users;
TRUNCATE TABLE users;  -- 清空数据，保留表结构
```

### [](#DML-数据操作语言)DML（数据操作语言）

用于操作表中的数据：

```sql
-- 插入
INSERT INTO users (username, password, email) VALUES ('zhangsan', '123456', 'zhangsan@example.com');
INSERT INTO users (username, password) VALUES 
    ('lisi', '111111'),
    ('wangwu', '222222');

-- 更新
UPDATE users SET email = 'new@example.com' WHERE id = 1;
UPDATE users SET age = age + 1 WHERE username LIKE 'zhang%';

-- 删除
DELETE FROM users WHERE id = 1;
DELETE FROM users;  -- 删除所有数据（逐行删除，可回滚）
TRUNCATE TABLE users;  -- 清空表（DDL操作，不可回滚，速度快）
```

### [](#DQL-数据查询语言)DQL（数据查询语言）

用于查询数据（最复杂、最重要）：

```sql
SELECT 
    column1, column2, 
    COUNT(*) as cnt,
    AVG(age) as avg_age
FROM table_name
WHERE condition
GROUP BY column1, column2
HAVING cnt > 10
ORDER BY avg_age DESC
LIMIT 10 OFFSET 20;
```

**执行顺序**（面试常考）：
```
FROM → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT
```

### [](#DCL-数据控制语言)DCL（数据控制语言）

用于权限控制：

```sql
-- 创建用户
CREATE USER 'app_user'@'%' IDENTIFIED BY 'password';

-- 授权
GRANT SELECT, INSERT ON mydb.* TO 'app_user'@'%';
GRANT ALL PRIVILEGES ON *.* TO 'admin'@'localhost';

-- 撤销权限
REVOKE INSERT ON mydb.* FROM 'app_user'@'%';

-- 删除用户
DROP USER 'app_user'@'%';
```

## [](#范式理论)范式理论

范式（Normalization）是关系型数据库设计时遵循的规范，目的是**减少数据冗余、避免数据异常**。

> 💡 **面试核心**：理解每个范式的含义、判断表是否满足某范式、知道如何进行规范化。

### [](#第一范式-1NF)第一范式（1NF）

**定义**：表的每个列都是**原子性**的（不可再分）。

❌ **不满足1NF的例子**：
```
学生表：
| 学号 | 姓名  | 联系电话              |
|------|--------|-----------------------|
| 001  | 张三  | 13800138000,13900139000 |
```

一个学生有多个电话，放在同一列中，不满足1NF。

✅ **满足1NF的改造**：
```
方案1：拆成多行
| 学号 | 姓名  | 联系电话    |
|------|--------|-------------|
| 001  | 张三  | 13800138000|
| 001  | 张三  | 13900139000|

方案2：拆成另一张表
学生表：| 学号 | 姓名 |
电话表：| 学号 | 联系电话 |
```

### [](#第二范式-2NF)第二范式（2NF）

**定义**：在满足1NF的基础上，**非主键列必须完全依赖于主键**（不能只依赖主键的一部分）。

❌ **不满足2NF的例子**（联合主键）：
```
选课表（学号，课程号，学生姓名，课程名称，成绩）
主键：（学号，课程号）
```
问题：`学生姓名` 只依赖于 `学号`（主键的一部分），`课程名称` 只依赖于 `课程号`（主键的一部分）。这叫"部分依赖"。

✅ **满足2NF的改造**（拆表）：
```
学生表：（学号，学生姓名）
课程表：（课程号，课程名称）
选课表：（学号，课程号，成绩）  ← 成绩完全依赖于（学号，课程号）
```

### [](#第三范式-3NF)第三范式（3NF）

**定义**：在满足2NF的基础上，**非主键列不能传递依赖于主键**。

❌ **不满足3NF的例子**：
```
学生表：（学号，姓名，学院编号，学院名称，学院地址）
主键：学号
```
问题：`学院名称` 和 `学院地址` 依赖于 `学院编号`，而 `学院编号` 依赖于 `学号`。这叫"传递依赖"（学号 → 学院编号 → 学院名称）。

✅ **满足3NF的改造**（拆表）：
```
学生表：（学号，姓名，学院编号）
学院表：（学院编号，学院名称，学院地址）
```

### [](#BC范式-BCNF)BC范式（BCNF）

**定义**：在满足3NF的基础上，**每个 determinant 都是候选键**。

（detriminant = 决定因素，即可以唯一决定其他列的列）

❌ **不满足BCNF的例子**：
```
学生表：（学号，课程号，导师）
```
假设：
- 一个学生可以选多门课（学号 → 课程号）
- 一门课只有一个导师（课程号 → 导师）
- 一个导师只教一门课（导师 → 课程号）

问题：导师可以决定课程号，但导师不是候选键。

✅ **满足BCNF的改造**：
```
选课表：（学号，课程号）
课程导师表：（课程号，导师）
```

### [](#第四范式-4NF)第四范式（4NF）

**定义**：在满足BCNF的基础上，**消除多值依赖**。

多值依赖例子：一个学生有多个联系电话、多个紧急联系人，这两组多值之间无关系，但放在同一张表中会产生冗余。

### [](#第五范式-5NF)第五范式（5NF）

**定义**：在满足4NF的基础上，**消除连接依赖**，确保表在逻辑上不可再分。

> 📝 **面试提示**：实际面试中，1NF、2NF、3NF是必考的，BCNF偶尔考，4NF和5NF几乎不考。

## [](#反范式化)反范式化

范式化可以减少冗余，但会增加表的数量，导致查询时需要**多表JOIN**，影响性能。

**反范式化（Denormalization）** 是有意在表中保留冗余数据，以**空间换时间**：

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 冗余字段 | 在订单表中冗余用户名 | 避免每次查订单都要JOIN用户表 |
| 汇总表 | 提前计算并存储统计结果 | 报表查询 |
| 宽表 | 将多表JOIN结果存为一张大表 | OLAP分析场景 |

**原则**：
- 读多写少的场景适合反范式化
- 写多读少的场景应保持范式化（避免更新异常）

## [](#ER图与数据库设计)ER图与数据库设计

### [](#ER图基础)ER图基础

```
[实体] ---<关系>--- [实体]
   |                      |
 属性                    属性
```

- **实体（Entity）**：现实世界中的对象（如：学生、课程）
- **属性（Attribute）**：实体的特征（如：学号、姓名）
- **关系（Relationship）**：实体之间的联系
  - 一对一（1:1）：一个学生只有一个学生证
  - 一对多（1:N）：一个学院有多个学生
  - 多对多（M:N）：一个学生选多门课，一门课有多个学生

### [](#数据库设计步骤)数据库设计步骤

```
需求分析 → 概念设计（ER图） → 逻辑设计（关系模式） → 物理设计（建表） → 运行维护
```

**需求分析**：了解业务需求，确定系统要管理哪些数据

**概念设计**：画ER图，确定实体、属性、关系

**逻辑设计**：将ER图转换为关系模式（表结构），并进行规范化（范式化）

**物理设计**：选择合适的存储引擎、索引策略、分区策略

## [](#常见数据库模型)常见数据库模型

### [](#层次模型)层次模型

- 树形结构，父子关系
- 缺点：只能表示1:N关系，不够灵活

### [](#网状模型)网状模型

- 允许节点有多个父节点
- 缺点：结构复杂，不易维护

### [](#关系模型)关系模型

- 二维表结构，是目前最主流的模型
- 优点：简单、灵活、有坚实的数学基础（关系代数）

### [](#面向对象模型)面向对象模型

- 支持对象、继承、多态
- 代表：ObjectDB

### [](#NoSQL模型)NoSQL模型

- **KV存储**：Redis
- **文档存储**：MongoDB
- **列族存储**：Cassandra、HBase
- **图数据库**：Neo4j

## [](#CAP定理与数据库选择)CAP定理与数据库选择

CAP定理：分布式系统中，一致性（Consistency）、可用性（Availability）、分区容错性（Partition tolerance）**三者不可兼得**。

| 类型 | 说明 | 代表数据库 |
|------|------|------------|
| CA | 一致性和可用性（无网络分区） | 单机MySQL、PostgreSQL |
| CP | 一致性和分区容错性 | ZooKeeper、HBase、Redis Cluster |
| AP | 可用性和分区容错性 | Cassandra、DynamoDB、Riak |

**面试常问**：为什么ACID数据库（如MySQL）通常选择CA？因为在分布式环境中，P是必须要保证的（网络分区不可避免），所以实际上是在C和A之间权衡。
