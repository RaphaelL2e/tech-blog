# 分布式缓存与一致性哈希深度解析

## 一、引言

分布式缓存是提升系统性能的核心手段，而一致性哈希是分布式缓存中数据分片的关键算法。理解缓存策略和一致性哈希，对设计高性能分布式系统至关重要。

**本文要点**：
1. 分布式缓存架构
2. 缓存策略（Cache Aside、Read/Write Through、Write Behind）
3. 数据分片策略（哈希取模、范围分片、一致性哈希）
4. 一致性哈希算法详解
5. 虚拟节点与平衡性
6. 缓存迁移与扩缩容
7. 面试高频问题

---

## 二、分布式缓存架构

### 2.1 为什么需要分布式缓存

```
单机缓存（本地缓存）：
  - 容量有限（受 JVM 内存限制）
  - 多节点数据不一致
  - 节点故障缓存丢失

分布式缓存（Redis Cluster）：
  - 容量可扩展（添加节点即可）
  - 数据全局一致
  - 高可用（主从复制 + 故障转移）
```

### 2.2 多级缓存架构

```
请求 → 本地缓存（Caffeine/Guava）
         ↓ 未命中
       分布式缓存（Redis Cluster）
         ↓ 未命中
         数据库
         ↓ 回写
       分布式缓存
         ↓ 回写
       本地缓存
```

**各级缓存对比**：

| 缓存层级 | 延迟 | 容量 | 一致性 | 典型实现 |
|---------|------|------|--------|---------|
| 本地缓存 | ~1μs | ~100MB | 最弱 | Caffeine、Guava |
| 分布式缓存 | ~1ms | ~TB | 弱 | Redis、Memcached |
| 数据库 | ~10ms | ~PB | 强 | MySQL、PostgreSQL |

### 2.3 缓存穿透/击穿/雪崩回顾

> 详见本系列 Redis 缓存问题篇，此处简要回顾：

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 穿透 | 查询不存在的数据 | 布隆过滤器 + 缓存空值 |
| 击穿 | 热点 Key 过期 | 互斥锁 / 逻辑过期 |
| 雪崩 | 大量 Key 同时过期 | 随机 TTL + 多级缓存 |

## 三、缓存策略

### 3.1 Cache Aside（旁路缓存）

**最常用的缓存策略**。

```
读操作：
1. 先读缓存
2. 缓存命中 → 返回
3. 缓存未命中 → 读数据库 → 写入缓存 → 返回

写操作：
1. 先更新数据库
2. 再删除缓存（不是更新缓存！）
```

**为什么是删除而不是更新缓存**：

```
并发场景（更新缓存）：
T1: A 更新 DB → name = "Alice"
T2: B 更新 DB → name = "Bob"  
T3: B 更新缓存 → cache = "Bob"
T4: A 更新缓存 → cache = "Alice" ← 脏数据！

并发场景（删除缓存）：
T1: A 更新 DB → name = "Alice"
T2: B 更新 DB → name = "Bob"
T3: B 删除缓存
T4: A 删除缓存
T5: C 读缓存未命中 → 读 DB → "Bob" → 写入缓存 ✅
```

**延迟双删策略**：

```
问题：先更新 DB，后删除缓存，中间可能读到旧缓存

解决：延迟双删
1. 删除缓存
2. 更新数据库
3. 延迟一段时间后，再次删除缓存

延迟时间 = 读业务逻辑的耗时 + 几百毫秒

代码实现：
public void update(String key, Object value) {
    // 1. 先删缓存
    cache.delete(key);
    
    // 2. 更新数据库
    db.update(key, value);
    
    // 3. 延迟双删（异步）
    CompletableFuture.runAsync(() -> {
        try {
            Thread.sleep(500);  // 延迟 500ms
            cache.delete(key);
        } catch (InterruptedException e) {
            // ...
        }
    });
}
```

### 3.2 Read/Write Through

**缓存层负责读写数据库，应用层只与缓存交互**。

```
读操作：
1. 读缓存
2. 缓存命中 → 返回
3. 缓存未命中 → 缓存层从数据库加载 → 写入缓存 → 返回

写操作：
1. 写缓存
2. 缓存层同步更新数据库
```

**优点**：应用层逻辑简单
**缺点**：缓存层实现复杂

### 3.3 Write Behind（异步写回）

**先更新缓存，异步批量写入数据库**。

```
写操作：
1. 只更新缓存
2. 异步批量写入数据库

读操作：
1. 读缓存（缓存中有最新数据）
```

**优点**：
- 写性能极高（只写缓存）
- 可批量写入数据库

**缺点**：
- 数据可能丢失（缓存故障）
- 数据一致性弱

**适用场景**：写多读少、容忍数据丢失（如浏览量计数器）

## 四、数据分片策略

### 4.1 哈希取模分片

**公式**：`shard = hash(key) % N`（N 为节点数）

**示例**：

```
3 个 Redis 节点
key = "user:1"  → hash("user:1") % 3 = 0 → 节点 0
key = "user:2"  → hash("user:2") % 3 = 2 → 节点 2
key = "order:1" → hash("order:1") % 3 = 1 → 节点 1
```

**问题：扩缩容时数据大量迁移**

```
3 节点 → 4 节点：
  key = "user:1" → hash % 3 = 0 → hash % 4 = ? 
  → 大量 Key 的节点发生变化

  迁移比例 ≈ 1 - 3/4 = 75% 的 Key 需要迁移！

3 节点 → 5 节点：
  迁移比例 ≈ 1 - 3/5 = 60%
```

### 4.2 范围分片

**按 Key 的范围分片**。

```
节点 0: key ∈ [0, 1000000)
节点 1: key ∈ [1000000, 2000000)
节点 2: key ∈ [2000000, 3000000)
```

**优点**：扩容只需增加新范围，不影响旧数据
**缺点**：热点问题（某些范围访问量大）

### 4.3 一致性哈希分片

**解决哈希取模的扩缩容问题**。

**核心思想**：将哈希值空间组织为虚拟的圆环（哈希环），扩缩容时只需迁移少量数据。

## 五、一致性哈希算法

### 5.1 基本原理

**哈希环**：

```
将整个哈希值空间组织为 0 ~ 2^32 - 1 的圆环

1. 计算节点的哈希值，放置在环上
   hash("Node-A") = 1000
   hash("Node-B") = 3000
   hash("Node-C") = 5000

2. 计算 Key 的哈希值，顺时针找到最近的节点
   hash("key-1") = 1500 → 顺时针 → Node-B
   hash("key-2") = 4000 → 顺时针 → Node-C
   hash("key-3") = 800  → 顺时针 → Node-A
```

**哈希环示意图**：

```
           0
           │
     Node-A(1000)
      ╱          ╲
   ╱                ╲
 Node-C(5000)    Node-B(3000)
   ╲                ╱
      ╲          ╱
           │
        2^32-1

key-1(1500) → 顺时针 → Node-B
key-2(4000) → 顺时针 → Node-C  
key-3(800)  → 顺时针 → Node-A
```

### 5.2 扩容

**添加 Node-D（hash = 4000）**：

```
扩容前：
  key-2(4000) → Node-C

扩容后：
  key-2(4000) → Node-D（更近）

迁移量：只有 Node-B 和 Node-C 之间的 Key 需要迁移
迁移比例 ≈ 1/(N+1)（理想情况）
```

**对比哈希取模**：
```
3→4 节点：
  哈希取模：迁移 ~75%
  一致性哈希：迁移 ~25%（理想）
```

### 5.3 缩容

**移除 Node-B**：

```
缩容前：
  key-1(1500) → Node-B

缩容后：
  key-1(1500) → Node-C（顺时针下一个节点）

迁移量：只有 Node-B 上的 Key 迁移到 Node-C
```

### 5.4 数据倾斜问题

**问题**：如果节点较少，哈希值分布不均匀，导致某些节点负载过高。

```
极端情况：
  Node-A(100), Node-B(200), Node-C(8000)
  
  大部分 Key 的哈希值在 200~8000 之间 → 都落到 Node-C
  → Node-C 负载 >> Node-A、Node-B
```

**解决方案：虚拟节点**

## 六、虚拟节点

### 6.1 虚拟节点的原理

**核心思想**：每个物理节点对应多个虚拟节点，虚拟节点均匀分布在哈希环上。

```
物理节点 Node-A → 虚拟节点 Node-A-1, Node-A-2, ..., Node-A-100
物理节点 Node-B → 虚拟节点 Node-B-1, Node-B-2, ..., Node-B-100
物理节点 Node-C → 虚拟节点 Node-C-1, Node-C-2, ..., Node-C-100

每个物理节点 100 个虚拟节点
→ 总共 300 个虚拟节点
→ 均匀分布在哈希环上
→ 负载均衡
```

**虚拟节点的计算**：

```java
// 为每个物理节点创建 N 个虚拟节点
for (int i = 0; i < virtualNodeCount; i++) {
    String virtualNodeName = "Node-" + physicalNodeId + "-VN" + i;
    int hash = hash(virtualNodeName);
    ring.put(hash, physicalNodeId);
}
```

### 6.2 虚拟节点数量选择

| 虚拟节点数/物理节点 | 负载均衡 | 内存占用 |
|---------------------|---------|---------|
| 10 | 较差 | 低 |
| 100 | 较好 | 中 |
| 200 | 好 | 较高 |
| 500+ | 很好 | 高 |

**经验值**：150-200 个虚拟节点/物理节点

### 6.3 一致性哈希的 Java 实现

```java
public class ConsistentHash<T> {
    
    // 哈希环（SortedMap 代替环）
    private final SortedMap<Integer, T> ring = new TreeMap<>();
    
    // 哈希函数
    private final HashFunction hashFunction;
    
    // 虚拟节点数
    private final int virtualNodeCount;
    
    public ConsistentHash(HashFunction hashFunction, int virtualNodeCount, 
                          Collection<T> nodes) {
        this.hashFunction = hashFunction;
        this.virtualNodeCount = virtualNodeCount;
        for (T node : nodes) {
            addNode(node);
        }
    }
    
    // 添加节点
    public void addNode(T node) {
        for (int i = 0; i < virtualNodeCount; i++) {
            int hash = hashFunction.hash(node.toString() + "-VN" + i);
            ring.put(hash, node);
        }
    }
    
    // 移除节点
    public void removeNode(T node) {
        for (int i = 0; i < virtualNodeCount; i++) {
            int hash = hashFunction.hash(node.toString() + "-VN" + i);
            ring.remove(hash);
        }
    }
    
    // 查找 Key 对应的节点
    public T getNode(String key) {
        if (ring.isEmpty()) {
            return null;
        }
        
        int hash = hashFunction.hash(key);
        
        // 顺时针查找第一个节点
        SortedMap<Integer, T> tailMap = ring.tailMap(hash);
        
        // 如果没有比 hash 大的节点，取环的第一个节点
        int nodeHash = tailMap.isEmpty() ? ring.firstKey() : tailMap.firstKey();
        
        return ring.get(nodeHash);
    }
}
```

## 七、哈希函数选择

### 7.1 常见哈希函数对比

| 哈希函数 | 分布均匀性 | 性能 | 冲突率 |
|---------|-----------|------|--------|
| **CRC32** | 一般 | 高 | 较高 |
| **MurmurHash3** | 优秀 | 高 | 低 |
| **FNV** | 好 | 高 | 低 |
| **Ketama** | 优秀 | 中 | 最低 |
| **MD5** | 优秀 | 低 | 极低 |

**推荐**：**MurmurHash3**（性能好，分布均匀）

### 7.2 MurmurHash3 示例

```java
// 使用 Guava 的 MurmurHash3
int hash = Hashing.murmur3_32().hashString(key, StandardCharsets.UTF_8).asInt();

// 确保为正数
hash = hash & 0x7FFFFFFF;
```

## 八、Redis Cluster 的哈希槽

### 8.1 Redis Cluster 的分片方式

Redis Cluster 使用 **哈希槽（Hash Slot）** 而不是一致性哈希。

**16384 个哈希槽**：

```
slot = CRC16(key) % 16384

节点 A: slot 0 ~ 5460
节点 B: slot 5461 ~ 10922
节点 C: slot 10923 ~ 16383
```

### 8.2 哈希槽 vs 一致性哈希

| 对比项 | 一致性哈希 | 哈希槽 |
|--------|-----------|--------|
| **分片方式** | 哈希环 | 固定数量槽 |
| **扩缩容** | 自动迁移 | 手动迁移槽 |
| **负载均衡** | 依赖虚拟节点 | 槽均匀分配 |
| **灵活性** | 高 | 中 |
| **实现复杂度** | 高 | 低 |

### 8.3 哈希槽的扩缩容

```
添加节点 D：
1. 从 A、B、C 各迁移部分槽给 D
   节点 A: slot 0 ~ 4095 (迁移 1365 个槽)
   节点 D: slot 4096 ~ 5460 (来自 A)
   节点 B: slot 5461 ~ 10922
   节点 C: slot 10923 ~ 16383

2. 迁移过程中，客户端请求被重定向
   → ASK 重定向：迁移中的槽
   → MOVED 重定向：迁移完成的槽
```

## 九、缓存迁移实战

### 9.1 扩容迁移步骤

```
1. 新增缓存节点
2. 修改路由规则（哈希槽/一致性哈希）
3. 迁移数据
4. 验证数据一致性
5. 切换流量
```

### 9.2 平滑迁移（双写）

```
阶段一：双写
  → 同时写入旧缓存和新缓存
  → 读从旧缓存读

阶段二：数据迁移
  → 将旧缓存数据迁移到新缓存

阶段三：切读
  → 读从新缓存读
  → 写继续双写

阶段四：停止双写
  → 只写新缓存
  → 下线旧缓存
```

## 十、面试高频问题

### Q1: 一致性哈希的原理？

**答案要点**：
- 哈希环，Key 顺时针找到最近的节点
- 扩缩容只影响相邻节点
- 虚拟节点解决数据倾斜
- 迁移比例约 1/N

### Q2: 一致性哈希的数据倾斜如何解决？

**答案要点**：
- 虚拟节点：每个物理节点对应多个虚拟节点
- 虚拟节点均匀分布在环上
- 推荐每个物理节点 150-200 个虚拟节点

### Q3: Redis Cluster 为什么用哈希槽而不是一致性哈希？

**答案要点**：
- 哈希槽实现简单
- 槽可以手动迁移，控制更精确
- 16384 个槽足够均匀
- 扩缩容时可以灵活调整槽分配

### Q4: Cache Aside 为什么是删除缓存而不是更新？

**答案要点**：
- 并发写时更新缓存可能导致脏数据
- 删除缓存后，下次读自动加载最新数据
- 复杂数据更新缓存的代价可能比删除高

### Q5: 延迟双删是什么？

**答案要点**：
- 先删缓存 → 更新数据库 → 延迟后再删缓存
- 解决更新数据库和删除缓存之间的短暂不一致
- 延迟时间 = 读业务耗时 + 几百毫秒

## 十一、总结

本文系统梳理了分布式缓存与一致性哈希的核心知识：

1. **缓存策略**：Cache Aside（推荐）、Read/Write Through、Write Behind
2. **数据分片**：哈希取模（扩缩容差）、范围分片、一致性哈希
3. **一致性哈希**：哈希环 + 顺时针查找，虚拟节点解决倾斜
4. **Redis Cluster**：16384 哈希槽，手动迁移

**核心记忆**：
- Cache Aside：先更新 DB，再删除缓存
- 一致性哈希扩缩容迁移量 ≈ 1/N
- 虚拟节点数推荐 150-200/物理节点
- Redis Cluster 用哈希槽（16384），不用一致性哈希
- 延迟双删：删缓存 → 更新 DB → 延迟 → 再删缓存
