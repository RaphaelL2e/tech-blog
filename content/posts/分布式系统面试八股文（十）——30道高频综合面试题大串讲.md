## 第一部分：理论基础（Q1-Q5）

### Q1：说说你对 CAP 定理的理解，工作中如何权衡？

**参考答案：**

CAP 定理是分布式系统的基石：Consistency（一致性）、Availability（可用性）、Partition Tolerance（分区容错性），三者最多同时满足两个。

```
关键理解：

1. P（分区容错）是"必选项"——网络分区必然发生，无法回避
   所以本质上是 CP vs AP 的选择

2. CP 系统（强一致性）：
   - 发生分区时，牺牲可用性保证一致性
   - 典型：ZooKeeper、etcd、Consul
   - 场景：配置管理、选主、分布式锁

3. AP 系统（高可用）：
   - 发生分区时，牺牲一致性保证可用性
   - 典型：Eureka、Cassandra、DynamoDB
   - 场景：服务发现、用户状态、社交Feed

4. "三选二"是误导——其实是"发生分区时，选C还是选A"
   - 网络正常时，C和A可以兼得
   - 网络分区时，必须在C和A之间取舍

实战取舍：
- 支付/交易 → CP（宁可不可用，不能算错钱）
- 服务注册 → AP（宁可数据短暂不一致，不能服务不可用）
- 中间地带 → BASE理论（最终一致性）
```

### Q2：BASE 理论和 ACID 有什么区别？什么场景用哪个？

**参考答案：**

```
ACID（传统关系型数据库）：
├── Atomicity（原子性）：要么全做，要么全不做
├── Consistency（一致性）：事务前后数据满足约束
├── Isolation（隔离性）：并发事务互不干扰
└── Durability（持久性）：提交后数据永久保存

BASE（分布式系统）：
├── Basically Available（基本可用）：允许部分故障，核心功能可用
├── Soft State（软状态）：允许中间状态，不要求强一致
└── Eventually Consistent（最终一致性）：经过一段时间后数据达成一致

对比：

| 维度     | ACID                     | BASE                      |
|----------|--------------------------|---------------------------|
| 适用场景 | 单机数据库、金融交易       | 分布式系统、互联网应用     |
| 一致性   | 强一致性（同步）           | 最终一致性（异步）         |
| 可用性   | 受一致性约束，可能降级     | 优先保证可用性             |
| 典型实现 | MySQL事务、PostgreSQL     | 异步消息、补偿事务、TCC    |

实战选择：
- 银行转账、扣库存 → ACID 思维（先保证正确）
- 点赞计数、文章阅读量 → BASE 思维（允许短暂不一致）
- 电商下单 → 混合：核心流程用ACID保证，非核心流程用BASE提升吞吐
```

### Q3：分布式系统的 8 个谬误是什么？你踩过哪些坑？

**参考答案：**

```
分布式计算的8个谬误（Fallacies of Distributed Computing）：

1. 网络是可靠的           → 实际：网络分区、丢包、延迟随时发生
2. 延迟为零               → 实际：同机房1ms，跨机房10-50ms，跨地域100ms+
3. 带宽是无限的           → 实际：序列化开销、大对象传输成为瓶颈
4. 网络是安全的           → 实际：中间人攻击、数据泄露风险
5. 拓扑不会改变           → 实际：扩缩容、服务上下线频繁
6. 只有一个管理员         → 实际：多团队运维，策略冲突
7. 传输成本为零           → 实际：序列化/反序列化CPU开销不可忽视
8. 网络是同质的           → 实际：移动网络、WiFi、有线差异巨大

踩坑实录：
- "服务调用超时，以为是下游挂了，结果是网络抖动 → 加上重试+幂等"
- "RPC传输大对象，序列化吃掉30%CPU → 只传ID，让下游自己查"
- "跨机房调用延迟几百ms，用户等不及 → 就近部署+异步化"
```

### Q4：什么是"拜占庭将军问题"？和普通分布式一致性有什么区别？

**参考答案：**

```
拜占庭将军问题（Byzantine Generals Problem）：

场景：多个将军包围一座城，需要就是否进攻达成一致
- 忠将：诚实传令，执行一致策略
- 叛徒：可能发送假消息、选择性不响应、故意作恶

拜占庭容错 vs 非拜占庭容错：

| 维度         | 非拜占庭容错（CFT）      | 拜占庭容错（BFT）         |
|--------------|------------------------|--------------------------|
| 故障模型     | 节点可能宕机/断网        | 节点可能主动作恶/欺骗     |
| 容错数       | 2f+1 节点容忍 f 个故障   | 3f+1 节点容忍 f 个叛徒    |
| 复杂度       | 较低（Paxos/Raft）       | 很高（PBFT通信O(n²)）     |
| 应用场景     | 数据中心内部、企业系统   | 区块链、跨组织协作         |
| 典型协议     | Raft、Paxos、ZAB         | PBFT、Tendermint          |

关键理解：
- 绝大多数企业级分布式系统只考虑CFT（非拜占庭）
- 因为节点全部由自己运维，不会主动作恶
- 区块链/数字货币场景才需要BFT
- Raft/Paxos解决的是"部分节点宕机如何达成一致"
- PBFT解决的是"部分节点撒谎如何达成一致"
```

### Q5：什么是分布式系统的"脑裂"问题？如何解决？

**参考答案：**

```
脑裂（Split-Brain）：

    正常情况下：                分区发生后（脑裂）：
    
    Master ──→ Slave1          Master ──→ Slave1
      │                         (分区1认为自己是Master)
      └────→ Slave2
                                 Master ──→ Slave2
    一个集群，一个Master        (分区2也认为自己是Master！)
                                 → 两个Master同时写入 → 数据冲突！

解决方案：

1. Quorum 投票机制（过半原则）
   - 只有获得超过半数节点投票的才能成为 Master
   - 分区1有3个节点中的2个 → 能选出Master
   - 分区2只有1个节点 → 选不出Master → 拒绝服务
   - ZooKeeper、etcd、Redis Sentinel 都基于此

2. 隔离（Fencing）机制
   - 每次选主分配递增的 epoch/fencing token
   - 旧Master发送写请求时，存储系统校验token
   - token过期 → 拒绝写入
   - 如 HDFS NameNode 的 fencing

3. 仲裁盘（Quorum Disk）
   - 两个节点 + 一个共享存储
   - 谁抢到共享存储的锁谁就是Master

4. 第三方仲裁
   - 引入独立的仲裁服务（如 ZooKeeper）
   - 解决只有2个节点时的"偶数节点困境"

实战经验：
- 永远用奇数个节点（3/5/7）
- 过半原则是解决脑裂最经典的方法
- Redis Cluster 使用 gossip + 配置纪元
```

## 第二部分：一致性协议（Q6-Q9）

### Q6：Paxos 和 Raft 的区别？为什么 Raft 更流行？

**参考答案：**

```
Paxos（学术派）           Raft（工程派）

┌─────────────────────┐   ┌─────────────────────┐
│ 角色：Proposer/      │   │ 角色：Leader/Follower│
│ Acceptor/Learner     │   │ /Candidate          │
│ 多个Proposer可并存   │   │ 强Leader，唯一Leader  │
│                      │   │                      │
│ 两阶段提交           │   │ Leader选举 + 日志复制 │
│ Prepare → Accept     │   │ 也是两阶段，但更清晰  │
│                      │   │                      │
│ 难理解、难实现        │   │ 易理解、易实现        │
│ 需要额外机制解决      │   │ 内置成员变更          │
│ 活锁问题              │   │                      │
│                      │   │                      │
└─────────────────────┘   └─────────────────────┘

为什么 Raft 更流行：

1. 可理解性（Understandability）是 Raft 的首要设计目标
2. 问题分解清晰：
   - Leader Election（选主）
   - Log Replication（日志复制）
   - Safety（安全性保证）
   - Membership Changes（成员变更）
3. 工程实现友好：etcd、TiKV、Consul 都基于 Raft
4. 论文质量高，社区资料丰富

一句话：Raft 是 "工程化的 Paxos"
```

### Q7：Raft 协议中 Leader 选举的过程是怎样的？

**参考答案：**

```
Raft 选举流程图：

┌─────────────────────────────────────────────────────────┐
│                    Raft Leader 选举                      │
│                                                          │
│  所有节点启动时为 Follower                                 │
│       │                                                  │
│       │ election timeout 超时 (150-300ms 随机)            │
│       ↓                                                  │
│   变为 Candidate                                         │
│       │                                                  │
│       ├── term++                                         │
│       ├── 投票给自己                                      │
│       └── 向所有节点发送 RequestVote RPC                  │
│       │                                                  │
│       ├── 获得多数票 → 成为 Leader                        │
│       │   └── 立即发送心跳，阻止新的选举                   │
│       │                                                  │
│       ├── 收到合法 Leader 的心跳 → 退回 Follower          │
│       │                                                  │
│       └── 选举超时 → term++ 重新选举                      │
│                                                          │
│  关键机制：                                               │
│  1. 随机超时（避免多个Candidate同时发起选举）               │
│  2. 每个Term每个节点只能投一票                             │
│  3. Candidate只投票给日志至少和自己一样新的节点             │
│  4. 选出的Leader一定包含所有已提交的日志                   │
└─────────────────────────────────────────────────────────┘

// Raft选举的核心状态
enum State { FOLLOWER, CANDIDATE, LEADER }

// Leader 心跳
class AppendEntriesRequest {
    long term;           // Leader的当前任期
    int leaderId;        // 让Follower知道谁是Leader
    long prevLogIndex;   // 前一条日志的索引
    long prevLogTerm;    // 前一条日志的任期
    List<LogEntry> entries;  // 需要复制的日志（心跳时为空）
    long leaderCommit;   // Leader已提交的日志索引
}
```

### Q8：ZooKeeper 的 ZAB 协议和 Raft 有什么异同？

**参考答案：**

```
ZAB（ZooKeeper Atomic Broadcast） vs Raft：

相同点：
✓ 都是强 Leader 模型
✓ 都有任期（epoch/term）概念
✓ 日志复制都采用两阶段方式
✓ 过半提交原则
✓ Leader 需要获得多数派确认

不同点：

| 维度         | ZAB                       | Raft                      |
|--------------|--------------------------|---------------------------|
| 读操作       | 可以直接从 Follower 读    | 强一致性读需从 Leader 读   |
| 日志提交     | 可以批量提交               | 一条一条提交               |
| 成员变更     | 较复杂，需要专门处理       | 联合共识（Joint Consensus）|
| 设计目标     | 为 ZooKeeper 定制          | 通用共识算法               |
| 实现生态     | ZooKeeper 专用             | etcd/Consul/TiKV 等多实现  |

关键差异——ZAB 的快速选举：
- ZAB 允许上一轮的 Follower 参与投票给当前轮
- Raft 严格要求 Candidate 只能投给日志更新者
- 因此 ZAB 选举可能稍快，但安全性需要额外保证
```

### Q9：什么是分布式共识（Consensus）？和分布式事务有什么区别？

**参考答案：**

```
分布式共识（Consensus）：
目标：多个节点对"某一个值"达成一致
例如：
- "谁是 Leader？" → 选主
- "这条日志提交了没有？" → 日志复制
- "version=5 的配置生效了吗？" → 配置一致性
核心协议：Paxos、Raft、ZAB

分布式事务（Distributed Transaction）：
目标：多个节点在"各自的数据"上执行一致的操作
例如：
- 订单服务减库存 + 账户服务扣款 → 要么都成功，要么都失败
- 跨库转账
核心协议：2PC、3PC、TCC、Saga、XA

区别对比：

| 维度       | 分布式共识            | 分布式事务              |
|------------|----------------------|------------------------|
| 关注点     | 多个节点对同一值一致   | 多个操作原子性执行      |
| 典型场景   | 选主、配置同步、日志   | 跨服务数据修改          |
| 协议举例   | Paxos/Raft           | 2PC/TCC/Saga           |
| 复杂度     | 协议本身复杂          | 场景/补偿逻辑复杂       |
| 系统角色   | 对等节点              | Coordinator + 参与者    |
```

## 第三部分：分布式事务（Q10-Q13）

### Q10：请详细说说 2PC 和 3PC 的区别？为什么 3PC 很少用？

**参考答案：**

```
2PC（两阶段提交）：

    Coordinator                        Participants
         │                                   │
    阶段1 │ ──── Prepare ────────────────→   │ ① 执行事务但不提交
         │                                   │ ② 写入 undo/redo 日志
         │ ←─── Vote Yes/No ─────────────    │ ③ 回复Yes（就绪）或No（失败）
         │                                   │
    阶段2 │ ──── Commit/Rollback ────────→   │ ④ 收到Commit则提交
         │                                   │ ⑤ 收到Rollback则回滚
         │ ←─── ACK ────────────────────     │
         
    问题：
    - 同步阻塞：Prepare后参与者必须等待Coordinator决策
    - 单点故障：Coordinator挂了，参与者不知该提交还是回滚
    - 数据不一致：第二阶段部分节点收到Commit，部分没收到

3PC（三阶段提交）：

    阶段1 │ ──── CanCommit ────────────→     │ ① 询问是否可以提交
         │ ←─── Yes/No ────────────────     │
         │                                   │
    阶段2 │ ──── PreCommit ────────────→     │ ② 执行事务但不提交
         │ ←─── ACK ───────────────────     │
         │                                   │
    阶段3 │ ──── DoCommit ────────────→     │ ③ 最终提交
         │ ←─── ACK ───────────────────     │
         
    改进：
    - 引入超时机制：参与者超时后可以自行提交（降低阻塞）
    - 但引入新的问题：网络分区时可能"误提交"

为什么 3PC 很少用：

1. 网络分区场景下仍不能完全保证一致性
2. 多一次网络交互，性能更差
3. 超时提交可能在网络恢复后造成数据不一致
4. 实际生产中 2PC + 完善的异常恢复机制 比 3PC 更实用
```

### Q11：TCC 事务是怎么工作的？和 2PC 有什么不同？

**参考答案：**

```
TCC（Try-Confirm-Cancel）：

以转账为例：

┌──────────────────────────────────────────────────────────────┐
│                    TCC 转账流程                               │
│                                                              │
│  Try 阶段（预留资源）：                                       │
│  ┌─────────────────────────────────────────────┐            │
│  │ 账户A 服务：冻结 100元                      │            │
│  │ 账户B 服务：预增加额度（不实际增加）          │            │
│  │ 状态：A余额100→0(冻结)  B冻结额度+100        │            │
│  └─────────────────────────────────────────────┘            │
│       │                                                      │
│       │ Try 全部成功？                                        │
│       │                                                      │
│  ┌────┴────┐                                                │
│  │ YES     │ NO                                              │
│  ↓         ↓                                                 │
│  Confirm   Cancel                                            │
│  A扣减确认  A冻结释放                                         │
│  B实际到账  B冻结取消                                         │
│  完成转账   回滚到初始状态                                    │
└──────────────────────────────────────────────────────────────┘

TCC vs 2PC：

| 维度       | 2PC                      | TCC                       |
|------------|--------------------------|---------------------------|
| 资源锁定   | 数据库层面锁（悲观）      | 业务层面预留（灵活）       |
| 实现层次   | 基础设施层                | 业务代码层                 |
| 性能       | 全局锁，性能差            | 无全局锁，性能好           |
| 侵入性     | 低，对业务透明            | 高，需实现Try/Confirm/Cancel|
| 应用场景   | 单体→分布式过渡           | 高并发互联网业务           |
| 空回滚处理 | 框架处理                  | 需自己处理（Cancel接到Try前的请求）|

// TCC 关键注意事项
1. 空回滚：Cancel 比 Try 先到 → 记录已回滚，忽略后续 Try
2. 悬挂：Cancel 后 Try 才到 → 记录已回滚，拒绝 Try
3. 幂等：Try/Confirm/Cancel 都可能被重试 → 必须幂等
```

### Q12：Saga 事务模式有哪两种实现？各有什么优劣？

**参考答案：**

```
Saga 模式：长事务拆分为多个本地事务，每个事务有对应的补偿操作

模式一：编排式（Choreography）——事件驱动

    Order Service          Payment Service        Inventory Service
         │                       │                       │
         │ ① 创建订单            │                       │
         │ ② 发布"订单已创建"     │                       │
         │ ───────────────────→  │                       │
         │                       │ ③ 扣款               │
         │                       │ ④ 发布"付款成功"      │
         │                       │ ───────────────────→  │
         │                       │                       │ ⑤ 减库存
         │                       │                       │
    失败补偿：                                              │
         │                       │                  减库存失败！
         │                       │ ←── 发布"库存不足" ── │
         │ ←── 发布"退款" ────── │                       │
         │ ⑥ 取消订单            │ ⑦ 退款               │
         
    优点：松耦合、无中心节点、服务自治
    缺点：流程分散、难以追踪、循环依赖风险

模式二：编排式（Orchestration）——中心化

    Saga Orchestrator
         │
         │ ① ──→ Order Service: 创建订单
         │ ② ──→ Payment Service: 扣款
         │ ③ ──→ Inventory Service: 减库存
         │
         │ 如果③失败：
         │ ④ ──→ Payment Service: 退款（补偿②）

- 复杂流程（>5步、多分支条件）→ 编排式
```

### Q13：分布式事务中如何处理"空回滚"和"悬挂"问题？

**参考答案：**

```java
// 空回滚和悬挂是 TCC 模式的特有挑战

/**
 * 空回滚（Empty Rollback）：
 * 场景：Cancel 请求先于 Try 请求到达
 * 原因：Try 超时 → Coordinator 发起 Cancel → 原 Try 请求延迟到达
 * 
 * 处理：记录事务状态，Cancel 到达时标记事务为"已回滚"，
 *       后续 Try 到达时发现已回滚，直接拒绝
 */
public class TccAccountService {
    
    // 事务状态表
    /**
     * CREATE TABLE tcc_transaction (
     *   tx_id      VARCHAR(64) PRIMARY KEY,
     *   status     VARCHAR(16),  -- TRYING/CONFIRMED/CANCELLED
     *   created_at TIMESTAMP,
     *   updated_at TIMESTAMP
     * );
     */
    
    // Try 阶段
    public void tryFreeze(String txId, long amount) {
        // 检查是否已回滚（空回滚防护）
        TccTransaction tx = transactionMapper.selectById(txId);
        if (tx != null && "CANCELLED".equals(tx.getStatus())) {
            // 本次Try已经被Cancel过了 → 空回滚
            logger.warn("空回滚检测: txId={}", txId);
            return;
        }
        
        // 插入事务记录（防悬挂）
        int inserted = transactionMapper.insertIgnore(
            new TccTransaction(txId, "TRYING")
        );
        if (inserted == 0) {
            // 插入失败意味着已有记录 → 可能是悬挂
            TccTransaction existing = transactionMapper.selectById(txId);
            if ("CANCELLED".equals(existing.getStatus())) {
                logger.warn("悬挂检测: txId={}", txId);
                return;
            }
        }
        
        // 执行冻结逻辑
        accountMapper.freeze(accountId, amount);
    }
    
    // Cancel 阶段
    public void cancel(String txId, long amount) {
        TccTransaction tx = transactionMapper.selectById(txId);
        
        if (tx == null) {
            // Try 还没到，先插入 Cancelled 记录（防悬挂）
            transactionMapper.insert(new TccTransaction(txId, "CANCELLED"));
            return; // 不需要回滚（因为Try还没执行）
        }
        
        if ("CANCELLED".equals(tx.getStatus())) {
            return; // 幂等：已经取消过了
        }
        
        if ("CONFIRMED".equals(tx.getStatus())) {
            throw new IllegalStateException("已确认的事务不能取消");
        }
        
        // 执行解冻
        accountMapper.unfreeze(accountId, amount);
        transactionMapper.updateStatus(txId, "CANCELLED");
    }
}

/**
 * 解决悬挂的完整方案：
 * 
 * 1. 事务状态表（核心）
 *    - Try 到达时检查状态
 *    - Cancel 先于 Try 到达时插入 Cancelled 记录
 * 
 * 2. 防悬挂执行器
 *    - Try 发现已有 Cancelled 记录 → 跳过业务逻辑
 *    - 但 Try 要返回成功（让 Coordinator 认为 Try 完成了）
 * 
 * 3. 定期清理
 *    - 定时清理超时的 TRYING 状态记录
 *    - 避免长期占用资源
 */
```

### Q15：MySQL 主从复制延迟怎么解决？

**参考答案：**

```
主从延迟的原因：

1. 网络延迟（跨机房/跨地域）
2. 从库性能不足（从库承担读压力 + 回放压力）
3. 大事务（一个事务操作百万行，回放慢）
4. 从库并行复制能力（MySQL 5.6前单线程回放）
5. 锁冲突（从库有查询锁住表）

解决方案：

① 强制走主库（最直接）
   - 写后立刻读 → 走主库
   - Sharding-JDBC 的 HintManager 强制路由
   - 适合"写完立刻要读"的场景

② 并行复制（MySQL 5.7+）
   -- 基于组提交的并行复制
   SET GLOBAL slave_parallel_type = 'LOGICAL_CLOCK';
   SET GLOBAL slave_parallel_workers = 8;
   -- 基于DATABASE的并行复制（跨库事务串行）
   SET GLOBAL slave_parallel_type = 'DATABASE';

③ 半同步复制（Semi-Sync）
   - 主库提交后等待至少一个从库确认收到binlog
   - 降低数据丢失风险，但增加写入延迟
   - MySQL 5.7+ AFTER_SYNC模式

④ 缓存一致性方案
   // 写操作：先更新DB，再删除缓存
   // 读操作：先查缓存，miss再查DB
   // 主从延迟时可能出现：主库已更新，从库还是旧值
   // 解决：延迟双删策略
   //   写 → 删除缓存 → 等待Nms → 再次删除缓存

⑤ 读写分离中间件
   - Sharding-JDBC: 同一线程写后读自动走主库
   - MyCat/ProxySQL: 配置SQL hint强制走主

⑥ 架构优化
   - 敏感业务不分库（单库单表）
   - 异步化：写完后不立即要结果
   - 最终一致性：接受短暂延迟
```

### Q17：Redis 集群模式下，如何保证数据一致性？

**参考答案：**

```
Redis Cluster 数据一致性分析：

架构特点：
- 16384 个哈希槽，每个节点负责部分槽
- 主从结构，每个分片有 1 Master + N Slave
- 使用 Gossip 协议进行节点通信
- 客户端可以重定向（MOVED/ASK）

一致性问题：

1. 主从异步复制导致的数据丢失

   → Slave 提升为 Master → 数据丢失！

   # 要求至少同步到1个从节点
   MIN-SLAVES-TO-WRITE 1
   # 从节点延迟不超过10秒
   MIN-SLAVES-MAX-LAG 10
   # 注意：这只是"尽量"保证，不是强一致

2. 脑裂场景
   Master 网络分区 → 旧 Master 还在写入 → 新 Master 被选出
   → 旧 Master 上的数据丢失！
   
   解决：同机房部署 + 配置 min-slaves-to-write

3. 为什么 Redis 不提供强一致性？
   - 设计哲学：高性能 > 强一致
   - 适用场景：缓存（丢了可以重建）
   - 不要用 Redis Cluster 做数据唯一存储！

强一致性需求的替代方案：
- 需要持久化 + 强一致 → 考虑 TiKV 或 etcd
- Redis 缓存层 + MySQL 持久层是最佳实践
- Redis 只做缓存，挂了从数据库重建
```

### Q21：负载均衡的六种算法及其适用场景？

**参考答案：**

```

① 随机（Random）
   - 随机选一个节点
   - Dubbo 默认策略
   - 适用：节点性能相同

   - 按顺序依次分配
   - 适用：节点性能相同，长连接

       // 每轮所有节点 currentWeight += effectiveWeight
   }

④ 最小连接数（Least Connections）
   - 选择当前活跃连接数最少的节点
   - 适用：长连接场景，请求处理时间差异大
   
⑤ 一致性哈希（Consistent Hash）
   - 相同参数的请求总是路由到同一节点
   - 适用：有状态服务、缓存亲和性
   
   虚拟节点优化：
   ┌───────────────────────┐
   │    一致性哈希环        │
   │                       │
   │  物理节点A → 150个虚拟节点 │
   │  物理节点B → 150个虚拟节点 │
   │  物理节点C → 150个虚拟节点 │
   │                       │
   │  虚拟节点数越多，分布越均匀 │
   └───────────────────────┘

⑥ 最短响应时间（Least Response Time）
   - 选择响应时间最短的节点
   - 动态感知节点性能
   - 适用：节点性能差异大

选型决策：

| 场景                     | 推荐算法         | 原因                 |
|--------------------------|-----------------|----------------------|
| 无状态、短连接            | 加权轮询         | 简单、均匀            |
| 长连接、耗时差异大         | 最小连接数       | 动态感知负载          |
| 需要会话保持/分片亲和性    | 一致性哈希       | 相同请求到相同节点    |
| 节点异构（不同配置）       | 加权轮询/最短RT   | 按能力分配           |
| 秒杀/高并发              | 随机             | 无状态开销，性能最好  |
```

### Q24：消息重复消费怎么解决？

**参考答案：**

```

1. Producer 重复发送（确认超时重试）
2. Broker 重复投递（Consumer确认超时）
3. Consumer 重复消费（业务处理成功但确认失败）

解决核心：Consumer 幂等！

幂等方案：

① 数据库唯一约束（最推荐）

   VALUES (?, ?, ?);

   boolean success = redis.setnx(key, "1", 60, TimeUnit.MINUTES);
   if (!success) {
       return; // 已处理过
   }
   processMessage();
   
   问题：Redis setnx 和业务操作不在一个事务 → 需要补偿

③ 业务状态机

   // PENDING → PAID → SHIPPED → COMPLETED
   public void handlePaymentMessage(String orderId) {
       Order order = orderMapper.selectById(orderId);
       if (order.getStatus() != OrderStatus.PENDING) {
           // 已经处理过了，直接返回
           return;
       }
       // 更新状态（CAS：只有 PENDING 才更新）
       int updated = orderMapper.updateStatus(orderId, 
           OrderStatus.PENDING, OrderStatus.PAID);
       if (updated == 0) {
           // 并发更新失败 → 已被其他线程处理
           return;
       }
       // 后续处理...
   }

④ 版本号/乐观锁
   UPDATE account SET balance = balance - 100, version = version + 1
   WHERE id = ? AND version = 5;
   -- 如果 version 已变，更新0行 → 说明已处理过

方案对比：

| 方案         | 复杂度 | 可靠性 | 适用场景              |
|--------------|-------|--------|----------------------|
| 唯一约束      | 低    | 最高   | 所有场景（最推荐）     |
| Redis去重     | 中    | 中     | 高并发、允许少量重复   |
| 业务状态机    | 低    | 高     | 业务有明确状态流转     |
| 版本号        | 低    | 高     | 有 version 字段的表   |
```

### Q25：Kafka 的高吞吐是如何实现的？

**参考答案：**

```
Kafka 高吞吐的六大支柱：

1. 顺序写入（Sequential I/O）
   ┌────────────────────────────────────┐
   │  随机写入：磁盘寻道（机械盘10ms+）  │
   │  顺序写入：追加写日志（类似WAL）    │
   │  速度差异：100倍+                   │
   └────────────────────────────────────┘
   
   每个 Partition 是一个不可变的、有序的日志
   只追加、不修改 = 天然顺序写入

2. 零拷贝（Zero Copy）
   传统方式：磁盘 → 内核缓冲区 → 用户缓冲区 → Socket缓冲区 → 网卡
   Kafka:    磁盘 → 内核缓冲区 ──sendfile()→ Socket缓冲区 → 网卡
   
   节省了 2 次内核态↔用户态切换 + 2 次数据拷贝

3. Page Cache
   - Kafka 依赖 OS 的 Page Cache 而非 JVM 堆
   - 热数据在内存中，读写极快
   - 避免 GC 压力（与 RocketMQ 的差异）
   - 即使 Kafka 进程重启，Page Cache 数据还在

4. 批量处理（Batching）
   // Producer 批量发送
   props.put("batch.size", 16384);     // 16KB 一批
   props.put("linger.ms", 5);          // 最多等5ms凑一批
   
   // Consumer 批量拉取
   props.put("fetch.min.bytes", 1024); // 至少1KB才返回
   props.put("fetch.max.wait.ms", 500);
   
   批量 = 更少的网络往返 + 更高的吞吐

5. 分区并行（Partition）
   ┌─────────────────────────────────┐
   │ Topic: order-events              │
   │ ├── Partition 0 → Consumer 1    │  ← 并行消费
   │ ├── Partition 1 → Consumer 2    │
   │ └── Partition 2 → Consumer 3    │
   └─────────────────────────────────┘
   
   分区内有序，分区间并行
   消费者组内每个分区只能被一个Consumer消费

6. 稀疏索引
   - Kafka 不为每条消息建索引
   - 只在 Log Segment 中稀疏地记录 offset ↔ position 映射
   - 查找时：二分找到最近的索引条目 → 顺序扫描到目标offset
   - 索引文件很小，可以全部加载到内存

性能数据：
- 单机吞吐：百万条/秒（小消息）
- 延迟：2-5ms（内存命中时）
- 相比 RabbitMQ：吞吐高1-2个数量级
```

```
public SeckillResult trySeckill(String userId, String productId) {
    // 单用户限购
    String userKey = "seckill:user:" + productId;
    Long userCount = redis.incr(userKey);
    redis.expire(userKey, 300);  // 5分钟过期
    if (userCount > 1) {
        return SeckillResult.repeat();
    }
    
    // 库存预减
    String stockKey = "seckill:stock:" + productId;
    Long stock = redis.decr(stockKey);
    if (stock < 0) {
        redis.incr(stockKey);    // 恢复库存
        redis.decr(userKey);    // 恢复用户计数
        return SeckillResult.soldOut();
    }
    
    // 异步创建订单
    mq.send("seckill-order", new SeckillMessage(userId, productId));
    return SeckillResult.queuing();
}

// 2. 异步下单（MQ Consumer）
@RabbitListener(queues = "seckill-order")
public void createOrder(SeckillMessage msg) {
    // 幂等检查
    if (orderMapper.existsByUserIdAndProductId(msg.userId, msg.productId)) {
        return;
    }
    
    // 分布式锁（防超卖）
    String lockKey = "lock:seckill:order:" + msg.productId;
    boolean locked = redisLock.tryLock(lockKey, 5, TimeUnit.SECONDS);
    if (!locked) return;
    
    try {
        // 真实库存校验
        Product product = productMapper.selectForUpdate(msg.productId);
        if (product.getStock() <= 0) {
            // 超卖补偿：更新Redis库存为0
            redis.set("seckill:stock:" + msg.productId, "0");
            return;
        }
        
        // 减库存 + 创建订单（DB事务）
        productMapper.decreaseStock(msg.productId);
        orderMapper.insert(new Order(msg.userId, msg.productId));
    } finally {
        redisLock.unlock(lockKey);
    }
}
```

### Q27：如何实现一个分布式锁？需要考虑哪些问题？

**参考答案：**

```
分布式锁的演进：

V1：Redis SETNX + EXPIRE（有问题！）
    SETNX lock:order:123 "value"    ← 获取锁
    EXPIRE lock:order:123 30        ← 设置过期
    问题：SETNX 和 EXPIRE 不是原子操作！
    如果 SETNX 之后进程崩溃 → 死锁！

V2：SET key value NX EX 30（原子操作）
    SET lock:order:123 unique_value NX EX 30
    解决：获取锁+设置过期时间原子化
    新问题：锁过期了但我还在执行 → 释放了别人的锁！

V3：看门狗 + 唯一标识
    String lockValue = UUID.randomUUID().toString();
    
    // 获取锁
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent("lock:order:123", lockValue, 30, TimeUnit.SECONDS);
    
    // 看门狗自动续期（Redisson 实现）
    // 每 10 秒检查：如果锁还存在且是当前线程持有 → 续期到 30 秒
    
    // 释放锁（Lua脚本保证原子性）
    String script = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "    return redis.call('del', KEYS[1]) " +
        "else " +
        "    return 0 " +
        "end";
    redisTemplate.execute(new DefaultRedisScript<>(script, Long.class),
        Collections.singletonList("lock:order:123"), lockValue);

V4：RedLock（多节点Redis）
    问题：Redis 主从切换可能丢锁
    RedLock 方案：
    1. 获取当前时间 T1
    2. 依次向 N 个独立的 Redis 节点获取锁
    3. 如果大多数（N/2+1）节点获取成功 且 总耗时 < 锁有效期
       → 获取成功
    4. 获取失败 → 向已获取的所有节点释放锁
    
    争议：Martin Kleppmann 指出 RedLock 也不能保证绝对安全
    结论：如果确实需要强一致性 → ZooKeeper/etcd

分布式锁必备特性 Checklist：

- [ ] 互斥性：任意时刻只有一个客户端持有锁
- [ ] 防死锁：锁持有者崩溃后锁能自动释放（过期机制）
- [ ] 防误解锁：只能解自己的锁（unique value + Lua 原子释放）
- [ ] 可重入：同一个客户端可以重复获取（ThreadLocal 计数）
- [ ] 自动续期：业务执行时间超过锁有效期，自动延长（Watch Dog）
- [ ] 高可用：锁服务本身不能单点故障（集群/哨兵）
- [ ] 阻塞/非阻塞：获取失败是立即返回还是自旋等待

ZooKeeper 实现分布式锁：

// 利用 ZK 的临时顺序节点 + Watch 机制
// 1. 所有客户端在 /lock 下创建临时顺序节点
// 2. 序号最小的节点获得锁
// 3. 其他节点 Watch 前一个节点的删除事件
// 4. 释放锁：删除自己的节点（或 session 断开自动删除）

优点：强一致性，自动释放（Session过期）
缺点：性能不如 Redis，写操作需要过半确认
```

### Q28：如何设计一个高可用的分布式定时任务系统？

**参考答案：**

```

   - 第一个：固定选择第一个执行器
   - 最后一个：固定选择最后一个

   - 随机：随机选择

4. 任务分片（关键）

   int pageSize = totalCount / shardTotal;
   int start = shardIndex * pageSize;
   int end = (shardIndex == shardTotal - 1) ? 
       totalCount : start + pageSize;
   
   // 每个节点只处理自己的分片
   List<Data> data = queryByRange(start, end);
   processBatch(data);

5. 路由策略选型
   - 单机任务 → 故障转移（保证高可用）
   - 定时批量处理 → 分片广播（并行加速）
   - 实时性要求高 → 忙碌转移（找空闲节点）
```

### Q29：线上系统 CPU 飙升到 100%，你的排查思路是什么？

**参考答案：**

```
CPU 100% 排查六步法：

Step 1: 确认问题
    # 找到 CPU 最高的进程
    top -c
    
    # 假设 Java 进程 PID=12345，CPU=200%（2核满）

Step 2: 找到问题线程
    # 找到进程内 CPU 最高的线程
    top -Hp 12345
    
    # 假设 线程 TID=12380 CPU=99%
    
    # 转换成 16 进制（jstack 用 16 进制）
    printf "%x\n" 12380
    → 305c

Step 3: 查看线程堆栈
    # 多次 dump 对比
    jstack 12345 > jstack1.txt
    sleep 2
    jstack 12345 > jstack2.txt
    
    # 搜索 nid=0x305c 的线程
    grep "nid=0x305c" jstack1.txt -A 20
    
    示例输出：
    "pool-1-thread-5" #15 prio=5 os_prio=0 tid=0x00007f...
       nid=0x305c runnable [0x00007f...]
       at com.example.OrderService.calculatePrice(OrderService.java:42)
       at com.example.OrderController.createOrder(OrderController.java:20)

Step 4: 常见原因分析

| CPU高类型      | 常见原因                        | 排查方向                  |
|----------------|--------------------------------|--------------------------|
| 持续100%       | 死循环/无限递归                  | jstack 直接定位线程       |
| 尖峰脉冲       | 流量高峰/秒杀                    | 查看QPS监控              |
| GC频繁         | 频繁 Full GC/Young GC           | jstat -gcutil 查看        |
| 线程数暴涨     | 线程池配置不当                   | jstack 统计线程数         |
| 序列化耗时     | JSON 序列化大对象               | Arthas trace定位          |

Step 5: Arthas 定位（推荐）
    # 查看 CPU 最高的 N 个线程
    thread -n 5
    
    # 监控方法调用耗时
    trace com.example.OrderService calculatePrice
    
    # 火焰图
    profiler start
    profiler stop --format html

Step 6: 根本原因归类
    1. 代码问题：死循环、递归、算法复杂度O(n²)
       → 修代码
    
    2. 流量问题：突然的流量尖峰
       → 限流 + 扩容
    
    3. 配置问题：线程池不合理、GC参数不当
       → 调参数
    
    4. 框架问题：序列化框架性能差
       → 换框架/升级版本

// 完整的一站式诊断命令
#!/bin/bash
PID=$(jps | grep -i app | awk '{print $1}')
echo "=== Top CPU Threads ==="
top -Hp $PID -bn1 | head -20
echo "=== GC Stats ==="
jstat -gcutil $PID 1000 3
echo "=== Thread Count ==="
jstack $PID | grep "java.lang.Thread.State" | sort | uniq -c | sort -nr
echo "=== Memory ==="
jmap -heap $PID 2>/dev/null | head -30
```

十篇文章，从 CAP 定理讲到电商系统架构，从 Paxos 讲到 Snowflake，从 2PC 讲到 Saga。

回顾整个系列：

| 篇目 | 主题 | 核心收获 |
|------|------|---------|
| 一 | 理论基础 | CAP 不是"三选二"，是发生分区时在 C 和 A 之间取舍 |
| 二 | 一致性协议 | Raft 是可理解的 Paxos，ZAB 是为 ZooKeeper 定制的 |
| 三 | 分布式事务 | TCC 是业务层面的 2PC，Saga 是长事务的异步方案 |
| 四 | 分布式存储 | 分库分表不是银弹，全局 ID 和分布式事务是配套难题 |
| 五 | 服务治理 | 注册中心+负载均衡是微服务通信的基础设施 |
| 六 | 消息队列 | 可靠投递和幂等消费是消息系统的一体两面 |
| 七 | 定时任务 | 分片+故障转移+错过补偿是分布式调度的核心 |

| 十 | 综合串讲 | 30道高频题，从理论到实战的系统化整合 |

**记住：分布式不是堆组件，而是在可用性和一致性之间做取舍的艺术。** 每引入一个中间件，就在为解决一个问题而付出一个代价。理解每个决策背后的 trade-off，才是从"会用"到"会设计"的关键跨越。
