### 2.2 核心架构决策（呼应第四篇架构设计）

| 决策点 | 方案 | 理由 |
|--------|------|------|
| 静态化 | CDN + Nginx | 减少后端压力，商品详情页提前静态化 |
| 限流 | 令牌桶 + 滑动窗口（双层） | API网关层令牌桶防突发，业务层滑动窗口精确控量 |

## 三、核心技术攻坚——代码实战

### 3.1 Redis库存预扣——防超卖的核心

**面试高频：为什么用Lua脚本？**

> 因为Redis执行Lua脚本是原子性的，整个脚本作为一个整体执行，中间不会被其他命令插入。如果分开用GET+DECR，会有竞态条件导致超卖。

**Java实现——Spring Boot + Redisson：**

```java
@Service
public class SeckillService {
    
    @Autowired
    private RedissonClient redissonClient;
    
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    // Lua脚本：原子性检查库存+扣减
    private static final String STOCK_LUA = 
        "local stock = redis.call('get', KEYS[1]) " +
        "if stock == false then return -1 end " +  // key不存在
        "if tonumber(stock) <= 0 then return 0 end " + // 库存不足
        "redis.call('decr', KEYS[1]) " +
        "return 1 "; // 扣减成功
    
    /**
     * 秒杀核心逻辑
     */
    public SeckillResult executeSeckill(Long userId, Long goodsId) {
        // 1. 幂等性检查——防止重复秒杀
        String recordKey = "seckill:record:" + goodsId + ":" + userId;
        RBucket<String> bucket = redissonClient.getBucket(recordKey);
        if (bucket.isExists()) {
            return SeckillResult.repeat();
        }
        
        // 2. 原子库存扣减
        String stockKey = "seckill:stock:" + goodsId;
        RScript script = redissonClient.getScript();
        Long result = script.eval(
            RScript.Mode.READ_WRITE,
            STOCK_LUA,
            RScript.ReturnType.INTEGER,
            Collections.singletonList(stockKey)
        );
        
        if (result == -1) {
            return SeckillResult.notStarted();
        }
        if (result == 0) {
            return SeckillResult.soldOut();
        }
        
        // 3. 标记用户已参与
        bucket.set("1", 30, TimeUnit.MINUTES);
        
        // 4. 发送MQ异步创建订单
        SeckillMessage message = new SeckillMessage(userId, goodsId);
        rocketMQTemplate.asyncSend("seckill-order-topic", message, 
            new SendCallback() {
                @Override
                public void onSuccess(SendResult result) {
                    log.info("秒杀消息发送成功: {}", message);
                }
                @Override
                public void onException(Throwable e) {
                    // 发送失败：回滚Redis库存
                    rollbackStock(goodsId);
                    bucket.delete(); // 清除参与标记
                    log.error("秒杀消息发送失败，已回滚: {}", message, e);
                }
            }
        );
        
        return SeckillResult.success();
    }
    
    /**
     * 回滚库存——MQ发送失败时
     */
    private void rollbackStock(Long goodsId) {
        String stockKey = "seckill:stock:" + goodsId;
        redissonClient.getAtomicLong(stockKey).incrementAndGet();
    }
}
```

**Go实现——Gin + go-redis：**

```go
package seckill

import (
    "context"
    "fmt"
    "github.com/go-redis/redis/v8"
    "time"
)

const stockLuaScript = `
local stock = redis.call('get', KEYS[1])
if stock == false then return -1 end
if tonumber(stock) <= 0 then return 0 end
redis.call('decr', KEYS[1])
return 1
`

type SeckillService struct {
    rdb       *redis.Client
    producer  *RocketMQProducer
}

func (s *SeckillService) ExecuteSeckill(ctx context.Context, userId, goodsId int64) (*SeckillResult, error) {
    // 1. 幂等性检查
    recordKey := fmt.Sprintf("seckill:record:%d:%d", goodsId, userId)
    exists, err := s.rdb.Exists(ctx, recordKey).Result()
    if err != nil {
        return nil, err
    }
    if exists > 0 {
        return &SeckillResult{Code: 2, Msg: "请勿重复秒杀"}, nil
    }

    // 2. 原子库存扣减
    stockKey := fmt.Sprintf("seckill:stock:%d", goodsId)
    result, err := s.rdb.Eval(ctx, stockLuaScript, []string{stockKey}).Int()
    if err != nil {
        return nil, err
    }

    switch result {
    case -1:
        return &SeckillResult{Code: 3, Msg: "秒杀未开始"}, nil
    case 0:
        return &SeckillResult{Code: 4, Msg: "已售罄"}, nil
    }

    // 3. 标记已参与
    s.rdb.SetEX(ctx, recordKey, "1", 30*time.Minute)

    // 4. 异步发送MQ
    msg := &SeckillMessage{UserId: userId, GoodsId: goodsId}
    if err := s.producer.SendAsync(ctx, "seckill-order-topic", msg); err != nil {
        // 发送失败回滚
        s.rdb.Incr(ctx, stockKey)
        s.rdb.Del(ctx, recordKey)
        return nil, fmt.Errorf("MQ发送失败: %w", err)
    }

    return &SeckillResult{Code: 0, Msg: "秒杀成功，订单处理中"}, nil
}
```

### 3.2 限流器——多层防线

**面试高频：令牌桶 vs 漏桶 vs 滑动窗口的区别？**

| 算法 | 原理 | 特点 | 适用场景 |
|------|------|------|----------|
| 固定窗口 | 每个时间窗口固定请求数 | 简单，但有临界突刺问题 | 粗粒度限流 |
| 滑动窗口 | 窗口随时间滑动，精确计数 | 精度高，内存开销稍大 | API限流 |
| 漏桶 | 固定速率流出，超出丢弃 | 强制平滑，无法应突增 | 保护下游 |
| 令牌桶 | 固定速率生成令牌，可积累 | 允许突发，兼顾平滑 | 网关限流 |

**Java实现——滑动窗口+令牌桶双层限流：**

```java
@Component
public class RateLimiterInterceptor implements HandlerInterceptor {
    
    // 滑动窗口：精确控制每用户每秒请求数
    private final ConcurrentHashMap<String, SlidingWindow> userWindows = new ConcurrentHashMap<>();
    
    // 令牌桶：网关层整体限流
    private final RateLimiter globalLimiter = RateLimiter.create(100000); // 10万QPS
    
    @Override
    public boolean preHandle(HttpServletRequest request, 
                            HttpServletResponse response, 
                            Object handler) {
        // 第一层：全局令牌桶
        if (!globalLimiter.tryAcquire(100, TimeUnit.MILLISECONDS)) {
            response.setStatus(429);
            return false;
        }
        
        // 第二层：用户级滑动窗口
        String userId = request.getHeader("X-User-Id");
        String key = "rate:" + userId + ":" + request.getRequestURI();
        
        SlidingWindow window = userWindows.computeIfAbsent(key, 
            k -> new SlidingWindow(10, 1000)); // 每秒10个请求
        
        if (!window.tryAcquire()) {
            response.setStatus(429);
            response.getWriter().write("{\"code\":429,\"msg\":\"请求过于频繁\"}");
            return false;
        }
        
        return true;
    }
}

// 滑动窗口实现
class SlidingWindow {
    private final int limit;
    private final long windowSizeMs;
    private final LinkedList<Long> timestamps = new LinkedList<>();
    
    public SlidingWindow(int limit, long windowSizeMs) {
        this.limit = limit;
        this.windowSizeMs = windowSizeMs;
    }
    
    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        // 移除过期时间戳
        while (!timestamps.isEmpty() && now - timestamps.getFirst() > windowSizeMs) {
            timestamps.removeFirst();
        }
        if (timestamps.size() < limit) {
            timestamps.addLast(now);
            return true;
        }
        return false;
    }
}
```

### 3.3 分布式锁——防止重复下单

**Redisson分布式锁实战：**

```java
@Service
public class OrderService {
    
    @Autowired
    private RedissonClient redissonClient;
    
    /**
     * 基于分布式锁的安全下单
     */
    @Transactional
    public Order createOrder(Long userId, Long goodsId) {
        String lockKey = "order:lock:" + userId + ":" + goodsId;
        RLock lock = redissonClient.getLock(lockKey);
        
        try {
            // 尝试加锁，最多等待3秒，锁持有时间10秒
            if (!lock.tryLock(3, 10, TimeUnit.SECONDS)) {
                throw new BusinessException("系统繁忙，请稍后重试");
            }
            
            // 二次校验：是否已有该商品的订单
            Order existOrder = orderMapper.selectByUserAndGoods(userId, goodsId);
            if (existOrder != null) {
                throw new BusinessException("已参与该商品秒杀");
            }
            
            // 查询商品信息
            Goods goods = goodsMapper.selectById(goodsId);
            if (goods == null || goods.getStock() <= 0) {
                throw new BusinessException("商品已售罄");
            }
            
            // 创建订单
            Order order = Order.builder()
                .userId(userId)
                .goodsId(goodsId)
                .price(goods.getSeckillPrice())
                .status(OrderStatus.PENDING)
                .createTime(new Date())
                .build();
            orderMapper.insert(order);
            
            // 扣减MySQL库存（最终一致性兜底）
            goodsMapper.decrStock(goodsId);
            
            return order;
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new BusinessException("系统异常");
        } finally {
            // 确保释放锁
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

### 3.4 消息队列削峰——从100万QPS到1万TPS

### 3.5 分布式事务——Seata AT模式

**场景**：下单服务调用库存服务和积分服务，需要保证分布式事务一致性。

```java
@Service
public class SeckillOrderService {
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Autowired
    private StockService stockService;  // Feign远程调用
    
    @Autowired
    private PointsService pointsService; // Feign远程调用
    
    /**
     * Seata AT模式——@GlobalTransactional注解搞定分布式事务
     */
    @GlobalTransactional(name = "seckill-create-order", timeoutMills = 300000)
    public void createOrderWithTx(Long userId, Long goodsId) {
        // 1. 本地事务：创建订单
        Order order = new Order();
        order.setUserId(userId);
        order.setGoodsId(goodsId);
        order.setStatus(OrderStatus.CREATED);
        orderMapper.insert(order);
        
        // 2. 远程调用：扣减库存
        stockService.deduct(goodsId, 1);
        
        // 3. 远程调用：增加积分
        pointsService.addPoints(userId, 100);
        
        // 任何一步失败，Seata自动回滚全局事务
    }
}
```

**Seata AT模式原理**：
1. **一阶段**：各服务执行本地事务，Seata记录undo_log
2. **二阶段-提交**：全局事务提交，异步删除undo_log
3. **二阶段-回滚**：根据undo_log逆向回滚各分支事务

### 3.6 缓存架构——多级缓存设计

```
请求 → CDN(边缘节点) → Nginx本地缓存 → Caffeine(JVM本地) → Redis(分布式) → MySQL
                             ↓                 ↓                  ↓               ↓
                         静态HTML         热点商品详情         商品库存          持久数据
                        (不经过后端)      (1min过期)          (实时)          (最终数据)
```

**Caffeine + Redis多级缓存实现：**

```java
@Service
public class GoodsCacheService {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    // JVM本地缓存：热点商品
    private final Cache<Long, Goods> localCache = Caffeine.newBuilder()
        .maximumSize(1000)
        .expireAfterWrite(1, TimeUnit.MINUTES)
        .recordStats()
        .build();
    
    /**
     * 多级缓存查询：本地 → Redis → MySQL
     */
    public Goods getGoods(Long goodsId) {
        // L1：JVM本地缓存
        Goods goods = localCache.getIfPresent(goodsId);
        if (goods != null) {
            return goods;
        }
        
        // L2：Redis分布式缓存
        String cacheKey = "goods:detail:" + goodsId;
        String json = redisTemplate.opsForValue().get(cacheKey);
        if (json != null) {
            goods = JSON.parseObject(json, Goods.class);
            localCache.put(goodsId, goods); // 回填L1
            return goods;
        }
        
        // L3：数据库
        goods = goodsMapper.selectById(goodsId);
        if (goods != null) {
            // 回写L2+L1
            redisTemplate.opsForValue().set(cacheKey, 
                JSON.toJSONString(goods), 5, TimeUnit.MINUTES);
            localCache.put(goodsId, goods);
        }
        
        return goods;
    }
    
    /**
     * 缓存预热——秒杀前批量加载
     */
    @Scheduled(cron = "0 0 9 * * ?") // 每天9点预热
    public void warmUp() {
        List<Goods> seckillGoods = goodsMapper.selectTodaySeckill();
        for (Goods goods : seckillGoods) {
            String cacheKey = "goods:detail:" + goods.getId();
            redisTemplate.opsForValue().set(cacheKey, 
                JSON.toJSONString(goods), 2, TimeUnit.HOURS);
            localCache.put(goods.getId(), goods);
        }
        log.info("缓存预热完成，预热商品数: {}", seckillGoods.size());
    }
}
```

### 3.7 分库分表——订单表水平扩展

**ShardingSphere分片策略**：

public class SeckillController {

## 五、面试高频——分布式系统核心问题

### Q1：Redis库存扣减为什么用Lua脚本？

**A**：Redis单线程执行Lua脚本保证原子性。如果用GET判断库存 + DECR扣减两步操作，之间存在竞态条件，会导致超卖。Lua脚本将"检查+扣减"打包为一个原子操作。

### Q2：100万并发请求如何做到不超卖？

**A**：三层防线：
1. **Redis层**：Lua原子扣减，只有1万人通过
2. **MQ层**：异步削峰，避免DB压力
3. **DB层**：乐观锁version字段兜底 `UPDATE goods SET stock=stock-1, version=version+1 WHERE id=? AND stock>0 AND version=?`

### Q3：分布式事务Seata AT模式的优缺点？

**优点**：对业务无侵入（一个注解搞定），自动回滚
**缺点**：性能有损耗（需要记录undo_log），不适合高并发写场景
**替代方案**：TCC模式（Try-Confirm-Cancel），性能更好但开发复杂

### Q4：如何保证消息不丢失？

**A**：三重确认机制：
1. **生产者**：同步发送 + 失败重试 + 本地消息表兜底
2. **Broker**：同步刷盘 + 主从复制
3. **消费者**：手动ACK + 消费完再确认

### Q5：系统设计题的答题框架（面试必备）

回答任何系统设计题的结构：
1. **需求澄清**（5min）：场景、DAU、QPS、存储量
2. **顶层设计**（5min）：画架构图、服务拆分
3. **组件深挖**（10min）：核心模块详细设计
4. **质量属性**（5min）：扩展性、可用性、一致性

## 六、知识总结

### 核心知识矩阵

| 层次 | 核心技术 | 关键原理 | 落地要点 |
|------|----------|----------|----------|
| 接入层 | Nginx + API网关 | 动静分离、SSL卸载、限流 | 网关层是防线第一关 |
| 缓存层 | Redis + Caffeine | 缓存穿透/击穿/雪崩三件套 | Lua原子操作是关键 |
| 消息层 | RocketMQ/Kafka | 削峰、解耦、异步 | ACK机制确保不丢消息 |
| 数据层 | MySQL + 分库分表 | 读写分离、ShardingSphere | user_id分片避免跨库查询 |
| 事务层 | Seata AT/TCC | 2PC、undo_log、全局锁 | 高并发场景优选TCC |
| 可观测 | Prometheus + Grafana | 指标、链路、日志三支柱 | 先建监控再上线 |

**分布式系统设计四字诀：**

1. **分（分而治之）**：分库分表、服务拆分，但不要过度拆分
2. **缓（缓存为王）**：多级缓存是性能提升的最有效手段，但注意一致性
3. **异（异步解耦）**：用MQ削峰填谷，不要同步等待非核心链路
4. **兜（兜底容错）**：限流、熔断、降级、重试，假定一切都会挂
