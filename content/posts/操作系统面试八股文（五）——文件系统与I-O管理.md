## 一、文件系统概述

### 1.1 文件的概念

**文件**是操作系统中具有名字的、逻辑上具有完整意义的信息集合。文件包含两个视角：
- **逻辑文件结构**：用户眼中的文件形态（记录式、流式）
- **物理文件结构**：文件在磁盘上的存储形态

**文件的属性**（至少掌握以下核心属性）：
1. 文件名（name）：用户可见，操作系统通过文件名访问
2. 文件类型（type）：普通文件、目录文件、设备文件、链接文件等
3. 文件大小（size）：字节数
4. 创建时间（ctime）：文件创建时间
5. 修改时间（mtime）：文件内容最后修改时间
6. 访问权限（permission）：读/写/执行权限
7. 文件位置（location）：文件数据在磁盘上的物理地址
8. 链接计数（link count）：指向该文件的硬链接数

> ⚠️ **面试区分**：ctime是inode的最后修改时间（包括属性和数据），mtime是数据内容最后修改时间，atime是最后访问时间。修改数据会更新mtime和ctime，修改属性会更新ctime。

### 1.2 文件系统的层次结构

```
┌─────────────────────────────────────────┐
│        用户/应用程序                      │
├─────────────────────────────────────────┤
│    命令接口（系统调用：open/read/write）   │
├─────────────────────────────────────────┤
│    文件组织模块（逻辑→物理地址转换）         │
├─────────────────────────────────────────┤
│    基本文件系统（物理块操作）               │
├─────────────────────────────────────────┤
│    设备驱动程序（磁盘驱动）                 │
├─────────────────────────────────────────┤
│    物理设备（磁盘）                        │
└─────────────────────────────────────────┘
```

### 1.3 文件系统的功能

| 功能 | 说明 |
|------|------|
| 文件创建/删除 | 创建新文件，删除不需要的文件 |
| 目录管理 | 创建、删除、重命名目录 |
| 文件操作 | 打开、关闭、读写、定位 |
| 空间管理 | 磁盘空间的分配与回收 |
| 安全保护 | 访问权限控制 |
| 完整性保护 | 故障恢复、文件备份 |

## 二、文件组织与目录结构

### 2.1 文件的逻辑结构

#### 记录式文件

数据以**记录**为单位组织，适合数据库系统：

```
记录1 | 记录2 | 记录3 | ... | 记录N
```

- **定长记录**：每条记录长度相同，定位简单
- **变长记录**：记录长度可变，定位复杂但节省空间

#### 流式文件

数据以**字节流**为单位组织，操作系统不关心内容含义：

```
字节1 字节2 字节3 ... 字节N
```

Unix/Linux系统采用流式文件模型。

### 2.2 文件的物理结构

#### 连续分配（顺序结构）

```
┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐
│ F1  │     │ F2  │     │     │ F3  │     │     │
│ ABC │     │ DEF │     │     │ GHI │     │     │
└─────┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘
```

**优点**：
- 顺序访问速度快（磁头移动少）
- 只需记录起始块号和长度

**缺点**：
- 外部碎片（磁盘空间利用率低）
- 文件长度固定，难以扩展

**适用场景**：磁带、CD-ROM等顺序访问设备

#### 链接分配（串联结构）

```
块1 → 块2 → 块3 → 块4 → 块5 → ...
```

**优点**：
- 无外部碎片
- 文件大小可动态增长

**缺点**：
- 只能顺序访问，无法随机访问
- 指针占用存储空间
- 指针丢失导致文件损坏

**改进**：FAT文件系统（File Allocation Table）

```
┌──────┬──────┬──────┬──────┬──────┬──────┐
│ FAT  │ 块1  │ 块2  │ 块3  │ 块4  │ 块5  │
│ -1   │ 2    │ 3    │ -1   │ -1   │ 0    │
└──────┴──────┴──────┴──────┴──────┴──────┘
     ↑
    FAT存放所有块的链接关系
```

#### 索引分配

每个文件有一个**索引块**（Index Block），存放该文件所有数据块的块号：

```
┌────────────┐
│  索引块     │
│  ┌──┬──┬──┐│
│  │0 │1 │2 ││  → 数据块0,1,2
│  └──┴──┴──┘│
│  ┌──┬──┐   ││
│  │3 │4 │   ││  → 数据块3,4
│  └──┴──┘   ││
└────────────┘
```

**优点**：
- 支持随机访问
- 无外部碎片
- 文件长度可动态增长

**缺点**：
- 小文件浪费索引块空间
- 大文件需要多层索引

**改进方案**：
1. **i-node结构**：Unix采用，每个文件一个i-node，固定大小（如256字节），支持直接块+间接块+双间接块+三间接块

```
i-node 结构（以Unix ext2为例）：
├── 直接块（12个）        → 12 × 4KB = 48KB
├── 一级间接块（1个）     → 1024 × 4KB = 4MB
├── 二级间接块（1个）     → 1024² × 4KB = 4GB
├── 三级间接块（1个）     → 1024³ × 4KB = 4TB
└── 文件元信息（权限、时间等）
```

### 2.3 目录结构

| 类型 | 特点 | 代表系统 |
|------|------|----------|
| 单层目录 | 所有文件在同一目录 | 早期DOS |
| 两层目录 | 用户目录+系统目录 | 早期CP/M |
| 层次目录 | 树形结构 | Windows、Unix |
| 图形化目录 | 支持快捷方式 | Windows |
| 平面目录 | 无层级，所有文件平等 | 某些邮件系统 |

**目录项的存放方式**：
1. **线性查找**：目录项顺序存放，查找O(n)
2. **Hash表**：目录项通过Hash组织，查找O(1)
3. **B+树**：适合大型目录系统

## 三、虚拟文件系统（VFS）

### 3.1 为什么需要VFS？

```
┌──────────────┐
│   用户程序    │
│ open/read    │
└──────┬───────┘
       ↓
┌──────────────┐
│   VFS层       │ ← 统一抽象层
│ (sys_open等)  │
└──────┬───────┘
       ↓
┌──────┴───────┬───────────┬──────────┐
│   Ext4       │   NTFS     │   FAT32  │
│   XFS        │   Btrfs    │   NFS    │
│   (具体文件系统实现)                │
└──────────────────────────────────────┘
```

**核心思想**：VFS对用户程序提供**统一的文件系统接口**，屏蔽底层不同文件系统的差异。

### 3.2 VFS的核心数据结构

#### 超级块（Superblock）

存储文件系统的**全局信息**：
- 文件系统类型（ext4、NTFS等）
- 逻辑块大小
- 空闲块数量和位图位置
- i-node总数和位图位置
- 挂载时间、修改时间等

#### i-node（索引节点）

每个文件在文件系统中对应一个**i-node**，存储文件的**元数据**：

```c
struct inode {
    unsigned long i_ino;        // i-node号
    umode_t       i_mode;       // 文件类型和权限
    uid_t         i_uid;        // 所有者UID
    gid_t         i_gid;        // 所有者GID
    loff_t        i_size;       // 文件大小
    struct timespec i_atime;    // 最后访问时间
    struct timespec i_mtime;    // 最后修改时间
    struct timespec i_ctime;    // 最后状态改变时间
    unsigned long i_blocks;     // 文件块数
    unsigned long i_blkbits;    // 块大小位数
    unsigned long i_flags;      // 文件系统标志
    struct super_block *i_sb;   // 指向超级块的指针
    unsigned long i_nlink;      // 硬链接数
    // ... 其他字段
};
```

#### 目录项（dentry）

**目录项**是路径解析过程中的临时数据结构，串联成路径：

```c
struct dentry {
    struct qstr d_name;         // 目录项名称
    struct inode *d_inode;      // 指向i-node
    struct dentry *d_parent;     // 父目录
    struct list_head d_subdirs; // 子目录链表
    // ...
};
```

> ⚠️ **面试区分**：目录项是VFS的缓存结构，存放在**内存**中。目录项缓存（dentry cache）加速路径解析。目录项和i-node的关系是：一个i-node可以被多个目录项引用（硬链接）。

### 3.3 文件系统注册与挂载

```
mount -t ext4 /dev/sda1 /mnt/data
         │        │        │
         │        │        └── 挂载点（VFS中的目录路径）
         │        └── 设备路径
         └── 文件系统类型（ext4由VFS识别并调用对应驱动）
```

挂载过程：
1. 根据文件系统类型，找到对应的**file_system_type**
2. 读取磁盘超级块，创建**super_block**结构
3. 将超级块与挂载点目录的**dentry**关联
4. 将新文件系统加入VFS的**mount**树

## 四、磁盘调度算法

### 4.1 磁盘结构与访问时间

```
┌─────────────────────────────────┐
│          磁盘结构                │
│                                 │
│   磁道（Track）← 外圈同心圆      │
│   ┌───────────────────────┐    │
│   │  扇区（Sector）← 512B  │    │
│   │  扇区（Sector）← 512B  │    │
│   │  扇区（Sector）← 512B  │    │
│   └───────────────────────┘    │
│                                 │
│   磁头（Head）← 读写磁头         │
│   柱面（Cylinder）← 同半径磁道   │
└─────────────────────────────────┘
```

**磁盘访问时间构成**：

| 组成部分 | 说明 | 耗时 |
|----------|------|------|
| 寻道时间 Tseek | 磁头移动到目标柱面 | 3-10ms |
| 旋转延迟 Tlatency | 等待目标扇区旋转到磁头下 | 0-8ms |
| 传输时间 Ttransfer | 数据读写时间 | 0.1-0.5ms |

**总访问时间 = Tseek + Tlatency + Ttransfer**

寻道时间是最主要的延迟来源，因此磁盘调度算法的核心目标是**减少磁头移动距离**。

### 4.2 先来先服务（FCFS）

**算法描述**：按请求到达顺序服务

**示例**：请求访问柱面顺序：98, 183, 37, 122, 14, 124, 65, 67，磁头起始位置：53

```
磁头移动：53→98→183→37→122→14→124→65→67
总移动距离 = 45+85+146+85+108+110+59+2 = 640
```

**评价**：
- ✅ 公平，简单
- ❌ 平均寻道距离大，性能差

### 4.3 最短寻道时间优先（SSTF）

**算法描述**：总是选择距当前磁头位置最近的请求

**示例**：同上

```
当前：53
最近：65（距离12）→ 67（距离2）→ 37（距离30）→ 14（距离23）→ 98（距离84）→ 122（距离24）→ 124（距离2）→ 183（距离59）
总移动距离 = 12+2+30+23+84+24+2+59 = 236
```

**评价**：
- ✅ 比FCFS性能好
- ❌ 会产生**饥饿**（远端请求可能永远得不到服务）

### 4.4 扫描算法（SCAN）

**算法描述**：磁头从外向内（或从内向外）移动，途中处理所有经过的请求，到达边界后反向

**别名**：电梯算法（Elevator Algorithm）

**示例**：同上，假设磁头从外向内移动

```
磁头移动：53→65→67→98→122→124→183→（到边界）→ 37→14
总移动距离 = 12+2+31+24+2+59+146+23 = 299
```

**评价**：
- ✅ 磁头移动距离可控，无饥饿
- ❌ 极端情况下请求分布不均匀导致不均衡

### 4.5 循环扫描算法（C-SCAN）

**算法描述**：磁头单向扫描，到达一端后快速返回起点，重新开始扫描

```
移动方向：只能从外向内（或从内向外）
回程时不处理请求，直接返回起点
```

**示例**：从外向内

```
磁头移动：53→65→67→98→122→124→183→（快速返回）→ 53→（再开始）...
```

**评价**：
- ✅ 提供更均匀的响应时间
- ❌ 磁头回程时浪费

### 4.6 LOOK与C-LOOK

**LOOK**：与SCAN类似，但不移动到边界，而是在最后一个请求处反向

```
实际路径：53→65→67→98→122→124→183→（反向，不到达183）→ 37→14
```

**C-LOOK**：与C-SCAN类似，但不移动到边界

**实际应用**：Linux的deadline I/O调度器、noop调度器等都有SCAN变种

### 4.7 算法对比总结

| 算法 | 平均寻道距离 | 公平性 | 饥饿 | 实现 |
|------|-------------|--------|------|------|
| FCFS | 高 | 公平 | 无 | 简单 |
| SSTF | 低 | 不公平 | 有 | 较简单 |
| SCAN | 中等 | 较公平 | 无 | 中等 |
| C-SCAN | 中等 | 公平 | 无 | 中等 |
| LOOK | 较低 | 较公平 | 无 | 中等 |

**面试口诀**：FCFS最简单，SSTF最短但会饿，SCAN像电梯，C-SCAN单向扫，LOOK是SCAN改进版。

## 五、I/O控制方式

### 5.1 四种I/O控制方式对比

| 方式 | CPU干预 | 数据传输 | 特点 | 适用场景 |
|------|---------|----------|------|----------|
| 程序直接控制 | 完全干预 | 程序循环读状态位 | 简单，效率低 | 微控制器 |
| 中断驱动 | 单字节中断 | 中断驱动传输 | 效率高，实现复杂 | 键盘、鼠标等字符设备 |
| DMA（直接存储器访问） | 开始/结束干预 | DMA控制器传输整块数据 | 高效，适合块设备 | 磁盘、网络设备 |
| 通道方式 | 极低干预 | 通道程序执行 | 最灵活，成本高 | 大型机、超级计算机 |

### 5.2 DMA工作原理

```
┌────────┐      ┌────────┐      ┌────────┐
│  CPU   │      │  DMA   │      │  磁盘   │
└───┬────┘      └───┬────┘      └───┬────┘
    │               │               │
    │ 1.设置DMA寄存器│               │
    ├──────────────→│               │
    │               │               │
    │ 2.启动DMA     │               │
    ├──────────────→│               │
    │               │               │
    │ 3.数据→内存   │  4.读/写数据  │
    │               ├──────────────→│
    │               │               │
    │ 5.完成中断    │               │
    │←──────────────┤               │
```

**DMA寄存器**：
- MAR（Memory Address Register）：内存地址
- DRT（Data Register）：传输字数
- CR（Command Register）：命令
- SR（Status Register）：状态

### 5.3 缓冲区管理

**单缓冲（Single Buffer）**：
```
┌─────────┐     ┌─────────┐
│  T 设备  │ ─→ │ 缓冲区   │ ─→ 工作区 ─→ 进程处理
└─────────┘     └─────────┘
```
- 处理时间T，输入时间M
- 处理一个数据块时间：MAX(C,T) + M

**双缓冲（Double Buffer / 循环缓冲）**：
```
┌─────────┐     ┌─────────┐     ┌─────────┐
│  T 设备  │ ─→ │ 缓冲区A  │ ─→ │ 工作区A  │ → 处理
└─────────┘     └─────────┘     └─────────┘
                   ↑
┌─────────┐     切换
│  T 设备  │ ─→ │ 缓冲区B  │ ─→ │ 工作区B  │ → 处理
└─────────┘     └─────────┘     └─────────┘
```
- 理想情况下：MAX(C,T)

**缓冲池（Buffer Pool）**：
- 空闲缓冲区队列
- 输入缓冲区队列
- 输出缓冲区队列
- 动态分配，效率最高

## 六、页面置换算法进阶

### 6.1 全局置换 vs 局部置换

- **局部置换**：每个进程有自己的候选页框集（进程固定大小）
- **全局置换**：所有物理页框都可被置换（进程大小可变）

| 类型 | 特点 | 代表算法 |
|------|------|----------|
| 局部置换 | 页面抖动不传染，内存利用率低 | LRU、FIFO、Clock |
| 全局置换 | 页面抖动会传染，内存利用率高 | 工作集算法、PFF |

### 6.2 抖动（Thrashing）

**定义**：频繁的页面置换，导致CPU时间用于换页而非计算

**原因**：
1. 进程数量过多，物理内存不足
2. 进程工作集 > 物理内存
3. 全局置换导致页面在进程间"颠簸"

**解决思路**：
1. **工作集模型**：跟踪每个进程的工作集，保持在内存
2. **页面故障频率控制**（PFF）：当故障率过高，分配更多页框
3. **控制并发进程数**：减少竞争

### 6.3 页面置换算法详解

#### Clock算法（第二次机会）

```
每个页有访问位（A）和修改位（M）：

A=0, M=0 → 最佳，可置换
A=0, M=1 → 可置换，但需写回
A=1, M=0 → 置A=0，继续找
A=1, M=1 → 置A=0，继续找
```

Clock指针环形扫描，已访问的清A，未访问的置换。

#### 改进Clock算法（NRU）

按(A,M)分组优先级：

| 优先级 | A | M | 说明 |
|--------|---|---|------|
| 1 | 0 | 0 | 最近未访问且未修改，最佳置换候选 |
| 2 | 0 | 1 | 未访问但已修改，置换需写回 |
| 3 | 1 | 0 | 最近访问过，未修改 |
| 4 | 1 | 1 | 最近访问过且已修改 |

从低优先级组开始扫描。

#### 工作集算法

**工作集 W(t, Δ)**：进程在时间t过去Δ时间内访问的页面集合

```
W(t, Δ) = {p | p在[t-Δ+1, t]内被访问}
```

**驻留集**：当前在内存中的页面集合

**置换策略**：
- 驻留集 < 工作集 → 从空闲链表分配
- 驻留集 > 工作集 → 置换超出部分

**时间窗口Δ的选择**：
- Δ太小 → 工作集包含近期局部信息
- Δ太大 → 包含过多页面，失去局部性意义

### 6.4 Belady异常

**定义**：增加页框数反而导致页面故障率增加（违反常理）

**FIFO算法的Belady异常**：

```
访问序列：1,2,3,4,1,2,5,1,2,3,4,5

3个页框：故障9次
4个页框：故障10次 ← Belady异常！
```

**哪些算法没有Belady异常？**
- LRU（最近最久未使用）
- OPT（最优算法）
- Clock算法在大多数情况下

### 面试高频问题

1. **硬链接和软链接的区别？**
   - 硬链接：共享inode，跨文件系统限制，删原文件不影响
   - 软链接：独立inode，存原文件路径，可跨文件系统，删原文件变死链

2. **为什么要引入VFS？**
   - 统一不同文件系统的接口
   - 简化用户程序（只调用一套API）
   - 便于添加新文件系统

3. **为什么FIFO会有Belady异常？**
   - FIFO基于物理页面驻留时间，与程序的局部性原理无关
   - 内存增加可能装入"新"的页面，恰好是之前被淘汰的活跃页面
