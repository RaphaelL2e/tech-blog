### [](#Q1-从浏览器输入URL到页面显示的完整过程)Q1：从浏览器输入URL到页面显示的完整过程

这是计算机网络面试中**最经典**的问题，几乎必问：

```
1. URL解析
   ├── 协议解析（HTTP/HTTPS）
   ├── 域名解析
   └── 端口解析（默认80/443）

2. DNS解析
   ├── 浏览器DNS缓存
   ├── 操作系统DNS缓存
   ├── 本地DNS服务器
   ├── 根DNS服务器 → 顶级域DNS → 权威DNS
   └── 返回目标服务器IP

3. TCP连接
   ├── 三次握手：SYN → SYN-ACK → ACK
   └── 若HTTPS，继续TLS握手

4. 发送HTTP请求
   ├── 构造请求行、请求头、请求体
   └── 经过网络层（IP）、数据链路层（MAC）传输

5. 服务器处理
   ├── 负载均衡（如Nginx）
   ├── 应用服务器处理
   ├── 数据库查询
   └── 返回HTTP响应

6. 浏览器渲染
   ├── 解析HTML构建DOM树
   ├── 解析CSS构建CSSOM树
   ├── 合并为渲染树（Render Tree）
   ├── 布局（Layout）
   ├── 绘制（Paint）
   └── 合成（Composite）

7. TCP断开（四次挥手）
   └── FIN → ACK → FIN → ACK
```

**加分项**：
- 提到HTTP缓存机制（强缓存、协商缓存）
- 提到CDN加速
- 提到浏览器预检请求（preflight）和CORS
- 提到连接复用（Keep-Alive、HTTP/2多路复用）

### [](#Q2-HTTP1.0-1.1-2.0-3.0的区别)Q2：HTTP/1.0、1.1、2.0、3.0的区别

| 特性 | HTTP/1.0 | HTTP/1.1 | HTTP/2.0 | HTTP/3.0 |
|------|----------|----------|----------|----------|
| 连接方式 | 短连接 | Keep-Alive长连接 | 多路复用 | 多路复用 |
| 请求方式 | 串行 | 管道化 | 并行多路复用 | 并行多路复用 |
| 头部压缩 | 无 | 无 | HPACK | QPACK |
| 传输协议 | TCP | TCP | TCP | QUIC(UDP) |
| 队头阻塞 | 有 | 有（管道化也有） | 无（应用层） | 无（传输层） |
| 服务器推送 | 无 | 无 | 支持 | 支持 |
| 二进制传输 | 文本 | 文本 | 二进制帧 | 二进制帧 |
| 错误恢复 | 慢 | 慢 | 快（帧级别） | 更快（包级别） |

**重点**：
- HTTP/1.1的管道化虽然有队头阻塞问题，但支持长连接
- HTTP/2.0解决了应用层的队头阻塞，但TCP层面的队头阻塞仍然存在
- HTTP/3.0基于QUIC（UDP），彻底解决了队头阻塞

### [](#Q3-TCP和UDP的区别及适用场景)Q3：TCP和UDP的区别及适用场景

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠传输 | 不可靠传输 |
| 有序性 | 保证顺序 | 不保证顺序 |
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有（慢启动等） | 无 |
| 传输方式 | 字节流 | 数据报文 |
| 速度 | 较慢 | 较快 |
| 头部开销 | 20字节 | 8字节 |
| 广播 | 不支持 | 支持 |

**TCP适用场景**：文件传输、HTTP/HTTPS、邮件（SMTP）、数据库连接

**UDP适用场景**：视频通话、直播、DNS查询、游戏、IoT

**现代UDP协议**：QUIC（HTTP/3.0底层协议）使用UDP实现了可靠传输和拥塞控制

### [](#Q4-TCP三次握手为什么不能是两次)Q4：TCP三次握手为什么不能是两次？

- **防止已失效的连接请求到达服务器**：如果客户端发送了一个SYN，由于网络延迟很久才到达服务器，客户端已经超时重传并建立了连接。此时旧的SYN到达服务器，如果只需要两次握手，服务器就会建立一个无用的连接
- **确认双方的收发能力**：三次握手确保客户端和服务端都能发送和接收数据
- **同步序列号**：双方交换初始序列号（ISN）

### [](#Q5-TCP四次挥手为什么TIME_WAIT是2MSL)Q5：TCP四次挥手为什么TIME_WAIT是2MSL？

- **MSL**（Maximum Segment Lifetime）：报文最大生存时间，通常2分钟
- **2MSL**：一个MSL是报文在网络中的最长生存时间，两个MSL确保：
  - 最后一个ACK能到达对方（如果对方没收到会重发FIN）
  - 网络中所有延迟的报文都过期，不会影响新连接

### [](#Q6-HTTPS的加密过程)Q6：HTTPS的加密过程

1. **TCP三次握手**建立连接
2. **TLS握手**：
   - 客户端发送支持的加密套件列表和随机数
   - 服务器选择加密套件，发送证书和随机数
   - 客户端验证证书，生成预主密钥（Pre-Master Secret），用服务器公钥加密发送
   - 双方根据预主密钥和随机数生成会话密钥
3. 使用**对称加密**（会话密钥）进行HTTP通信

### [](#Q7-如何设计一个短链接服务)Q7：如何设计一个短链接服务？

```
1. 核心功能
   ├── 长链接 → 短链接（生成）
   ├── 短链接 → 长链接（跳转）
   └── 点击统计

2. 生成策略
   ├── 发号器（自增ID）+ Base62编码
   ├── MD5/SHA256哈希（取前N位）
   └── 分布式ID（Snowflake）

3. 存储设计
   ├── Redis缓存热点短链
   ├── MySQL持久化存储
   └── key=短链hash, value=原始长链接

4. 技术细节
   ├── 302重定向（便于统计点击）
   ├── 布隆过滤器防止重复生成
   └── 定时清理过期短链
```

### [](#Q8-高并发下的负载均衡方案)Q8：高并发下的负载均衡方案

```
1. DNS负载均衡
   ├── 最简单的方案
   └── 缺点：DNS缓存导致更新慢，无法感知服务器状态

2. 四层负载均衡（L4）
   ├── 工作在传输层（TCP/UDP）
   ├── 性能高，只转发IP+端口
   ├── 代表：LVS、Nginx stream
   └── 算法：轮询、加权轮询、最小连接、源地址哈希

3. 七层负载均衡（L7）
   ├── 工作在应用层（HTTP）
   ├── 可以根据URL、Cookie、Header等路由
   ├── 代表：Nginx、HAProxy
   └── 支持会话保持（Session Affinity）

4. 常见架构
   DNS → CDN → L4负载均衡 → L7负载均衡 → 应用服务器
```

### [](#Q9-如何实现WebSocket)Q9：如何实现WebSocket？

```
1. 握手阶段（基于HTTP升级）
   客户端：
   GET /chat HTTP/1.1
   Upgrade: websocket
   Connection: Upgrade
   Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==

   服务端：
   HTTP/1.1 101 Switching Protocols
   Upgrade: websocket
   Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

2. 数据传输
   ├── 全双工通信
   ├── 帧格式：FIN+RSV+Opcode+Mask+Payload
   ├── 支持文本帧和二进制帧
   └── 心跳检测（Ping/Pong）

3. 与HTTP长轮询对比
   ├── WebSocket：全双工、低延迟、持久连接
   ├── 长轮询：半双工、延迟高、频繁建立连接
   └── SSE：服务端推送、单向通信
```

### [](#Q11-CookieSessionToken的区别)Q11：Cookie、Session、Token的区别

| 特性 | Cookie | Session | Token (JWT) |
|------|--------|---------|-------------|
| 存储位置 | 客户端浏览器 | 服务器内存 | 客户端（通常localStorage） |
| 大小限制 | 4KB | 无限制 | 自定义 |
| 安全性 | 较低（可被篡改） | 较高 | 较高（签名验证） |
| 跨域 | 需要配置 | 不支持跨域 | 支持跨域 |
| 服务器压力 | 无 | 有（存储Session） | 无（无状态） |
| 分布式 | 需要Session共享 | 需要Session复制 | 天然支持 |

### [](#Q12-HTTP状态码常考总结)Q12：HTTP状态码常考总结

**2xx成功**：
- **200 OK**：请求成功
- **201 Created**：资源创建成功
- **204 No Content**：成功但无返回内容

**3xx重定向**：
- **301 Moved Permanently**：永久重定向（SEO权重转移）
- **302 Found**：临时重定向（不转移权重）
- **304 Not Modified**：协商缓存命中

**4xx客户端错误**：
- **400 Bad Request**：请求语法错误
- **401 Unauthorized**：未认证
- **403 Forbidden**：权限不足
- **404 Not Found**：资源不存在
- **405 Method Not Allowed**：请求方法不支持
- **429 Too Many Requests**：请求过多（限流）

**5xx服务端错误**：
- **500 Internal Server Error**：服务器内部错误
- **502 Bad Gateway**：网关/代理错误
- **503 Service Unavailable**：服务不可用
- **504 Gateway Timeout**：网关超时

### [](#Q13-输入网址后如果出现502怎么排查)Q13：输入网址后如果出现502怎么排查？

```
1. 确认502的含义
   └── 网关/反向代理（如Nginx）收到了无效的响应

2. 排查步骤
   ├── 检查Nginx错误日志：tail -f /var/log/nginx/error.log
   ├── 检查后端服务是否正常：systemctl status app / ps aux | grep app
   ├── 检查后端端口是否监听：ss -tlnp | grep 8080
   ├── 直接访问后端：curl http://127.0.0.1:8080
   ├── 检查防火墙：iptables -L -n
   └── 检查资源：df -h（磁盘）、free -m（内存）、top（CPU）

3. 常见原因
   ├── 后端服务挂了
   ├── 后端服务启动慢/超时
   ├── 后端端口被占用
   ├── Nginx配置的upstream地址错误
   └── 资源耗尽（OOM、连接数满）
```

### [](#Q14-如何实现接口限流)Q14：如何实现接口限流？

```
1. 固定窗口算法
   ├── 单位时间内固定数量请求
   └── 缺点：临界点突发流量（如10:00:59和10:01:01各100请求）

2. 滑动窗口算法
   ├── 将窗口细分为小格子
   └── 更平滑，但有内存开销

3. 漏桶算法（Leaky Bucket）
   ├── 请求进入桶中，以恒定速率处理
   ├── 输出速率固定
   └── 适合平滑流量

4. 令牌桶算法（Token Bucket）
   ├── 以恒定速率向桶中放入令牌
   ├── 请求需要获取令牌才能处理
   ├── 桶满则丢弃令牌
   └── 允许突发流量（桶中有积累的令牌）

5. 实现方式
   ├── Nginx：limit_req_zone
   ├── Redis + Lua：基于令牌桶
   ├── Sentinel：基于滑动窗口
   └── Guava RateLimiter：基于令牌桶
```

### [](#Q15-跨域问题及解决方案)Q15：跨域问题及解决方案？

**什么是跨域**：浏览器的同源策略限制，协议+域名+端口任一不同即为跨域。

**解决方案**：

| 方案 | 原理 | 适用场景 |
|------|------|---------|
| CORS | 服务端设置Access-Control-Allow-Origin | 最推荐 |
| JSONP | 利用script标签不受同源策略限制 | 仅GET，老旧方案 |
| 代理 | 前端请求同源代理，代理转发到目标 | Nginx反向代理 |
| WebSocket | 不受同源策略限制 | 双向通信场景 |
| postMessage | 窗口间通信 | iframe场景 |

**CORS预检请求**：
```
OPTIONS /api HTTP/1.1
Origin: http://example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type
```

非简单请求（PUT/DELETE/自定义Header）会先发OPTIONS预检请求。
