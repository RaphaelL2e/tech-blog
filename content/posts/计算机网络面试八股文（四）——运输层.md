# 计算机网络面试八股文（四）——运输层

## [](#运输层概述)运输层概述

### [](#运输层的功能)运输层的功能

运输层位于网络层之上，应用层之下，是端到端通信的核心层：

- **进程到进程的通信**：提供端口号标识应用进程
- **复用和分用**：多个进程可同时使用运输层协议，接收方能正确分用
- **差错检测**：对数据和首部进行校验
- **提供可靠的传输服务**（TCP）或不可靠的传输服务（UDP）

### [](#端口的概念)端口的概念

- 端口号范围：0 ~ 65535
- **熟知端口**（0-1023）：HTTP(80)、HTTPS(443)、FTP(21)、SSH(22)、SMTP(25)、DNS(53)
- **登记端口**（1024-49151）：MySQL(3306)、Redis(6379)
- **客户端端口**（49152-65535）：临时使用

### [](#套接字)套接字

套接字（Socket）= IP地址 + 端口号

唯一标识网络中一个进程的通信端点，是运输层与应用层的接口。

## [](#UDP协议)UDP协议

### [](#UDP的特点)UDP的特点

- **无连接**：发送数据前不需要建立连接
- **不可靠**：不保证数据到达、不保证顺序
- **面向报文**：应用层交下来的报文不合并也不拆分，保留边界
- **无拥塞控制**：网络拥塞不会降低发送速率
- **首部开销小**：仅8字节
- **支持一对一、一对多、多对多通信**

### [](#UDP首部格式)UDP首部格式

| 字段 | 长度 | 说明 |
|------|------|------|
| 源端口 | 16bit | 发送方端口（可选填0） |
| 目的端口 | 16bit | 接收方端口 |
| 长度 | 16bit | UDP数据报总长度（首部+数据） |
| 校验和 | 16bit | 检测数据在传输过程中是否有错 |

**UDP校验和**：
- 计算时需要加上12字节的伪首部（源IP、目的IP、协议、UDP长度）
- 伪首部只在计算校验和时使用，不实际传输
- 如果校验和计算结果为0，则填入全1（0xFFFF）

### [](#UDP的应用场景)UDP的应用场景

- 实时音视频通信（延迟敏感）
- DNS查询
- 简单文件传输（TFTP）
- 在线游戏
- 广播/多播通信

## [](#TCP协议)TCP协议

### [](#TCP的特点)TCP的特点

- **面向连接**：通信前必须建立连接
- **可靠传输**：保证数据无差错、不丢失、不重复、按序到达
- **面向字节流**：把数据看作无结构的字节流
- **全双工通信**：双方可以同时发送和接收数据
- **点对点通信**：只能一对一，不支持广播和多播
- **首部开销大**：最小20字节

### [](#TCP首部格式)TCP首部格式

| 字段 | 长度 | 说明 |
|------|------|------|
| 源端口 | 16bit | 发送方端口 |
| 目的端口 | 16bit | 接收方端口 |
| 序号 | 32bit | 字节流编号 |
| 确认号 | 32bit | 期望收到的下一个字节序号 |
| 数据偏移 | 4bit | 首部长度（以4字节为单位） |
| URG | 1bit | 紧急指针有效 |
| ACK | 1bit | 确认号有效（建立连接后ACK=1） |
| PSH | 1bit | 接收方尽快交付应用层 |
| RST | 1bit | 重置连接 |
| SYN | 1bit | 同步位，建立连接时使用 |
| FIN | 1bit | 终止位，释放连接时使用 |
| 窗口 | 16bit | 接收窗口大小 |
| 校验和 | 16bit | 检验首部和数据 |
| 紧急指针 | 16bit | 紧急数据的偏移量 |

### [](#TCP连接管理)TCP连接管理

#### [](#三次握手)三次握手

```
客户端                    服务器
  |--- SYN, seq=x -------->|      第一次握手
  |<-- SYN+ACK, seq=y, ----|      第二次握手
  |    ack=x+1             |
  |--- ACK, seq=x+1, ----->|      第三次握手
  |    ack=y+1             |
```

**为什么需要三次握手？**

1. **确认双方的发送和接收能力**：
   - 第一次握手：服务器确认客户端发送能力正常
   - 第二次握手：客户端确认服务器接收和发送能力正常
   - 第三次握手：服务器确认客户端接收能力正常

2. **防止失效的连接请求到达服务器**：如果只用两次握手，一个延迟的旧SYN到达服务器后，服务器会立即建立连接浪费资源

3. **同步初始序号**：双方交换并确认对方的初始序号

**面试高频问题**：第三次握手可以携带数据吗？
- 可以。第三次握手时客户端已经处于ESTABLISHED状态，可以携带数据

#### [](#四次挥手)四次挥手

```
客户端                    服务器
  |--- FIN, seq=u -------->|      第一次挥手
  |<-- ACK, ack=u+1 -------|      第二次挥手
  |                        |      (服务器可能还有数据要发)
  |<-- FIN, seq=w ---------|      第三次挥手
  |--- ACK, ack=w+1 ------->|      第四次挥手
  |   [进入TIME_WAIT]      |
```

**为什么需要四次挥手？**

TCP是全双工通信，每个方向都需要单独关闭：
- 服务器收到FIN后，可能还有数据要发送，所以先回复ACK
- 服务器数据发送完毕后，再发送自己的FIN
- 因此FIN和ACK分两次发送

**TIME_WAIT状态**：

客户端发送最后一个ACK后进入TIME_WAIT状态，等待2MSL（最大报文段生存时间）后关闭：
- **原因1**：确保最后一个ACK能到达服务器（如果丢失，服务器会重发FIN）
- **原因2**：让本连接的所有报文在网络中消失，避免影响新连接

**为什么TIME_WAIT是2MSL？**
- 1个MSL：ACK到达服务器的最大时间
- 1个MSL：服务器重发FIN的最大时间
- 总共2MSL可确保连接可靠关闭

### [](#TCP可靠传输)TCP可靠传输

TCP通过以下机制实现可靠传输：

#### [](#序号与确认)序号与确认

- 每个字节都有序号
- 确认号为期望收到的下一个字节序号
- 累计确认：确认号表示该序号之前的所有数据都已正确收到

#### [](#超时重传)超时重传

- 发送方在发送数据后启动定时器
- 如果超时未收到确认，则重传数据
- **RTO（重传超时时间）**：动态计算，略大于RTT

#### [](#快速重传)快速重传

- 不等待超时，收到3个重复ACK后立即重传
- 比超时重传更快发现丢包

### [](#TCP流量控制)TCP流量控制

流量控制是**发送方根据接收方的处理能力控制发送速率**：

- **滑动窗口机制**：接收方通过窗口字段告知发送方自己的接收缓冲区大小
- **零窗口探测**：当接收方窗口为0时，发送方定期发送零窗口探测报文，防止死锁
- **糊涂窗口综合征**：接收方缓冲区快满时，每次只允许发送很少的数据，效率很低
  - 解决方案：Nagle算法（发送方）、Clark方案（接收方）

### [](#TCP拥塞控制)TCP拥塞控制

拥塞控制是**防止过多数据注入网络，导致网络过载**：

#### [](#慢开始)慢开始

- 连接建立后，拥塞窗口cwnd从1个MSS开始
- 每收到一个确认，cwnd翻倍（指数增长）
- 增长到慢开始门限ssthresh时改为拥塞避免

#### [](#拥塞避免)拥塞避免

- cwnd每经过一个RTT加1（线性增长）
- 更加平缓地接近网络容量

#### [](#快重传与快恢复)快重传与快恢复

- **快重传**：收到3个重复ACK后立即重传丢失的报文段
- **快恢复**：不执行慢开始，而是将ssthresh和cwnd设为当前cwnd的一半

**拥塞控制状态转换**：

```
                    超时
    ┌──────────────────────────────┐
    │                              ▼
    │     ┌─────────────────────────────┐
    │     │         慢开始              │
    │     │  cwnd从1开始指数增长        │
    │     └──────────┬──────────────────┘
    │                │ cwnd ≥ ssthresh
    │                ▼
    │     ┌─────────────────────────────┐
    │     │        拥塞避免             │
    │     │  cwnd每个RTT加1（线性增长） │
    │     └──────────┬──────────────────┘
    │                │ 收到3个重复ACK
    │                ▼
    │     ┌─────────────────────────────┐
    │     │        快恢复               │
    │     │  ssthresh = cwnd/2          │
    │     │  cwnd = ssthresh            │
    │     └─────────────────────────────┘
    │                │
    └────────────────┘
```

**超时 vs 3个重复ACK的区别**：
- 超时：说明网络严重拥塞 → 从慢开始重新来（cwnd=1）
- 3个重复ACK：说明个别报文丢失，网络还能正常工作 → 快恢复（cwnd减半）

### [](#TCP与UDP对比)TCP与UDP对比

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接方式 | 面向连接 | 无连接 |
| 可靠性 | 可靠 | 不可靠 |
| 传输方式 | 字节流 | 报文 |
| 通信模式 | 点对点 | 一对一/多对多 |
| 首部开销 | 20-60字节 | 8字节 |
| 流量控制 | 有（滑动窗口） | 无 |
| 拥塞控制 | 有 | 无 |
| 传输速度 | 较慢 | 较快 |
| 应用场景 | 文件传输、网页浏览 | 实时音视频、DNS |

### [](#TCP粘包问题)TCP粘包问题

**什么是粘包？**
- TCP面向字节流，不保留报文边界
- 多次发送的数据可能被合并为一次接收（粘包）
- 或者一次发送的数据被拆分为多次接收（拆包）

**产生原因**：
- 发送方：Nagle算法可能将多个小数据合并发送
- 接收方：应用层读取不及时，多个报文在缓冲区中合并

**解决方案**：
1. 固定长度消息
2. 使用特殊分隔符
3. 消息头中包含长度字段（最常用）
4. TLV格式（Type-Length-Value）

## [](#TCP拥塞控制算法进阶)TCP拥塞控制算法进阶

### [](#BBR算法)BBR算法

Google提出的BBR（Bottleneck Bandwidth and RTT）算法：
- 不依赖丢包判断拥塞
- 通过测量带宽和RTT来确定最优发送速率
- 在高延迟、有丢包的网络中表现更好

### [](#CUBIC算法)CUBIC算法

Linux默认的拥塞控制算法：
- 基于窗口的拥塞控制
- 使用三次函数调整窗口大小
- 比Reno在高带宽长时延网络中更高效

## [](#总结)总结

运输层是面试的重中之重，必须掌握：

1. **TCP与UDP的区别**：连接性、可靠性、传输方式
2. **三次握手**：流程、原因、为什么不能两次
3. **四次挥手**：流程、TIME_WAIT的作用和时长
4. **TCP可靠传输**：序号、确认、超时重传、快重传
5. **流量控制vs拥塞控制**：滑动窗口 vs 慢开始/拥塞避免
6. **TCP粘包**：原因和解决方案
