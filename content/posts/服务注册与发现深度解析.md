# 服务注册与发现深度解析

## 一、引言

服务注册与发现是微服务的基石，没有它，服务之间无法相互调用。理解 Eureka 和 Nacos 的原理与差异，是微服务开发和面试的核心考点。

**本文要点**：
1. 服务发现原理（客户端 vs 服务端）
2. Eureka 深度解析
3. Nacos 深度解析
4. Eureka 与 Nacos 对比
5. 服务发现实战
6. 面试高频问题

## 二、服务发现原理

### 2.1 没有服务发现的问题

```
传统静态配置：
  # 订单服务调用用户服务
  user.service.url=http://192.168.1.100:8080
  
问题：
  1. IP 写死，服务迁移要改配置
  2. 无法动态扩缩容
  3. 单点故障（一个服务实例挂了，调用方不知道）
```

### 2.2 服务发现架构

```
方案一：客户端发现（Client-Side Discovery）

  服务实例启动 → 注册到注册中心
  服务消费者 → 从注册中心拉取服务列表
               → 本地缓存 + 负载均衡
               → 直接调用服务提供者

  代表：Eureka + Ribbon、Nacos


方案二：服务端发现（Server-Side Discovery）

  服务实例启动 → 注册到注册中心
  服务消费者 → 请求发给负载均衡器/API 网关
               → 负载均衡器查询注册中心
               → 转发请求到服务提供者

  代表：Kubernetes Service、AWS ELB
```

### 2.3 核心流程

```
                注册中心
              （Eureka/Nacos）
                    │
    ① 注册 ─────────┤───────── ③ 拉取服务列表
    ② 心跳 ─────────┤───────── ④ 缓存 + 负载均衡
                    │
              ┌─────┴─────┐
              │   服务提供者 │
              └────────────┘
                    │
              ┌─────┴─────┐
              │   服务消费者 │
              └────────────┘
                    │
                ⑤ 远程调用
```

## 三、Eureka 深度解析

### 3.1 Eureka 架构

```
                    ┌─────────────────────┐
                    │   Eureka Server      │
                    │   （注册中心）        │
                    │                     │
                    │  ┌─────┐  ┌─────┐│
                    │  │Node1│  │Node2││
                    │  └─────┘  └─────┘│
                    │  Peer-to-Peer 复制  │
                    └─────────────────────┘
     注册/心跳 │            │ 拉取服务列表
       ┌──────┴────────────┴──────┐
       ↓                           ↓
┌─────────────┐            ┌─────────────┐
│  Service A  │            │  Service B  │
│ (Provider)  │            │ (Consumer)  │
└─────────────┘            └─────────────┘
```

### 3.2 Eureka Server（注册中心）

**搭建 Eureka Server**：

```java
@SpringBootApplication
@EnableEurekaServer  // ← 开启 Eureka Server
public class EurekaServerApplication {
    public static void main(String[] args) {
        SpringApplication.run(EurekaServerApplication.class, args);
    }
}
```

```yaml
# application.yml
server:
  port: 8761

eureka:
  client:
    register-with-eureka: false  # 不注册自己
    fetch-registry: false       # 不从其他 Eureka 拉取注册表
    service-url:
      defaultZone: http://localhost:8761/eureka/
```

### 3.3 Eureka Client（服务提供者/消费者）

**服务提供者注册**：

```java
@SpringBootApplication
@EnableEurekaClient  // ← 开启 Eureka Client（可省略，自动开启）
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

```yaml
# application.yml
spring:
  application:
    name: order-service  # ← 服务名，非常重要！

eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka/
  instance:
    prefer-ip-address: true  # 注册 IP 而非主机名
    instance-id: ${spring.cloud.client.ip-address}:${server.port}  # 实例 ID
```

### 3.4 Eureka 核心机制

**心跳与续约（Renew/Heartbeat）**：

```
提供者 → 每 30 秒发送心跳 → Eureka Server
Eureka Server → 超过 90 秒未收到心跳 → 剔除实例

配置：
  # 提供者配置
  eureka.instance.lease-renewal-interval-in-seconds: 30   # 心跳间隔
  eureka.instance.lease-expiration-duration-in-seconds: 90 # 过期时间

  # Server 配置
  eureka.server.eviction-interval-timer-in-ms: 60000  # 剔除任务间隔
```

**服务剔除（Eviction）**：

```
Eureka Server 后台任务：
  每隔 eureka.server.eviction-interval-timer-in-ms（默认 60 秒）
  → 扫描过期实例（超过 90 秒未心跳）
  → 从注册表剔除
```

**自我保护模式（Self-Preservation）**：

```
问题：网络分区时，大量服务心跳失败
  → Eureka 误剔除所有服务 → 系统崩溃

解决：自我保护模式
  → 若心跳失败比例 > 15%（15 分钟内）
  → 进入自我保护模式
  → 不再剔除任何实例

配置：
  eureka.server.enable-self-preservation: true  # 默认开启
  eureka.server.renewal-percent-threshold: 0.85  # 心跳阈值（85%）

生产建议：
  ✓ 生产环境：开启自我保护
  ✗ 测试环境：关闭自我保护（方便测试）
```

**服务拉取（Fetch Registry）**：

```
消费者 → 每 30 秒拉取服务列表 → 本地缓存

配置：
  eureka.client.registry-fetch-interval-seconds: 30  # 拉取间隔
  eureka.client.fetch-registry: true                   # 是否拉取注册表
```

### 3.5 Eureka 的 AP 特性

```
CAP 理论：
  C（Consistency）：一致性
  A（Availability）：可用性
  P（Partition Tolerance）：分区容错性

Eureka 选择 AP：
  ✓ 高可用（节点故障不影响服务调用）
  ✗ 一致性弱（可能获取到已下线的服务实例）

场景：
  服务 A 下线 → Eureka Server 标记剔除
  → 消费者 30 秒内仍可能调用到已下线服务 A
  → 需要 Ribbon 重试机制配合
```

## 四、Nacos 深度解析

### 4.1 Nacos 架构

```
                      ┌─────────────────────┐
                      │    Nacos Server      │
                      │   （注册中心 +      │
                      │    配置中心）        │
                      │                     │
                      │  ┌─────┐  ┌─────┐│
                      │  │Node1│  │Node2││
                      │  └─────┘  └─────┘│
                      │  Raft 协议（CP）   │
                      │  或 Distro（AP）   │
                      └─────────────────────┘
         注册/心跳 │            │ 拉取服务列表
           ┌──────┴────────────┴──────┐
           ↓                           ↓
    ┌─────────────┐            ┌─────────────┐
    │  Service A  │            │  Service B  │
    │ (Provider)  │            │ (Consumer)  │
    └─────────────┘            └─────────────┘
```

### 4.2 Nacos Server 搭建

**方式一：独立部署（推荐）**

```bash
# 下载 Nacos
wget https://github.com/alibaba/nacos/releases/download/2.2.3/nacos-server-2.2.3.tar.gz
tar -xzf nacos-server-2.2.3.tar.gz
cd nacos/bin

# 单机模式启动（测试）
sh startup.sh -m standalone

# 集群模式启动（生产）
sh startup.sh

# 访问控制台
# http://localhost:8848/nacos
# 默认账号/密码：nacos/nacos
```

**方式二：Docker 部署**

```bash
docker run -d \
  --name nacos \
  -p 8848:8848 \
  -p 9848:9848 \
  -e MODE=standalone \
  nacos/nacos-server:v2.2.3
```

### 4.3 服务注册（Spring Cloud Alibaba）

```java
@SpringBootApplication
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

```yaml
# application.yml
spring:
  application:
    name: order-service
  cloud:
    nacos:
      discovery:
        server-addr: localhost:8848
        namespace: dev  # 命名空间（环境隔离）
        group: DEFAULT_GROUP  # 分组
```

**依赖**：

```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

### 4.4 Nacos 核心机制

**心跳机制**：

```
提供者 → 每 5 秒发送心跳 → Nacos Server
Nacos Server → 超过 15 秒未收到心跳 → 标记为不健康
             → 超过 30 秒未收到心跳 → 剔除实例

配置（客户端）：
  spring.cloud.nacos.discovery.heart-beat-interval: 5   # 心跳间隔（秒）
  spring.cloud.nacos.discovery.heart-beat-timeout: 15  # 心跳超时（秒）
  spring.cloud.nacos.discovery.ip-delete-timeout: 30   # 剔除超时（秒）
```

**服务拉取**：

```
消费者 → 长轮询（Long Polling）← Nacos Server
         → 每 10 秒拉取一次（可配置）
         → 服务变更时，Server 立即推送

配置：
  spring.cloud.nacos.discovery.watch-delay: 10000  # 拉取间隔（毫秒）
```

**临时实例 vs 持久化实例**：

```
临时实例（Temporary Instance）：
  - 客户端上报心跳
  - 心跳超时 → 剔除
  - 对应 AP 模式
  
持久化实例（Persistent Instance）：
  - Server 主动健康检查（TCP/HTTP/MySQL）
  - 实例信息持久化到 Server
  - 对应 CP 模式

配置：
  spring.cloud.nacos.discovery.ephemeral: true  # true=临时，false=持久化
```

### 4.5 Nacos CP/AP 模式切换

```
Nacos 支持两种模式：

CP 模式（Raft 协议）：
  - 强一致性
  - 选举期间集群不可用
  - 适合：配置中心、核心服务注册

AP 模式（Distro 协议）：
  - 高可用
  - 最终一致性
  - 适合：服务发现

切换方式：
  # 启动时指定模式
  startup.sh -m cp   # CP 模式
  startup.sh -m ap   # AP 模式（默认）
```

## 五、Eureka 与 Nacos 对比

### 5.1 功能对比

| 对比项 | Eureka | Nacos |
|--------|---------|-------|
| **服务发现** | ✓ | ✓ |
| **配置管理** | ✗（需配合 Config） | ✓（内置） |
| **CAP** | AP | CP/AP 可切换 |
| **健康检查** | 心跳（客户端） | 心跳 + Server 主动检查 |
| **保护模式** | 自我保护（AP） | 保护阈值（可配置） |
| **控制台** | 无 | 有（功能强大） |
| **动态配置刷新** | 需手动调用 /actuator/refresh | 自动推送 |
| **命名空间** | 无 | 有（环境隔离） |
| **分组** | 无 | 有（服务分组） |
| **权重** | 无 | 有（负载均衡权重） |
| **元数据** | 有 | 有（更强） |
| **集群** | Peer-to-Peer 复制 | Raft/Distro |
| **维护性** | 维护模式（不推荐新项目） | 活跃（强烈推荐） |

### 5.2 性能对比

```
Eureka：
  - 服务实例上限：约 1 万（单 Server）
  - 心跳压力：每 30 秒一次
  - 适合小规模集群

Nacos：
  - 服务实例上限：约 10 万（单 Server）
  - 心跳压力：每 5 秒一次（可调整）
  - 适合大规模集群
```

### 5.3 选型建议

```
新项目：
  ✓ 推荐 Nacos（功能全、性能高、维护活跃）
  
老项目（已用 Eureka）：
  1. 继续使用 Eureka（迁移成本高）
  2. 或逐步迁移到 Nacos

配置管理：
  ✓ 推荐 Nacos（服务发现 + 配置中心 二合一）
  ✗ 不推荐 Eureka + Config（两个组件）
```

## 六、服务发现实战

### 6.1 高可用部署

**Eureka 集群**：

```yaml
# application-peer1.yml
eureka:
  client:
    service-url:
      defaultZone: http://peer2:8762/eureka/,http://peer3:8763/eureka/

# application-peer2.yml
eureka:
  client:
    service-url:
      defaultZone: http://peer1:8761/eureka/,http://peer3:8763/eureka/

# application-peer3.yml
eureka:
  client:
    service-url:
      defaultZone: http://peer1:8761/eureka/,http://peer2:8762/eureka/
```

**Nacos 集群**：

```bash
# 1. 修改 cluster.conf
#   192.168.1.101:8848
#   192.168.1.102:8848
#   192.168.1.103:8848

# 2. 使用外部数据库（MySQL）
#   修改 application.properties
#   spring.datasource.platform=mysql
#   db.url.0=jdbc:mysql://localhost:3306/nacos?...
#   db.user.0=nacos
#   db.password.0=nacos

# 3. 启动所有节点
sh startup.sh
```

### 6.2 服务发现最佳实践

```
1. 本地缓存
   → 消费者缓存服务列表（Eureka/Nacos Client 自动缓存）
   → 注册中心故障时，仍可从缓存调用

2. 重试机制
   → 调用失败 → 重试其他实例
   → Ribbon/LoadBalancer 自动重试

3. 熔断降级
   → 服务实例不健康 → 熔断
   → 降级返回默认值

4. 监控告警
   → 服务实例上下线告警
   → 心跳丢失告警

5. 灰度发布
   → Nacos 权重配置
   → 新版本权重调低，逐步放量
```

### 6.3 常见坑点

```
坑点 1：Eureka 自我保护导致问题
  现象：服务下线了，但消费者仍能拉取到
  解决：测试环境关闭自我保护
        eureka.server.enable-self-preservation=false

坑点 2：服务名写错
  现象：No instances available for xxx
  解决：检查 spring.application.name 是否一致

坑点 3：网络分区
  现象：服务心跳失败，被大量剔除
  解决：开启自我保护 + 调整心跳超时时间

坑点 4：Nacos 配置不生效
  现象：修改配置后，服务未刷新
  解决：检查是否引入了 @RefreshScope
       检查配置格式是否正确（YAML/Properties）

坑点 5：Eureka 和 Nacos 混用
  现象：服务无法发现
  解决：统一使用一种注册中心
```

## 七、面试高频问题

### Q1: Eureka 自我保护模式是什么？

**答案要点**：
- 网络分区时，防止误剔除健康实例
- 心跳失败比例 > 15%（15 分钟内），进入保护模式
- 生产环境建议开启，测试环境可关闭

### Q2: Eureka 和 Nacos 的区别？

**答案要点**：
- Eureka：AP 系统，功能简单，已进入维护模式
- Nacos：CP/AP 可切换，功能全面（服务发现 + 配置中心），维护活跃
- 新项目强烈推荐 Nacos

### Q3: 服务发现的原则？

**答案要点**：
- 客户端发现（Eureka/Nacos）：消费者拉取服务列表，自己负载均衡
- 服务端发现（K8s Service）：负载均衡器转发
- 高可用：本地缓存 + 重试 + 熔断

### Q4: Nacos 如何保证高可用？

**答案要点**：
- 集群部署（至少 3 节点）
- CP 模式使用 Raft 协议选举
- AP 模式使用 Distro 协议最终一致
- 持久化到 MySQL（防止数据丢失）

### Q5: 微服务中，服务注册与发现的核心流程？

**答案要点**：
1. 提供者启动 → 注册到注册中心
2. 提供者定时发送心跳
3. 消费者拉取服务列表并缓存
4. 消费者调用时，从缓存负载均衡
5. 注册中心检测心跳，剔除不健康实例

## 八、总结

本文深度解析了服务注册与发现的核心知识：

1. **服务发现原理**：客户端发现（Eureka/Nacos）vs 服务端发现（K8s）
2. **Eureka**：AP 系统，自我保护模式，心跳续约
3. **Nacos**：CP/AP 可切换，功能全面（服务发现 + 配置中心）
4. **Eureka vs Nacos**：Nacos 全面优于 Eureka，新项目推荐 Nacos

**核心记忆**：
- Eureka：AP、自我保护、心跳 30 秒、维护模式
- Nacos：CP/AP 可切换、内置配置中心、心跳 5 秒、维护活跃
- 新项目选 Nacos，老项目 Eureka 可继续用
- 服务发现三板斧：注册、心跳、拉取
