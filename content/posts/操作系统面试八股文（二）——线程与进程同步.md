# 操作系统面试八股文（二）——线程与进程同步

## 线程

### [](#线程的引入)线程的引入

为了减小进程切换的开销，提高并发性能，引入了**线程**。

- **进程**：资源分配的基本单位
- **线程**：CPU调度的基本单位，是进程中的一个执行流

引入线程后，进程只作为资源的拥有者，不再作为调度的基本单位。

### [](#线程与进程的对比)线程与进程的对比

| 维度 | 进程 | 线程 |
|------|------|------|
| 调度 | 传统OS中是调度的基本单位 | 调度的基本单位 |
| 资源 | 拥有独立的地址空间和系统资源 | 不拥有系统资源，共享所属进程的资源 |
| 并发性 | 进程间可并发 | 线程间也可并发，开销更小 |
| 系统开销 | 创建/切换开销大 | 创建/切换开销小 |
| 通信 | 需要IPC机制（管道、消息队列等） | 可直接读写进程数据段 |
| 地址空间 | 每个进程有独立地址空间 | 同一进程的线程共享地址空间 |

> **线程共享的资源**：代码段、数据段、打开的文件、信号处理等
>
> **线程私有的资源**：线程ID、寄存器、栈、程序计数器PC、errno变量

### [](#线程的实现方式)线程的实现方式

#### 用户级线程（User-Level Thread, ULT）

- 由**用户空间的线程库**管理，操作系统内核不知道线程的存在
- 优点：线程切换不需要内核参与，速度快；跨平台
- 缺点：一个线程阻塞导致整个进程阻塞；不能利用多核CPU（内核只看到进程）
- 示例：早期的Java绿色线程

#### 内核级线程（Kernel-Level Thread, KLT）

- 由**操作系统内核**直接管理
- 优点：一个线程阻塞不影响其他线程；可以利用多核CPU；内核可调度线程
- 缺点：线程切换需要从用户态转到内核态，开销较大
- 示例：Linux的pthread、Windows线程

#### 组合方式

- 多对一模型：多个用户线程映射到一个内核线程
- 一对一模型：每个用户线程对应一个内核线程
- 多对多模型：n个用户线程映射到m个内核线程（n ≥ m）

> **面试高频问**：Linux采用的是哪种线程模型？
> Linux采用的是**一对一模型**，每个用户线程对应一个内核线程，由内核统一调度。Linux中线程也称为**轻量级进程（LWP）**。

### [](#线程的状态)线程的状态

与进程类似，线程也有以下基本状态：
- **就绪态**：等待被调度执行
- **运行态**：正在CPU上执行
- **阻塞态**：等待某一事件

线程状态的转换与进程类似，区别在于线程的阻塞可能不会导致整个进程阻塞（取决于线程实现方式）。

## 进程同步

### [](#进程同步的基本概念)进程同步的基本概念

**进程同步**的主要任务是对多个相关进程在执行次序上进行协调，使并发执行的进程之间能按照一定的规则共享系统资源，并能很好地相互合作。

**两个基本关系**：

| 关系 | 说明 | 举例 |
|------|------|------|
| 互斥 | 间接制约关系，对临界资源的竞争 | 多个进程访问共享打印机 |
| 同步 | 直接制约关系，为完成某任务而协作 | 生产者生产后消费者才能消费 |

### [](#临界资源与临界区)临界资源与临界区

- **临界资源**：一次仅允许一个进程使用的资源（如打印机、共享变量）
- **临界区**：访问临界资源的那段代码

```c
// 临界区访问的通用模式
enterCriticalSection();   // 进入区
// 临界区 —— 访问临界资源的代码
exitCriticalSection();    // 退出区
// 剩余区 —— 其余代码
```

**临界区访问原则**：
1. **空闲让进**：临界区空闲时，允许进程进入
2. **忙则等待**：已有进程进入临界区时，其他进程必须等待
3. **有限等待**：等待进入临界区的进程不能无限等待（避免饥饿）
4. **让权等待**：不能进入临界区的进程应释放CPU

### [](#进程互斥的软件实现)进程互斥的软件实现

#### 单标志法

```c
int turn = 0;  // 表示哪个进程可以进入临界区

// 进程P0
while(turn != 0);  // 进入区
// 临界区
turn = 1;          // 退出区

// 进程P1
while(turn != 1);  // 进入区
// 临界区
turn = 0;          // 退出区
```

**缺点**：必须交替进入，违反"空闲让进"原则。

#### 双标志先检查法

```c
bool flag[2] = {false, false};

// 进程Pi
while(flag[j]);    // 先检查对方是否在临界区
flag[i] = true;    // 后标记自己想进入
// 临界区
flag[i] = false;   // 退出区
```

**缺点**：可能同时进入临界区（检查和标记不是原子操作）。

#### 双标志后检查法

```c
bool flag[2] = {false, false};

// 进程Pi
flag[i] = true;    // 先标记自己想进入
while(flag[j]);    // 后检查对方是否在临界区
// 临界区
flag[i] = false;   // 退出区
```

**缺点**：可能产生死锁（双方同时标记后互相等待）。

#### Peterson算法

```c
bool flag[2] = {false, false};
int turn;

// 进程Pi
flag[i] = true;
turn = j;
while(flag[j] && turn == j);  // 对方想进且轮到对方→等待
// 临界区
flag[i] = false;
```

**优点**：满足临界区访问的四条原则，是软件方法中的最优解。
**缺点**：仍然可能因为编译器优化导致问题（需要内存屏障）。

### [](#进程互斥的硬件实现)进程互斥的硬件实现

#### 中断屏蔽

```c
// 进入临界区前关中断
disableInterrupt();
// 临界区
// 退出临界区后开中断
enableInterrupt();
```

**缺点**：只适用于单核CPU；关中断时间过长影响系统响应；不适合用户进程使用。

#### TestAndSet（TSL指令）

```c
// 原子操作
bool TestAndSet(bool *lock) {
    bool old = *lock;
    *lock = true;
    return old;
}

// 使用
while(TestAndSet(&lock));  // 进入区
// 临界区
lock = false;              // 退出区
```

#### Swap指令

```c
// 原子操作
void Swap(bool *a, bool *b) {
    bool temp = *a;
    *a = *b;
    *b = temp;
}

// 使用
bool key = true;
while(key == true) Swap(&lock, &key);
// 临界区
lock = false;
```

> **硬件方法的共同缺点**：不满足"让权等待"原则，进入临界区失败的进程会忙等（Busy Waiting），浪费CPU时间。

### [](#信号量机制)信号量机制（⭐⭐⭐⭐⭐）

信号量是Dijkstra提出的进程同步工具，是**面试最重要的考点之一**。

#### 整型信号量

```c
int S = 1;  // 信号量

void wait(int S) {      // P操作
    while(S <= 0);      // 忙等
    S--;
}

void signal(int S) {    // V操作
    S++;
}
```

**缺点**：不满足"让权等待"（忙等）。

#### 记录型信号量

```c
typedef struct {
    int value;              // 资源数量
    struct process *L;      // 等待队列
} semaphore;

void wait(semaphore S) {       // P操作
    S.value--;
    if (S.value < 0) {
        // |S.value| = 等待队列中的进程数
        block(S.L);           // 自我阻塞，挂到等待队列
    }
}

void signal(semaphore S) {     // V操作
    S.value++;
    if (S.value <= 0) {
        wakeup(S.L);          // 唤醒等待队列中的第一个进程
    }
}
```

**优点**：满足"让权等待"原则，不会忙等。

> **P操作的含义**：申请一个资源，如果S.value < 0说明没有资源了，进程自我阻塞
>
> **V操作的含义**：释放一个资源，如果S.value ≤ 0说明还有进程在等待，唤醒一个

#### 信号量的应用

**1. 实现进程互斥**

```c
semaphore mutex = 1;  // 互斥信号量，初值为1

// 进程Pi
P(mutex);    // 进入区
// 临界区
V(mutex);    // 退出区
```

> 注意：P、V操作必须**成对出现**，且在同一进程中。

**2. 实现进程同步**

确保进程的执行顺序：

```c
semaphore sync = 0;  // 同步信号量，初值为0

// 进程P1（先执行）
// 代码1
V(sync);     // 通知P2可以执行了

// 进程P2（后执行）
P(sync);     // 等待P1的信号
// 代码2
```

> 同步信号量初值为0！先V后P，保证P1先执行代码1，P2再执行代码2。

**3. 实现前驱关系**

对于前驱图中的每一对前驱关系，设置一个初值为0的同步信号量。

### [](#管程)管程

管程（Monitor）是一种高级同步机制，将共享变量及对共享变量的操作封装起来。

**管程的特点**：
1. 管程中的共享变量只能被管程内的过程访问
2. 每次仅允许一个进程进入管程（由编译器保证互斥）
3. 条件变量用于解决阻塞/唤醒问题

**条件变量**：

```c
condition x, y;

// x.wait()：将自己阻塞在x条件队列上，释放管程互斥权
// x.signal()：唤醒x条件队列上的一个进程
```

> **条件变量 vs 信号量**：条件变量没有值，仅实现阻塞/唤醒功能；信号量有值，反映资源数量。

### [](#经典同步问题)经典同步问题（⭐⭐⭐⭐⭐）

#### 生产者-消费者问题

**问题描述**：生产者向缓冲区放产品，消费者从缓冲区取产品。缓冲区大小为n。

```c
semaphore mutex = 1;     // 互斥访问缓冲区
semaphore empty = n;     // 空缓冲区数量
semaphore full = 0;      // 满缓冲区数量

// 生产者
Producer() {
    while(true) {
        生产一个产品;
        P(empty);        // 申请空位（同步）
        P(mutex);        // 互斥访问
        将产品放入缓冲区;
        V(mutex);        // 释放互斥
        V(full);         // 增加产品数量（同步）
    }
}

// 消费者
Consumer() {
    while(true) {
        P(full);         // 申请产品（同步）
        P(mutex);        // 互斥访问
        从缓冲区取出产品;
        V(mutex);        // 释放互斥
        V(empty);        // 增加空位（同步）
        消费产品;
    }
}
```

> ⚠️ **关键**：P操作的顺序不能颠倒！必须先P同步信号量，再P互斥信号量。否则可能死锁。
>
> 例如：如果消费者先P(mutex)再P(full)，当缓冲区为空时，消费者拿到mutex后因full=0被阻塞，而生产者因mutex=0也无法进入，导致**死锁**。
>
> V操作的顺序不影响正确性，但建议先V(mutex)。

#### 读者-写者问题

**问题描述**：多个读者可以同时读，但写者必须独占。

```c
semaphore rw = 1;        // 读写/写写互斥
semaphore mutex = 1;     // 保护readcount
int readcount = 0;       // 当前读者数量

// 读者
Reader() {
    P(mutex);
    readcount++;
    if (readcount == 1) P(rw);  // 第一个读者负责加锁
    V(mutex);
    // 读操作
    P(mutex);
    readcount--;
    if (readcount == 0) V(rw);  // 最后一个读者负责解锁
    V(mutex);
}

// 写者
Writer() {
    P(rw);
    // 写操作
    V(rw);
}
```

> **问题**：读者优先可能导致写者饥饿（只要有读者在读，写者就一直等待）
>
> **写者优先方案**：增加一个写优先信号量 `w = 1`，写者在P(rw)之前先P(w)，读者在P(mutex)之前先P(w)。

#### 哲学家进餐问题

**问题描述**：5个哲学家围坐，每两人之间一支筷子，需要同时拿两支才能进餐。

**可能导致死锁的解法**：每个哲学家先拿左边筷子，再拿右边。

**避免死锁的方案**：

1. **最多允许4个哲学家同时进餐**（限制并发数）
2. **奇数号先拿左后拿右，偶数号先拿右后拿左**（破坏循环等待）
3. **仅当左右筷子都可用时才拿起**（AND信号量 / 一次性申请所有资源）

```c
// 方案3：使用AND信号量
semaphore chopstick[5] = {1, 1, 1, 1, 1};

Philosopher(i) {
    while(true) {
        think;
        Swait(chopstick[i], chopstick[(i+1)%5]);  // AND型P操作
        eat;
        Ssignal(chopstick[i], chopstick[(i+1)%5]); // AND型V操作
    }
}
```

## [](#经典面试题)经典面试题

### ⭐⭐⭐⭐⭐ 用信号量实现两个进程的交替打印

要求：进程P1打印A，进程P2打印B，交替输出ABABAB...

```c
semaphore s1 = 1;   // P1先执行
semaphore s2 = 0;   // P2后执行

P1() {
    while(true) {
        P(s1);
        print("A");
        V(s2);
    }
}

P2() {
    while(true) {
        P(s2);
        print("B");
        V(s1);
    }
}
```

### ⭐⭐⭐⭐⭐ 生产者-消费者中P操作的顺序能颠倒吗？

**不能！** 必须先P同步信号量（empty/full），再P互斥信号量（mutex）。

如果先P(mutex)：
- 生产者：P(mutex) → P(empty)，如果缓冲区满了，生产者持有mutex等empty，消费者需要mutex才能V(full)，**死锁**
- 消费者：P(mutex) → P(full)，如果缓冲区空了，消费者持有mutex等full，生产者需要mutex才能V(empty)，**死锁**

### ⭐⭐⭐⭐ 用户级线程和内核级线程的区别

| 维度 | 用户级线程 | 内核级线程 |
|------|-----------|-----------|
| 管理者 | 线程库 | 操作系统内核 |
| 切换速度 | 快（不需要陷入内核） | 慢（需要内核参与） |
| 阻塞影响 | 一个线程阻塞→整个进程阻塞 | 一个线程阻塞→其他线程不受影响 |
| 多核利用 | 不能（内核只看到进程） | 能（内核调度不同线程到不同核心） |

### ⭐⭐⭐⭐ 什么是管程？和信号量有什么区别？

- **管程**：高级同步机制，封装了共享变量和操作，由编译器保证互斥
- **区别**：信号量需要程序员自己保证P/V操作的正确性（容易出错），管程由语言/编译器保证互斥（更安全）
