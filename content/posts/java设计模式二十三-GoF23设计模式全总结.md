## 一、设计模式全景图

### 1.1 三大分类

GoF 23 种设计模式按目的分为三大类：

| 分类 | 关注点 | 模式 |
|------|--------|------|
| **创建型** | 对象的创建 | 单例、工厂、建造者、原型、抽象工厂 |
| **结构型** | 对象的组合 | 适配器、装饰器、代理、外观、桥接、组合、享元 |
| **行为型** | 对象的交互 | 策略、观察者、命令、模板方法、迭代器、中介者、备忘录、访问者、状态、责任链、解释器 |

### 1.2 创建型模式（5种）

| 模式 | 核心思想 | 一句话总结 |
|------|---------|-----------|
| **单例模式** | 全局唯一实例 | 一个类只有一个对象 |
| **工厂模式** | 将创建逻辑封装 | 用工厂方法代替 new |
| **抽象工厂** | 创建产品族 | 一组工厂方法的集合 |
| **建造者模式** | 分步构建复杂对象 | 链式调用构建对象 |
| **原型模式** | 克隆创建对象 | 用 clone 代替 new |

### 1.3 结构型模式（7种）

| 模式 | 核心思想 | 一句话总结 |
|------|---------|-----------|
| **适配器模式** | 转换接口 | 让不兼容的接口协同工作 |
| **装饰器模式** | 动态添加职责 | 套娃式增强功能 |
| **代理模式** | 控制访问 | 替身控制真实对象 |
| **外观模式** | 简化接口 | 封装复杂性 |
| **桥接模式** | 分离抽象与实现 | 组合替代继承 |
| **组合模式** | 树形结构统一接口 | 叶子和容器一视同仁 |
| **享元模式** | 共享内部状态 | 用时间换空间 |

### 1.4 行为型模式（11种）

| 模式 | 核心思想 | 一句话总结 |
|------|---------|-----------|
| **策略模式** | 封装算法族 | 运行时选择算法 |
| **观察者模式** | 一对多依赖 | 状态变化自动通知 |
| **命令模式** | 封装请求 | 把请求变成对象 |
| **模板方法** | 定义算法骨架 | 父类搭框架，子类填细节 |
| **迭代器模式** | 顺序访问集合 | 不暴露内部结构遍历 |
| **中介者模式** | 封装对象交互 | 网状变星状 |
| **备忘录模式** | 保存对象状态 | 撤销/恢复的利器 |
| **访问者模式** | 分离操作与数据结构 | 双重分发的艺术 |
| **状态模式** | 状态即类 | 消灭状态 if-else |
| **责任链模式** | 链式处理请求 | 沿着链条找处理者 |
| **解释器模式** | 文法规则映射为类 | 用类实现 DSL |

## 二、设计原则回顾

设计模式背后的灵魂是六大设计原则：

### 2.1 SOLID 原则

| 原则 | 含义 | 体现的模式 |
|------|------|-----------|
| **单一职责 (SRP)** | 一个类只有一个变化的理由 | 策略、观察者、访问者 |
| **开闭原则 (OCP)** | 对扩展开放，对修改关闭 | 工厂、策略、观察者、访问者 |
| **里氏替换 (LSP)** | 子类可以替换父类 | 模板方法、状态模式 |
| **接口隔离 (ISP)** | 接口应该小而专 | 适配器、桥接 |
| **依赖倒置 (DIP)** | 依赖抽象而非具体 | 工厂、策略、桥接 |

### 2.2 其他原则

| 原则 | 含义 | 体现的模式 |
|------|------|-----------|
| **迪米特法则 (LoD)** | 最少知道原则 | 外观、中介者 |
| **组合优于继承** | 优先使用组合 | 桥接、装饰器、策略 |

## 三、模式之间的关系

### 3.1 经常联用的模式

| 组合 | 场景 |
|------|------|
| 组合 + 访问者 | 对树形结构执行操作 |
| 策略 + 工厂 | 运行时选择策略 |
| 命令 + 备忘录 | 实现可撤销的命令 |
| 装饰器 + 代理 | 增强功能 + 控制访问 |
| 状态 + 策略 | 状态驱动策略选择 |
| 中介者 + 观察者 | 中介者使用观察者通信 |
| 迭代器 + 组合 | 遍历树形结构 |

### 3.2 容易混淆的模式

| 对比 | 关键区别 |
|------|---------|
| 代理 vs 装饰器 | 代理控制访问，装饰器增强功能 |
| 代理 vs 适配器 | 代理不改变接口，适配器转换接口 |
| 外观 vs 中介者 | 外观单向简化，中介者双向协调 |
| 策略 vs 状态 | 策略由客户端选，状态自动切换 |
| 桥接 vs 策略 | 桥接分离维度，策略选择算法 |
| 建造者 vs 工厂 | 建造者分步构建，工厂一步创建 |
| 命令 vs 策略 | 命令封装请求，策略封装算法 |
| 备忘录 vs 命令 | 备忘录保存状态，命令保存操作 |

## 四、模式选择指南

### 4.1 按问题选择模式

| 你想解决什么问题 | 推荐模式 |
|----------------|---------|
| 对象创建太复杂 | 工厂 / 建造者 |
| 全局只需要一个实例 | 单例 |
| 需要克隆大量相似对象 | 原型 |
| 接口不兼容 | 适配器 |
| 需要动态添加功能 | 装饰器 |
| 需要控制对对象的访问 | 代理 |
| 子系统太复杂 | 外观 |
| 两个维度独立变化 | 桥接 |
| 树形结构统一处理 | 组合 |
| 大量重复对象占内存 | 享元 |
| 需要运行时切换算法 | 策略 |
| 状态变化需要通知 | 观察者 |
| 需要把请求变成对象 | 命令 |
| 算法骨架固定，细节可变 | 模板方法 |
| 需要遍历集合 | 迭代器 |
| 对象间网状交互 | 中介者 |
| 需要撤销/恢复 | 备忘录 |
| 对不同元素执行不同操作 | 访问者 |
| 行为随状态改变 | 状态 |
| 请求需要链式处理 | 责任链 |
| 需要解释特定语言 | 解释器 |

### 4.2 按场景选择模式

**Web 应用开发**：
- 控制器层：策略模式（不同请求处理策略）
- 服务层：代理模式（AOP/事务）、装饰器模式
- 数据层：工厂模式（DAO 创建）、迭代器模式（结果遍历）

**框架/工具开发**：
- 插件体系：策略 + 工厂
- 事件系统：观察者 + 命令
- 模板引擎：解释器 + 组合
- 依赖注入：代理 + 工厂

**游戏开发**：
- 角色状态：状态模式
- 技能系统：策略 + 命令
- 场景管理：组合模式
- 粒子系统：享元模式
- 存档系统：备忘录模式

## 五、Spring 中的设计模式

Spring 框架是设计模式的最佳实践库：

| 模式 | Spring 应用 |
|------|------------|
| **单例** | Bean 默认 scope=singleton |
| **工厂** | BeanFactory / FactoryBean |
| **代理** | AOP 动态代理 |
| **模板方法** | JdbcTemplate / RestTemplate |
| **观察者** | ApplicationEvent / @EventListener |
| **适配器** | HandlerAdapter / MessageConverter |
| **策略** | Resource 接口的不同实现 |
| **装饰器** | BeanWrapper / 各种 Wrapper |
| **责任链** | Filter Chain / Interceptor Chain |
| **外观** | JdbcTemplate 封装 JDBC |
| **建造者** | ResponseEntity.Builder |
| **迭代器** | CompositeIterator |

## 六、设计模式的误区

### 6.1 过度设计

**误区**：处处用设计模式，简单问题复杂化。

**正确做法**：遵循"三法则"——当一个设计问题第三次出现时，再考虑使用模式。

### 6.2 生搬硬套

**误区**：看到某个场景就套用某个模式，不考虑是否真正适合。

**正确做法**：先理解问题本质，再选择合适的模式。模式是解决问题的工具，不是目标。

### 6.3 忽视简单性

**误区**：认为用了设计模式的代码一定比不用的好。

**正确做法**：简单性永远是第一原则。如果一个简单的 if-else 就能解决问题，就不要引入策略模式。

### 6.4 不考虑场景

**误区**：照书抄代码，不考虑具体业务场景。

**正确做法**：根据实际需求变通。模式的实现不是一成不变的，理解思想比记住代码更重要。

## 七、学习路线

### 7.1 初级（必须掌握）

1. **单例模式** — 最简单，最常用
2. **工厂模式** — 创建对象的基础
3. **策略模式** — 消灭 if-else 的利器
4. **观察者模式** — 事件驱动的基础
5. **模板方法** — 复用代码骨架
6. **代理模式** — 理解 Spring AOP 的关键

### 7.2 中级（重要掌握）

7. **建造者模式** — 复杂对象的优雅创建
8. **装饰器模式** — 动态增强功能
9. **适配器模式** — 接口兼容
10. **外观模式** — 封装复杂性
11. **命令模式** — 请求的封装
12. **责任链模式** — 链式处理

### 7.3 高级（理解思想）

13. **桥接模式** — 多维度解耦
14. **组合模式** — 树形结构
15. **享元模式** — 内存优化
16. **迭代器模式** — 集合遍历
17. **中介者模式** — 对象间解耦
18. **备忘录模式** — 状态恢复
19. **状态模式** — 状态机
20. **访问者模式** — 双重分发
21. **解释器模式** — DSL 实现
22. **原型模式** — 克隆创建
23. **抽象工厂** — 产品族创建

## 九、总结

设计模式不是银弹，而是解决特定问题的经验总结。掌握设计模式的关键不在于记住每种模式的代码实现，而在于：

1. **理解思想**：每个模式背后的设计原则和动机
2. **识别场景**：在实际开发中识别可以用模式解决的问题
3. **灵活变通**：根据具体需求调整模式的实现
4. **适度使用**：避免过度设计，简单性优先

> "设计模式是对特定环境下重复出现的问题的解决方案。" — GoF

希望这个系列能帮助你建立完整的设计模式知识体系，在日常开发中写出更优雅、更灵活、更可维护的代码。
