## 一、SQL注入攻防

### 1.1 SQL注入的本质

```
SQL注入的本质：用户输入被当作SQL代码执行

攻击原理:
  输入: ' OR '1'='1' --
  SELECT * FROM users WHERE username = '' OR '1'='1' --' AND password = 'xxx'
                                    ↑
                          WHERE条件恒为真，绕过认证
```

**常见注入类型：**

| 注入类型 | 攻击方式 | 示例 |
|---------|---------|------|
| 联合查询注入 | 利用UNION窃取数据 | `' UNION SELECT user,password FROM mysql.user --` |
| 布尔盲注 | 根据返回结果真假推断 | `' AND SUBSTRING(password,1,1)='a' --` |
| 时间盲注 | 根据响应时间推断 | `' AND IF(SUBSTRING(password,1,1)='a', SLEEP(5), 0) --` |
| 报错注入 | 利用数据库报错信息 | `' AND extractvalue(1,concat(0x7e,database())) --` |
| 堆叠注入 | 执行多条SQL语句 | `'; DROP TABLE users; --` |
| 二次注入 | 存储恶意数据后被二次执行 | 注册时用户名含注入代码，后续查询触发 |

### 1.2 防御手段全解析

**Q1: 如何彻底防御SQL注入？**

**（1）预编译语句（Prepared Statement）—— 最核心防线**

```sql
-- ❌ 错误：拼接SQL
String sql = "SELECT * FROM users WHERE id = " + userId;

-- ✅ 正确：预编译
String sql = "SELECT * FROM users WHERE id = ?";
PreparedStatement pstmt = connection.prepareStatement(sql);
pstmt.setInt(1, userId);
```

**原理：** 预编译将SQL结构与数据分离，参数永远不会被当作SQL代码执行。

```
执行流程对比:

拼接SQL:
  "SELECT * FROM users WHERE id = 1 OR 1=1"
  解析器看到一个WHERE条件，正常执行 → 注入成功

预编译:
  SQL模板: "SELECT * FROM users WHERE id = ?"
  参数: "1 OR 1=1"
  解析器把参数当普通字符串 → id = "1 OR 1=1" → 查不到记录 → 注入失败
```

**（2）ORM框架的防护边界**

```java
// ✅ JPA安全: 使用参数绑定
@Query("SELECT u FROM User u WHERE u.username = :username")
User findByUsername(@Param("username") String username);

// ❌ JPA不安全: 原生查询拼接
@Query(value = "SELECT * FROM users WHERE username = '" + username + "'", nativeQuery = true)

// ⚠️ MyBatis ${} vs #{}:
// #{} 使用预编译 → 安全
// ${} 直接拼接   → 危险（除非确定输入可信）
SELECT * FROM users WHERE username = #{username}  -- 安全
SELECT * FROM users ORDER BY ${column}  -- 需要白名单校验column
```

**（3）输入验证与过滤**

```java
// 白名单验证
public boolean isValidUsername(String username) {
    return username != null && username.matches("^[a-zA-Z0-9_]{3,20}$");
}

// 敏感字符转义（第二道防线，不能替代预编译）
public String escape(String input) {
    return input.replace("\\", "\\\\")
                .replace("'", "\\'")
                .replace("\"", "\\\"");
}
```

**（4）最小权限原则**

```sql
-- 应用账号只给必要权限
CREATE USER 'app_user'@'%' IDENTIFIED BY 'xxx';
GRANT SELECT, INSERT, UPDATE, DELETE ON mydb.* TO 'app_user'@'%';
-- ❌ 不给 DROP、ALTER、CREATE、FILE 等危险权限

-- 不同场景不同账号
-- 只读报表 → 只有SELECT
-- 数据写入 → INSERT + UPDATE
-- 管理后台 → 全面但限制来源IP
GRANT ALL ON mydb.* TO 'admin'@'192.168.1.%';
```

**防御层次总结：**

```
第一道防线: 预编译 (根本解决方案)
第二道防线: 输入验证与白名单
第三道防线: 最小权限原则
第四道防线: WAF/数据库防火墙
第五道防线: 审计告警
```

### 1.3 面试高频题

**Q2: MyBatis中 `#{}` 和 `${}` 的区别？**

| 对比维度 | `#{}` | `${}` |
|---------|-------|-------|
| 处理方式 | 预编译占位符 | 字符串替换 |
| SQL注入 | 安全 | 存在风险 |
| 类型转换 | 自动 | 无 |
| 适用场景 | WHERE值、参数 | 表名、列名、ORDER BY |
| 底层实现 | `PreparedStatement.setXxx()` | 字符串拼接 |

**$安全使用**：需配合白名单校验。

```java
private static final Set<String> ALLOWED_COLUMNS = Set.of("id", "name", "age", "create_time");
private static final Set<String> ALLOWED_DIRECTIONS = Set.of("ASC", "DESC");

public List<User> getUsersByOrder(String column, String direction) {
    if (column == null || !ALLOWED_COLUMNS.contains(column)) {
        throw new IllegalArgumentException("Invalid column: " + column);
    }
    if (direction == null || !ALLOWED_DIRECTIONS.contains(direction.toUpperCase())) {
        throw new IllegalArgumentException("Invalid direction: " + direction);
    }
    return mapper.getUsersByOrder(column, direction.toUpperCase());
}
```

## 二、数据加密

### 2.1 传输加密（TLS/SSL）

**Q3: 数据库TLS加密的握手过程？**

```
MySQL TLS握手:
  客户端                    服务端
    |                         |
    |--- ClientHello -------->|  (支持的TLS版本、密码套件)
    |<-- ServerHello ---------|  (选定TLS版本、密码套件)
    |<-- Certificate ---------|  (服务端证书)
    |<-- ServerHelloDone -----|
    |--- ClientKeyExchange -->|  (用服务端公钥加密的预主密钥)
    |--- ChangeCipherSpec --->|
    |--- Finished ----------->|  (加密握手完成消息)
    |<-- ChangeCipherSpec ----|
    |<-- Finished ------------|  (加密握手完成消息)
    |                         |
    |<== 对称加密通信 ========>|
```

### 2.2 存储加密（TDE与列级加密）

**Q4: 透明数据加密（TDE）的原理？**

```
TDE (Transparent Data Encryption) 架构:

  +---------------------------+
  |        应用层              |  ← 应用无感知
  +---------------------------+
  |    SQL解析 → 执行引擎       |
  +---------------------------+
  |    存储引擎 (InnoDB)        |
  |  ┌─────────────────────┐  |
  |  │  加密/解密层 (TDE)    │  |  ← 在此层透明加解密
  |  │  主密钥 → 表空间密钥  │  |
  |  └─────────────────────┘  |
  +---------------------------+
  |    文件系统 (加密.ibd)      |  ← 落盘已是密文
  +---------------------------+

TDE加密粒度:
  表空间级别: MySQL 8.0+支持，每个表一个独立密钥
  日志文件:   Redo Log / Undo Log也可加密
  备份文件:   MySQL Enterprise Backup支持加密备份
```

**MySQL 8.0 TDE配置：**

```sql
-- 1. 安装密钥环插件
INSTALL PLUGIN keyring_file SONAME 'keyring_file.so';
SET GLOBAL keyring_file_data = '/var/lib/mysql-keyring/keyring';

-- 2. 创建加密表
CREATE TABLE sensitive_data (
    id INT PRIMARY KEY,
    secret_info VARCHAR(255)
) ENCRYPTION='Y';

-- 3. 加密已有的表
ALTER TABLE existing_table ENCRYPTION='Y';

-- 4. 查看加密状态
SELECT TABLE_SCHEMA, TABLE_NAME, CREATE_OPTIONS 
FROM information_schema.TABLES 
WHERE CREATE_OPTIONS LIKE '%ENCRYPTION%';
```

### 2.3 应用层列级加密

```java
// 不适合全表加密的场景：只加密敏感字段
public class UserService {
    
    private static final String AES_KEY = System.getenv("DB_COLUMN_ENCRYPTION_KEY");
    
    public void saveUser(User user) {
        user.setPhone(encrypt(user.getPhone()));
        user.setIdCard(encrypt(user.getIdCard()));
        // name字段不加密，需要索引和搜索
        mapper.insert(user);
    }
    
    private String encrypt(String plaintext) {
        // AES-256-GCM 带认证加密
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        SecretKeySpec keySpec = new SecretKeySpec(AES_KEY.getBytes(), "AES");
        GCMParameterSpec gcmSpec = new GCMParameterSpec(128, generateIV());
        cipher.init(Cipher.ENCRYPT_MODE, keySpec, gcmSpec);
        byte[] encrypted = cipher.doFinal(plaintext.getBytes());
        return Base64.getEncoder().encodeToString(encrypted);
    }
}
```

**列级加密vs TDE对比：**

| 维度 | 列级加密 | TDE |
|------|---------|-----|
| 加密范围 | 指定列 | 整个表空间/数据库 |
| 应用感知 | 需要改代码 | 完全透明 |
| 性能影响 | 精确控制 | 整体开销 |
| 索引支持 | 加密列难以索引 | 索引正常使用 |
| 密钥管理 | 应用自行管理 | 数据库统一管理 |
| 防DBA | 能防御（密钥不在数据库） | 不能（DBA有密钥权限） |

### 2.4 密钥管理

```
密钥层次结构:

  根密钥 (Root Key / KMS)
    ↓ 加密
  主密钥 (Master Encryption Key)
    ↓ 加密
  数据密钥 (Data Encryption Key) — 实际加密数据的密钥

密钥轮换:
  1. 生成新数据密钥
  2. 用新密钥写新数据
  3. 旧数据用旧密钥读，逐步重新加密（rekey）
  4. 旧密钥保留到所有数据重新加密完毕

最佳实践:
  - 密钥与数据物理分离（密钥管理服务KMS独立部署）
  - 定期密钥轮换（如90天）
  - 密钥访问审计
  - 绝不硬编码密钥
```

## 三、审计日志与合规

### 3.1 数据库审计体系

**Q5: 数据库审计需要审计什么？**

```
数据库审计四要素:
  WHO   — 谁操作的（用户名、来源IP）
  WHEN  — 什么时候（精确时间戳）
  WHERE — 在哪里（数据库实例、表名）
  WHAT  — 做了什么（具体SQL、影响行数、执行时长）

审计矩阵:
  ┌──────────┬──────┬──────┬──────┬──────┐
  │  操作     │ SELECT │ INSERT │ UPDATE │ DELETE │ DDL  │
  ├──────────┼──────┼──────┼──────┼──────┤
  │ 普通用户  │  ✓   │  ✓   │  ✓   │  ✓   │  ✗   │
  │ 管理员    │  ✗   │  ✗   │  ✓   │  ✓   │  ✓   │
  │ 敏感表    │  ✓   │  ✓   │  ✓   │  ✓   │  ✓   │
  │ 非工作时间│  ✓   │  ✓   │  ✓   │  ✓   │  ✓   │
  └──────────┴──────┴──────┴──────┴──────┴──────┘
```

**MySQL审计实现：**

```sql
-- MySQL Enterprise Audit (企业版)
INSTALL PLUGIN audit_log SONAME 'audit_log.so';

SET GLOBAL audit_log_policy = ALL;  -- 审计所有操作
SET GLOBAL audit_log_format = JSON;  -- JSON格式便于分析

-- 查看审计日志
SELECT * FROM mysql.audit_log WHERE command_class = 'delete';

-- 第三方方案: MariaDB Audit Plugin (开源)
INSTALL SONAME 'server_audit';
SET GLOBAL server_audit_events = 'CONNECT,QUERY,TABLE';
SET GLOBAL server_audit_logging = ON;
```

**应用层审计：**

```java
@Aspect
@Component
public class DatabaseAuditAspect {
    
    @Around("@annotation(AuditLog)")
    public Object audit(ProceedingJoinPoint joinPoint) throws Throwable {
        long start = System.currentTimeMillis();
        String user = SecurityContextHolder.getContext().getAuthentication().getName();
        String method = joinPoint.getSignature().getName();
        Object[] args = joinPoint.getArgs();
        
        try {
            Object result = joinPoint.proceed();
            long duration = System.currentTimeMillis() - start;
            
            // 记录审计日志
            auditLogService.log(AuditEntry.builder()
                .user(user)
                .action(method)
                .parameters(sanitize(args))  // 脱敏后记录
                .result("SUCCESS")
                .duration(duration)
                .timestamp(Instant.now())
                .build());
            
            return result;
        } catch (Exception e) {
            auditLogService.log(AuditEntry.builder()
                .user(user)
                .action(method)
                .result("FAILURE: " + e.getMessage())
                .build());
            throw e;
        }
    }
}
```

### 3.2 GDPR与数据合规

**Q6: GDPR对数据库设计有哪些影响？**

```
GDPR核心原则在数据库层面的体现:

1. 数据最小化（Data Minimization）
   设计原则: 只存储必要的数据
   实践: 定期清理过期数据、匿名化历史数据

2. 存储限制（Storage Limitation）
   设计原则: 数据有明确的保留期限
   实践: 分区表 + 定时任务自动删除过期数据

3. 被遗忘权（Right to Erasure）
   设计原则: 用户可要求删除所有个人数据
   实践: 软删除+硬删除双机制，级联删除关联数据

4. 数据可移植性（Data Portability）
   设计原则: 用户可导出自己的数据
   实践: 标准格式导出接口（JSON/CSV）

5. 假名化与匿名化
   设计原则: 降低数据泄露风险
   实践: 生产环境数据脱敏到测试环境
```

**数据库层面的GDPR实现：**

```sql
-- 1. 数据保留策略: 分区表自动清理
CREATE TABLE user_activity_log (
    id BIGINT AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    action VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (UNIX_TIMESTAMP(created_at)) (
    PARTITION p202501 VALUES LESS THAN (UNIX_TIMESTAMP('2025-02-01')),
    PARTITION p202502 VALUES LESS THAN (UNIX_TIMESTAMP('2025-03-01'))
    -- ... 每月自动创建新分区，删除旧分区
);

-- 2. 软删除: 标记删除 + 定时物理删除
ALTER TABLE users ADD COLUMN deleted_at TIMESTAMP NULL;
ALTER TABLE users ADD COLUMN anonymized_at TIMESTAMP NULL;

-- 30天后物理删除
DELETE FROM users WHERE deleted_at < DATE_SUB(NOW(), INTERVAL 30 DAY);

-- 3. 被遗忘权: 级联删除/匿名化
CREATE PROCEDURE forget_user(IN uid BIGINT)
BEGIN
    -- 匿名化敏感字段而非直接删除（保留业务数据完整性）
    UPDATE users SET 
        email = CONCAT('anonymized_', uid, '@deleted.local'),
        phone = NULL,
        real_name = '[已删除]',
        anonymized_at = NOW()
    WHERE id = uid;
    
    -- 删除纯个人信息记录
    DELETE FROM user_sessions WHERE user_id = uid;
    DELETE FROM user_login_history WHERE user_id = uid;
END;
```

### 3.3 数据脱敏

```java
// 动态数据脱敏
public class DataMaskingUtil {
    
    // 手机号脱敏: 138****1234
    public static String maskPhone(String phone) {
        if (phone == null || phone.length() < 7) return phone;
        return phone.substring(0, 3) + "****" + phone.substring(7);
    }
    
    // 身份证: 320***********1234
    public static String maskIdCard(String idCard) {
        if (idCard == null || idCard.length() < 8) return idCard;
        return idCard.substring(0, 3) + "***********" + idCard.substring(idCard.length() - 4);
    }
    
    // 邮箱: u***@example.com
    public static String maskEmail(String email) {
        if (email == null || !email.contains("@")) return email;
        String[] parts = email.split("@");
        String name = parts[0];
        if (name.length() <= 2) return "*@" + parts[1];
        return name.charAt(0) + "***@" + parts[1];
    }
}
```

## 四、权限管理最佳实践

### 4.1 权限模型设计

```
数据库权限模型对比:

RBAC (基于角色的访问控制):
  用户 → 角色 → 权限
  优点: 管理方便，适合企业场景
  示例: "财务角色"有财务表的SELECT/INSERT权限

ABAC (基于属性的访问控制):
  用户(属性) + 资源(属性) + 环境(属性) → 决策
  优点: 灵活精细
  示例: "工作时间 + 内网 + 高级工程师" 可以访问

最佳实践: RBAC为主 + ABAC补充
```

**MySQL权限体系：**

```sql
-- 权限粒度从粗到细:
-- 全局级 → 数据库级 → 表级 → 列级 → 存储过程级

-- 数据库级权限
GRANT SELECT, INSERT, UPDATE, DELETE ON mydb.* TO 'dev_user'@'%';

-- 表级权限（敏感表单独控制）
GRANT SELECT ON mydb.products TO 'analyst'@'%';
-- salary表不给analyst角色访问

-- 列级权限
GRANT SELECT (id, name, department) ON mydb.employees TO 'manager'@'%';
-- salary列不给manager看

-- 限制来源IP
CREATE USER 'dba'@'10.0.0.%' IDENTIFIED BY 'xxx';
CREATE USER 'dba'@'localhost' IDENTIFIED BY 'yyy';

-- 查看用户权限
SHOW GRANTS FOR 'dev_user'@'%';
```

### 4.2 权限管理反模式与修正

| 反模式 | 风险 | 修正方案 |
|-------|------|---------|
| `GRANT ALL ON *.*` | 一个账号可删库跑路 | 按需最小授权 |
| 多人共用root | 无法审计追溯 | 每人独立账号 |
| 应用使用root | SQL注入后果放大 | 应用账号最小权限 |
| 密码硬编码 | 泄露后无法轮换 | 密钥管理服务/环境变量 |
| `%` 通配host | 任意来源均可登录 | 限制IP段 |
| 无密码复杂度要求 | 弱口令被爆破 | 强制密码策略 |

## 五、数据库安全攻防全景

### 5.1 攻击面分析

```
数据库攻击面全景图:

  网络层:
    ├── 端口暴露 (3306默认端口被扫)
    ├── 中间人攻击 (未启用TLS)
    └── DDoS连接耗尽

  认证层:
    ├── 弱口令/默认口令爆破
    ├── 认证绕过漏洞 (CVE)
    └── 凭证泄露 (代码仓库、配置文件)

  应用层:
    ├── SQL注入
    ├── 权限提升 (越权访问其他表)
    └── 逻辑漏洞 (绕过业务限制)

  存储层:
    ├── 备份文件未加密
    ├── 磁盘未加密 (物理窃取)
    └── 日志文件含敏感数据

  管理面:
    ├── 第三方工具后门
    ├── 运维失误 (误删数据)
    └── 内部威胁 (恶意DBA)
```

### 5.2 纵深防御体系

```
安全层次 (Defense in Depth):

  第一层: 网络隔离
    数据库部署在私有子网，不暴露公网IP
    通过堡垒机/跳板机访问
    防火墙白名单IP + 端口

  第二层: 传输加密
    TLS 1.2+ 强制加密
    证书校验 (避免中间人)

  第三层: 认证授权
    强密码策略 + 多因素认证 (MFA)
    最小权限原则
    定期权限审计

  第四层: 数据加密
    TDE 存储加密
    敏感列应用层加密
    备份加密

  第五层: 审计监控
    全量SQL审计
    异常行为告警 (如批量导出、非工作时间DDL)
    定期渗透测试

  第六层: 灾备恢复
    异地备份
    延迟复制 (防误删)
    恢复演练
```

**Q7: 发现数据库被攻击后，应急处置的步骤是什么？**

```
应急响应流程:

1. 隔离 (Isolate) — 30秒内
   - 断开数据库外网连接（如已暴露）
   - 保留现场，不要重启/关机
   
2. 止损 (Contain) — 5分钟内
   - 冻结可疑账号
   - 阻断可疑来源IP
   - 如发现数据被篡改，从延迟从库恢复

3. 取证 (Evidence) — 1小时内
   - 保存数据库全量日志
   - dump连接列表和慢查询
   - 保存binlog、general log
   - 系统层面抓取进程、网络连接快照

4. 排查 (Investigation) — 2小时内
   - 审计日志回溯：首次异常操作时间
   - 确定攻击路径（SQL注入？弱口令？漏洞？）
   - 评估影响范围（哪些表/数据被触及）

5. 恢复 (Recovery)
   - 从备份恢复到攻击前时间点
   - 修复漏洞（更新补丁/修复SQL注入/加强密码）
   - 渐进式放开访问

6. 复盘 (Post-Mortem)
   - 攻击时间线
   - 根因分析 (5 Whys)
   - 改进措施 & 落实到安全检查清单
```

## 六、面试高频20问

**Q1: SQL注入如何防御？**
A: 预编译是第一道防线。配合输入验证、最小权限、WAF构成纵深防御。

**Q2: PreparedStatement为什么能防注入？**
A: 将SQL结构与数据分离，数据永远不会被当作SQL代码解析。

**Q3: TDE和列级加密的区别？**
A: TDE在存储引擎层透明加密，应用无感知但DBA可见。列级加密在应用层，精确控制但不能索引。

**Q4: 数据库审计等保合规要求？**
A: 等保三级要求全量操作审计、审计日志保留6个月、不可篡改、异常告警。

**Q5: MySQL如何限制来源IP？**
A: `CREATE USER 'user'@'10.0.0.%'` 指定host，或MySQL 8.0+使用 `caching_sha2_password` 插件。

**Q6: 如何安全地存储用户密码？**
A: 用bcrypt/scrypt/Argon2，加盐，绝不明文存储，不用MD5/SHA1。

**Q7: 动态数据脱敏的实现方式？**
A: 应用层脱敏（代码中处理）、数据库视图脱敏、中间件脱敏（如ShardingSphere）。

**Q8: 数据库备份安全如何保障？**
A: 备份文件加密存储、传输加密、访问控制分离、定期恢复验证。

**Q9: 如何实现数据库操作的不可否认性？**
A: 审计日志 + 数字签名，确保操作记录不可篡改。

**Q10: TLS连接与不加密连接的性能差距？**
A: 计算开销约5-15%CPU，再加上初次握手的一次性开销。现代硬件上影响较小，安全性收益远大于性能损失。

## 七、总结

| 安全领域 | 核心要点 | 面试关键词 |
|---------|---------|-----------|
| SQL注入 | 预编译、输入验证、最小权限、WAF | PreparedStatement, #{} vs ${}, 二次注入 |
| 传输加密 | TLS握手、证书链校验、JDBC配置 | SSL, TLS 1.2, 中间人攻击 |
| 存储加密 | TDE表空间加密、列级AES加密、密钥层次 | TDE, AES-256-GCM, Key Rotation |
| 审计合规 | 审计四要素、日志保留、GDPR被遗忘权 | Audit Log, GDPR, 数据脱敏, 匿名化 |
| 权限管理 | RBAC、最小权限、IP限制、定期审计 | GRANT, role, host限制, 密码策略 |
| 应急响应 | 隔离→止损→取证→排查→恢复→复盘 | 延迟从库, binlog回溯, 攻击面分析 |

**数据库安全三大原则：**
1. **纵深防御**：不依赖单点安全措施，层层设防
2. **最小权限**：只给必要的权限，多余的不要给
3. **审计全覆盖**：所有操作可追溯，异常行为可告警
