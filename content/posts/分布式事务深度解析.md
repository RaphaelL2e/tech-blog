## 一、引言

在微服务架构下，一个业务操作往往涉及多个服务，每个服务有自己的数据库。如何保证跨服务的数据一致性，是分布式系统中最复杂的问题之一。分布式事务是面试中的核心考点，也是架构设计中必须面对的挑战。

**本文要点**：
1. 分布式事务的背景与问题
2. 2PC（两阶段提交）
3. 3PC（三阶段提交）
4. TCC（Try-Confirm-Cancel）
5. Saga 模式
6. 本地消息表
7. MQ 事务消息
8. 方案对比与选型
9. 面试高频问题

## 二、分布式事务的背景

### 2.1 为什么需要分布式事务

**单体应用**：一个数据库，本地事务保证一致性。

```
@Transactional
public void createOrder(Order order) {
    // 1. 创建订单
    orderMapper.insert(order);
    // 2. 扣减库存
    stockMapper.decrease(order.getProductId(), order.getQuantity());
    // 3. 扣减余额
    accountMapper.decrease(order.getUserId(), order.getAmount());
}
// 全部成功或全部回滚 ✅
```

**微服务架构**：多个数据库，本地事务无法保证跨服务一致性。

```
订单服务 → 订单数据库
库存服务 → 库存数据库
账户服务 → 账户数据库

createOrder() 需要跨三个服务：
1. 订单服务：创建订单 ✅
2. 库存服务：扣减库存 ❌（库存不足）
3. 账户服务：还没调用

→ 订单已创建但库存未扣减 → 数据不一致！
```

### 2.2 CAP 与分布式事务

```
CP 系统（强一致性）：
  → 使用 2PC/3PC
  → 宁可不可用，也要一致
  → 适合金融场景

AP 系统（最终一致性）：
  → 使用 TCC/Saga/消息
  → 宁可短暂不一致，也要可用
  → 适合电商场景
```

## 三、2PC（两阶段提交）

### 3.1 基本原理

**2PC**（Two-Phase Commit）是最经典的分布式事务协议，由一个**协调者（Coordinator）**和多个**参与者（Participant）**组成。

**两个阶段**：

```
阶段一：Prepare（准备/投票）
  1. 协调者向所有参与者发送 Prepare 请求
  2. 参与者执行事务操作，但不提交
  3. 参与者将 Undo/Redo 信息写入日志
  4. 参与者返回 Yes 或 No

阶段二：Commit/Rollback（提交/回滚）
  - 如果所有参与者返回 Yes → 协调者发送 Commit
  - 如果任一参与者返回 No → 协调者发送 Rollback
  - 参与者执行 Commit 或 Rollback
```

**时序图**：

```
协调者           参与者A          参与者B
  │                │                │
  │─── Prepare ──→ │                │
  │─── Prepare ────────────────→ │
  │                │                │
  │←── Yes ───────│                │
  │←── Yes ────────────────────── │
  │                │                │
  │─── Commit ───→ │                │
  │─── Commit ────────────────→ │
  │                │                │
  │←── ACK ──────│                │
  │←── ACK ────────────────────── │
```

### 3.2 异常情况分析

**情况一：参与者 A 在 Prepare 后宕机**

```
协调者           参与者A          参与者B
  │                │                │
  │─── Prepare ──→ │                │
  │─── Prepare ────────────────→ │
  │                │                │
  │←── Yes ───────│                │
  │←── Yes ────────────────────── │
  │                │                │
  │─── Commit ──→ ✘（宕机）        │
  │─── Commit ────────────────→ │
  │                │                │

问题：参与者 A 的事务状态不确定
→ 需要等待参与者 A 恢复后重新 Commit（阻塞）
```

**情况二：协调者在发送 Commit 前宕机**

```
协调者 ✘         参与者A          参与者B
  │                │                │
  │─── Prepare ──→ │                │
  │─── Prepare ────────────────→ │
  │                │                │
  │←── Yes ───────│                │
  │←── Yes ────────────────────── │
  │                │                │
  ✘（宕机，未发送 Commit）
  │                │                │
                  （锁定资源，等待）

问题：所有参与者都阻塞等待协调者恢复
→ 资源长时间被锁定
```

### 3.3 2PC 的问题

| 问题 | 描述 |
|------|------|
| **同步阻塞** | Prepare 后参与者锁定资源，直到 Commit/Rollback |
| **单点故障** | 协调者宕机导致所有参与者阻塞 |
| **数据不一致** | 部分参与者收到 Commit，部分未收到 |
| **太保守** | 任一参与者返回 No 就回滚，没有超时机制 |

## 四、3PC（三阶段提交）

### 4.1 基本原理

**3PC**（Three-Phase Commit）是 2PC 的改进版，增加了 **CanCommit** 阶段和**超时机制**。

**三个阶段**：

```
阶段一：CanCommit（询问）
  1. 协调者向所有参与者发送 CanCommit 请求
  2. 参与者评估是否能执行事务（不执行，不锁资源）
  3. 参与者返回 Yes 或 No

阶段二：PreCommit（预提交）
  1. 如果所有参与者返回 Yes → 协调者发送 PreCommit
  2. 参与者执行事务操作（但不提交），写入 Undo/Redo 日志
  3. 参与者返回 ACK

阶段三：DoCommit（提交）
  1. 协调者发送 DoCommit
  2. 参与者提交事务
  3. 参与者返回 ACK
```

**超时机制**：

```
参与者的超时：
  - PreCommit 后，如果等待 DoCommit 超时 → 自动提交
  → 避免协调者宕机导致的永久阻塞

协调者的超时：
  - CanCommit 后，如果等待参与者响应超时 → 中断事务
  - PreCommit 后，如果等待参与者 ACK 超时 → 中断事务
```

### 4.2 3PC vs 2PC

| 对比项 | 2PC | 3PC |
|--------|-----|-----|
| 阶段数 | 2 | 3 |
| 阻塞问题 | 严重 | 缓解（超时自动提交） |
| 数据不一致 | 可能 | 可能（超时自动提交导致） |
| 资源锁定 | Prepare 后锁定 | PreCommit 后锁定（延迟了） |
| 实际使用 | XA 协议 | 几乎不用 |

**3PC 为什么几乎不用**：
- 仍然存在数据不一致的可能
- 增加了一轮 RPC，延迟更高
- 超时自动提交可能导致错误提交
- 工程实现复杂度高

## 五、TCC（Try-Confirm-Cancel）

### 5.1 基本原理

**TCC** 是一种业务层面的分布式事务方案，将一个事务拆分为三个阶段：

| 阶段 | 描述 | 操作 |
|------|------|------|
| **Try** | 资源预留 | 检查资源，冻结部分资源 |
| **Confirm** | 确认提交 | 使用预留资源，完成业务操作 |
| **Cancel** | 取消回滚 | 释放预留资源，回滚业务操作 |

**示例（下单场景）**：

```
Try 阶段：
  订单服务：创建订单（状态=待确认）
  库存服务：冻结库存（available → frozen）
  账户服务：冻结余额（available → frozen）

Confirm 阶段：
  订单服务：更新订单状态（待确认 → 已确认）
  库存服务：扣减冻结库存（frozen → 0，总库存 -1）
  账户服务：扣减冻结余额（frozen → 0，总余额 -100）

Cancel 阶段：
  订单服务：更新订单状态（待确认 → 已取消）
  库存服务：释放冻结库存（frozen → available）
  账户服务：释放冻结余额（frozen → available）
```

### 5.2 TCC 的数据库设计

```sql
-- 库存表
CREATE TABLE stock (
    product_id BIGINT PRIMARY KEY,
    total      INT,      -- 总库存
    frozen     INT       -- 冻结库存
);

-- Try：冻结库存
UPDATE stock SET frozen = frozen + 1 WHERE product_id = 1 AND total - frozen >= 1;

-- Confirm：扣减冻结库存
UPDATE stock SET total = total - 1, frozen = frozen - 1 WHERE product_id = 1;

-- Cancel：释放冻结库存
UPDATE stock SET frozen = frozen - 1 WHERE product_id = 1;
```

### 5.3 TCC 代码实现

```java
public class OrderTccService {
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Autowired
    private StockTccService stockTccService;
    
    @Autowired
    private AccountTccService accountTccService;
    
    // Try 阶段
    @Transactional
    public void tryCreateOrder(Order order) {
        // 1. 创建订单（状态=待确认）
        order.setStatus("TRYING");
        orderMapper.insert(order);
        
        // 2. 冻结库存
        stockTccService.tryFreeze(order.getProductId(), order.getQuantity());
        
        // 3. 冻结余额
        accountTccService.tryFreeze(order.getUserId(), order.getAmount());
    }
    
    // Confirm 阶段
    @Transactional
    public void confirmCreateOrder(Long orderId) {
        Order order = orderMapper.selectById(orderId);
        
        // 1. 确认订单
        order.setStatus("CONFIRMED");
        orderMapper.updateById(order);
        
        // 2. 扣减库存
        stockTccService.confirmDeduct(order.getProductId(), order.getQuantity());
        
        // 3. 扣减余额
        accountTccService.confirmDeduct(order.getUserId(), order.getAmount());
    }
    
    // Cancel 阶段
    @Transactional
    public void cancelCreateOrder(Long orderId) {
        Order order = orderMapper.selectById(orderId);
        
        // 1. 取消订单
        order.setStatus("CANCELLED");
        orderMapper.updateById(order);
        
        // 2. 释放库存
        stockTccService.cancelFreeze(order.getProductId(), order.getQuantity());
        
        // 3. 释放余额
        accountTccService.cancelFreeze(order.getUserId(), order.getAmount());
    }
}
```

### 5.4 TCC 的异常处理

**空回滚**：

```
问题：Try 请求因网络丢失，未到达参与者 → TM 触发 Cancel
→ 参与者的 Cancel 没有 Try 对应

解决：记录事务日志，Cancel 时检查是否执行过 Try
```

**幂等性**：

```
问题：Confirm/Cancel 可能因网络重试被多次调用
→ 必须保证幂等

解决：每个事务分支记录全局事务 ID + 分支事务 ID
→ 判断是否已执行过
```

**悬挂**：

```
问题：Cancel 比 Try 先到达参与者
→ Cancel 执行后，Try 才到达 → 资源被冻结且无法释放

解决：Cancel 执行时记录状态
→ Try 执行时检查是否已 Cancel → 如果已 Cancel，不再执行 Try
```

### 5.5 TCC 框架

| 框架 | 描述 |
|------|------|
| **Seata** | 阿里开源，支持 AT/TCC/Saga/XA 四种模式 |
| **Hmily** | 金融级 TCC 框架，支持异步 Confirm/Cancel |
| **ByteTCC** | 字节跳动开源，高性能 TCC |

## 六、Saga 模式

### 6.1 基本原理

**Saga** 是一种长事务解决方案，将一个大事务拆分为多个本地小事务，每个小事务都有对应的补偿操作。

**两种编排方式**：

**1. 串联式（Choreography）**：

```
T1 → T2 → T3 → T4（正向）
                  ↓ 失败
C4 ← C3 ← C2 ← C1（补偿）

示例：
  T1: 创建订单 → C1: 取消订单
  T2: 扣减库存 → C2: 恢复库存
  T3: 扣减余额 → C3: 恢复余额
  T4: 发货 → C4: 取消发货
```

**2. 编排式（Orchestration）**：

```
         协调器
        ↙  ↓  ↘
    T1   T2   T3
    ↓    ↓    ↓
    C1   C2   C3

协调器决定执行顺序和补偿策略
```

### 6.2 Saga 的补偿机制

**补偿操作 vs 回滚**：

```
2PC/3PC：回滚（Undo）→ 恢复到事务开始前的状态
Saga：补偿（Compensate）→ 执行一个反向操作，语义上等价于取消

示例：
  正向：账户 A 转账 100 元给账户 B
  回滚：恢复 A 和 B 的余额到转账前
  补偿：账户 B 扣减 100 元（可能余额不足！）
```

**补偿的局限**：
- 补偿操作不一定能完美恢复
- 可能产生脏数据
- 业务逻辑需要自行处理

## 七、本地消息表

### 7.1 基本原理

**本地消息表**是 eBay 提出的方案，利用本地事务保证消息和业务操作的原子性。

**流程**：

```
1. 业务操作和写入消息表在同一个本地事务中
2. 后台线程定时扫描消息表，发送消息到 MQ
3. 消费者消费消息，执行下游业务
4. 消费者执行成功，更新消息状态
```

### 7.2 数据库设计

```sql
-- 本地消息表
CREATE TABLE local_message (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    message_id  VARCHAR(64) UNIQUE,      -- 消息唯一 ID
    topic       VARCHAR(128),             -- MQ Topic
    content     TEXT,                      -- 消息内容
    status      TINYINT,                  -- 0=待发送, 1=已发送, 2=已消费, 3=失败
    retry_count INT DEFAULT 0,            -- 重试次数
    create_time DATETIME DEFAULT NOW(),
    update_time DATETIME DEFAULT NOW()
);
```

### 7.3 代码实现

```java
@Transactional
public void createOrder(Order order) {
    // 1. 创建订单（业务操作）
    orderMapper.insert(order);
    
    // 2. 写入消息表（同一事务）
    LocalMessage message = new LocalMessage();
    message.setMessageId(UUID.randomUUID().toString());
    message.setTopic("order-created");
    message.setContent(JSON.toJSONString(order));
    message.setStatus(0);  // 待发送
    localMessageMapper.insert(message);
}

// 定时任务：扫描消息表，发送到 MQ
@Scheduled(fixedDelay = 5000)
public void sendPendingMessages() {
    List<LocalMessage> messages = localMessageMapper.selectByStatus(0);
    for (LocalMessage msg : messages) {
        try {
            rocketMQTemplate.syncSend(msg.getTopic(), msg.getContent());
            msg.setStatus(1);  // 已发送
            localMessageMapper.updateById(msg);
        } catch (Exception e) {
            msg.setRetryCount(msg.getRetryCount() + 1);
            if (msg.getRetryCount() > 5) {
                msg.setStatus(3);  // 失败
            }
            localMessageMapper.updateById(msg);
        }
    }
}
```

## 八、MQ 事务消息

### 8.1 RocketMQ 事务消息

**流程**：

```
1. 生产者发送半消息（Half Message）到 MQ
   → 半消息对消费者不可见

2. MQ 返回发送成功

3. 生产者执行本地事务

4. 根据本地事务结果：
   - Commit → 半消息变为可消费
   - Rollback → 半消息被删除
   - Unknown → MQ 回查本地事务状态

5. 消费者消费消息，执行下游业务
```

### 8.2 代码实现

```java
// RocketMQ 事务消息生产者
@RestController
public class OrderController {
    
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    public void createOrder(Order order) {
        // 发送事务消息
        rocketMQTemplate.sendMessageInTransaction(
            "order-created",
            MessageBuilder.withPayload(order).build(),
            order  // 传递给本地事务的参数
        );
    }
}

// 本地事务监听器
@RocketMQTransactionListener
public class OrderTransactionListener implements RocketMQLocalTransactionListener {
    
    @Autowired
    private OrderMapper orderMapper;
    
    @Override
    public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            Order order = (Order) arg;
            orderMapper.insert(order);  // 执行本地事务
            return RocketMQLocalTransactionState.COMMIT;
        } catch (Exception e) {
            return RocketMQLocalTransactionState.ROLLBACK;
        }
    }
    
    @Override
    public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
        // 回查本地事务状态
        String orderId = (String) msg.getHeaders().get("orderId");
        Order order = orderMapper.selectById(orderId);
        if (order != null) {
            return RocketMQLocalTransactionState.COMMIT;
        }
        return RocketMQLocalTransactionState.ROLLBACK;
    }
}
```

## 九、方案对比与选型

### 9.1 全面对比

| 方案 | 一致性 | 性能 | 实现复杂度 | 业务侵入 | 适用场景 |
|------|--------|------|-----------|---------|---------|
| **2PC/XA** | 强一致 | 低 | 低 | 无 | 数据库层面 |
| **3PC** | 强一致 | 低 | 高 | 无 | 几乎不用 |
| **TCC** | 最终一致 | 中 | 高 | 高 | 资金/库存 |
| **Saga** | 最终一致 | 高 | 中 | 中 | 长流程业务 |
| **本地消息表** | 最终一致 | 高 | 中 | 低 | 异步通知 |
| **MQ 事务消息** | 最终一致 | 高 | 低 | 低 | 异步通知 |

### 9.2 选型建议

```
场景一：强一致性要求（金融、支付）
→ 2PC/XA（数据库层面）
→ TCC（业务层面）

场景二：最终一致性可接受（电商、社交）
→ Saga（长流程）
→ 本地消息表 / MQ 事务消息（短流程）

场景三：异步通知（订单创建后发短信/邮件）
→ MQ 事务消息（推荐）
→ 本地消息表（备选）

场景四：高并发（秒杀、抢购）
→ TCC（资源预留）
→ MQ 异步（削峰填谷）
```

## 十、Seata 框架实战

### 10.1 Seata 概述

**Seata**（Simple Extensible Autonomous Transaction Architecture）是阿里开源的分布式事务框架，支持四种模式：

| 模式 | 描述 | 一致性 | 性能 |
|------|------|--------|------|
| **AT** | 自动补偿（拦截 SQL） | 最终一致 | 高 |
| **TCC** | 手动补偿 | 最终一致 | 中 |
| **Saga** | 长事务补偿 | 最终一致 | 高 |
| **XA** | 2PC 协议 | 强一致 | 低 |

### 10.2 AT 模式（最常用）

**原理**：

```
1. 拦截业务 SQL
2. 在执行 SQL 前，查询修改前的数据（Before Image）
3. 执行业务 SQL
4. 查询修改后的数据（After Image）
5. 生成 Undo Log（Before Image + After Image）
6. 全局提交 → 删除 Undo Log
7. 全局回滚 → 根据 Undo Log 恢复数据
```

**Undo Log 表**：

```sql
CREATE TABLE undo_log (
    id            BIGINT PRIMARY KEY AUTO_INCREMENT,
    branch_id     BIGINT NOT NULL,
    xid           VARCHAR(100) NOT NULL,
    context       VARCHAR(128) NOT NULL,
    rollback_info LONGBLOB NOT NULL,
    log_status    INT NOT NULL,
    log_created   DATETIME NOT NULL,
    log_modified  DATETIME NOT NULL,
    UNIQUE KEY ux_undo_log (xid, branch_id)
);
```

**使用方式**：

```java
// 只需加一个注解！
@GlobalTransactional
public void createOrder(Order order) {
    orderMapper.insert(order);
    stockClient.deduct(order.getProductId(), order.getQuantity());
    accountClient.deduct(order.getUserId(), order.getAmount());
}
```

**AT 模式的限制**：
- 只支持关系数据库
- 需要创建 undo_log 表
- 不支持 NoSQL
- 全局锁可能导致性能下降

## 十一、面试高频问题

### Q1: 2PC 和 3PC 的区别？

**答案要点**：
- 3PC 增加了 CanCommit 阶段（减少资源锁定时间）
- 3PC 增加了超时机制（协调者宕机不永久阻塞）
- 3PC 可能导致数据不一致（超时自动提交）
- 实际工程中几乎不用 3PC

### Q2: TCC 和 Saga 的区别？

**答案要点**：
- TCC：Try 阶段预留资源，隔离性好
- Saga：没有预留，直接执行，补偿操作反向抵消
- TCC 业务侵入大（需要写 Try/Confirm/Cancel 三个方法）
- Saga 适合长流程，TCC 适合短流程

### Q3: 如何保证消息和本地事务的原子性？

**答案要点**：
- 方案一：本地消息表（业务和消息在同一事务）
- 方案二：MQ 事务消息（半消息 + 本地事务 + 回查）
- 方案三：Outbox 模式（CDC 监听数据库变更）

### Q4: Seata 的 AT 模式原理？

**答案要点**：
- 拦截 SQL，生成 Before Image 和 After Image
- 写入 undo_log
- 全局提交时删除 undo_log
- 全局回滚时根据 undo_log 恢复数据
- 全局锁保证写隔离

### Q5: 分布式事务如何选型？

**答案要点**：
- 强一致 → 2PC/XA
- 最终一致 + 短流程 → TCC
- 最终一致 + 长流程 → Saga
- 异步通知 → MQ 事务消息 / 本地消息表
- 快速接入 → Seata AT 模式

## 十二、总结

本文系统梳理了分布式事务的主流方案：

1. **2PC/XA**：强一致，但阻塞严重，适合数据库层面
2. **3PC**：2PC 改进，但几乎不用
3. **TCC**：业务侵入大，隔离性好，适合资金/库存场景
4. **Saga**：长事务补偿，适合流程编排
5. **本地消息表**：简单可靠，适合异步通知
6. **MQ 事务消息**：最优雅的异步方案

**核心记忆**：
- 2PC 问题：同步阻塞 + 单点故障 + 数据不一致
- TCC 三步：Try（预留）→ Confirm（确认）→ Cancel（补偿）
- Saga：正向 + 补偿，无预留，适合长流程
- Seata AT：拦截 SQL + Undo Log，最易接入
- 选型原则：强一致用 XA，最终一致用 TCC/Saga/消息
