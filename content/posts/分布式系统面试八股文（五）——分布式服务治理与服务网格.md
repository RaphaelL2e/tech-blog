## 一、服务治理全景图

在分布式/微服务架构中，服务治理是保证系统稳定运行的核心基础设施。六大领域形成完整的治理闭环：

```
                        ┌──────────────┐
                        │   配置中心    │  ← 动态配置下发
                        │ (Nacos/Apollo)│
                        └──────┬───────┘
                               │
用户 ──→ API网关 ──→ 负载均衡 ──→ 服务A ──→ 服务B
         (Gateway)   (Ribbon/LB)     │          │
                                     │          │
              ┌──────────────────────┘          │
              ↓                                  ↓
        限流熔断降级                        链路追踪
        (Sentinel)                      (SkyWalking)
              │                                  │
              └──────────┬───────────────────────┘
                         ↓
                服务注册与发现 (Nacos/Eureka)
```

### 面试追问应答

**Q: 为什么Eureka选择AP而不是CP？**

> 服务注册发现场景中，**可用性高于一致性**。即使注册中心短暂数据不一致（A节点有服务、B节点没有），客户端通过重试和负载均衡也能正常工作。但若注册中心不可用，整个微服务体系直接瘫痪。这就是"宁可数据短暂不一致，也不能不可用"的设计哲学。

**Q: Eureka自我保护机制触发后会怎样？**

> 当Eureka Server在短时间内丢失超过85%的心跳时，进入自我保护模式：(1) 不再剔除任何服务实例 (2) 保护已有的注册信息 (3) 新注册不受影响。退出条件：心跳数恢复到85%以上。这是为了防止网络分区导致服务被大规模误剔除。

### Nacos配置刷新原理

```
Nacos Server
    │
    │ (1) 客户端发起长轮询请求(hold 30s)
    │ (2) 配置无变化 → 返回空，客户端立即发起下次长轮询
    │ (3) 配置有变化 → 返回变更的配置
    │
    ↓
Spring Cloud Nacos Config
    │
    │ 接收变更 → ContextRefresher.refresh()
    │
    ↓
@RefreshScope / @ConfigurationProperties
    │
    │ 重新绑定配置值
    ↓
应用运行时参数更新
```

### 面试追问应答

**Q: 配置热更新对数据库连接池这样的资源有效吗？**

> **不会自动生效！** 数据库连接池(如Druid/HikariCP)在初始化时已建立连接，仅修改配置参数不会重建连接池。需要自定义监听器：监听到配置变更 → 关闭旧连接池 → 用新配置创建连接池。同理，线程池参数也需要自定义刷新逻辑。

### GlobalFilter 代码示例

```java
@Component
public class AuthFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest().getHeaders().getFirst("Authorization");
        if (StringUtils.isEmpty(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }
        // 解析JWT → 获取用户信息 → 添加到请求头
        // exchange.getRequest().mutate().header("userId", userId);
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        return -100; // 优先级最高
    }
}
```

### 面试追问应答

**Q: Gateway的线程模型是怎样的？为什么比Zuul1性能好？**

> Gateway基于Spring WebFlux + Netty，采用**事件驱动 + 非阻塞I/O**模型。使用少量EventLoop线程处理大量请求，避免了Servlet容器的**每请求一线程**模型。Zuul1基于Servlet 2.5（阻塞），每个请求占用一个线程，线程上下文切换成本高。Netty的零拷贝和直接内存也减少了数据拷贝开销。

| 4 | 一致性哈希如何解决数据倾斜？ | ⭐⭐⭐ | 虚拟节点(每物理节点映射150虚拟节点) |
| 5 | Spring Cloud LoadBalancer vs Ribbon？ | ⭐⭐⭐ | Ribbon停止维护 → LoadBalancer替代(响应式) |

### 负载均衡算法对比

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| 轮询(Round Robin) | 依次分配 | 简单均匀 | 不关心性能差异 | 同构服务 |
| 加权轮询 | 按权重分配 | 考虑性能差异 | 权重需手动设 | 异构服务 |
| 随机(Random) | 随机选择 | 简单、统计均匀 | 个别请求不均 | 大量请求 |
| 最少连接 | 选连接数最少的 | 动态适应负载 | 需维护连接数 | 长连接场景 |
| 一致性哈希 | Hash环路由 | 扩缩容影响小 | 可能数据倾斜 | 有状态/缓存服务 |

### 面试追问应答

**Q: 一致性哈希为什么适合缓存集群？虚拟节点解决什么问题？**

> 普通Hash取模在节点增减时会导致**大量缓存失效**（缓存雪崩）。一致性哈希将节点映射到Hash环上，Key落到环上后顺时针找到第一个节点，增减节点只影响相邻节点间的数据。虚拟节点解决的是**数据倾斜**问题：物理节点在环上分布不均时，通过每个物理节点映射多个虚拟节点（通常150个），使分布更均匀。

### 三种保护机制的区别

```
┌──────────────────────────────────────────────────────┐
│  限流 (Rate Limiting)                                │
│  "你来得太快了，慢一点"                                  │
│  流量入口控制 → 令牌桶/滑动窗口 → 拒绝多余请求           │
├──────────────────────────────────────────────────────┤
│  熔断 (Circuit Breaking)                             │
│  "他出问题了，先别去了"                                 │
│  故障隔离 → 异常比例/慢调用 → 快速失败+降级              │
├──────────────────────────────────────────────────────┤
│  降级 (Fallback)                                     │
│  "做不到完美的，给个差不多的"                            │
│  功能回退 → 返回缓存数据/默认值/友好提示                  │
└──────────────────────────────────────────────────────┘
```

### Sentinel 熔断器状态转换

```
   请求正常处理
   ┌──────────┐
   │          ↓
┌──┴─────┐  达到熔断阈值  ┌──────────┐  等待时间窗口结束  ┌──────────┐
│ CLOSED │──────────────→│   OPEN   │─────────────────→│ HALF_OPEN│
│ (关闭)  │               │  (打开)   │                   │  (半开)   │
└──┬─────┘               └──────────┘                   └────┬─────┘
   │                                                        │
   │          探测成功(恢复正常) ←─────────────────────────────┤
   │                                                        │
   └────────────────────────────────────────────────────────┘
                        探测失败(继续保持OPEN)
```

### Redis + Lua 分布式限流示例

```lua
-- 滑动窗口限流 Lua脚本
local key = KEYS[1]           -- 限流Key
local limit = tonumber(ARGV[1]) -- 限制QPS
local window = tonumber(ARGV[2])-- 窗口大小(秒)
local now = redis.call('TIME')
local current = now[1] * 1000 + now[2] / 1000

-- 清除过期窗口
redis.call('ZREMRANGEBYSCORE', key, 0, current - window * 1000)

-- 统计当前窗口请求数
local count = redis.call('ZCARD', key)

if count < limit then
    redis.call('ZADD', key, current, current)
    redis.call('EXPIRE', key, window)
    return 1  -- 通过
else
    return 0  -- 限流
end
```

### 面试追问应答

**Q: Sentinel 和 Hystrix 的核心区别？**

> (1) **隔离策略**：Hystrix默认线程池隔离（重量级），Sentinel默认信号量隔离（轻量） (2) **熔断策略**：Hystrix仅错误比例，Sentinel支持慢调用比例/异常比例/异常数三种 (3) **控制台**：Sentinel有实时监控控制台，支持动态规则下发 (4) **扩展性**：Sentinel支持网关限流、热点参数限流、系统自适应限流 (5) **维护状态**：Hystrix已停维，Sentinel是阿里开源持续活跃。

### 面试追问应答

**Q: SkyWalking Agent 的零侵入是如何实现的？**

> 使用 **Java Agent + ByteBuddy 字节码增强**技术。通过 `premain` 方法在JVM加载类时拦截目标类（如Spring MVC的DispatcherServlet、HttpClient、JDBC Driver），在方法前后插入埋点代码，记录调用耗时、参数、异常等。对业务代码完全透明，无需任何依赖或注解。

### 面试追问应答

**Q: Service Mesh 的核心价值是什么？什么情况下不需要 Service Mesh？**

> **核心价值**：(1) **治理与业务解耦** — 业务代码中不再需要Sentinel/Ribbon/Feign等治理库 (2) **多语言统一治理** — Go/Python/Node.js与Java享受同样的治理能力 (3) **可观察性** — 零侵入的全链路追踪和指标采集 (4) **安全** — 自动mTLS零信任网络。
>
> **不需要的场景**：(1) 技术栈统一为Java + Spring Cloud且无多语言需求 (2) 服务规模小(<20个服务) (3) 运维能力不足以支撑Kubernetes+Istio (4) 对延迟极度敏感(每跳增加1-3ms)

**Q: 南北向流量和东西向流量有什么区别？**

> **南北向(North-South)**：外部请求进入集群的流量（用户→API网关→服务），由API网关管理。**东西向(East-West)**：集群内部服务间调用的流量（服务A→服务B），由Service Mesh管理。两者互补：网关负责边界安全+路由，Mesh负责内部治理。

### 大规模集群下的优化方案

| 层次 | 优化方案 | 效果 |
|------|---------|------|
| 注册中心 | Nacos元数据缓存 + 推送精简 | 减少30%网络开销 |
| 配置中心 | 本地文件兜底 + 异步刷新 | 不可用时保证启动 |
| API网关 | 多级限流(网关+Sentinel) | 第一层拦截无效流量 |
| 负载均衡 | 自定义ZoneAvoidanceRule | 优先同机房/同区域 |
| 链路追踪 | 自适应采样(正常低+异常高) | 减少50%存储量 |

## 十、总结

| 领域 | 核心组件 | 关键机制 | 一句话总结 |
|------|---------|---------|-----------|
| 注册发现 | Nacos/Eureka | 心跳+AP/CP取舍 | 宁可数据短暂不一致，不能不可用 |
| 配置中心 | Nacos/Apollo | 长轮询+热更新 | 配置动、服务不动 |

| 负载均衡 | LoadBalancer/Ribbon | 拉取列表+本地缓存 | 用一致性哈希解决有状态路由 |
| 限流熔断 | Sentinel | 信号量隔离+滑动窗口 | 三把刀：限流、熔断、降级 |
| 链路追踪 | SkyWalking | Agent零侵入+TraceId传递 | 看得见的调用链路 |
| 服务网格 | Istio+Envoy | Sidecar+iptables劫持 | 治理下沉到基础设施，业务零感知 |
