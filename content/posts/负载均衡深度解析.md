# 负载均衡深度解析

## 一、引言

在微服务架构中，服务消费者需要调用服务提供者，而服务提供者通常会有多个实例。如何选择一个合适的实例进行调用？这就是**负载均衡（Load Balancing）**要解决的问题。

负载均衡是微服务流量管控的核心组件，它决定了请求如何分发到多个服务实例，直接影响系统的**吞吐量、可用性和响应延迟**。

Spring Cloud 提供了两套负载均衡解决方案：
1. **Ribbon**（Netflix 开源，已进入维护模式）
2. **Spring Cloud LoadBalancer**（Spring 官方推荐，Spring Cloud 2020.0.0+ 默认）

理解负载均衡的原理、Ribbon 与 Spring Cloud LoadBalancer 的差异，是微服务开发和面试的核心考点。

**本文要点**：
1. 负载均衡原理（客户端 vs 服务端）
2. Ribbon 深度解析（IRule、负载均衡策略、自定义策略）
3. Spring Cloud LoadBalancer 深度解析（新版本推荐）
4. Ribbon vs Spring Cloud LoadBalancer 对比
5. 负载均衡策略详解
6. 自定义负载均衡实战
7. 面试高频问题

## 二、负载均衡原理

### 2.1 什么是负载均衡

**负载均衡（Load Balancing）** 是将请求分摊到多个操作单元（服务器、服务实例）上执行，从而：
- **提升系统吞吐量**：并行处理更多请求
- **提高系统可用性**：避免单点故障
- **优化资源利用**：合理分配请求，避免部分实例过载

### 2.2 负载均衡的类型

#### 2.2.1 服务端负载均衡（Server-Side Load Balancing）

**原理**：客户端发送请求到负载均衡器（如 Nginx、F5），由负载均衡器选择后端服务器。

```
客户端 → 负载均衡器 (Nginx) → 后端服务器1
                           → 后端服务器2
                           → 后端服务器3
```

**代表产品**：
- **硬件负载均衡**：F5、A10
- **软件负载均衡**：Nginx、HAProxy、LVS

**优点**：
- 客户端无感知，无需配置
- 集中管理，方便监控

**缺点**：
- 负载均衡器本身可能成为单点故障（需主备）
- 额外网络跳转，增加延迟

#### 2.2.2 客户端负载均衡（Client-Side Load Balancing）

**原理**：客户端从注册中心获取服务实例列表，自己选择调用哪个实例。

```
客户端（内置负载均衡算法）→ 服务实例1
                      → 服务实例2
                      → 服务实例3
```

**代表产品**：
- **Ribbon**（Netflix）
- **Spring Cloud LoadBalancer**（Spring）

**优点**：
- 无单点故障，客户端直接调用
- 减少网络跳转，降低延迟
- 可定制化强，灵活配置

**缺点**：
- 客户端需要维护服务列表和负载均衡算法
- 配置相对复杂

### 2.3 微服务中的负载均衡

在微服务架构中，**客户端负载均衡**是主流方案，因为：
1. 避免单点故障（服务端负载均衡器挂了全挂）
2. 减少网络跳转（客户端直连服务实例）
3. 与注册中心无缝集成（动态感知服务上下线）

**Spring Cloud 负载均衡架构**：

```
服务消费者 (Consumer)          服务提供者 (Provider)
        ↓                             ↑
        ↓                             │
   ① 从注册中心拉取服务列表   ④ 注册服务实例
        ↓                             │
   ② 本地缓存服务列表                    │
        ↓                             │
   ③ 负载均衡算法选择实例 ────────→ 调用服务
```

**工作流程**：
1. 服务消费者从注册中心拉取服务提供者实例列表
2. 本地缓存服务列表（定期更新）
3. 负载均衡算法选择一个实例
4. 发起远程调用

## 三、Ribbon 深度解析

### 3.1 Ribbon 简介

**Ribbon** 是 Netflix 开源的客户端负载均衡器，Spring Cloud 早期默认集成。

**现状**：Spring Cloud 2020.0.0（Ilford）开始，Ribbon 进入**维护模式**，不再接受新功能，推荐使用 Spring Cloud LoadBalancer。

**核心功能**：
- 服务实例列表获取（从注册中心）
- 负载均衡算法（轮询、随机、重试等）
- 健康检查（ping）
- 故障实例剔除

### 3.2 Ribbon 核心组件

#### 3.2.1 IRule（负载均衡规则）

**IRule** 是 Ribbon 的负载均衡策略接口，内置多种实现：

| 策略类 | 说明 |
|--------|------|
| RoundRobinRule | 轮询（默认） |
| RandomRule | 随机 |
| RetryRule | 重试（先按轮询，失败重试其他实例） |
| WeightedResponseTimeRule | 响应时间加权（响应越快，权重越高） |
| BestAvailableRule | 选择并发请求最少的实例 |
| AvailabilityFilteringRule | 过滤熔断/高并发实例，轮询剩余 |
| ZoneAvoidanceRule | 复合判断（区域 + 可用性） |

**默认策略**：ZoneAvoidanceRule（区域感知轮询）

#### 3.2.2 IPing（健康检查）

**IPing** 用于检查服务实例是否存活：

| 实现类 | 说明 |
|--------|------|
| DummyPing | 默认返回 true（不检查） |
| NoOpPing | 默认返回 true（不检查） |
| PingConstant | 固定返回 true/false |
| PingUrl | HTTP 请求检查（真实 ping） |

**默认实现**：DummyPing（不检查，依赖注册中心心跳）

#### 3.2.3 ServerList（服务列表）

**ServerList** 获取服务实例列表：

| 实现类 | 说明 |
|--------|------|
| ConfigurationBasedServerList | 从配置文件加载（静态配置） |
| DiscoveryEnabledNIWSServerList | 从 Eureka 拉取（动态发现） |

**默认实现**：DiscoveryEnabledNIWSServerList（配合 Eureka）

### 3.3 Ribbon 配置

#### 3.3.1 全局配置

```yaml
# application.yml
ribbon:
  ConnectTimeout: 2000        # 连接超时（ms）
  ReadTimeout: 5000           # 读取超时（ms）
  OkToRetryOnAllOperations: false  # 是否所有操作都重试
  MaxAutoRetries: 0           # 同一实例重试次数
  MaxAutoRetriesNextServer: 1 # 切换实例重试次数
```

#### 3.3.2 指定服务配置

```yaml
# application.yml
# 格式：<service-name>.ribbon.<property>
user-service:
  ribbon:
    ConnectTimeout: 3000
    ReadTimeout: 6000
    NFLoadBalancerRuleClassName: com.netflix.loadbalancer.RandomRule  # 随机策略
```

#### 3.3.3 自定义 IRule

```java
// 自定义负载均衡策略
@Configuration
public class RibbonConfig {
    
    @Bean
    public IRule myRule() {
        return new RandomRule();  // 使用随机策略
    }
}
```

```java
// 在启动类指定配置
@SpringBootApplication
@RibbonClient(name = "user-service", configuration = RibbonConfig.class)
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

### 3.4 Ribbon  retry机制

**Ribbon 重试**：当调用失败时，自动切换到其他实例重试。

**配置示例**：

```yaml
ribbon:
  MaxAutoRetries: 1           # 同一实例重试1次
  MaxAutoRetriesNextServer: 2 # 切换2个不同实例
  OkToRetryOnAllOperations: true  # 所有操作都重试（包括POST，危险！）
```

**重试流程**：

```
第1次调用 → 实例1失败
    ↓
重试实例1 → 失败
    ↓
切换实例2 → 失败
    ↓
重试实例2 → 成功 ✅
```

**注意**：`OkToRetryOnAllOperations=true` 时，**POST 请求也会重试**，可能导致**重复提交**！建议设置为 `false`，仅 GET 请求重试。

### 3.5 Ribbon 与 RestTemplate 集成

**使用 Ribbon 的 RestTemplate**：

```java
@Configuration
public class RestTemplateConfig {
    
    @Bean
    @LoadBalanced  // 关键注解：启用 Ribbon 负载均衡
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

```java
@Service
public class OrderService {
    
    @Autowired
    private RestTemplate restTemplate;
    
    public User getUser(Long userId) {
        // 使用服务名（而非 IP:Port），Ribbon 会自动选择实例
        String url = "http://user-service/users/" + userId;
        return restTemplate.getForObject(url, User.class);
    }
}
```

**工作原理**：
1. `@LoadBalanced` 注解给 RestTemplate 添加拦截器 `LoadBalancerInterceptor`
2. 拦截器从 URL 中提取服务名（`user-service`）
3. 通过 Ribbon 获取该服务的实例列表
4. 负载均衡算法选择一个实例
5. 将服务名替换为 `IP:Port`，发起 HTTP 请求


## 四、Spring Cloud LoadBalancer 深度解析

### 4.1 Spring Cloud LoadBalancer 简介

**Spring Cloud LoadBalancer** 是 Spring 官方推出的客户端负载均衡器，用于**替代 Ribbon**。

**背景**：
- Netflix Ribbon 进入维护模式（不再接受新功能）
- Spring Cloud 2020.0.0（Ilford）开始，默认使用 Spring Cloud LoadBalancer
- 支持 **Reactive**（WebFlux）和 **Servlet**（Web MVC）两种环境

**核心特性**：
- 轻量级，非阻塞（基于 Project Reactor）
- 支持响应式编程（WebFlux）
- 可扩展性强（自定义负载均衡算法）
- 与 Spring Cloud 生态深度集成

### 4.2 Spring Cloud LoadBalancer 核心组件

#### 4.2.1 ReactorLoadBalancer

**Reactive 负载均衡器接口**，支持 Mono/Flux：

```java
public interface ReactorLoadBalancer<T> extends ReactiveLoadBalancer<T> {
    Mono<Response<T>> choose(Request request);
}
```

#### 4.2.2 LoadBalancer 实现

Spring Cloud LoadBalancer 内置两种实现：

| 实现类 | 说明 |
|--------|------|
| RoundRobinLoadBalancer | **轮询**（默认） |
| RandomLoadBalancer | **随机** |

**默认策略**：RoundRobinLoadBalancer（轮询）

#### 4.2.3 ServiceInstanceListSupplier

**服务实例列表提供者**，替代 Ribbon 的 ServerList：

```java
public interface ServiceInstanceListSupplier extends Supplier<Flux<List<ServiceInstance>>> {
    // 从注册中心获取实例列表
}
```

**内置实现**：
- `DiscoveryClientServiceInstanceListSupplier`：从 `DiscoveryClient` 获取（Eureka、Nacos 等）
- `CachingServiceInstanceListSupplier`：带缓存的列表（提升性能）

### 4.3 Spring Cloud LoadBalancer 配置

#### 4.3.1 Maven 依赖

```xml
<!-- Spring Cloud LoadBalancer Starter -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-loadbalancer</artifactId>
</dependency>

<!-- 如果使用 Eureka -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
</dependency>
```

**注意**：Spring Cloud 2020.0.0+ 默认不包含 Ribbon，如需使用需单独引入。

#### 4.3.2 配置文件

```yaml
spring:
  cloud:
    loadbalancer:
      configurations: round-robin  # 负载均衡策略（round-robin | random）
      retry:
        enabled: true               # 启用重试
        max-retries-on-same-service-instance: 1  # 同实例重试次数
        max-retries-on-next-service-instance: 1  # 切换实例重试次数
```

#### 4.3.3 自定义负载均衡策略

**方式1：配置类**

```java
@Configuration
@LoadBalancerClientConfiguration
public class CustomLoadBalancerConfig {
    
    @Bean
    public ReactorLoadBalancer<ServiceInstance> customLoadBalancer(
            Environment environment,
            LoadBalancerClientFactory factory) {
        String name = environment.getProperty(LoadBalancerClientFactory.PROPERTY_NAME);
        return new RandomLoadBalancer(
            factory.getLazyProvider(name, ServiceInstanceListSupplier.class), 
            name
        );
    }
}
```

**方式2：注解指定**

```java
@SpringBootApplication
@LoadBalancerClient(name = "user-service", configuration = CustomLoadBalancerConfig.class)
public class OrderServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderServiceApplication.class, args);
    }
}
```

### 4.4 Spring Cloud LoadBalancer 与 RestTemplate/WebClient 集成

#### 4.4.1 RestTemplate（Web MVC）

```java
@Configuration
public class RestTemplateConfig {
    
    @Bean
    @LoadBalanced  // 启用 Spring Cloud LoadBalancer
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

#### 4.4.2 WebClient（WebFlux，推荐）

```java
@Configuration
public class WebClientConfig {
    
    @Bean
    @LoadBalanced  // 启用 Spring Cloud LoadBalancer
    public WebClient.Builder webClientBuilder() {
        return WebClient.builder();
    }
}
```

```java
@Service
public class UserService {
    
    @Autowired
    private WebClient.Builder webClientBuilder;
    
    public Mono<User> getUser(Long userId) {
        return webClientBuilder.build()
                .get()
                .uri("http://user-service/users/" + userId)
                .retrieve()
                .bodyToMono(User.class);
    }
}
```

**推荐 WebClient**：
- 支持异步非阻塞（性能更好）
- 与 Spring Cloud LoadBalancer 深度集成
- 适合高并发场景


## 五、Ribbon vs Spring Cloud LoadBalancer 对比

### 5.1 核心差异对比

| 对比项 | Ribbon | Spring Cloud LoadBalancer |
|--------|--------|----------------------------|
| **维护状态** | 维护模式（不推荐新项目） | 活跃开发（官方推荐） |
| **响应式支持** | 不支持（阻塞式） | 支持（WebFlux） |
| **负载均衡策略** | 7种内置（IRule） | 2种内置（轮询/随机） |
| **扩展性** | 中等 | 强（基于 Reactor） |
| **性能** | 阻塞式，性能一般 | 非阻塞，性能更好 |
| **依赖** | Netflix 生态 | Spring 生态 |
| **Spring Cloud 默认** | 2020.0.0 之前 | 2020.0.0 之后 |

### 5.2 功能对比

#### 5.2.1 负载均衡策略

**Ribbon（丰富）**：

```java
// 7种内置策略
1. RoundRobinRule           // 轮询
2. RandomRule               // 随机
3. RetryRule                // 重试
4. WeightedResponseTimeRule // 响应时间加权
5. BestAvailableRule        // 最少并发
6. AvailabilityFilteringRule // 可用过滤
7. ZoneAvoidanceRule        // 区域感知（默认）
```

**Spring Cloud LoadBalancer（精简）**：

```java
// 2种内置策略
1. RoundRobinLoadBalancer   // 轮询（默认）
2. RandomLoadBalancer       // 随机
```

**差异分析**：
- Ribbon 策略更丰富（特别是 `WeightedResponseTimeRule` 响应时间加权）
- Spring Cloud LoadBalancer 需要自定义实现复杂策略

#### 5.2.2 重试机制

**Ribbon 重试**：

```yaml
ribbon:
  MaxAutoRetries: 1           # 同一实例重试1次
  MaxAutoRetriesNextServer: 2 # 切换2个实例
  OkToRetryOnAllOperations: false  # 是否所有操作重试（危险！）
```

**Spring Cloud LoadBalancer 重试**（需引入 `spring-retry`）：

```yaml
spring:
  cloud:
    loadbalancer:
      retry:
        enabled: true
        max-retries-on-same-service-instance: 1
        max-retries-on-next-service-instance: 1
```

**差异分析**：
- Ribbon 重试配置简单，但 `OkToRetryOnAllOperations=true` 可能导致 POST 重复提交
- Spring Cloud LoadBalancer 重试需额外依赖，但更安全

#### 5.2.3 健康检查

**Ribbon**：通过 `IPing` 检查实例存活（默认 DummyPing，不检查）。

**Spring Cloud LoadBalancer**：依赖注册中心健康检查（如 Eureka 心跳、Nacos 心跳）。

**差异分析**：
- Ribbon 可主动 ping（但默认不开启）
- Spring Cloud LoadBalancer 完全依赖注册中心（更简洁）

### 5.3 迁移指南（Ribbon → Spring Cloud LoadBalancer）

**步骤1：移除 Ribbon 依赖**

```xml
<!-- 删除 -->
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-ribbon</artifactId>
</dependency>
```

**步骤2：添加 Spring Cloud LoadBalancer 依赖**

```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-loadbalancer</artifactId>
</dependency>
```

**步骤3：修改配置**

```yaml
# 删除 Ribbon 配置
# ribbon:
#   ConnectTimeout: 2000

# 添加 Spring Cloud LoadBalancer 配置
spring:
  cloud:
    loadbalancer:
      configurations: round-robin
```

**步骤4：验证功能**

启动应用，调用服务，观察日志：

```
DEBUG o.s.c.l.core.RoundRobinLoadBalancer - Chose instance 192.168.1.100:8080
```

### 5.4 如何选择？

**使用 Ribbon**：
- 老项目（已使用 Ribbon，无特别需求）
- 需要复杂的负载均衡策略（如响应时间加权）
- 暂时不想迁移

**使用 Spring Cloud LoadBalancer**：
- 新项目（Spring Cloud 2020.0.0+）
- 使用 WebFlux（响应式编程）
- 追求更好的性能
- 希望与 Spring 生态深度集成

**推荐**：**新项目使用 Spring Cloud LoadBalancer**，老项目逐步迁移。


## 六、负载均衡策略详解

### 6.1 轮询策略（Round Robin）

**原理**：按顺序依次选择实例。

```
实例列表：[A, B, C]
请求1 → A
请求2 → B
请求3 → C
请求4 → A（循环）
```

**优点**：
- 简单公平，均匀分布
- 无状态，实现简单

**缺点**：
- 无视实例性能差异（性能差的实例会拖累整体）
- 无视实例当前负载（可能把请求发给高负载实例）

**适用场景**：
- 实例性能相近
- 无状态服务

**Ribbon 实现**：

```java
public class RoundRobinRule extends AbstractLoadBalancerRule {
    
    private AtomicInteger nextServerCyclicCounter;
    
    public Server choose(ILoadBalancer lb, Object key) {
        // 自旋锁 + CAS 实现线程安全
        int count = 0;
        for (;;) {
            int current = nextServerCyclicCounter.get();
            int next = (current + 1) % allServers.size();
            if (nextServerCyclicCounter.compareAndSet(current, next)) {
                return allServers.get(next);
            }
            count++;
            if (count >= 10) {
                return null;  // 重试10次失败，返回null
            }
        }
    }
}
```

**Spring Cloud LoadBalancer 实现**：

```java
public class RoundRobinLoadBalancer implements ReactorLoadBalancer<ServiceInstance> {
    
    private final AtomicInteger position;
    
    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        ServiceInstanceListSupplier supplier = ...
        return supplier.get(request)
                .map(instances -> {
                    int pos = Math.abs(this.position.getAndIncrement());
                    return instances.get(pos % instances.size());
                });
    }
}
```

### 6.2 随机策略（Random）

**原理**：随机选择实例。

```java
// Ribbon 实现
public class RandomRule extends AbstractLoadBalancerRule {
    
    public Server choose(ILoadBalancer lb, Object key) {
        int index = ThreadLocalRandom.current().nextInt(allServers.size());
        return allServers.get(index);
    }
}
```

**优点**：
- 简单，分布均匀
- 避免"热点"（某些实例一直被轮询到）

**缺点**：
- 无视实例性能差异
- 无视实例当前负载

**适用场景**：
- 实例性能相近
- 需要简单快速的负载均衡

### 6.3 响应时间加权策略（Weighted Response Time）

**原理**：根据实例的平均响应时间计算权重，响应越快权重越高。

**Ribbon 实现**：`WeightedResponseTimeRule`

**权重计算**：

```
实例A 平均响应时间 100ms → 权重 10
实例B 平均响应时间 200ms → 权重 5
实例C 平均响应时间 500ms → 权重 2

总权重 = 10 + 5 + 2 = 17

实例A 被选中的概率 = 10 / 17 ≈ 59%
实例B 被选中的概率 = 5 / 17 ≈ 29%
实例C 被选中的概率 = 2 / 17 ≈ 12%
```

**优点**：
- 自动感知实例性能
- 性能好的实例承担更多流量

**缺点**：
- 启动阶段无统计数据，退化成轮询
- 权重计算开销（需定时更新）

**适用场景**：
- 实例性能差异较大
- 服务响应时间差异明显

### 6.4 最少并发策略（Best Available）

**原理**：选择当前并发请求最少的实例。

**Ribbon 实现**：`BestAvailableRule`

```java
public class BestAvailableRule extends ClientConfigEnabledRoundRobinRule {
    
    @Override
    public Server choose(Object key) {
        // 遍历所有实例，选择并发请求最少的
        int minConcurrent = Integer.MAX_VALUE;
        Server bestServer = null;
        for (Server server : allServers) {
            int concurrent = serverStats.getConcurrentRequests();
            if (concurrent < minConcurrent && server.isAlive()) {
                minConcurrent = concurrent;
                bestServer = server;
            }
        }
        return bestServer;
    }
}
```

**优点**：
- 动态感知实例负载
- 避免高负载实例

**缺点**：
- 需要统计每个实例的并发数（开销）
- 可能选择到性能差的实例（并发少但响应慢）

**适用场景**：
- 实例性能相近
- 需要动态感知负载

### 6.5 可用过滤策略（Availability Filtering）

**原理**：先过滤掉故障/高并发实例，再从剩余实例中轮询。

**Ribbon 实现**：`AvailabilityFilteringRule`

**过滤条件**：
1. 实例熔断（断路器打开）
2. 实例并发请求数 > 阈值（默认 2^31-1，实际无限）
3. 实例故障（连接失败）

```java
public class AvailabilityFilteringRule extends PredicateBasedRule {
    
    @Override
    public boolean apply(Server server) {
        ServerStats stats = lbStats.getSingleServerStat(server);
        // 过滤条件1：实例熔断
        if (stats.isCircuitBreakerTripped()) {
            return false;
        }
        // 过滤条件2：并发请求数 > 阈值
        if (stats.getActiveRequestsCount() >= server.getMaxTotalHttpConnections()) {
            return false;
        }
        return true;
    }
}
```

**优点**：
- 自动剔除故障实例
- 提升系统可用性

**缺点**：
- 过滤后可能只剩少量实例（甚至0个）
- 阈值配置需调优

**适用场景**：
- 实例可能故障
- 需要高可用保障

### 6.6 区域感知策略（Zone Avoidance）

**原理**：先过滤掉不可用区域，再从可用区域中轮询选择实例。

**Ribbon 默认策略**：`ZoneAvoidanceRule`

**区域感知**：
- 实例按区域（Zone）分组（如 `us-east-1a`、`us-west-2b`）
- 优先选择同区域的实例（减少跨区延迟）
- 如果同区域实例故障，才跨区调用

**优点**：
- 减少跨区延迟（同区域调用）
- 提升容灾能力（跨区故障切换）

**缺点**：
- 需要实例配置区域信息
- 区域划分需合理

**适用场景**：
- 多区域部署（如 AWS 多 AZ）
- 跨区延迟明显

### 7.1 自定义 Ribbon 负载均衡策略

#### 7.1.1 实现 IRule 接口

```java
// 自定义策略：加权随机（根据实例元数据中的权重）
public class WeightedRandomRule extends AbstractLoadBalancerRule {
    
    private final Random random = new Random();
    
    @Override
    public void initWithNiwsConfig(IClientConfig clientConfig) {
        // 初始化配置（可选）
    }
    
    @Override
    public Server choose(Object key) {
        ILoadBalancer lb = getLoadBalancer();
        List<Server> servers = lb.getReachableServers();
        
        if (servers.isEmpty()) {
            return null;
        }
        
        // 读取实例元数据中的权重（从 Eureka metadata 或 Nacos metadata）
        Map<Server, Integer> weightMap = new HashMap<>();
        int totalWeight = 0;
        for (Server server : servers) {
            int weight = getWeightFromMetadata(server);  // 自定义方法
            weightMap.put(server, weight);
            totalWeight += weight;
        }
        
        // 加权随机选择
        int randomWeight = random.nextInt(totalWeight);
        int current = 0;
        for (Map.Entry<Server, Integer> entry : weightMap.entrySet()) {
            current += entry.getValue();
            if (current > randomWeight) {
                return entry.getKey();
            }
        }
        
        return servers.get(0);  // fallback
    }
    
    private int getWeightFromMetadata(Server server) {
        // 从 Eureka metadata 读取 "weight" 配置
        // 默认权重 1
        String weightStr = ((DiscoveryEnabledServer) server)
                .getInstanceInfo()
                .getMetadata()
                .get("weight");
        return weightStr == null ? 1 : Integer.parseInt(weightStr);
    }
}
```

```java
    }
}
```

### 7.2 自定义 Spring Cloud LoadBalancer 负载均衡策略

#### 7.2.1 实现 ReactorLoadBalancer 接口

```java
// 自定义策略：灰度发布（根据请求头中的版本号路由）
public class GrayReleaseLoadBalancer implements ReactorLoadBalancer<ServiceInstance> {
    
    private final String serviceId;
    private final ServiceInstanceListSupplier supplier;
    
    public GrayReleaseLoadBalancer(String serviceId, ServiceInstanceListSupplier supplier) {
        this.serviceId = serviceId;
        this.supplier = supplier;
    }
    
    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        // 从 Request 中提取请求头（灰度版本号）
        String grayVersion = null;
        if (request instanceof DefaultRequest) {
            Object context = ((DefaultRequest<?>) request).getContext();
            if (context instanceof HttpRequest) {
                HttpRequest httpRequest = (HttpRequest) context;
                grayVersion = httpRequest.getHeaders().getFirst("X-Version");
            }
        }
        
        final String finalGrayVersion = grayVersion;
        
        return supplier.get()
                .map(instances -> {
                    // 过滤出版本匹配的实例
                    List<ServiceInstance> matchedInstances = instances.stream()
                            .filter(instance -> {
                                String instanceVersion = instance.getMetadata().get("version");
                                if (finalGrayVersion == null) {
                                    // 无版本头，选择默认版本（无 version 标签或 version=stable）
                                    return instanceVersion == null || "stable".equals(instanceVersion);
                                } else {
                                    // 有版本头，选择匹配版本
                                    return finalGrayVersion.equals(instanceVersion);
                                }
                            })
                            .collect(Collectors.toList());
                    
                    if (matchedInstances.isEmpty()) {
                        return new EmptyResponse();  // 无可用实例
                    }
                    
                    // 从匹配的实例中随机选择一个
                    int index = ThreadLocalRandom.current().nextInt(matchedInstances.size());
                    return new DefaultResponse(matchedInstances.get(index));
                });
    }
}
```

#### 7.2.2 配置自定义策略

```java

    public ReactorLoadBalancer<ServiceInstance> grayReleaseLoadBalancer(
            Environment environment,

            serviceId,
            factory.getLazyProvider(serviceId, ServiceInstanceListSupplier.class)
        );
    }
}
```

```java
    }
}
```

## 八、面试高频问题

### 8.1 Ribbon 相关

#### Q1: Ribbon 的负载均衡策略有哪些？

**答**：Ribbon 内置 7 种策略：

1. **RoundRobinRule**：轮询（默认 ZoneAvoidanceRule 内部使用）
2. **RandomRule**：随机
3. **RetryRule**：重试（先按轮询，失败重试其他实例）
4. **WeightedResponseTimeRule**：响应时间加权
5. **BestAvailableRule**：最少并发请求
6. **AvailabilityFilteringRule**：可用实例过滤 + 轮询
7. **ZoneAvoidanceRule**：区域感知（默认策略）

**记忆口诀**：
> 轮询随机重试三大将，
> 响应时间加权重，
> 最少并发可用过滤，
> 区域感知是默认。

#### Q2: Ribbon 的重试机制是什么？

**答**：Ribbon 重试分两层：

1. **同一实例重试**（`MaxAutoRetries`）：对同一个实例重试 N 次
2. **切换实例重试**（`MaxAutoRetriesNextServer`）：切换 N 个不同实例

**配置示例**：

```yaml
ribbon:
  MaxAutoRetries: 1           # 同实例重试1次
  MaxAutoRetriesNextServer: 2 # 切换2个实例
```

**注意**：
- `OkToRetryOnAllOperations=true` 时，POST 请求也会重试，可能导致**重复提交**！
- 生产环境建议 `OkToRetryOnAllOperations=false`，仅 GET 请求重试。

#### Q3: @LoadBalanced 注解的作用是什么？

**答**：
- `@LoadBalanced` 给 `RestTemplate` 添加拦截器 `LoadBalancerInterceptor`
- 拦截器从 URL 中提取**服务名**（如 `user-service`）
- 通过 Ribbon/Spring Cloud LoadBalancer 获取该服务的实例列表
- 负载均衡算法选择一个实例
- 将服务名替换为 `IP:Port`，发起 HTTP 请求

**源码流程**：

```
RestTemplate 发起请求
    ↓
LoadBalancerInterceptor 拦截
    ↓
从 URL 提取服务名（user-service）
    ↓
LoadBalancerClient.choose(serviceId) 选择实例
    ↓
将 URL 中的服务名替换为 IP:Port
    ↓
发起真实 HTTP 请求
```

### 8.2 Spring Cloud LoadBalancer 相关

#### Q4: Spring Cloud LoadBalancer 与 Ribbon 的区别？

**答**：

| 对比项 | Ribbon | Spring Cloud LoadBalancer |
|--------|--------|----------------------------|
| 维护状态 | 维护模式 | 活跃开发 |
| 响应式支持 | 不支持 | 支持（WebFlux） |
| 负载均衡策略 | 7种 | 2种（轮询、随机） |
| 性能 | 阻塞式 | 非阻塞（Reactor） |
| Spring Cloud 默认 | 2020.0.0 之前 | 2020.0.0 之后 |

### 8.3 负载均衡原理相关

#### Q6: 客户端负载均衡 vs 服务端负载均衡？

**答**：

| 对比项 | 客户端负载均衡 | 服务端负载均衡 |
|--------|----------------|----------------|
| 代表 | Ribbon、Spring Cloud LoadBalancer | Nginx、HAProxy、F5 |
| 位置 | 客户端内部 | 独立服务器/硬件 |
| 单点故障 | 无（客户端直连） | 有（需主备） |
| 网络跳转 | 无（直连） | 有（多一次跳转） |
| 灵活性 | 高（客户端可定制） | 低（集中配置） |

**微服务推荐**：客户端负载均衡（避免单点故障）。

#### Q7: 负载均衡算法有哪些？

**答**：
1. **轮询（Round Robin）**：按顺序依次选择
2. **随机（Random）**：随机选择
3. **加权轮询（Weighted Round Robin）**：根据权重轮询
4. **加权随机（Weighted Random）**：根据权重随机
5. **最小连接数（Least Connections）**：选择并发最少的实例
6. **源 IP 哈希（IP Hash）**：同一 IP 总是路由到同一实例（会话保持）
7. **响应时间加权（Response Time Weighted）**：响应越快，权重越高

#### Q8: 如何实现灰度发布（Gray Release）？

**答**：
灰度发布的核心是根据**请求特征**（如请求头、用户 ID）将部分流量路由到新版本。

**实现方案1：自定义 Spring Cloud LoadBalancer**

```java
public class GrayReleaseLoadBalancer implements ReactorLoadBalancer<ServiceInstance> {
    
    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        // 1. 从 Request 中提取灰度标记（如请求头 X-Version）
        // 2. 过滤出匹配版本的服务实例
        // 3. 从匹配实例中随机选择一个
    }
}
```

**实现方案2：服务网格（Service Mesh）**

使用 Istio、Linkerd 等服务网格，通过 **Virtual Service** 和 **Destination Rule** 配置流量分配：

```yaml
# Istio VirtualService 配置
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: user-service
spec:
  hosts:
    - user-service
  http:
  - match:
    - headers:
        x-version:
          exact: gray
    route:
    - destination:
        host: user-service
        subset: gray
  - route:
    - destination:
        host: user-service
        subset: stable
```

**推荐**：小团队用自定义负载均衡，大团队用服务网格。


## 九、总结

本文深度解析了微服务架构中的负载均衡技术，核心知识点如下：

### 9.1 核心知识点回顾

1. **负载均衡类型**：
   - 服务端负载均衡（Nginx、F5）
   - 客户端负载均衡（Ribbon、Spring Cloud LoadBalancer）✅ 微服务推荐

2. **Ribbon**（维护模式）：
   - 7种负载均衡策略（轮询、随机、响应时间加权等）
   - 重试机制（`MaxAutoRetries`、`MaxAutoRetriesNextServer`）
   - 与 `@LoadBalanced` + `RestTemplate` 集成

3. **Spring Cloud LoadBalancer**（官方推荐）：
   - 2种内置策略（轮询、随机）
   - 支持响应式编程（WebFlux）
   - 非阻塞，性能更好

4. **Ribbon vs Spring Cloud LoadBalancer**：
   - 新项目用 Spring Cloud LoadBalancer
   - 老项目可逐步迁移

5. **自定义负载均衡**：
   - Ribbon：实现 `IRule` 接口
   - Spring Cloud LoadBalancer：实现 `ReactorLoadBalancer` 接口
   - 实战案例：灰度发布（Gray Release）

6. **面试高频问题**：
   - Ribbon 7种策略
   - `@LoadBalanced` 原理
   - 客户端 vs 服务端负载均衡
   - 灰度发布实现方案

### 9.2 负载均衡策略选择指南

| 场景 | 推荐策略 |
|------|----------|
| 实例性能相近 | 轮询（Round Robin） |
| 实例性能差异大 | 响应时间加权（WeightedResponseTime） |
| 需要快速实现 | 随机（Random） |
| 需要高可用 | 可用过滤（Availability Filtering） |
| 多区域部署 | 区域感知（Zone Avoidance） |

### 9.3 最佳实践

1. **新项目使用 Spring Cloud LoadBalancer**（官方推荐，性能更好）
2. **合理配置重试**（`OkToRetryOnAllOperations=false`，避免 POST 重复提交）
3. **自定义策略要谨慎**（确保线程安全、性能达标）
4. **结合注册中心健康检查**（自动剔除故障实例）
5. **监控负载均衡效果**（通过 Metrics 监控各实例请求分布）

### 9.4 核心记忆口诀

> 负载均衡两类型，
> 客户端式最流行。
> Ribbon 七策维护中，
> Spring 官方推新宠。
> 轮询随机响应加权，
> 最少并发可用过滤。
> 自定义策要实现接口，
> 灰度发布是实战。
