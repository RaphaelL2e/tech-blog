# AI大模型面试八股文（二）——大模型训练与微调技术

> 面试高频问题：大模型预训练怎么做？SFT和RLHF有什么区别？LoRA为什么能高效微调？PPO和DPO怎么选？本文带你深入理解大模型从预训练到对齐的完整流程。

## 引言

上一篇我们理解了Transformer架构——大模型的"骨架"。但一个强大的大模型，光有架构远远不够。从预训练的海量数据喂养，到指令微调的"教化"，再到RLHF的"价值观对齐"，每一步都至关重要。理解训练流程，是理解大模型能力边界的关键。

**本文要点**：
1. 预训练：数据、目标与工程
2. 有监督微调（SFT）：让模型听懂指令
3. RLHF：人类反馈强化学习
4. DPO：直接偏好优化
5. 参数高效微调：LoRA/QLoRA/Adapter
6. 面试高频问题精讲

## 一、预训练（Pre-training）

### 1.1 预训练的目标

预训练是大模型训练的第一阶段，核心目标是：**让模型学会语言的统计规律和世界知识**。

```
预训练目标：下一个token预测（Causal Language Modeling）

给定: "今天天气很"
预测: "好" / "差" / "热" / ...

数学形式：
maximize Π P(x_t | x_1, x_2, ..., x_{t-1}; θ)

θ: 模型参数
整个训练就是在最大化"预测下一个token"的概率
```

### 1.2 预训练数据

| 数据类型 | 典型来源 | 规模 | 用途 |
|---------|---------|------|------|
| 网页文本 | CommonCrawl | 数十TB | 通用知识 |
| 书籍 | Books3等 | 数百GB | 深度知识 |
| 代码 | GitHub | 数百GB | 逻辑推理 |
| 学术论文 | arXiv | 数十GB | 专业知识 |
| 百科 | Wikipedia | 数十GB | 事实知识 |
| 对话 | Reddit/论坛 | 数百GB | 对话能力 |

**数据处理流水线**：

```
原始数据 → 去重 → 过滤 → 清洗 → 分词 → 预训练数据

1. 去重：MinHash/LSH去重，去除重复文档
2. 过滤：语言检测、质量评分、毒性过滤
3. 清洗：去除HTML标签、模板文本、乱码
4. 分词：BPE/SentencePiece分词
5. 混合：按比例混合不同来源数据
```

### 1.3 关键训练超参数

| 参数 | 典型值 | 说明 |
|------|--------|------|
| 学习率 | 1e-4 ~ 3e-4 | 预训练峰值 |
| LR调度 | Cosine decay | 先warmup再余弦衰减 |
| Batch size | 数百万token | 梯度累积实现 |
| 训练步数 | 100K ~ 1M步 | 取决于数据量 |
| 权重衰减 | 0.1 | 防止过拟合 |
| 梯度裁剪 | 1.0 | 防止梯度爆炸 |
| 序列长度 | 4096 ~ 8192 | 逐渐增长策略 |

### 1.4 预训练的工程挑战

```
挑战1: 显存不够
方案: 
  - 数据并行 (Data Parallelism)
  - 张量并行 (Tensor Parallelism, Megatron-LM)
  - 流水线并行 (Pipeline Parallelism)
  - ZeRO优化 (DeepSpeed ZeRO-1/2/3)

挑战2: 训练不稳定
方案:
  - 梯度裁剪
  - 学习率warmup
  - 损失尖峰监控与重启
  - BF16混合精度（比FP16更稳定）

挑战3: 训练效率
方案:
  - Flash Attention（IO优化）
  - FSDP（全分片数据并行）
  - 序列并行（Sequence Parallelism）
```

### 1.5 预训练 vs 继续预训练

```
预训练 (Pre-training):
  从随机初始化开始，用海量数据训练
  → 成本极高（百万~千万美元级）

继续预训练 (Continual Pre-training):
  在已有预训练模型基础上，用领域数据继续训练
  → 成本较低，适合垂直领域
  → 注意：数据混合比例很重要，防止灾难性遗忘

典型场景：
  - 医疗领域：混入医疗文献继续预训练
  - 法律领域：混入法律法规和案例
  - 金融领域：混入研报和财报
```

## 二、有监督微调（SFT）

### 2.1 为什么需要SFT？

```
预训练模型：学会了"续写"，但不理解"指令"

用户: "请帮我写一首关于春天的诗"
模型: "请帮我写一首关于夏天的诗" ← 续写模式

SFT后模型：
用户: "请帮我写一首关于春天的诗"
模型: "春风拂面花满枝..." ← 指令遵循模式

核心变化：从"续写"到"对话"的能力转换
```

### 2.2 SFT数据构造

```
SFT数据格式：

{
  "instruction": "将以下句子翻译成英文",
  "input": "今天天气很好",
  "output": "The weather is nice today"
}

或更简单的对话格式：

[
  {"role": "user", "content": "解释什么是量子计算"},
  {"role": "assistant", "content": "量子计算是利用量子力学原理..."}
]
```

**SFT数据来源**：

| 来源 | 特点 | 代表 |
|------|------|------|
| 人工标注 | 质量高，成本高 | OpenAI的RLHF数据 |
| 模型生成 | 规模大，需过滤 | Alpaca (GPT-3.5生成) |
| 开源数据 | 免费可用 | ShareGPT, Open-Assistant |
| 自我指令 | 自动化，需验证 | Self-Instruct |

### 2.3 SFT训练细节

```python
# SFT训练核心代码（伪代码）

for batch in sft_dataloader:
    # 只对assistant的回复计算loss
    input_ids = batch["input_ids"]
    labels = input_ids.clone()
    
    # 将user部分的label设为-100（忽略）
    labels[user_positions] = -100
    
    # 前向传播
    logits = model(input_ids)
    
    # 计算交叉熵损失（只在assistant部分）
    loss = cross_entropy(logits, labels, ignore_index=-100)
    
    # 反向传播
    loss.backward()
    optimizer.step()
```

**关键要点**：
- **Loss Mask**：只对模型的回复部分计算损失，不对指令部分计算
- **学习率**：比预训练小1-2个数量级（1e-5 ~ 5e-5）
- **Epoch**：通常2-3轮，过多容易过拟合
- **数据质量 > 数据数量**：几万条高质量数据 > 百万条低质量数据

### 2.4 SFT常见问题

```
问题1: 过拟合
  症状：模型死记硬背训练数据，泛化差
  解决：减少epoch、增加数据多样性、正则化

问题2: 灾难性遗忘
  症状：微调后丧失预训练能力
  解决：学习率小、数据混合（混入通用数据）

问题3: 指令遵循不稳定
  症状：有时遵循指令，有时不
  解决：增加指令多样性、数据去重

问题4: 输出格式不统一
  症状：同一指令输出格式不一致
  解决：统一输出格式模板、增加格式示例
```

## 三、RLHF：人类反馈强化学习

### 3.1 为什么需要RLHF？

```
SFT的问题：
  - 模型学会了"模仿"人类回复
  - 但不知道什么是"好的"回复
  - 可能生成有害、偏见、不准确的内容

RLHF的目标：
  - 让模型对齐人类的偏好
  - 有帮助 (Helpful)
  - 无害 (Harmless) 
  - 诚实 (Honest)

类比：
  SFT = 教小孩说话（模仿）
  RLHF = 教小孩什么该说什么不该说（价值观）
```

### 3.2 RLHF三步流程

```
Step 1: SFT模型（已有）
  ↓
Step 2: 训练奖励模型 (Reward Model)
  ↓  
Step 3: 用强化学习优化 (PPO)
```

### 3.3 奖励模型训练

```
数据格式：人类偏好排序

prompt: "解释量子计算"
response_A: "量子计算利用量子叠加..."  ← 人类偏好
response_B: "量子计算就是很快的计算机"  ← 人类不偏好

训练目标：
  maximize σ(r(prompt, A) - r(prompt, B))

r(·): 奖励模型，对回复打分
σ: sigmoid函数
目标：让好回复的得分高于差回复
```

```python
# 奖励模型训练（伪代码）

def train_reward_model(rm, preference_data):
    for batch in preference_data:
        prompt = batch["prompt"]
        chosen = batch["chosen"]      # 人类偏好的回复
        rejected = batch["rejected"]  # 人类不偏好的回复
        
        # 奖励模型打分
        score_chosen = rm(prompt, chosen)
        score_rejected = rm(prompt, rejected)
        
        # Bradley-Terry偏好损失
        loss = -log(sigmoid(score_chosen - score_rejected))
        
        loss.backward()
        optimizer.step()
```

### 3.4 PPO强化学习

```
PPO (Proximal Policy Optimization) 流程：

1. SFT模型生成多个回复
2. 奖励模型对每个回复打分
3. 用PPO更新SFT模型参数

关键约束：不能偏离SFT模型太远
→ KL散度惩罚项

总奖励 = 奖励模型分数 - β × KL(当前策略 || SFT策略)
                        ↑
              防止奖励黑客（reward hacking）
```

```
PPO训练示意：

SFT模型 (Actor)          奖励模型 (Critic)
    │                        │
    ├─ 生成回复 ───────────→ 打分
    │                        │
    ├─ 计算优势函数 ←────────┤
    │   A(s,a) = R(s,a) - V(s)
    │                        │
    ├─ PPO裁剪目标           │
    │   L = min(r·A, clip(r,1-ε,1+ε)·A)
    │                        │
    └─ 更新Actor参数         │
```

### 3.5 RLHF的问题

```
问题1: 奖励黑客 (Reward Hacking)
  模型学会"钻空子"，生成高分但不好的回复
  解决：KL惩罚、对抗性数据收集

问题2: 训练不稳定
  PPO有4个模型在跑（Actor, Critic, RM, Ref）
  解决：仔细调参、梯度裁剪

问题3: 人类标注成本高
  偏好标注需要大量人工
  解决：AI辅助标注（RLAIF）

问题4: 价值观不一致
  不同标注者偏好不同
  解决：标注规范、多人投票
```

## 四、DPO：直接偏好优化

### 4.1 DPO的核心思想

```
RLHF的痛点：需要训练奖励模型 + PPO训练复杂

DPO的洞察：
  可以直接从偏好数据优化策略模型，
  不需要显式训练奖励模型！

数学推导：
  RLHF的目标 = max E[r(x,y)] - β·KL(π || π_ref)
  
  最优策略的闭式解：
  r(x,y) = β · log(π*(y|x) / π_ref(y|x)) + const
  
  代入偏好损失，得到DPO损失：
  L_DPO = -E[log σ(β · log(π(y_w|y)/π_ref(y_w|x)) 
                    - β · log(π(y_l|x)/π_ref(y_l|x)))]
  
  y_w: 偏好回复, y_l: 不偏好回复
```

### 4.2 DPO vs RLHF对比

| 维度 | RLHF | DPO |
|------|------|-----|
| 奖励模型 | 需要单独训练 | 不需要 |
| 训练复杂度 | 高（PPO+4个模型） | 低（直接优化） |
| 训练稳定性 | 不稳定 | 稳定 |
| 数据需求 | 偏好数据 | 偏好数据 |
| 计算成本 | 高 | 低 |
| 效果 | 成熟，上限高 | 接近RLHF |
| 代表模型 | ChatGPT, Claude | LLaMA-2-Chat, Qwen |

### 4.3 DPO变体

```
IPO (Identity Preference Optimization):
  用更鲁棒的损失替代Bradley-Terry假设
  解决DPO对偏好数据的过拟合问题

KTO (Kahneman-Tversky Optimization):
  只需要"好/坏"标签，不需要成对偏好
  数据收集更简单

ORPO (Odds Ratio Preference Optimization):
  将SFT和偏好优化合二为一
  一步完成指令微调和对齐

SimPO (Simple Preference Optimization):
  去掉参考模型，用序列长度归一化
  更简单、更高效
```

### 4.4 实际选择建议

```
选RLHF：
  - 大规模生产环境
  - 需要精细控制对齐效果
  - 有充足的计算和标注资源

选DPO：
  - 研究和实验阶段
  - 计算资源有限
  - 快速迭代需求

选KTO/SimPO：
  - 只有二元反馈数据
  - 追求最简实现
```

## 五、参数高效微调（PEFT）

### 5.1 为什么需要PEFT？

```
全参数微调的问题：
  - 7B模型需要约28GB显存（仅参数）
  - 加上梯度、优化器状态，需要100GB+
  - 每个任务保存一份完整模型 → 存储爆炸

PEFT的解决思路：
  - 冻结大部分预训练参数
  - 只训练少量新增参数
  - 效果接近全参数微调
```

### 5.2 LoRA（Low-Rank Adaptation）

```
核心思想：权重更新是低秩的

原始: Y = X × W     W ∈ R^(d×d)

LoRA: Y = X × (W + ΔW)
           ΔW = A × B
           A ∈ R^(d×r), B ∈ R^(r×d)
           r << d (通常r=4~64)

参数量对比：
  全参数：d × d = 4096 × 4096 ≈ 16M
  LoRA(r=16)：d × r × 2 = 4096 × 16 × 2 ≈ 131K
  压缩比：~120倍
```

```
LoRA架构示意：

           输入X
          ┌──┴──┐
          │     │
     ┌────▼─┐ ┌─▼────┐
     │ 原始W │ │  A×B │
     │(冻结) │ │(训练) │
     └────┬─┘ └─┬────┘
          │     │
          └──┬──┘
             │
          ┌──▼──┐
          │  +   │ ← 两条路径相加
          └──┬──┘
             │
           输出Y
```

**LoRA关键配置**：

| 参数 | 含义 | 典型值 |
|------|------|--------|
| r | 秩 | 8, 16, 32, 64 |
| alpha | 缩放因子 | 16, 32 |
| dropout | 正则化 | 0.05 |
| target_modules | 应用LoRA的层 | q_proj, v_proj, k_proj, o_proj |

```
alpha/r 的比值决定LoRA的有效学习率缩放：
  缩放因子 = alpha / r
  
  例：r=16, alpha=16 → 缩放=1（无缩放）
      r=16, alpha=32 → 缩放=2（LoRA更新放大2倍）
```

### 5.3 QLoRA

```
QLoRA = 量化 + LoRA

步骤：
1. 将预训练模型4-bit量化（NF4格式）
2. 冻结量化后的模型
3. 在量化模型上添加LoRA适配器
4. 只训练LoRA参数

显存对比（7B模型）：
  全参数微调：~120GB
  LoRA：~40GB
  QLoRA：~10GB（4-bit量化）

关键创新：
  - NF4量化：正态分布最优4-bit格式
  - 双重量化：量化常数本身也量化
  - 分页优化器：CPU/GPU内存分页防OOM
```

### 5.4 其他PEFT方法

```
Adapter:
  在Transformer层中插入小模块
  ┌─────────────┐
  │ Attention   │
  └──────┬──────┘
  ┌──────▼──────┐
  │ Adapter ↓   │ ← 下投影 + 非线性 + 上投影
  └──────┬──────┘
  ┌──────▼──────┐
  │ FFN         │
  └──────┬──────┘
  ┌──────▼──────┐
  │ Adapter ↓   │
  └──────┬──────┘
  缺点：增加推理延迟

Prefix-Tuning:
  在每层前添加可训练的虚拟token前缀
  推理无额外延迟
  但占用上下文长度

Prompt Tuning:
  只在输入层添加可训练向量
  最简单，但效果较弱

P-Tuning v2:
  在每层添加可训练向量
  效果接近全参数微调
```

### 5.5 PEFT方法对比

| 方法 | 可训练参数 | 推理开销 | 效果 | 适用场景 |
|------|-----------|---------|------|---------|
| LoRA | ~0.1-1% | 极小 | 好 | 通用推荐 |
| QLoRA | ~0.1-1% | 极小 | 好 | 显存受限 |
| Adapter | ~1-5% | 有延迟 | 好 | 多任务切换 |
| Prefix-Tuning | ~0.1% | 占上下文 | 中等 | 生成任务 |
| Prompt Tuning | ~0.01% | 无 | 较弱 | 极简场景 |

## 六、大模型训练全流程

### 6.1 训练流水线

```
┌─────────────┐
│  预训练       │ ← 海量数据，学习语言和知识
│  (Pre-train)  │    耗时：数周到数月
└──────┬──────┘
       │
┌──────▼──────┐
│  SFT         │ ← 指令数据，学会对话
│  (监督微调)   │    耗时：数小时到数天
└──────┬──────┘
       │
┌──────▼──────┐
│  RLHF/DPO    │ ← 偏好数据，对齐价值观
│  (对齐)       │    耗时：数天
└──────┬──────┘
       │
┌──────▼──────┐
│  部署上线     │ ← 量化、推理优化
└─────────────┘
```

### 6.2 各阶段数据量

| 阶段 | 数据量 | 数据特点 |
|------|--------|---------|
| 预训练 | 1-15T tokens | 通用、大规模、自动获取 |
| SFT | 10K-1M条 | 高质量、人工标注或筛选 |
| RLHF/DPO | 10K-500K对 | 偏好排序、人工标注 |

### 6.3 实际案例：LLaMA 2训练

```
LLaMA 2-Chat的训练路径：

1. 预训练：2T tokens, 公开数据
   ↓
2. SFT：27,540条指令数据
   ↓
3. RLHF迭代：
   - 第1轮：100K偏好对 → 训练RM → PPO
   - 第2轮：收集更多偏好对 → 更新RM → PPO
   - 第3轮：继续迭代...
   ↓
4. 安全对齐：额外的安全偏好数据
   ↓
5. LLaMA 2-Chat发布

关键发现：
  - RLHF迭代轮次越多，模型越好
  - 偏好数据的质量比数量更重要
  - 安全是持续迭代的过程
```

## 七、面试高频问题

### Q1: 预训练和SFT有什么区别？

| 维度 | 预训练 | SFT |
|------|--------|-----|
| 目标 | 学习语言统计规律 | 学习指令遵循 |
| 数据量 | T级别 | 万~百万条 |
| 学习率 | 1e-4 | 1e-5 |
| 损失函数 | 全序列CLM | 只对回复部分 |
| 数据来源 | 自动获取 | 人工标注/筛选 |
| 训练时间 | 周~月 | 小时~天 |

### Q2: LoRA为什么有效？

```
1. 内在低秩假设：
   权重更新矩阵ΔW本身就是低秩的
   → 大参数空间中的变化只发生在低维子空间

2. 实验验证：
   研究发现ΔW的有效秩通常只有几十到几百
   → 用r=16~64就能捕获大部分变化

3. 正则化效果：
   低秩约束天然防止过拟合
   → 参数少=泛化好

4. 实践证据：
   大量实验表明LoRA在多数任务上接近全参数微调
   → 不是理论近似，而是实践中确实有效
```

### Q3: RLHF和DPO怎么选？

```
选RLHF的场景：
  - 超大规模模型（70B+），需要精细控制
  - 在线学习场景（模型持续从反馈中学习）
  - 复杂的多目标对齐

选DPO的场景：
  - 中小规模模型（7B-70B）
  - 离线偏好数据充足
  - 迭代速度快比极致效果重要

趋势：DPO及其变体正在成为主流
  - 实现简单
  - 训练稳定
  - 效果接近RLHF
```

### Q4: 什么是灾难性遗忘？怎么解决？

```
定义：模型在学习新任务时，忘记了旧任务的知识

原因：
  - 微调时新数据的梯度覆盖了旧知识的权重
  - 参数空间被新任务"占据"

解决方案：

1. 数据混合：微调时混入预训练数据
   ratio = 95%微调数据 + 5%预训练数据

2. 学习率控制：使用小学习率
   全参数：1e-5
   LoRA：5e-4~1e-3（因为只训练少量参数）

3. LoRA/Adapter：冻结原始参数
   新增参数不覆盖旧知识

4. EWC (Elastic Weight Consolidation)：
   对重要参数施加正则化约束

5. 渐进式学习：逐步增加新任务难度
```

### Q5: 如何评估微调后的模型？

```
评估维度：

1. 能力评估
   - 通用能力：MMLU, C-Eval, AGI-Eval
   - 代码能力：HumanEval, MBPP
   - 数学能力：GSM8K, MATH
   - 推理能力：BBH, ARC

2. 对齐评估
   - 有用性：AlpacaEval, MT-Bench
   - 安全性：TruthfulQA, ToxiGen

3. 人类评估
   - 盲测对比（A/B测试）
   - 专家评审

4. 自动评估
   - GPT-4作为裁判
   - 奖励模型打分

关键：自动评估做初筛，人类评估做最终确认
```

## 八、总结

**本文核心知识**：

| 模块 | 关键要点 | 面试重点 |
|------|---------|---------|
| 预训练 | CLM目标、数据工程、分布式训练 | 数据处理流水线 |
| SFT | 指令微调、Loss Mask、质量>数量 | 只对回复部分计算Loss |
| RLHF | RM训练 + PPO优化 | 奖励黑客问题 |
| DPO | 直接偏好优化，无需RM | DPO vs RLHF |
| LoRA | 低秩分解ΔW=AB | r和alpha的含义 |
| QLoRA | 4-bit量化+LoRA | NF4量化格式 |

**核心记忆口诀**：

1. **训练三步曲**：预训练学知识 → SFT学指令 → RLHF学对齐
2. **LoRA精髓**：低秩分解，以小博大
3. **DPO优势**：无需奖励模型，直接优化偏好
4. **遗忘对策**：数据混合 + 小学习率 + LoRA
5. **质量为王**：微调数据质量 > 数量
