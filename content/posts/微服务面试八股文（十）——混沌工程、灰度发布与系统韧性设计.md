## 一、混沌工程：主动发现系统脆弱性

### 1.1 为什么需要混沌工程？

```
传统测试 vs 混沌工程：

传统测试：
  单元测试 → 函数逻辑正确
  集成测试 → 模块间接口正确
  端到端测试 → 全链路功能正确
  问题：都是在「正常路径」上验证，不测故障场景

混沌工程：
  目标：验证系统在「故障发生时」的行为是否正确
  方法：主动在生产/预发环境注入真实故障
  核心问题：分布式系统在部分组件故障时，整体表现如何？

经典案例：Netflix 的 Chaos Monkey
  → 随机关闭生产环境的 EC2 实例
  → 迫使工程师设计出「容单点故障」的系统
  → 最终演进为全套 Resilience Engineering 工具链
```

### 1.2 混沌工程实施五步法

```
Step 1：定义稳态假设
  "假设 MySQL 从库宕机，订单服务的读取成功率仍然 > 99.9%"

Step 2：设计故障实验
  注入：Kill 掉 MySQL 从库进程
  观测：订单服务读取成功率是否下降

Step 3：在低风险环境先验证
  先在 Staging 环境运行，发现问题再优化

Step 4：执行实验，逐步加大故障范围
  单机 → 单服务 → 多服务 → 核心服务

Step 5：自动化，持续运行
  纳入 CI/CD，自动化执行 + 结果报告

实验黄金法则：
  ⚠️ 故障注入必须有「Abort Condition」
  ⚠️ 影响范围必须有「上限」
  ⚠️ 任何时候「人工干预」可以立即停止实验
```

### 1.4 FMEA 故障模式与影响分析

```
FMEA（Failure Mode and Effects Analysis）：

目的：系统性地识别系统中所有可能的故障点，评估影响，制定应对策略

FMEA 表格结构：
┌──────────┬──────────────┬────────┬─────────┬──────────┬──────────────┐
│ 故障点   │ 故障模式      │ 严重度 │ 发生概率│ 检测难度│ 风险优先级RPN│
├──────────┼──────────────┼────────┼─────────┼──────────┼──────────────┤
│ MySQL主库│ 宕机         │ S=9   │ O=3    │ D=2     │ RPN=54      │
│ MySQL主库│ 响应慢(>5s)  │ S=7   │ O=4    │ D=3     │ RPN=84      │
│ Redis缓存│ 缓存穿透     │ S=8   │ O=5    │ D=4     │ RPN=160     │
│ 订单服务 │ OOM崩溃      │ S=9   │ O=2    │ D=2     │ RPN=36      │
│ API网关  │ 限流触发     │ S=6   │ O=7    │ D=1     │ RPN=42      │
└──────────┴──────────────┴────────┴─────────┴──────────┴──────────────┘

RPN = 严重度(S) × 发生概率(O) × 检测难度(D)

RPN 应对策略：
  RPN > 100：必须立即处理，优化设计或增加监控
  RPN 50-100：需要处理，制定应对方案
  RPN < 50：可接受，但需持续监控

常见故障模式及应对策略：

MySQL 主库宕机：
  → 影响：写入全部失败
  → 应对：主从自动切换（MHA/Orchestrator/Altas）
  → 监控：心跳检测 + 延迟告警

缓存穿透：
  → 影响：数据库被打满，所有请求变慢
  → 应对：布隆过滤器 + 空值缓存 + 限流
  → 监控：缓存命中率 < 80% 告警

服务雪崩：
  → 影响：级联失败，整站不可用
  → 应对：熔断器（Sentinel/Hystrix）+ 超时控制
  → 监控：5xx 错误率 > 1% 立即告警
```

### 2.3 金丝雀部署：渐进式流量迁移

```
核心思想：新版本只接收一小部分流量（10%），验证通过后再逐步放大

Istio 流量管理（金丝雀发布最佳实践）：

Step 1：部署新版本（金丝雀）
kubectl apply -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service-canary
spec:
  replicas: 1    # 金丝雀只部署 1 个副本（占总流量 10%）
  selector:
    matchLabels:
      app: order-service
      version: v2.0.0
template:
  metadata:
    labels:
      app: order-service
      version: v2.0.0
EOF

Step 2：Istio 流量规则（控制流量分配）
apiVersion: networking.istio.io/v1alpha3
kind: VirtualService
metadata:
  name: order-service
spec:
  hosts:
  - order-service
  http:
  - route:
    - destination:
        host: order-service
        subset: v1         # 稳定版本，接收 90% 流量
      weight: 90
    - destination:
        host: order-service
        subset: canary     # 金丝雀版本，接收 10% 流量
      weight: 10

---
apiVersion: networking.istio.io/v1alpha3
kind: DestinationRule
metadata:
  name: order-service
spec:
  host: order-service
  subsets:
  - name: v1
    labels:
      version: v1.0.0
  - name: canary
    labels:
      version: v2.0.0

Step 3：观察指标（Prometheus + Grafana）
  关注：错误率、延迟、P99、核心业务指标
  如果：金丝雀指标正常 → 逐步提升到 30% → 50% → 100%
  如果：异常 → 自动/手动回滚

Step 4：全量切换
  将流量 100% 切换到新版本，旧版本缩容

渐进式流量分配策略：
  10% → 30% → 50% → 80% → 100%
  每个阶段观察 5-30 分钟（根据流量规模调整）
```

## 三、SRE 实战：SLO/SLI/SLA 与错误预算

### 3.1 核心概念三角

```
SLO（Service Level Objective）—— 目标
  "我们的服务每月可用性目标是 99.95%"
  → 内部承诺的「目标」，用于指导工程决策

SLI（Service Level Indicator）—— 指标
  "可用性 = 成功请求数 / 总请求数 × 100%"
  → 实际可测量的指标

SLA（Service Level Agreement）—— 协议
  "服务可用性保证 99.9%，否则赔偿"
  → 对外承诺，与客户合同绑定，通常比 SLO 更宽松

关系图：
┌─────────────────────────────────────────────┐
│           SLA（对外承诺，99.9%）              │
│  ┌───────────────────────────────────────┐  │
│  │         SLO（内部目标，99.95%）        │  │
│  │  ┌─────────────────────────────────┐  │  │
│  │  │     SLI（实际测量值，99.97%）   │  │  │
│  │  └─────────────────────────────────┘  │  │
│  └───────────────────────────────────────┘  │
│                                             │
│  SLA > SLO > 实际测量值（安全边际）          │
└─────────────────────────────────────────────┘

为什么要「安全边际」？
  • SLA 99.9% = 每月 43.8 分钟允许停机
  • 如果工程师以 SLA 为目标，迟早会违约
  • SLO 设置为 99.95%，预留 21 分钟安全边际
  • 错误预算 = 21 分钟/月 = 收到告警 + 立即处理的缓冲时间
```

### 3.2 四大黄金指标与告警设置

```
Google SRE 四大黄金指标（RED/USE 方法）：

Request 指标（适用于有请求的服务，如 API）：
  • Rate（请求率）：QPS，每秒请求数
  • Error（错误率）：5xx / 总请求数 × 100%
  • Duration（延迟）：P50/P90/P99/P999 响应时间

Resource 指标（适用于基础设施，如数据库/缓存）：
  • Utilization（利用率）：CPU/内存/磁盘使用率
  • Saturation（饱和度）：队列深度、线程池饱和度
  • Errors（错误数）

告警设置原则（USE + RED 告警）：

┌─────────────────────────────────────────────────────────────┐
│ 基于 SLO 的告警（推荐）：                                      │
│ "过去 30 分钟错误率是否可能导致本月 SLO 违约？"                 │
│                                                              │
│ 简单计算：                                                    │
│ 月度错误预算 = (1 - SLO) × 30天 × 每天总请求数               │
│ 当前允许错误数 = 错误预算 × (已过时间占比)                    │
│ 如果当前错误数 > 当前允许错误数 → 告警                        │
│                                                              │
│ 即：基于「消耗了多少错误预算」来告警，而不是「错误率是否超过阈值」│
│ 优点：凌晨流量低时，1% 错误率很严重                          │
│      高峰期流量高时，0.5% 错误率可接受                       │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 错误预算计算实战

```python
# 错误预算计算器
class SLOBudgetCalculator:
    def __init__(self, slo: float, period_days: int):
        self.slo = slo
        self.period_days = period_days
        self.error_budget_pct = (1 - slo) * 100  # 允许的不可用时间百分比
    
    def monthly_budget(self):
        """月度错误预算（分钟）"""
        minutes_per_month = self.period_days * 24 * 60
        return minutes_per_month * (1 - self.slo)
    
    def daily_budget(self):
        """每日错误预算（分钟）"""
        return self.monthly_budget() / self.period_days
    
    def check_alert(self, consumed_budget_pct: float):
        """检查错误预算消耗是否触发告警"""
        if consumed_budget_pct >= 100:
            return "CRITICAL: 错误预算已耗尽，立即处理！"
        elif consumed_budget_pct >= 50:
            return "WARNING: 错误预算已消耗 > 50%，需要关注"
        else:
            return "HEALTHY: 错误预算充足"
    
    def remaining_budget_pct(self, elapsed_days: float) -> float:
        """当前剩余错误预算百分比"""
        if elapsed_days <= 0:
            return 100.0
        total_budget = self.monthly_budget()
        daily_rate = total_budget / self.period_days
        # 按时间比例允许消耗的错误预算
        allowed_budget = daily_rate * elapsed_days
        # 实际消耗（简化，实际用 SLI 数据）
        # 返回剩余比例
        return max(0, (1 - elapsed_days / self.period_days) * 100)


# 示例：SLO 99.95% = 月度错误预算 21.6 分钟
calc = SLOBudgetCalculator(slo=0.9995, period_days=30)
print(f"月度错误预算: {calc.monthly_budget():.1f} 分钟")   # 21.6 分钟
print(f"每日错误预算: {calc.daily_budget():.1f} 分钟")     # 0.72 分钟
print(calc.check_alert(55))  # WARNING

# 告警策略：
# PagerDuty / Alertmanager 规则（基于 Prometheus）：
# groups:
# - name: slo-error-budget
#   rules:
#   - alert: SLOErrorBudgetExhausted
#     expr: |
#       1 - (
#         sum(rate(http_requests_total{job="order-service",code=~"2.."}[5m]))
#         /
#         sum(rate(http_requests_total{job="order-service"}[5m]))
#       ) > (1 - 0.9995) * 0.5  # 消耗超过 50% 的月度预算
#     labels:
#       severity: critical
#     annotations:
#       summary: "SLO 错误预算消耗过半"
```

## 四、容量规划与压测实战

### 4.1 容量规划方法论

```
容量规划核心公式：

目标容量 = 峰值 QPS × 单请求资源消耗 × 冗余系数

步骤：
1. 确定基准指标：单实例 QPS 上限、CPU/内存基准
2. 计算当前容量：实例数 × 单实例上限
3. 规划扩容阈值：容量使用率 > 70% 触发扩容
4. 计算扩容成本：成本 = 实例数 × 单价 × 时间

容量规划决策树：
┌──────────────────────────────────────────────┐
│  当前容量使用率 < 60%                        │
│  → 暂不扩容，持续监控                        │
├──────────────────────────────────────────────┤
│  当前容量使用率 60%-80%                      │
│  → 规划扩容，预计 1-2 周内完成               │
├──────────────────────────────────────────────┤
│  当前容量使用率 > 80%                       │
```

## 五、面试高频问题汇总

**Q1：混沌工程和故障演练有什么区别？**

混沌工程是系统性的、持续性的工程实践，有完整的假设→实验→观察→学习循环，目的是发现系统未知弱点；故障演练通常是预定义的应急演练脚本，验证已知故障场景的应对流程。混沌工程强调「主动发现未知风险」，故障演练强调「验证已知流程有效性」。Netflix 的 Chaos Monkey 是混沌工程的鼻祖，而国内很多企业的「故障演练」更偏向于后者。

**Q2：如何设计一个安全的灰度发布流程？**

核心原则：**灰度发布必须可观测、可中断、可回滚**。

安全流程：
1. Staging 验证：新版本在 Staging 完整测试通过
2. 流量隔离：使用命名空间/标签隔离灰度流量
3. 渐进放量：10% → 30% → 50% → 100%，每阶段观察 5-30 分钟
4. 核心指标监控：错误率、延迟、P99、核心业务转化率
5. 自动回滚：指标异常时自动切回旧版本（Istio/Argo Rollout 内置）
6. 人工介入点：每个阶段前人工确认（灰度期间保持 oncall）

**Q3：SLO 应该如何设置？**

设置 SLO 的原则：
1. **基于历史数据**：分析过去 3-6 个月的实际可用性
2. **与 SLA 保持距离**：SLO 比 SLA 严格 10-50%
3. **与业务影响挂钩**：核心链路（SLO 99.99%）vs 非核心链路（SLO 99.5%）
4. **可测量**：选择明确的 SLI 指标（错误率、延迟）

常见 SLO 参考：
```
搜索服务：99.9%（月度错误预算 44 分钟）
核心下单：99.99%（月度错误预算 4.4 分钟）
推荐系统：99%（月度错误预算 7.3 小时）
日志服务：99%（月度错误预算 7.3 小时）
```

**Q4：金丝雀发布与滚动发布的区别？**

滚动发布（Rolling Update）是 Kubernetes Deployment 默认策略，逐步替换旧 Pod 为新 Pod，每次替换比例可配置，但新旧 Pod 混合存在一段时间，期间流量分配不可控。

金丝雀发布的流量分配由 Istio EnvoySidecar / nginx-ingress 等控制，可以精确控制百分比（10% 金丝雀，90% 稳定版），支持基于用户特征（用户 ID、地区、Cookie）的差异化灰度，且支持实时动态调整流量比例。滚动发布适合日常迭代，金丝雀发布适合重要变更和高风险特性。
