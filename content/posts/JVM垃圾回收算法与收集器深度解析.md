# JVM 垃圾回收算法与收集器深度解析

> 面试高频问题：GC 有哪些算法？CMS 和 G1 有什么区别？ZGC 为什么能做到几乎不停顿？

## 引言

Java 与 C++ 的核心区别之一是内存管理。在 C++ 中，程序员需要手动分配和释放内存；而在 Java 中，垃圾回收器（Garbage Collector）自动完成这项工作。理解 GC 算法和收集器是 Java 面试的必考知识点。本文将从基础算法到高级收集器，全面解析 JVM 的垃圾回收机制。

## 一、为什么要垃圾回收？

### 1.1 内存泄漏与溢出

**内存泄漏（Memory Leak）**：程序申请内存后，无法释放已申请的内存空间，导致内存浪费。

```java
// 内存泄漏示例：集合类持有对象引用
List<Object> list = new ArrayList<>();
while (true) {
    list.add(new Object());  // 只添加不清理，内存持续增长
}
```

**内存溢出（OutOfMemoryError）**：程序申请内存时，没有足够的内存空间供其使用。

### 1.2 GC 的价值

- **自动管理内存**：程序员无需手动释放，减少内存泄漏
- **提高开发效率**：减少 dangling pointer、double free 等错误
- **安全性提升**：防止缓冲区溢出等安全漏洞

## 二、对象存活判断

GC 之前需要判断哪些对象是"死去"的（不再被使用），只有"死去"的对象才能被回收。

### 2.1 引用计数法

给对象添加引用计数器，有引用就 +1，引用失效就 -1。计数器为 0 的对象可回收。

```
┌─────────┐
│ Object  │
│  count=3│ ← 被 3 个引用持有
└─────────┘
```

**优点**：实现简单，判定效率高
**缺点**：无法解决循环引用问题

```java
// 循环引用示例
Object a = new Object();  // a count=1
Object b = new Object();  // b count=1
a.ref = b;               // a count=1, b count=2
b.ref = a;               // a count=2, b count=2

// a = null 后，a count=1, b count=2（无法回收）
a = null;
b = null;
// 此时 a 和 b 相互引用，但已经无法访问
// 引用计数法无法回收，JVM 未采用
```

> 💡 **面试题**：Python 采用引用计数，为什么 Java 不采用？
>
> **答**：引用计数无法处理循环引用，且每次引用变化都需要更新计数器，有性能开销。Java 采用可达性分析。

### 2.2 可达性分析（JVM 采用）

通过一系列 "GC Roots" 对象作为起点，向下搜索，形成"引用链"。不在引用链上的对象可回收。

```
        GC Root
            │
    ┌───────┼───────┐
    │       │       │
    ▼       ▼       ▼
 ┌─────┐ ┌─────┐ ┌─────┐
 │  A  │ │  B  │ │  C  │
 └──┬──┘ └──┬──┘ └──┬──┘
    │       │       │
    ▼       │       ▼
 ┌─────┐    │    ┌─────┐
 │  D  │    │    │  E  │
 └──┬──┘    │    └─────┘
    │       │       ↑
    ▼       │       │
 ┌─────┐    │       └── 可达，但无引用链
 │  F  │ ← ┘           (假设 F 无外部引用)
 └─────┘               F 不可达，可回收
```

**GC Roots 对象包括**：

| 类型 | 示例 |
|------|------|
| 虚拟机栈中引用的对象 | 方法中的局部变量、参数 |
| 方法区中类静态属性引用的对象 | static Object obj |
| 方法区中常量引用的对象 | final Object CONST |
| 本地方法栈中 JNI 引用的对象 | native 方法中的对象 |
| JVM 内部引用 | ClassLoader、异常对象 |
| 同步锁持有的对象 | synchronized(obj) 中的 obj |

### 2.3 四种引用类型

| 引用类型 | 回收时机 | 场景 |
|---------|---------|------|
| **强引用** | 从不回收 | `Object obj = new Object()` |
| **软引用** | 内存不足时回收 | 缓存 |
| **弱引用** | 下次 GC 时回收 | 缓存、Map 引用 |
| **虚引用** | 随时可能回收 | 对象跟踪、堆外内存管理 |

```java
// 强引用
Object strong = new Object();  // 永远不会 GC

// 软引用
SoftReference<Object> soft = new SoftReference<>(new Object());
// 内存不足时回收

// 弱引用
WeakReference<Object> weak = new WeakReference<>(new Object());
// 下次 GC 时回收

// 虚引用（必须和 ReferenceQueue 配合）
ReferenceQueue<Object> queue = new ReferenceQueue<>();
PhantomReference<Object> phantom = new PhantomReference<>(new Object(), queue);
```

## 三、垃圾回收算法

### 3.1 标记-清除算法（Mark-Sweep）

**步骤**：
1. 标记：找出所有需要回收的对象并标记
2. 清除：统一回收被标记的对象

```
标记前：
┌────────────────────────────┐
│  可用  │ 已标记 │  可用   │
│  内存  │ (回收) │  内存   │
└────────────────────────────┘

标记后：
┌────────────────────────────┐
│  可用  │  清除   │  可用   │
│  内存  │ (空)   │  内存   │
└────────────────────────────┘
```

**缺点**：
- **效率问题**：标记和清除都需要遍历所有对象
- **空间问题**：产生大量不连续内存碎片

```
内存碎片问题：
┌──┐┌────┐┌──┐┌──────┐┌┐┌────┐┌──┐
│  ││    ││  ││      ││││    ││  │
└──┘└────┘└──┘└──────┘└┘└────┘└──┘
 ↑ 碎片   ↑ 碎片      ↑ 碎片
```

### 3.2 复制算法（Copying）

将内存分为两块，每次只使用一块。GC 时将存活对象复制到另一块，然后清空原块。

```
原区域：                      目标区域：
┌────────────────────┐        ┌────────────────────┐
│ A │ B │ C │   D   │        │        (空)         │
└────────────────────┘        └────────────────────┘
  ↑存活    ↑存活    ↑已死

复制后：
┌────────────────────┐        ┌────────────────────┐
│        (空)        │        │  A  │  B  │  C   │
└────────────────────┘        └────────────────────┘
```

**优点**：
- 无内存碎片
- 实现简单，运行高效

**缺点**：
- 浪费一半内存
- 如果存活对象多，复制成本高

**应用**：新生代 Eden:Survivor:Survivor = 8:1:1

### 3.3 标记-整理算法（Mark-Compact）

在标记-清除基础上，增加整理步骤，让存活对象向一端移动。

```
整理前：
┌────┬────────┬──┬────┬────┬──┐
│存活│  已死   │存活│已死│存活│死│
└────┴────────┴──┴────┴────┴──┘
      ↑碎片                   ↑碎片

整理后：
┌────────────────────────────┬──┐
│存活│存活│存活│存活│存活│存活│空│
└────────────────────────────┴──┘
                         ↑连续内存
```

**优点**：无内存碎片，充分利用内存
**缺点**：整理过程需要移动对象，有性能开销

**应用**：老年代（存活对象多，不适合复制算法）

### 3.4 分代收集算法（Generational Collection）

根据对象存活周期将内存分为新生代和老年代，不同区域采用不同算法。

```
┌─────────────────────────────────────┐
│              堆内存                   │
├─────────────────┬───────────────────┤
│     新生代       │      老年代        │
│   (Young Gen)   │   (Old/Tenured)   │
│                 │                   │
│ ┌───┬───┬───┐  │                   │
│ │Eden│ S0│ S1│  │                   │
│ └───┴───┴───┘  │                   │
│  8    1   1   │                   │
└─────────────────┴───────────────────┘
```

| 区域 | 对象特征 | 算法 | GC 频率 |
|------|---------|------|--------|
| 新生代 | 朝生夕死 | 复制算法 | 高 |
| 老年代 | 长寿对象 | 标记-整理 | 低 |

### 3.5 三种 GC 类型

| 类型 | 发生区域 | 触发条件 |
|------|---------|---------|
| **Minor GC** | 新生代 | Eden 满 |
| **Major GC** | 老年代 | 老年代满 |
| **Full GC** | 整堆 | 多种条件触发 |

```
触发 Full GC 的情况：
1. 老年代空间不足
2. System.gc()（不保证执行）
3. MetaSpace 空间不足
4. Minor GC 前的检查失败
```

## 四、垃圾收集器

JVM 提供多种垃圾收集器，不同收集器适用于不同场景。

```
┌─────────────────────────────────────────────────────────────┐
│                     垃圾收集器                               │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  新生代收集器              │        老年代收集器            │
│  ┌─────────┐               │        ┌─────────┐              │
│  │ Serial  │ ───────────────┼─────── │ Serial  │              │
│  └─────────┘               │        └─────────┘              │
│  ┌─────────┐               │        ┌─────────┐              │
│  │ParNew   │ ───────────────┼─────── │ CMS     │              │
│  └─────────┘               │        └─────────┘              │
│  ┌─────────┐               │        ┌─────────┐              │
│  │Parallel │ ──────────────┼─────── │Parallel │              │
│  │Scavenge │               │        │Old      │              │
│  └─────────┘               │        └─────────┘              │
│                            │        ┌─────────┐              │
│                            ├─────── │  ZGC    │              │
│                            │        └─────────┘              │
│                            │        ┌─────────┐              │
│                            ├─────── │ G1      │              │
│                            │        └─────────┘              │
│                            │        ┌─────────┐              │
│                            ├─────── │ Shenandoah │           │
└────────────────────────────┴────────┴─────────┘              │
```

### 4.1 Serial 收集器

最古老、最简单的收集器，单线程执行 GC。

```bash
# 启用
-XX:+UseSerialGC

# 工作模式
新生代：Serial（复制算法）
老年代：Serial Old（标记-整理）
```

**特点**：
- 单线程，Stop The World
- 简单高效，没有线程切换开销
- 适用于桌面应用（内存小，GC 时间可接受）

### 4.2 ParNew 收集器

Serial 的多线程版本，平行收集。

```bash
# 启用
-XX:+UseParNewGC

# 工作模式
新生代：ParNew（复制算法，多线程）
老年代：Serial Old 或 CMS
```

**特点**：
- 多线程并行收集
- 默认线程数 = CPU 核心数
- **只能与 CMS 配合**（JDK 9 已移除 ParNew + Serial Old 组合）

### 4.3 Parallel Scavenge 收集器

专注于吞吐量（Throughput）的收集器。

```bash
# 启用
-XX:+UseParallelGC

# 参数配置
-XX:MaxGCPauseMillis=<ms>   # 最大 GC 停顿时间（目标，不保证）
-XX:GCTimeRatio=<ratio>     # GC 时间占总时间比例（默认 99，即 1%）
-XX:+UseAdaptiveSizePolicy   # 自适应调节（自动调整 Eden/Survivor 比例）
```

**特点**：
- 吞吐量优先
- 自适应调节
- JDK 8 默认新生代收集器

### 4.4 Serial Old 收集器

Serial 的老年代版本，使用标记-整理算法。

### 4.5 Parallel Old 收集器

Parallel Scavenge 的老年代版本，使用标记-整理算法。

```bash
# 启用
-XX:+UseParallelOldGC

# 组合
新生代：Parallel Scavenge
老年代：Parallel Old
```

**JDK 8 默认组合**：
```bash
-XX:+UseParallelGC  # 等价于 Parallel Scavenge + Parallel Old
```

### 4.6 CMS 收集器（Concurrent Mark Sweep）

以最短停顿时间为目标的收集器。

```bash
# 启用
-XX:+UseConcMarkSweepGC

# 组合
新生代：ParNew
老年代：CMS
```

**工作过程**：

```
1. 初始标记（Initial Mark）
   └─ Stop The World
   └─ 标记 GC Roots 直接关联的对象
   
2. 并发标记（Concurrent Mark）
   └─ 与用户线程并发执行
   └─ 遍历引用链，标记所有存活对象
   
3. 重新标记（Remark）
   └─ Stop The World
   └─ 修正并发标记期间变动的对象
   
4. 并发清除（Concurrent Sweep）
   └─ 与用户线程并发执行
   └─ 清除死亡对象
```

```
时间轴：
──┬────────────────┬──────┬────────────────────┬──
  │  Initial Mark   │      │       Remark        │
  ▼                 ▼      ▼                     ▼
 STW               STW    STW                    运行
  │                 │      │                    │
 └─ 并发标记 ───────┘      └─ 并发清除 ──────────┘
```

**优点**：
- 并发收集，低停顿
- 适合对响应时间有要求的应用

**缺点**：
- CPU 敏感（占用 CPU 资源）
- 无法处理浮动垃圾
- 产生内存碎片（标记-清除算法）

```
CMS 缺点详解：

1. CPU 敏感
   └─ 并发阶段占用 25% CPU
   └─ 4 核机器，用户程序只能用 3 核

2. 浮动垃圾
   └─ 并发标记期间用户线程产生的垃圾
   └─ 本次无法回收，只能等下次

3. 内存碎片
   └─ 标记-清除算法，不整理
   └─ Full GC 时进行碎片整理（Serial Old）
   └─ 导致长时间停顿
```

**CMS 触发条件**：
```bash
# 老年代空间使用率达到阈值
-XX:CMSInitiatingOccupancyFraction=<percent>  # 默认 92%

# 老年代空间不足
-XX:+UseCMSInitiatingOccupancyOnly              # 只用阈值判断
```

### 4.7 G1 收集器（Garbage First）

JDK 7 引入，JDK 9 成为默认收集器。

**核心思想**：将堆划分为多个大小相等的 Region，按回收收益优先回收。

```bash
# 启用
-XX:+UseG1GC

# 参数配置
-XX:MaxGCPauseMillis=<ms>   # 最大停顿时间（目标）
-XX:G1HeapRegionSize=<size>  # Region 大小（1MB-32MB，必须是 2 的幂）
```

**Region 划分**：

```
┌────────────────────────────────────────────────────────────┐
│                        堆内存                               │
│  ┌────┐┌────┐┌────┐┌────┐┌────┐┌────┐┌────┐┌────┐┌────┐ │
│  │ Eden ││ Eden ││ S0 ││ S1 ││ Old ││ Old ││ H  ││    │ │
│  └────┘└────┘└────┘└────┘└────┘└────┘└────┘└────┘└────┘ │
│         新生代              │  老年代  │  │ Humongous│ 空  │
│           │                │          │  │  (大对象)│     │
│           └────────────────┴──────────┴──┴──────────┘     │
└────────────────────────────────────────────────────────────┘
```

**G1 的优势**：

| 特性 | Serial/ParNew/CMS | G1 |
|------|------------------|-----|
| 内存管理 | 新生代/老年代固定 | Region 动态划分 |
| 碎片问题 | 有（除 G1 外） | 整理减少碎片 |
| 停顿模型 | 整堆停顿 | 可预测停顿 |
| 吞吐量 | 较低 | 较高 |

**G1 工作过程**：

```
1. 初始标记（Initial Mark）
   └─ Stop The World
   └─ 标记 GC Roots，直接关联对象

2. 并发标记（Concurrent Mark）
   └─ 与用户线程并发
   └─ 遍历引用链

3. 最终标记（Final Mark）
   └─ Stop The World
   └─ 处理 SATB 缓冲区

4. 筛选回收（Live Data Counting & Evacuation）
   └─ Stop The World
   └─ 按回收价值和成本排序
   └─ 回收选定 Region
```

```
G1 vs CMS 对比：

CMS：
┌──────────────────────────────────────────┐
│  初始标记 │ 并发标记 │ 重新标记 │ 并发清除 │
└──────────────────────────────────────────┘
      ↑              ↑              ↑
     STW            STW            并发

G1：
┌──────────────────────────────────────────────────┐
│ 初始标记 │ 并发标记 │ 最终标记 │ 筛选回收 │
└──────────────────────────────────────────────────┘
     ↑            ↑            ↑        ↑
    STW          并发         STW      STW
                              (可预测停顿)
```

**适用场景**：
- 堆内存 > 6GB
- 需要可控停顿时间
- 应用注重响应速度

### 4.8 ZGC（Z Garbage Collector）

JDK 11 引入，JDK 15 成为正式特性。目标：停顿时间 < 1ms，停顿时间不随堆大小增加。

```bash
# 启用
-XX:+UseZGC

# JDK 21+ 推荐配置
-XX:+UseZGC -Xmx16g
```

**核心技术**：

1. **着色指针（Colored Pointers）**
   - 利用 64 位指针的低 4-5 位存储标记信息
   - 不需要 Stop The World 就能访问对象状态

2. **读屏障（Load Barrier）**
   - 读取对象时检查指针状态
   - 触发时快速处理（几乎无感知）

3. **并发处理**
   - 大部分阶段与用户线程并发执行
   - 只有短暂的初始化和最终标记阶段需要 STW

**ZGC vs G1 对比**：

| 特性 | G1 | ZGC |
|------|----|----|
| 最大停顿 | < 200ms | < 1ms |
| 停顿来源 | Root 扫描、Card 扫描 | 初始化、最终标记 |
| 内存占用 | 高 | 中 |
| 吞吐量 | 高 | 中 |
| 堆大小支持 | 数十 GB | 16TB+ |
| GC 阶段 | 4 阶段 | 7 阶段（大部分并发）|

```
ZGC 停顿时间：
┌────────────────────────────────────────────────────────┐
│ ZGC                                                      │
│ ├─ 初始标记   : ~1ms                                     │
│ ├─ 再标记     : ~1ms                                     │
│ ├─ 初始转移   : ~1ms                                     │
│ └─ 再转移     : ~1ms                                     │
│                                                          │
│ 全程停顿时间 < 10ms，停顿不随堆大小线性增长                │
└────────────────────────────────────────────────────────┘
```

### 4.9 Shenandoah（红岩）

Red Hat 开发的低停顿收集器，与 ZGC 类似，但非 Oracle 官方维护。

```bash
# 启用（JDK 12+）
-XX:+UseShenandoahGC
```

**与 ZGC 的区别**：
- ZGC 使用着色指针
- Shenandoah 使用转发指针（Brooks Pointer）

### 4.10 收集器选择指南

```
┌────────────────────────────────────────────────────────┐
│                   收集器选择决策树                       │
└────────────────────────────────────────────────────────┘

内存 < 100MB？
  └─ 是 → Serial（简单高效）

需要高吞吐量？
  └─ 是 → Parallel Scavenge + Parallel Old

对停顿时间敏感？
  ├─ 堆 < 6GB → CMS
  ├─ 堆 6-16GB → G1
  └─ 堆 > 16GB → ZGC / Shenandoah

JDK 11+，追求最低停顿？
  └─ ZGC

不确定？
  └─ G1（大多数场景的良好选择）
```

## 五、JVM 参数配置

### 5.1 常用 GC 参数

```bash
# 通用参数
-Xms512m              # 初始堆大小
-Xmx2g                # 最大堆大小
-Xmn256m              # 新生代大小
-XX:MetaspaceSize=256m   # 元空间大小
-XX:MaxMetaspaceSize=512m

# 新生代比例
-XX:NewRatio=2        # 老年代/新生代 = 2:1（默认）
-XX:SurvivorRatio=8   # Eden/Survivor = 8:1（默认）

# 日志参数
-XX:+PrintGCDetails   # 打印 GC 详情
-XX:+PrintGCDateStamps   # 打印时间戳
-Xloggc:gc.log       # GC 日志输出到文件

# 常用收集器组合
-XX:+UseSerialGC         # Serial + Serial Old
-XX:+UseParNewGC         # ParNew + CMS（已废弃）
-XX:+UseParallelGC        # Parallel Scavenge + Parallel Old（默认）
-XX:+UseConcMarkSweepGC   # ParNew + CMS（已废弃）
-XX:+UseG1GC             # G1
-XX:+UseZGC              # ZGC
```

## 六、面试高频问题

### Q1：Minor GC 和 Full GC 有什么区别？

**答**：
- **Minor GC**：发生在新生代，Eden 区满时触发，频率高，停顿时间短
- **Full GC**：发生在老年代或整堆，条件更多（老年代满、显式调用等），停顿时间长

### Q2：对象进入老年代的条件？

**答**：
1. **年龄达标**：经历过 N 次 Minor GC（默认 15 次，可配置 `-XX:MaxTenuringThreshold`）
2. **大对象**：对象大小超过 `-XX:PretenureSizeThreshold`（只对 Serial/ParNew 生效）
3. **动态年龄判断**：Survivor 中相同年龄对象总和 > Survivor 的 50%
4. **空间担保失败**：Minor GC 前的检查

### Q3：CMS 和 G1 的区别？

**答**：
| 特性 | CMS | G1 |
|------|-----|-----|
| 算法 | 标记-清除 | 标记-整理 |
| 内存管理 | 新生代/老年代固定 | Region 动态划分 |
| 停顿模型 | 不可预测 | 可预测 |
| 碎片问题 | 有 | 整理减少碎片 |
| 适用堆大小 | < 6GB | > 6GB |

### Q4：ZGC 为什么能做到低停顿？

**答**：
1. **着色指针**：利用指针位存储 GC 状态，读取时通过读屏障快速处理
2. **并发执行**：标记、转移等阶段与用户线程并发
3. **最短停顿**：只保留必要的 STW 阶段（初始化、最终标记）
4. **不整理**：采用染色整理而非移动对象

### Q5：如何减少 GC 停顿时间？

**答**：
1. **选择合适的收集器**：G1/ZGC 适合大堆
2. **合理设置停顿目标**：`MaxGCPauseMillis`
3. **减少对象分配**：对象池、复用、栈上分配
4. **减少大对象**：避免创建大数组
5. **优化代码**：避免频繁创建对象

### Q6：什么情况下会触发 Full GC？

**答**：
1. 老年代空间不足
2. System.gc()（可被禁用 `-XX:+DisableExplicitGC`）
3. MetaSpace 空间不足
4. Minor GC 前的检查失败
5. CMS 收集失败（Concurrent Mode Failure）
6. RMI 定时触发

## 七、实战案例

### 案例 1：频繁 Full GC

**现象**：应用响应缓慢，GC 日志显示频繁 Full GC

**排查步骤**：

```bash
# 1. 查看 GC 日志
-XX:+PrintGCDetails -Xloggc:gc.log

# 2. 使用工具分析
java -jar gcviewer.jar gc.log
```

**常见原因**：
- 内存泄漏
- 大对象分配
- 字符串常量池膨胀
- 元空间不足

**解决方案**：
```bash
# 增加堆大小
-Xmx4g -Xms4g

# 优化 CMS
-XX:CMSInitiatingOccupancyFraction=70

# 使用 G1
-XX:+UseG1GC -XX:MaxGCPauseMillis=200
```

### 案例 2：GC 时间过长

**排查**：
```bash
# 查看停顿时间
grep "real=" gc.log | awk '{sum+=$2; count++} END {print "平均停顿:", sum/count "s"}'
```

**优化方向**：
- 减少对象分配
- 使用大 Survivor 区
- 调整晋升阈值

## 八、总结

**GC 核心知识**：
- 对象存活判断：引用计数 vs 可达性分析
- GC 算法：标记-清除、复制、标记-整理、分代收集
- 收集器选择：根据场景选择合适的收集器

**收集器对比**：

| 收集器 | 停顿时间 | 吞吐量 | 适用场景 |
|--------|---------|--------|---------|
| Serial | 长 | 低 | 桌面应用 |
| Parallel | 长 | 高 | 后台批处理 |
| CMS | 短 | 中 | Web 应用 |
| G1 | 可控 | 高 | 大规模应用 |
| ZGC | < 1ms | 高 | 超大堆应用 |

**最佳实践**：
1. 了解应用特性（吞吐量 vs 响应时间）
2. 选择合适的收集器
3. 合理设置 JVM 参数
4. 持续监控 GC 状态
5. 根据实际情况调优
