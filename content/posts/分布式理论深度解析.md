# 分布式理论深度解析

## 一、引言

分布式系统是现代后端架构的基石。无论是微服务、云原生，还是大数据处理，都离不开分布式理论的支持。CAP 定理、BASE 理论、Paxos 和 Raft 共识算法是面试中的高频考点，也是理解分布式系统设计的基础。

**本文要点**：
1. 分布式系统的基本概念与挑战
2. CAP 定理：一致性、可用性、分区容错性
3. BASE 理论：基本可用、软状态、最终一致性
4. Paxos 共识算法：理论推导与工程实践
5. Raft 共识算法：易懂的共识协议
6. 一致性模型：强一致、最终一致、因果一致
7. 面试高频问题与最佳实践

---

## 二、分布式系统基础

### 2.1 什么是分布式系统

**定义**：分布式系统是由多台独立计算机通过网络协同工作的系统，对外表现为一个整体。

**核心特征**：
- **多节点**：多台机器协同工作
- **网络通信**：节点之间通过消息传递协调
- **无共享内存**：每个节点有自己的内存和存储
- **部分故障**：任意节点可能随时故障

**为什么要用分布式系统**：
1. **可扩展性**：单机性能有上限，分布式可以水平扩展
2. **高可用**：单点故障不影响整体服务
3. **高性能**：并行处理，提升吞吐
4. **地理分布**：就近服务，降低延迟

### 2.2 分布式系统的挑战

| 挑战 | 描述 | 影响 |
|------|------|------|
| **网络不可靠** | 消息延迟、丢失、乱序 | 数据不一致 |
| **时钟不同步** | 各节点时钟有偏差 | 事件排序困难 |
| **节点故障** | Crash、Byzantine 故障 | 服务不可用 |
| **并发冲突** | 多节点同时写同一数据 | 数据冲突 |
| **分区** | 网络分区导致节点隔离 | 脑裂问题 |

**FLP 不可能定理**（Fischer-Lynch-Paterson）：

> 在异步系统中，即使只有一个进程可能失败，也不存在一个确定性算法能保证达成共识。

**含义**：分布式共识问题没有完美解法，只能在一定条件下实现。这就是 CAP 定理的理论基础。

## 三、CAP 定理

### 3.1 CAP 三个属性

**CAP 定理**（Brewer's Theorem，2000 年提出，2002 年被证明）：

一个分布式系统最多只能同时满足以下三个属性中的两个：

| 属性 | 英文 | 描述 |
|------|------|------|
| **一致性** (Consistency) | 所有节点在同一时刻看到相同的数据 | 等价于线性一致性 |
| **可用性** (Availability) | 每个请求都能在有限时间内得到非错误的响应 | 不保证最新数据 |
| **分区容错性** (Partition tolerance) | 网络分区发生时，系统仍能继续运行 | 现实中必须保证 |

### 3.2 为什么只能三选二

**网络分区是客观现实**，无法避免。所以在 P 必须保证的前提下，实际选择是 **CP 或 AP**。

**CP（一致性 + 分区容错）**：
- 分区发生时，牺牲可用性，保证一致性
- 不一致的节点停止服务，返回错误
- 典型系统：ZooKeeper、etcd、HBase

**AP（可用性 + 分区容错）**：
- 分区发生时，牺牲一致性，保证可用性
- 各节点继续服务，但可能返回旧数据
- 典型系统：Cassandra、DynamoDB、Eureka

**CA（一致性 + 可用性）**：
- 不允许分区，即单机系统
- 典型系统：单机 MySQL、PostgreSQL
- 在分布式环境下不现实

### 3.3 CAP 的选择场景

```
                分区发生时
                    ↓
          ┌─────────┴─────────┐
          ↓                   ↓
     优先一致性           优先可用性
          ↓                   ↓
      CP 系统              AP 系统
          ↓                   ↓
   停止不一致节点       继续服务，返回旧数据
          ↓                   ↓
   金融/支付/库存       社交/内容/搜索
```

**场景举例**：

**金融系统 → CP**：
```
用户 A 转账 100 元给用户 B
→ 分区发生时，宁可拒绝交易（不可用）
→ 也不能让 A 和 B 余额不一致（一致性）
```

**社交系统 → AP**：
```
用户发一条微博
→ 分区发生时，继续允许发帖（可用性）
→ 其他节点暂时看不到新微博（不一致）
→ 分区恢复后，数据最终同步（最终一致性）
```

### 3.4 CAP 的常见误解

**误解一：CAP 是非此即彼的选择**

> 实际上：CAP 只在**分区发生时**才需要选择。正常运行时，系统可以同时满足 C 和 A。

**误解二：AP 系统不需要一致性**

> 实际上：AP 系统在分区恢复后会同步数据，实现**最终一致性**。不是完全不要一致性，而是允许短暂不一致。

**误解三：CP 系统完全不可用**

> 实际上：CP 系统只是**分区期间不一致的节点不可用**，多数节点仍然可用。

## 四、BASE 理论

### 4.1 BASE 是什么

**BASE** 是对 CAP 中 AP 方向的延伸，是对大规模分布式系统的实践总结。

| 缩写 | 英文 | 中文 | 描述 |
|------|------|------|------|
| **BA** | Basically Available | 基本可用 | 系统在故障时，允许响应时间增加或功能降级 |
| **S** | Soft State | 软状态 | 允许系统中的数据存在中间状态 |
| **E** | Eventually Consistent | 最终一致性 | 系统保证在没有新更新后，数据最终达到一致 |

### 4.2 基本可用（Basically Available）

**不是"完全可用"**，而是在故障时提供**降级服务**。

**示例**：
```
正常：搜索响应时间 100ms，返回 100 条结果
故障时：搜索响应时间 2s，返回 10 条结果（降级）
```

**降级策略**：
1. **响应时间降级**：增加超时时间
2. **功能降级**：关闭非核心功能
3. **数据降级**：返回缓存数据或默认值

### 4.3 软状态（Soft State）

**允许数据存在中间状态**，即数据副本之间可以暂时不一致。

**示例**：
```
MySQL 主从复制：
  主库写入 → 数据已更新
  从库同步中 → 数据还在传输
  从库同步完成 → 数据一致
  
中间状态：主库和从库的数据不一致（软状态）
```

### 4.4 最终一致性（Eventually Consistent）

**定义**：如果没有新的更新操作，经过一段时间后，所有副本的数据最终达到一致。

**关键点**：
- **不保证何时一致**：只保证最终一致
- **收敛时间**：取决于网络延迟、复制方式、负载
- **先写入先到达**：不保证（可能乱序）

**最终一致性的变种**：

| 变种 | 描述 | 示例 |
|------|------|------|
| **因果一致性** | 有因果关系的操作按顺序执行 | A 发消息后 B 回复，所有人看到的顺序一致 |
| **读己之写** | 自己写入的数据，自己立即能读到 | 发朋友圈后自己能立即看到 |
| **会话一致性** | 同一会话内保证读己之写 | 用户登录后的操作，在同一会话内一致 |
| **单调读一致性** | 不会读到比之前更旧的数据 | 不会出现"时间倒流" |
| **单调写一致性** | 写操作按顺序执行 | 不会出现写操作乱序 |

### 4.5 BASE vs ACID

| 对比项 | ACID | BASE |
|--------|------|------|
| **一致性** | 强一致性 | 最终一致性 |
| **可用性** | 降低可用性保证一致性 | 基本可用，允许降级 |
| **适用场景** | 金融、交易 | 社交、内容、搜索 |
| **典型系统** | 关系数据库 | NoSQL、微服务 |
| **设计哲学** | 悲观锁 | 乐观策略 |

## 五、Paxos 共识算法

### 5.1 为什么需要共识算法

**共识问题**：多个节点就某个值达成一致。

**应用场景**：
1. **Leader 选举**：选出一个主节点
2. **日志复制**：所有节点日志顺序一致
3. **状态机复制**：所有节点执行相同的命令序列
4. **分布式锁**：多个节点竞争同一把锁

### 5.2 Paxos 算法概述

**Paxos** 由 Leslie Lamport 于 1990 年提出（论文 1998 年发表），是第一个被证明正确的分布式共识算法。

**角色**：
- **Proposer（提案者）**：提出提案（value）
- **Acceptor（接受者）**：对提案进行投票
- **Learner（学习者）**：学习最终被选定的值

> 注意：一个节点可以同时扮演多个角色。

### 5.3 Basic Paxos 算法

**目标**：在多个 Proposer 和多个 Acceptor 之间，就一个值达成共识。

**两阶段协议**：

#### 阶段一：Prepare（准备）

```
Proposer:
  1. 生成全局唯一递增的提案编号 n
  2. 向所有 Acceptor 发送 Prepare(n) 请求

Acceptor:
  3. 收到 Prepare(n) 后：
     - 如果 n > 已承诺的最大编号（maxPromisedN）：
       → 承诺不再接受编号 < n 的提案
       → 返回已接受的最高编号提案（如果有的话）
       → 更新 maxPromisedN = n
     - 否则：拒绝
```

#### 阶段二：Accept（接受）

```
Proposer:
  4. 收到多数 Acceptor 的 Prepare 响应后：
     - 如果响应中有已接受的提案 → 选择编号最大的那个值
     - 否则 → 使用自己的提案值
  5. 向所有 Acceptor 发送 Accept(n, value) 请求

Acceptor:
  6. 收到 Accept(n, value) 后：
     - 如果 n >= maxPromisedN：
       → 接受该提案
       → 更新 maxAcceptedN = n, acceptedValue = value
       → 返回 Accepted
     - 否则：拒绝
```

**示例**：

```
3 个 Acceptor（A1, A2, A3），2 个 Proposer（P1, P2）

T1: P1 发送 Prepare(1)
    A1: 承诺（maxPromisedN=1），返回 null
    A2: 承诺（maxPromisedN=1），返回 null
    → P1 获得多数承诺

T2: P2 发送 Prepare(2)
    A2: 承诺（maxPromisedN=2），返回 null
    A3: 承诺（maxPromisedN=2），返回 null
    → P2 获得多数承诺

T3: P1 发送 Accept(1, "X")
    A1: 接受（1 >= 1）→ Accepted
    A2: 拒绝（1 < 2）← 已承诺更高编号
    → P1 未获多数接受

T4: P2 发送 Accept(2, "Y")
    A2: 接受（2 >= 2）→ Accepted
    A3: 接受（2 >= 2）→ Accepted
    → P2 获得多数接受 → "Y" 被选定
```

### 5.4 Multi-Paxos

**问题**：Basic Paxos 每确定一个值需要两轮 RPC，效率太低。

**Multi-Paxos 优化**：
1. 选出一个 Leader，只有 Leader 可以提出提案
2. Leader 选出后，省略 Prepare 阶段，直接进入 Accept 阶段
3. 只有 Leader 变更时才需要重新执行 Prepare

```
Basic Paxos：每个值都需要 2 轮 RPC
  Prepare → Accept → Prepare → Accept → ...

Multi-Paxos：Leader 稳定时每个值只需 1 轮 RPC
  Prepare（仅一次，选出 Leader）→ Accept → Accept → Accept → ...
```

**效率提升**：从 2 次 RPC 降为 1 次 RPC，性能翻倍。

### 5.5 Paxos 的工程实现

**Google Chubby**：
- 基于 Paxos 的分布式锁服务
- 用于 Google 内部的 Leader 选举和元数据存储

**Google Spanner**：
- 基于 Paxos 的全球分布式数据库
- 使用 TrueTime API 提供外部一致性

**Apache ZooKeeper**：
- 使用 ZAB 协议（Paxos 的变种）
- 用于分布式协调服务

## 六、Raft 共识算法

### 6.1 为什么需要 Raft

**Paxos 的问题**：
- 难以理解（Lamport 的论文以晦涩著称）
- 工程实现复杂（Multi-Paxos 没有标准实现）
- 不适合教学

**Raft**（2014 年）由 Diego Ongaro 和 John Ousterhout 提出，目标是**提供和 Paxos 相同的正确性保证，但更易理解**。

**Raft 的设计原则**：
1. **问题分解**：将共识问题分解为三个子问题
2. **状态空间简化**：减少需要考虑的状态
3. **领导选举**：强 Leader 模型

### 6.2 Raft 的三个子问题

1. **Leader Election（领导选举）**：选出一个 Leader
2. **Log Replication（日志复制）**：Leader 将日志复制到所有 Follower
3. **Safety（安全性）**：保证所有节点状态机执行相同的命令序列

### 6.3 节点角色

| 角色 | 描述 |
|------|------|
| **Leader** | 处理所有客户端请求，复制日志到 Follower |
| **Follower** | 接收 Leader 的日志复制，被动响应 |
| **Candidate** | 选举过程中的候选人 |

**角色转换**：

```
        定时器超时
Follower ──────────→ Candidate
    ↑                    │
    │ 选举超时/发现更高Term │ 收到多数投票
    │                    ↓
    └───────────── Leader
```

### 6.4 Term（任期）

**Term 是 Raft 的逻辑时钟**，用于检测过时信息。

```
Term 1       Term 2       Term 3       Term 4
┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐
│Leader│    │选举中│    │Leader│    │选举中│
│ 正常 │    │      │    │ 正常 │    │      │
└──────┘    └──────┘    └──────┘    └──────┘
```

**规则**：
- 每个 Term 最多一个 Leader
- 如果 Leader 故障，开始新 Term 的选举
- 如果某个节点发现更高的 Term，立即转为 Follower

### 6.5 Leader Election 详解

**选举触发**：
- Follower 在**选举超时（Election Timeout）**内未收到 Leader 的心跳
- 转为 Candidate，发起选举

**选举过程**：

```
1. Candidate 自增当前 Term
2. 投自己一票
3. 向所有节点发送 RequestVote RPC
   - candidateTerm: 自增后的 Term
   - candidateId: 自己的 ID
   - lastLogIndex: 自己日志的最后索引
   - lastLogTerm: 自己日志最后条目的 Term

4. 等待其他节点的投票响应：
   a. 收到多数投票 → 成为 Leader
   b. 收到更高 Term 的心跳 → 退回 Follower
   c. 选举超时 → 开始新一轮选举
```

**投票规则**（每个 Term 每个节点只能投一票）：

```
节点收到 RequestVote(candidateTerm, candidateId, lastLogIndex, lastLogTerm) 后：

1. 如果 candidateTerm < currentTerm → 拒绝
2. 如果 candidateTerm == currentTerm 且已投票给其他人 → 拒绝
3. 如果候选人的日志不比自己新 → 拒绝
   （比较 lastLogTerm 和 lastLogIndex）
4. 否则 → 投票
```

**日志新旧比较**：
```
先比较 lastLogTerm：大的更新
如果 lastLogTerm 相同，比较 lastLogIndex：大的更新
```

**选举安全性保证**：
- 每个 Term 最多一个 Leader（每个节点每 Term 只投一票 + 多数派）
- Leader 的日志是最完整的（投票规则保证）

**随机化选举超时**：

```
为什么需要随机化？
→ 避免多个 Follower 同时发起选举（Split Vote）
→ 每个节点的 Election Timeout 在 [150ms, 300ms] 之间随机
→ 先超时的节点先成为 Candidate，获得多数投票
```

### 6.6 Log Replication 详解

**流程**：

```
1. 客户端发送命令给 Leader
2. Leader 将命令追加到自己的日志
3. Leader 并行发送 AppendEntries RPC 给所有 Follower
4. 等待多数 Follower 确认
5. Leader 提交该日志条目（应用到状态机）
6. Leader 响应客户端
7. Leader 通过心跳通知 Follower 提交
```

**日志格式**：

```
Index:  1    2    3    4    5    6    7
       ┌────┬────┬────┬────┬────┬────┬────┐
Term:  │ 1  │ 1  │ 1  │ 2  │ 2  │ 3  │ 3  │
       ├────┼────┼────┼────┼────┼────┼────┤
Cmd:   │SET │GET │DEL │SET │SET │DEL │SET │
       │x=1 │y   │z   │x=2 │y=3 │x   │z=4 │
       └────┴────┴────┴────┴────┴────┴────┘
                              ↑
                         commitIndex = 5
```

**AppendEntries RPC**：

```
Leader → Follower:
  term:         Leader 的当前 Term
  leaderId:     Leader 的 ID
  prevLogIndex: 前一条日志的索引
  prevLogTerm:  前一条日志的 Term
  entries[]:    要追加的日志条目（可能多个）
  leaderCommit: Leader 的 commitIndex

Follower → Leader:
  term:    Follower 的当前 Term
  success: 是否成功
```

**日志一致性检查**：

```
Follower 收到 AppendEntries 后：
1. 如果 term < currentTerm → 返回 false
2. 如果 prevLogIndex 处没有日志 → 返回 false
3. 如果 prevLogIndex 处的 Term 不匹配 → 返回 false
4. 如果已有日志与新日志冲突（同 Index 不同 Term）→ 删除冲突及之后的日志
5. 追加新日志
6. 如果 leaderCommit > commitIndex → 更新 commitIndex
```

**日志不一致的修复**：

```
Leader 日志：  1  2  3  4  5  6  7
Follower A：   1  2  3  4  5         ← 缺少 6、7
Follower B：   1  2  3  4  5  6' 7'  ← 6'、7' 与 Leader 不同
Follower C：   1  2  3              ← 缺少 4-7

修复过程：
1. Leader 对每个 Follower 维护 nextIndex（下次发送的日志索引）
2. 如果 AppendEntries 返回 false → nextIndex 递减
3. 重试直到找到一致的位置
4. 从一致位置开始覆盖
```

### 6.7 Safety 详解

**Leader 完整性（Leader Completeness）**：

> 如果一个日志条目在某个 Term 被提交，那么所有更高 Term 的 Leader 都包含该条目。

**证明要点**：
1. 提交 = 多数节点确认
2. 新 Leader 必须获得多数投票
3. 投票规则要求候选人的日志至少和投票者一样新
4. 因此新 Leader 一定包含已提交的条目

**约束条件**：
- **Election Safety**：每个 Term 最多一个 Leader
- **Leader Append-Only**：Leader 只追加日志，不删除不覆盖
- **Log Matching**：如果两个节点的日志在某个索引上有相同的 Term，那么之前的日志也相同
- **Leader Completeness**：已提交的日志不会丢失
- **State Machine Safety**：任何节点在某个索引应用了日志，其他节点不会在同一索引应用不同的日志

### 6.8 Raft vs Paxos 对比

| 对比项 | Paxos | Raft |
|--------|-------|------|
| **理解难度** | 极高 | 较低 |
| **Leader 模型** | 多 Leader / 无 Leader | 强 Leader |
| **日志管理** | 允许日志空洞 | 不允许日志空洞（顺序追加）|
| **工程实现** | 无标准实现 | 有多个开源实现（etcd、Consul）|
| **修改提议** | 允许任意修改 | 只允许追加 |
| **性能** | 理论上更优 | 实际接近 |
| **正确性证明** | 数学证明 | 形式化验证 |

## 七、一致性模型

### 7.1 一致性模型层级

从强到弱：

```
线性一致性（Linearizability）
    ↑ 最强
顺序一致性（Sequential Consistency）
    ↑
因果一致性（Causal Consistency）
    ↑
读己之写一致性（Read-Your-Writes）
    ↑
会话一致性（Session Consistency）
    ↑
单调读一致性（Monotonic Read）
    ↑
最终一致性（Eventual Consistency）
    ↑ 最弱
```

### 7.2 线性一致性（Linearizability）

**定义**：所有操作看起来像是在一个单一的时间点上原子执行，且与实时顺序一致。

**特点**：
- 最强的一致性模型
- 等价于 CAP 中的 C
- 代价最高（需要同步通信）

**示例**：
```
时间线：
T1: 客户端 A 写入 x=1（在 T1 完成）
T2: 客户端 B 读取 x → 必须返回 1（在 T2 > T1）

如果 B 读到 x=0（旧值），则违反线性一致性
```

**实现方式**：
- 主从复制 + 同步写入（等所有副本确认）
- 分布式锁 + 原子广播
- ZooKeeper 的写操作

### 7.3 顺序一致性（Sequential Consistency）

**定义**：所有操作的结果等同于某种顺序执行，且每个客户端的操作按其发出顺序执行。但不要求与实时顺序一致。

**与线性一致性的区别**：
- 线性一致性：要求与实时顺序一致
- 顺序一致性：只要求客户端顺序，不要求实时

**示例**：
```
客户端 A: W(x,1) → W(x,2)
客户端 B: R(x) → R(x)

线性一致性：R(x) 可能返回 1 或 2（取决于实时顺序）
顺序一致性：R(x) 可能返回 0、1 或 2（只要 A 的写按顺序）
```

### 7.4 因果一致性（Causal Consistency）

**定义**：有因果关系的操作按顺序执行，没有因果关系的操作可以乱序。

**因果关系**：
- 写 → 读（读后写）
- 读 → 写（写后读）
- 写 → 写（写后写）

**示例**：
```
因果相关：
  A 发朋友圈 "今天天气真好" → B 评论 "是啊" 
  → 所有人看到评论时，一定能看到原帖

因果无关：
  A 发朋友圈 "今天天气真好"
  C 发朋友圈 "今天吃了火锅"
  → 两者没有因果关系，不同人看到的顺序可以不同
```

**实现方式**：
- 向量时钟（Vector Clock）
- 版本向量（Version Vector）

**向量时钟示例**：
```
3 个节点：N1, N2, N3

初始状态：VC = [0, 0, 0]

N1 写入 x=1：VC = [1, 0, 0]  → N1 的计数器 +1
N2 同步 N1 的写入后写 x=2：VC = [1, 1, 0]  → N2 的计数器 +1
N3 同步 N2 的写入后读 x=2：VC = [1, 1, 1]

比较规则：
  VC1 = [1, 0, 0], VC2 = [1, 1, 0]
  VC2[i] >= VC1[i] 对所有 i → VC2 是 VC1 的后继 → 有因果关系
```

## 八、分布式协调服务实战

### 8.1 ZooKeeper 的 ZAB 协议

**ZAB**（ZooKeeper Atomic Broadcast）是 ZooKeeper 使用的共识协议，基于 Paxos 思想但做了简化。

**两种模式**：
1. **恢复模式**：Leader 崩溃后重新选举
2. **广播模式**：Leader 正常工作时，广播消息

**ZAB vs Paxos**：
- ZAB 是强 Leader 模型（类似 Raft）
- ZAB 的 ZXID 包含 Term 和 Counter（类似 Raft 的 Term + Index）
- ZAB 在 Leader 选举时优先选择数据最完整的节点

### 8.2 etcd 的 Raft 实现

**etcd** 是 CoreOS 开发的分布式 KV 存储，使用 Raft 协议。

**架构**：
```
客户端 → etcd Server → Raft 层 → WAL 日志 + Snapshot
                          ↓
                      BoltDB（状态机）
```

**关键特性**：
- 强一致性（线性一致性读）
- Watch 机制（监听 Key 变化）
- Lease 机制（TTL 自动删除）
- 事务支持（Compare-And-Set）

**使用场景**：
- 服务注册与发现
- 分布式锁
- 配置中心
- Leader 选举

## 九、面试高频问题

### Q1: CAP 定理是什么？如何选择？

**答案要点**：
- C（一致性）、A（可用性）、P（分区容错性）
- 网络分区不可避免，实际选择 CP 或 AP
- CP：金融、支付、库存（宁可不可用，不能不一致）
- AP：社交、内容、搜索（宁可不一致，不能不可用）
- 正常运行时可以同时满足 C 和 A

### Q2: BASE 理论是什么？

**答案要点**：
- Basically Available（基本可用）
- Soft State（软状态）
- Eventually Consistent（最终一致性）
- 是 CAP 中 AP 方向的实践总结
- 适合大规模分布式系统

### Q3: Paxos 和 Raft 的区别？

**答案要点**：
- Paxos：理论优美但难理解，允许多 Leader
- Raft：工程友好，强 Leader 模型，不允许日志空洞
- Raft 将问题分解为选举、日志复制、安全性三个子问题
- Raft 通过随机化选举超时避免 Split Vote

### Q4: 什么是最终一致性？有什么变种？

**答案要点**：
- 定义：没有新更新后，数据最终达到一致
- 变种：因果一致性、读己之写、会话一致性、单调读、单调写
- 实现方式：向量时钟、版本向量、Gossip 协议

### Q5: 如何实现分布式一致性？

**答案要点**：
1. 强一致性：Paxos / Raft + 同步复制（ZooKeeper、etcd）
2. 最终一致性：异步复制 + 向量时钟（Cassandra、DynamoDB）
3. 因果一致性：向量时钟 + 因果跟踪

## 十、总结

本文系统梳理了分布式系统的理论基础：

1. **CAP 定理**：C/A/P 三选二，实际选择 CP 或 AP
2. **BASE 理论**：基本可用 + 软状态 + 最终一致性，是 AP 的实践总结
3. **Paxos**：理论证明的共识算法，难以理解但正确性有保证
4. **Raft**：工程友好的共识算法，强 Leader 模型，易于理解和实现
5. **一致性模型**：从线性一致性到最终一致性，不同场景选择不同级别

**核心记忆**：
- CAP 只在分区时需要选择，正常运行可同时满足
- Raft 的三个子问题：选举、日志复制、安全性
- 最终一致性有 5 个变种：因果、读己之写、会话、单调读、单调写
- 向量时钟用于实现因果一致性

**面试关键词**：CAP、BASE、Paxos、Raft、线性一致性、最终一致性、向量时钟
