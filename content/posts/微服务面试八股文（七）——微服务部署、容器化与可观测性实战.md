## 一、微服务容器化：Docker 最佳实践

### 1.1 为什么微服务必须容器化？

> 📌 **基础知识提示**：Docker多阶段构建、镜像优化、安全配置等通用容器化知识已在《系统分析与设计（九）——持续交付与DevOps实践》中深度覆盖，本文聚焦**微服务场景下**的容器化独特挑战与最佳实践。

```
单体 vs 微服务容器化对比：

单体（一个巨型JAR）：
  ├── 一个容器，1-2GB镜像
  ├── 启动慢，修改频繁需全量重新部署
  └── 所有模块共享资源，无隔离

微服务（多个独立JAR）：
  ├── 每个服务一个容器，100-300MB镜像（轻量化目标）
  ├── 启动快，修改单个服务独立部署
  └── 每个服务独立资源限制，安全隔离
  └── 数量多（10-100+个服务）→ 容器编排成为刚需

微服务容器化的核心挑战：
1. 镜像数量多 → 需要自动化构建流水线
2. 服务间通信 → 网络配置和安全
3. 配置管理 → 环境变量/ConfigMap
4. 健康检查 → 存活探针/就绪探针
5. 弹性伸缩 → 水平扩缩容
```

### 1.2 Spring Boot 微服务 Dockerfile 最佳实践

```dockerfile
# ===== 微服务镜像最佳实践 =====

# 阶段1：构建
FROM maven:3.9-eclipse-temurin-21-alpine AS builder
WORKDIR /app

# 利用层缓存：先复制依赖文件
COPY pom.xml .
RUN mvn dependency:go-offline -B

# 再复制源代码（源代码变化时才重新下载依赖）
COPY src ./src

# 构建参数：允许在运行时指定 profile
ARG SPRING_PROFILES_ACTIVE=prod
RUN mvn package -DskipTests -B \
    && echo "构建完成"

# 阶段2：运行（使用 JRE 而非 JDK，体积大幅减小）
FROM eclipse-temurin:21-jre-alpine AS runtime

# 安全：非root用户运行
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

WORKDIR /app

# 从 builder 阶段只复制 jar 文件（排除源码和构建产物）
COPY --from=builder /app/target/*.jar app.jar

# 入口脚本：优雅关闭
COPY <<'EOF' /app/entrypoint.sh
#!/bin/sh
exec java -XX:+UseZGC \
    -XX:MaxRAMPercentage=75.0 \
    -XX:+ExitOnOutOfMemoryError \
    -Djava.security.egd=file:/dev/./urandom \
    -jar app.jar
EOF
RUN chmod +x /app/entrypoint.sh

EXPOSE 8080
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
    CMD wget -qO- http://localhost:8080/actuator/health || exit 1

ENTRYPOINT ["/app/entrypoint.sh"]
```

**微服务镜像优化关键指标对比：**

| 优化策略 | 优化前 | 优化后 | 节省 |
|---------|-------|-------|------|
| 使用 JRE 而非 JDK | 500MB | 180MB | 64% |
| Alpine 基础镜像 | 180MB | 120MB | 33% |
| 多阶段构建 | 120MB | 80MB | 33% |
| 非 root 用户 | - | - | 安全加固 |
| 健康检查 | - | - | K8s 存活探测 |

### 1.4 容器化面试高频问题

**Q: 每个微服务一个镜像还是多个服务放一个镜像？**

```
答：每个微服务独立镜像。这是微服务区别于单体最核心的设计原则之一：

独立镜像的好处：
✅ 独立版本管理：order-service v1.2 和 stock-service v2.0 可以同时运行
✅ 独立扩缩容：订单高峰期只扩 order-service，不影响其他服务
✅ 独立故障隔离：一个服务OOM不会影响同机的其他服务
✅ 独立安全边界：每个镜像独立扫描漏洞，独立打补丁
✅ 独立资源限制：order-service 可以分配更多CPU，stock-service 分配更多内存

多个服务放一个镜像的场景（少数）：
⚠️ 边车模式（Sidecar）：主容器 + 边车容器（如日志收集、代理）
⚠️ Init Container：启动前的初始化任务
⚠️ 强耦合的模块被迫一起部署（应尽量避免）
```

### 2.2 微服务 Deployment 完整配置

```yaml
# k8s/order-service.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
  labels:
    app: order-service
    version: v1.5.0
    team: order-team
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  
  # Pod 选择器
  selector:
    matchLabels:
      app: order-service
  
  # Pod 模板
  template:
    metadata:
      labels:
        app: order-service
        version: v1.5.0
        # Istio 注入标签（注入 Sidecar Proxy）
        sidecar.istio.io/inject: "true"
    spec:
      # Pod 反亲和：同节点尽量不部署同一服务的多个实例
      affinity:
        podAntiAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              podAffinityTerm:
                labelSelector:
                  matchLabels:
                    app: order-service
                topologyKey: kubernetes.io/hostname
        
        # 软亲和：优先调度到有特定标签的节点（如高IO节点）
        nodeAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 80
              preference:
                matchExpressions:
                  - key: workload-type
                    operator: In
                    values: [io-intensive]
      
      # 优雅终止：给应用30秒时间处理正在处理的请求
      terminationGracePeriodSeconds: 30
      
      # Init Container：等待依赖服务就绪
      initContainers:
        - name: wait-for-mysql
          image: busybox:1.36
          command: ['sh', '-c', 'until nc -z mysql:3306; do echo "等待MySQL..."; sleep 2; done']
          resources:
            limits: { cpu: "100m", memory: "64Mi" }
      
      containers:
        - name: order-service
          image: ghcr.io/company/order-service:v1.5.0
          imagePullPolicy: Always
          ports:
            - containerPort: 8080
              name: http
          
          # 资源配置
          resources:
            requests:
              cpu: "500m"
              memory: "512Mi"
            limits:
              cpu: "2000m"
              memory: "1Gi"
          
          # 存活探针：Pod是否还活着（失败则重启）
          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 60   # JVM启动较慢
            periodSeconds: 10
            failureThreshold: 3
          
          # 就绪探针：Pod是否可以接收流量（失败则摘除）
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 5
            failureThreshold: 3
            successThreshold: 1
          
          # 启动探针：慢启动应用（10分钟内必须启动完成）
          startupProbe:
            httpGet:
              path: /actuator/health
              port: 8080
            failureThreshold: 30
            periodSeconds: 10
          
          env:
            - name: SPRING_PROFILES_ACTIVE
              value: "prod"
            - name: JAVA_OPTS
              value: "-XX:+UseZGC -XX:MaxRAMPercentage=75.0 -XX:+HeapDumpOnOutOfMemoryError"
            - name: POD_NAME
              valueFrom:
                fieldRef:
                  fieldPath: metadata.name
          
          # 从 ConfigMap 挂载配置
          volumeMounts:
            - name: config
              mountPath: /app/config
              readOnly: true
          
          # 从 Secret 读取敏感信息
          envFrom:
            - secretRef:
                name: order-service-secrets
          
          lifecycle:
            preStop:
              exec:
                command: ["/bin/sh", "-c", "sleep 10"]  # 等待K8s结束流量引流
          
          ports:
            - name: metrics
              containerPort: 8081  # Prometheus 指标端点

      volumes:
        - name: config
          configMap:
            name: order-service-config

---
# K8s Service（ClusterIP 服务发现）
apiVersion: v1
kind: Service
metadata:
  name: order-service
  labels:
    app: order-service
spec:
  type: ClusterIP
  selector:
    app: order-service
  ports:
    - name: http
      port: 80
      targetPort: 8080
    - name: metrics
      port: 8081
      targetPort: 8081

---
# Horizontal Pod Autoscaler（自动扩缩容）
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: order-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicas: 3
  maxReplicas: 20
  
  metrics:
    # 基于 CPU 自动扩缩
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    
    # 基于自定义指标（Prometheus Adapter）自动扩缩
    - type: Pods
      pods:
        metric:
          name: order_requests_per_second
        target:
          type: AverageValue
          averageValue: "100"  # 每秒100个请求时扩容

  # 扩缩容行为控制
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # 缩容等待5分钟（避免抖动）
      policies:
        - type: Percent
          value: 50
          periodSeconds: 60
    scaleUp:
      stabilizationWindowSeconds: 0    # 扩容立即执行
      policies:
        - type: Percent
          value: 100
          periodSeconds: 15
        - type: Pods
          value: 4
          periodSeconds: 15
      selectPolicy: Max  # 取扩容最快的策略
```

### 2.4 Kubernetes 面试高频问题

**Q: 就绪探针和存活探针有什么区别？为什么两个都需要？**

```
┌─────────────────────────────────────────────────────┐
│  存活探针 (Liveness Probe)                          │
│  问："这个容器还活着吗？"                            │
│  答："不" → 重启容器                               │
│  适用：应用死锁、无响应、OOM                         │
│  典型失败：JVM死锁、无限循环、外部依赖卡死            │
├─────────────────────────────────────────────────────┤
│  就绪探针 (Readiness Probe)                         │
│  问："这个容器准备好接收流量了吗？"                  │
│  答："不" → 从Service负载均衡中摘除                  │
│  适用：启动中、依赖未就绪、负载过高                   │
│  典型失败：启动未完成、数据库连接池未初始化           │
├─────────────────────────────────────────────────────┤
│  启动探针 (Startup Probe)                           │
│  问："应用启动完成了吗？"                            │
│  问："还没" → 存活探针暂不检查                       │
│  问："好了" → 交给存活探针接管                       │
│  适用：Spring Boot等慢启动应用（30秒~10分钟）         │
└─────────────────────────────────────────────────────┘

为什么需要两个：
1. 启动中（就绪失败）：不应接收流量，但不重启
2. 启动后死锁（存活失败）：应重启，不继续处理请求
3. 两者配合：先等就绪，再保存活

常见配置：
liveness:  initialDelay=60s,  period=10s,  failure=3   # 启动60秒后才开始检查
readiness: initialDelay=5s,   period=5s,   failure=3   # 启动5秒后就开始检查
startup:    failure=30,       period=10s                # 最多300秒启动时间
```

## 三、服务网格：Istio 深度实战

### 3.1 什么是服务网格？解决了什么问题？

```
没有服务网格时：
┌──────────────────────────────────────────────────────────┐
│  Order Service                                          │
│    │                                                    │
│    │ 手动在代码中处理：                                  │
│    │   - HTTP/REST调用库存服务                           │
│    │   - 熔断器（Hystrix/Resilience4j）                 │
│    │   - 重试策略                                        │
│    │   - 超时设置                                        │
│    │   - mTLS加密（手动配置SSL证书）                     │
│    │   - 链路追踪（手动埋点）                            │
│    │                                                    │
│    问题：每个服务都要重复实现这些"基础设施"代码          │
└──────────────────────────────────────────────────────────┘

有服务网格后（Sidecar 代理模式）：
┌──────────────────────────────────────────────────────────┐
│  Order Service Pod                                       │
│  ┌──────────────┬────────────────────────────────────┐  │
│  │  order-svc   │        Envoy Sidecar Proxy          │  │
│  │  (主容器)     │                                    │  │
│  │              │  • 熔断、重试、超时（自动）          │  │
│  │              │  • mTLS 加密（自动）                 │  │
│  │              │  • 链路追踪（自动）                  │  │
│  │              │  • 流量管理（金丝雀/蓝绿）           │  │
│  │              │  • 指标采集（Prometheus格式）         │  │
│  │              │                                    │  │
│  │  ┌──────────▼──────────┐                         │  │
│  │  │  流量劫持（iptables）│ ← 所有出口/入口流量      │  │
│  │  └────────────────────┘    经由Sidecar代理          │  │
│  └────────────────────────────────────────────────────┘  │
│                                                          │
│  库存服务 Pod ── Envoy ── Envoy ── 订单服务 Pod          │
│                    ↑                                      │
│               所有服务间通信自动被代理                    │
└──────────────────────────────────────────────────────────┘

服务网格的核心价值：
✅ 网络层功能下沉：熔断、重试、超时、限流 → 代码无需关心
✅ 零信任安全：mTLS 自动加密，服务身份认证
✅ 可观测性：无代码侵入的指标、日志、追踪
✅ 流量控制：细粒度的金丝雀、蓝绿、A/B测试
✅ 开发者专注业务：基础设施交给网格处理
```

**面试要点：服务网格 vs SDK 方式的区别**

| 维度 | 服务网格（Istio） | SDK方式（Hystrix/Sentinel） |
|------|-----------------|--------------------------|
| 代码侵入 | **零侵入** | 高侵入（每个服务都要集成） |
| 多语言支持 | **原生支持** | 需要各语言SDK，维护成本高 |
| 资源消耗 | Sidecar 占用资源（每个Pod一个） | 无额外资源 |
| 网络开销 | Sidecar 一次转发（可忽略） | 无额外开销 |
| 灵活性 | 网格级别统一配置 | 每个服务可独立配置 |
| 复杂度 | 高（网格本身是复杂系统） | 低（库集成） |
| 适用场景 | 多语言微服务集群 | 同语言单体或少量服务 |

### 4.2 OpenTelemetry 统一埋点

```java
// OpenTelemetry 是可观测性的"大一统"标准
// 统一了 Tracing、Metrics、Logging 的采集规范
// 支持自动埋点和手动埋点

// ===== 自动埋点（零代码侵入）=====
// Spring Boot + OpenTelemetry 自动埋点
// 引入依赖后，HTTP、JDBC、Kafka、Redis 调用自动埋点

// pom.xml
<dependency>
    <groupId>io.opentelemetry</groupId>
    <artifactId>opentelemetry-javaagent</artifactId>
    <version>1.32.0</version>
    <scope>provided</scope>
</dependency>

// 启动参数：挂载 Java Agent 自动增强
// java -javaagent:opentelemetry-javaagent.jar \
//      -Dotel.service.name=order-service \
//      -Dotel.exporter.otlp.endpoint=http://jaeger-collector:4317 \
//      -jar order-service.jar

// ===== 手动埋点：业务关键路径 =====
// OrderService.java
@Slf4j
@Component
public class OrderService {
    
    // OpenTelemetry API
    private final Tracer tracer;
    private final Meter meter;
    
    public OrderService(Tracer tracer, Meter meter) {
        this.tracer = tracer;
        this.meter = meter;
    }
    
    public Order createOrder(CreateOrderRequest request) {
        // 创建 Span（一个操作单元）
        Span span = tracer.spanBuilder("createOrder")
                .setAttribute("order.userId", request.getUserId())
                .setAttribute("order.amount", request.getAmount())
                .startSpan();
        
        try (Scope scope = span.makeCurrentScope()) {
            // 业务逻辑
            validateOrder(request);
            
            // 创建子Span：扣减库存
            Span stockSpan = tracer.spanBuilder("deductStock")
                    .setAttribute("stock.skuId", request.getSkuId())
                    .startSpan();
            try (Scope stockScope = stockSpan.makeCurrentScope()) {
                stockService.deduct(request.getSkuId(), request.getQuantity());
            } catch (Exception e) {
                stockSpan.recordException(e);
                throw e;
            } finally {
                stockSpan.end();
            }
            
            // 创建订单
            Order order = orderRepository.save(new Order(request));
            
            // 添加事件日志（属于该Span）
            span.addEvent("order.created", 
                Attributes.of(
                    AttributeKey.stringKey("orderId"), order.getId().toString(),
                    AttributeKey.longKey("amount"), order.getAmount()
                ));
            
            return order;
        } catch (Exception e) {
            span.recordException(e);
            span.setStatus(StatusCode.ERROR, e.getMessage());
            throw e;
        } finally {
            span.end();
        }
    }
}

// ===== 自定义 Metrics =====
// 业务指标（RED方法：Rate/Errors/Duration）
public class OrderMetrics {
    
    private final Counter orderCreatedCounter;
    private final Counter orderFailedCounter;
    private final Histogram orderDurationHistogram;
    
    public OrderMetrics(Meter meter) {
        // 请求速率：每秒创建订单数
        orderCreatedCounter = meter.counterBuilder("order_created_total")
                .setDescription("总创建订单数")
                .setUnit("1")
                .build();
        
        // 错误计数：按错误类型标签
        orderFailedCounter = meter.counterBuilder("order_failed_total")
                .setDescription("失败订单数")
                .build();
        
        // 延迟分布：P50/P90/P99
        orderDurationHistogram = meter.histogramBuilder("order_create_duration")
                .setDescription("订单创建耗时")
                .setUnit("ms")
                .ofLongs()
                .build();
    }
    
    public void recordOrderCreated(long durationMs) {
        orderCreatedCounter.increment();
        orderDurationHistogram.record(durationMs);
    }
    
    public void recordOrderFailed(String reason) {
        orderFailedCounter.increment(
            Attributes.of(AttributeKey.stringKey("reason"), reason));
    }
}
```

### 4.4 端到端故障排查工作流

```
故障排查工作流（从告警到定位）：

告警触发（PagerDuty/飞书）
  │
  ↓
Step 1: 确认影响范围（Grafana Dashboard）
  ├── 哪些服务的错误率飙升？
  ├── 是所有服务还是个别服务？
  └── 错误率从多少开始飙升？
  │
  ↓
Step 2: 查看调用链路（Jaeger/Zipkin）
  ├── 找到 traceId
  ├── 时间线：哪个服务的哪个调用耗时最长？
  └── span层级：具体到哪个方法
  │
  ↓
Step 3: 关联日志（ELK/Loki）
  ├── 用 traceId 搜索所有相关日志
  ├── 按时间排序
  └── 找到第一条异常日志
  │
  ↓
Step 4: 定位根因
  ├── 数据库慢查询？→ 查看 MySQL 慢日志
  ├── Redis 超时？  → 查看 Redis 日志
  ├── 网络抖动？  → 查看 K8s 事件日志
  └── 代码bug？  →  查看 GitLab CI 最近提交
  │
  ↓
Step 5: 止血操作
  ├── 限流降级 → 切到降级逻辑
  ├── 紧急回滚 → helm rollback
  └── 流量切换 → 修改 VirtualService 权重

Step 6: 复盘改进
  ├── 添加监控告警（防止同类问题漏报）
  ├── 添加单元/集成测试（防止同类bug）
  └── 添加限流/熔断（防止下次雪崩）
```

## 五、微服务部署与可观测性面试高频问题

**Q1: 如何设计一套适合中大型团队的微服务可观测性方案？**

```
答：可观测性方案的设计需要考虑数据采集、存储、查询三个环节：

采集层：
- 指标：Prometheus + OpenTelemetry SDK
- 日志：Fluentd/Fluent Bit → Loki/Elasticsearch
- 追踪：OpenTelemetry Collector → Jaeger

存储层：
- 时序数据（Metrics）：Thanos/VictoriaMetrics（长期存储+压缩）
- 日志数据：Loki（低成本）或 Elasticsearch（全文检索）
- 追踪数据：Jaeger + Elasticsearch 后端

查询层：
- Grafana（统一Dashboard，指标+追踪+日志关联）
- Kibana（日志全文检索）
- Jaeger（调用链分析）

多团队协作：
- 服务级 Dashboard 由各团队自己维护
- 公共 Dashboard（如全局拓扑、SLO达标率）由平台团队维护
- Alert 由 SRE 团队统一配置，避免告警风暴
```

**Q2: 服务网格的缺点是什么？什么时候不该用？**

```
服务网格的代价：
1. 资源开销：每个Pod多一个Sidecar（~50MB内存）
2. 延迟增加：每次调用多一跳（~1-2ms）
3. 复杂度提升：学习曲线陡，维护成本高
4. 调试困难：流量被代理，问题排查更复杂
5. 性能瓶颈：Sidecar可能成为吞吐瓶颈

不该用服务网格的场景：
❌ 小团队（<10人）+少量服务（<5个）
   → 用 SDK 方式（Hystrix/Sentinel）更简单
❌ 对延迟极度敏感（如高频交易）
   → Sidecar 的一次转发延迟不可接受
❌ 资源极度受限（如边缘计算、IoT设备）
   → 没有资源给 Sidecar
❌ 服务已经高度成熟，基础设施稳定
   → 引入网格的风险大于收益

应该用服务网格的场景：
✅ 多语言微服务集群（>10个服务）
✅ 需要统一的安全策略（零信任mTLS）
✅ 需要细粒度的流量控制（金丝雀/蓝绿）
✅ 需要统一的可观测性（跨语言统一Trace）
✅ 大型团队的DevOps平台建设
```
