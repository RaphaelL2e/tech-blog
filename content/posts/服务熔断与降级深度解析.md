## 一、引言

在微服务架构中，服务之间的调用关系错综复杂，一个服务的故障可能像多米诺骨牌一样引发连锁反应，导致整个系统崩溃——这就是所谓的**雪崩效应**。服务熔断与降级是应对雪崩效应的两大核心手段，也是微服务高可用架构的基石。

本文将从原理到实践，深度解析熔断与降级机制，并对比三大主流框架：Netflix Hystrix、Alibaba Sentinel 和 Resilience4j。

### 雪崩效应的形成过程

```
服务A → 服务B → 服务C → 服务D(故障)
                     ↑
              请求堆积，线程耗尽
                     ↑
          服务C也因线程池耗尽而不可用
                     ↑
          服务B因等待服务C响应而阻塞
                     ↑
          服务A也因等待服务B响应而阻塞
                     ↑
          整个调用链崩溃 = 雪崩效应
```

### 熔断与降级的关系

| 概念 | 定义 | 目的 | 触发条件 |
|------|------|------|----------|
| **熔断** | 当失败率达到阈值时，快速切断对故障服务的调用 | 防止故障扩散，快速失败 | 失败率/慢调用率超阈值 |
| **降级** | 当服务不可用时，返回兜底方案而非报错 | 保证核心功能可用 | 熔断触发/超时/异常 |
| **限流** | 限制单位时间内的请求数量 | 防止过载 | QPS超限 |

## 二、熔断器模式深度解析

### 2.1 熔断器的状态机

熔断器（Circuit Breaker）的核心是一个有限状态机，包含三个状态：

```
         失败率达阈值              冷却期结束
  ┌──────────────────┐      ┌──────────────────┐
  │                  │      │                  │
  │                  ▼      │                  │
┌───┐           ┌────────┐  │          ┌──────────┐
│CLOSED────────►│  OPEN  │──┘─────────►│HALF-OPEN │
│               └────────┘             └──────────┘
│                                          │  │
│                  探测成功 ◄────────────────┘  │
│                  (恢复CLOSED)                 │
│                                               │
└───────────────────────────────────────────────┘
                     探测失败(回到OPEN)
```

**三个状态详解**：

1. **CLOSED（关闭）**：正常状态，所有请求正常通过。统计窗口内持续监控失败率。
2. **OPEN（打开）**：熔断状态，所有请求被快速拒绝（走降级逻辑），不发起远程调用。
3. **HALF-OPEN（半开）**：试探状态，允许少量请求通过。如果这些请求成功，说明服务已恢复，切换回 CLOSED；如果失败，说明服务仍未恢复，切回 OPEN。

### 2.2 熔断器的关键参数

```java
// 熔断器核心配置
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    // 滑动窗口类型：COUNT_BASED（基于计数）或 TIME_BASED（基于时间）
    .slidingWindowType(SlidingWindowType.COUNT_BASED)
    // 滑动窗口大小：统计最近100次调用
    .slidingWindowSize(100)
    // 最小调用次数：达到10次才开始计算失败率
    .minimumNumberOfCalls(10)
    // 失败率阈值：超过50%则熔断
    .failureRateThreshold(50.0)
    // 慢调用阈值：超过5秒视为慢调用
    .slowCallDurationThreshold(Duration.ofSeconds(5))
    // 慢调用率阈值：超过60%则熔断
    .slowCallRateThreshold(60.0)
    // 熔断持续时间：OPEN状态保持30秒
    .waitDurationInOpenState(Duration.ofSeconds(30))
    // 半开状态允许的请求数
    .permittedNumberOfCallsInHalfOpenState(10)
    // 自动从OPEN转为HALF_OPEN
    .automaticTransitionFromOpenToHalfOpenEnabled(true)
    .build();
```

### 2.3 滑动窗口算法

滑动窗口是熔断器统计指标的核心算法，分为两种：

**基于计数的滑动窗口**：
```
| 请求1 | 请求2 | ... | 请求100 |  ← 窗口大小=100
                                ↑ 新请求进入
| 请求2 | 请求3 | ... | 请求101 |  ← 窗口滑动
```

**基于时间的滑动窗口**：
```
| 00:00-00:10 | 00:10-00:20 | ... | 00:50-01:00 |  ← 10分钟窗口
                                            ↑ 时间推进
| 00:01-00:11 | 00:11-00:21 | ... | 00:51-01:01 |  ← 窗口滑动
```

**实现对比**：

| 实现方式 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| 数组环形缓冲区 | O(1)时间复杂度，内存高效 | 精度固定 | 基于计数的窗口 |
| 时间桶聚合 | 时间维度更精确 | 需要维护多个桶 | 基于时间的窗口 |
| 时间窗口+计数 | 兼顾精度与性能 | 实现较复杂 | 需要精确统计 |

## 三、Netflix Hystrix 深度解析

### 3.1 Hystrix 核心设计

Hystrix 是 Netflix 开源的服务容错框架，虽然已停止维护（进入维护模式），但其设计理念仍是行业标杆。

**Hystrix 的核心机制**：

1. **资源隔离**：通过线程池或信号量隔离，防止故障蔓延
2. **熔断器**：基于失败率自动熔断
3. **降级回退**：提供 fallback 逻辑
4. **请求合并**：合并多个请求减少开销
5. **请求缓存**：缓存重复请求结果

### 3.2 资源隔离策略

#### 线程池隔离

```java
// Hystrix 线程池隔离配置
@HystrixCommand(
    groupKey = "userServiceGroup",
    commandKey = "getUserById",
    threadPoolKey = "userServicePool",
    threadPoolProperties = {
        @HystrixProperty(name = "coreSize", value = "10"),           // 核心线程数
        @HystrixProperty(name = "maximumSize", value = "20"),        // 最大线程数
        @HystrixProperty(name = "maxQueueSize", value = "100"),      // 队列大小
        @HystrixProperty(name = "keepAliveTimeMinutes", value = "1"),// 空闲线程存活时间
        @HystrixProperty(name = "queueSizeRejectionThreshold", value = "100") // 队列拒绝阈值
    },
    commandProperties = {
        @HystrixProperty(name = "execution.isolation.strategy", 
                        value = "THREAD")  // 线程池隔离
    }
)
public User getUserById(Long id) {
    return userClient.getUser(id);
}
```

**线程池隔离模型**：

```
Tomcat线程池          Hystrix线程池
┌─────────┐         ┌──────────────────┐
│ Thread-1 │────────►│ Hystrix-User-1   │──► 用户服务
│ Thread-2 │────────►│ Hystrix-User-2   │──► 用户服务
│ Thread-3 │────────►│ Hystrix-Order-1  │──► 订单服务
│ Thread-4 │────────►│ Hystrix-Order-2  │──► 订单服务
│ ...      │         │ ...              │
└─────────┘         └──────────────────┘

每个依赖服务有独立的线程池，互不影响
```

#### 信号量隔离

```java
@HystrixCommand(
    commandProperties = {
        @HystrixProperty(name = "execution.isolation.strategy", 
                        value = "SEMAPHORE"),  // 信号量隔离
        @HystrixProperty(name = "execution.isolation.semaphore.maxConcurrentRequests", 
                        value = "10")  // 最大并发数
    }
)
public User getUserFromCache(Long id) {
    return userCache.get(id);
}
```

**两种隔离策略对比**：

| 维度 | 线程池隔离 | 信号量隔离 |
|------|-----------|-----------|
| 线程开销 | 高（每个命令独立线程） | 低（在调用线程执行） |
| 超时控制 | 支持（可中断线程） | 不支持（无法中断） |
| 异步执行 | 支持 | 不支持 |
| 适用场景 | 远程调用（网络IO） | 本地调用（缓存/计算） |
| 隔离性 | 强（完全隔离） | 弱（共享调用线程） |

### 3.3 Hystrix 熔断器实现

```java
// Hystrix 熔断器配置
@HystrixCommand(
    commandProperties = {
        // 熔断器开启
        @HystrixProperty(name = "circuitBreaker.enabled", value = "true"),
        // 请求量阈值（窗口内达到此数量才开始计算）
        @HystrixProperty(name = "circuitBreaker.requestVolumeThreshold", value = "20"),
        // 错误百分比阈值
        @HystrixProperty(name = "circuitBreaker.errorThresholdPercentage", value = "50"),
        // 熔断器休眠时间（毫秒）
        @HystrixProperty(name = "circuitBreaker.sleepWindowInMilliseconds", value = "5000"),
        // 强制开启熔断器
        @HystrixProperty(name = "circuitBreaker.forceOpen", value = "false"),
        // 强制关闭熔断器
        @HystrixProperty(name = "circuitBreaker.forceClosed", value = "false"),
        // 统计窗口时间（毫秒）
        @HystrixProperty(name = "metrics.rollingStats.timeInMilliseconds", value = "10000"),
        // 统计窗口桶数
        @HystrixProperty(name = "metrics.rollingStats.numBuckets", value = "10")
    },
    fallbackMethod = "getUserFallback"
)
public User getUserById(Long id) {
    return userClient.getUser(id);
}

// 降级方法
private User getUserFallback(Long id) {
    log.warn("用户服务降级，userId={}", id);
    return User.defaultUser(id);  // 返回默认用户
}
```

**Hystrix 熔断器工作流程**：

```
1. 请求到达
     │
2. 检查熔断器状态
     ├── OPEN → 直接执行fallback
     ├── HALF-OPEN → 允许请求通过（试探）
     └── CLOSED → 继续执行
              │
3. 检查线程池/信号量
     ├── 满载 → 执行fallback
     └── 有余量 → 执行命令
              │
4. 执行远程调用
     ├── 成功 → 返回结果，更新统计
     └── 失败/超时 → 执行fallback，更新统计
              │
5. 统计计算
     └── 失败率 >= 阈值 → 切换为OPEN
```

### 3.4 Hystrix 请求合并

```java
// 请求合并器：将多个单条请求合并为批量请求
public class UserCollapseCommand extends HystrixCollapser<List<User>, User, Long> {

    private final Long userId;

    public UserCollapseCommand(Long userId) {
        super(Setter.withCollapserKey(HystrixCollapserKey.Factory.asKey("userCollapse"))
            .andCollapserPropertiesDefaults(
                HystrixCollapserProperties.Setter()
                    .withTimerDelayInMilliseconds(100)  // 合并窗口100ms
                    .withMaxRequestsInBatch(200)         // 最大合并数200
            ));
        this.userId = userId;
    }

    @Override
    public Long getRequestArgument() {
        return userId;
    }

    @Override
    protected HystrixCommand<List<User>> createCommand(
            Collection<CollapsedRequest<User, Long>> requests) {
        // 将多个userId合并为批量请求
        List<Long> userIds = requests.stream()
            .map(CollapsedRequest::getArgument)
            .collect(Collectors.toList());
        return new UserBatchCommand(userIds);
    }

    @Override
    protected void mapResponseToRequests(
            List<User> batchResponse,
            Collection<CollapsedRequest<User, Long>> requests) {
        // 将批量响应映射回单个请求
        Iterator<User> iter = batchResponse.iterator();
        requests.forEach(request -> request.response(iter.next()));
    }
}
```

### 3.5 Hystrix 的局限性

| 局限性 | 说明 |
|--------|------|
| 已停止维护 | 2018年进入维护模式，不再发布新版本 |
| 线程池开销 | 线程池隔离带来额外的线程切换和内存开销 |
| 不支持限流 | 只有熔断和降级，没有流量控制 |
| 配置复杂 | 大量注解和配置项，学习曲线陡峭 |
| 不支持热点参数 | 无法根据参数维度做细粒度控制 |
| 无实时监控 | 需要单独搭建 Hystrix Dashboard |

## 四、Alibaba Sentinel 深度解析

### 4.1 Sentinel 核心设计

Sentinel 是阿里巴巴开源的面向分布式服务架构的流量控制组件，以流量为切入点，提供流量控制、熔断降级、系统负载保护等多个维度来保障服务的稳定性。

**Sentinel 核心概念**：

1. **资源**：需要保护的内容（接口、方法、代码块）
2. **规则**：如何保护资源（限流、熔断、降级规则）
3. **Slot Chain**：责任链模式，每个Slot负责一个功能

**Sentinel vs Hystrix 对比**：

| 功能 | Hystrix | Sentinel |
|------|---------|----------|
| 熔断降级 | ✅ 基于异常比例 | ✅ 基于异常比例/慢调用比例 |
| 实时统计 | ✅ 滑动窗口 | ✅ 滑动窗口（LeapArray） |
| 流量控制 | ❌ | ✅ QPS/线程数/关联/链路 |
| 系统保护 | ❌ | ✅ Load/CPU/RT/线程数 |
| 热点参数 | ❌ | ✅ 参数级限流 |
| 控制台 | ✅ Dashboard | ✅ Sentinel Dashboard（更强大） |
| 扩展性 | 一般 | 强（SPI扩展机制） |
| 维护状态 | 停止维护 | 活跃维护 |

### 4.2 Sentinel 责任链（Slot Chain）

Sentinel 的核心处理流程是一个 Slot Chain（责任链），每个 Slot 负责不同的功能：

```
请求进入
   │
   ▼
┌──────────────────┐
│ NodeSelectorSlot │ ← 构建调用树（Context维度）
└────────┬─────────┘
         │
┌────────▼─────────┐
│ ClusterBuilderSlot│ ← 构建集群节点（Resource维度）
└────────┬─────────┘
         │
┌────────▼─────────┐
│    LogSlot       │ ← 记录日志
└────────┬─────────┘
         │
┌────────▼─────────┐
│  StatisticSlot   │ ← 实时数据统计（QPS、RT、线程数）
└────────┬─────────┘
         │
┌────────▼─────────┐
│  AuthoritySlot   │ ← 授权控制（黑白名单）
└────────┬─────────┘
         │
┌────────▼─────────┐
│  SystemSlot      │ ← 系统保护（Load、CPU、RT、线程数）
└────────┬─────────┘
         │
┌────────▼─────────┐
│  FlowSlot        │ ← 流量控制（QPS、线程数）
└────────┬─────────┘
         │
┌────────▼─────────┐
│  DegradeSlot     │ ← 熔断降级（异常比例/慢调用）
└────────┬─────────┘
         │
     执行业务逻辑
```

### 4.3 Sentinel 流量控制

#### QPS 限流

```java
// QPS限流规则
FlowRule flowRule = new FlowRule("getUser")
    .setCount(100)                           // QPS阈值100
    .setGrade(RuleConstant.FLOW_GRADE_QPS)   // 按QPS限流
    .setLimitApp("default")                  // 来源应用
    .setStrategy(RuleConstant.FLOW_CHAIN);   // 链路限流

FlowRuleManager.loadRules(Collections.singletonList(flowRule));
```

**流控模式**：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **直接** | 资源自身达到阈值时限流 | 接口自身保护 |
| **关联** | 关联资源达到阈值时限流当前资源 | 写操作影响读操作时 |
| **链路** | 只统计从指定入口进入的流量 | 精细化链路控制 |

**关联限流示例**：
```java
// 当写接口QPS达到阈值时，限制读接口
// 场景：数据库写操作压力大时，限制读操作保护数据库
FlowRule readRule = new FlowRule("readResource")
    .setCount(100)
    .setGrade(RuleConstant.FLOW_GRADE_QPS)
    .setStrategy(RuleConstant.FLOW_RELATE)        // 关联模式
    .setRefResource("writeResource");              // 关联写接口

FlowRule writeRule = new FlowRule("writeResource")
    .setCount(50)
    .setGrade(RuleConstant.FLOW_GRADE_QPS);

FlowRuleManager.loadRules(Arrays.asList(readRule, writeRule));
```

**流控效果**：

| 效果 | 说明 | 特点 |
|------|------|------|
| **快速失败** | 直接抛出 FlowException | 默认方式，适合大多数场景 |
| **Warm Up** | 冷启动，逐渐增大流量 | 适合突发流量场景 |
| **排队等待** | 请求匀速通过 | 适合削峰填谷场景 |

#### Warm Up（冷启动）

```java
// Warm Up：冷启动模式
FlowRule warmUpRule = new FlowRule("getUser")
    .setCount(100)                               // 最终QPS阈值
    .setGrade(RuleConstant.FLOW_GRADE_QPS)
    .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_WARM_UP)  // 冷启动
    .setWarmUpPeriodSec(10);                     // 预热时长10秒

// 效果：系统刚启动时QPS阈值为 100/3 ≈ 33，10秒内逐渐增大到100
// 适用场景：数据库连接池需要预热、缓存需要填充
```

#### 排队等待（匀速器）

```java
// 排队等待：请求匀速通过
FlowRule rateLimiterRule = new FlowRule("processOrder")
    .setCount(100)                                    // QPS阈值
    .setGrade(RuleConstant.FLOW_GRADE_QPS)
    .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_RATE_LIMITER)  // 排队等待
    .setMaxQueueingTimeMs(5000);                      // 最大排队时间5秒

// 效果：请求以匀速方式通过，间隔10ms一个请求
// 适用场景：消息队列消费者、定时任务触发
```

### 4.4 Sentinel 熔断降级

Sentinel 提供三种熔断策略：

#### 慢调用比例

```java
// 慢调用比例熔断规则
DegradeRule slowCallRule = new DegradeRule("getUser")
    .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType())
    .setCount(0.5)           // 慢调用比例阈值：50%
    .setSlowRatioThreshold(0.5)
    .setTimeWindow(10)       // 熔断时长10秒
    .setMinRequestAmount(5)  // 最小请求数5
    .setStatIntervalMs(1000);// 统计时长1秒
```

#### 异常比例

```java
// 异常比例熔断规则
DegradeRule exceptionRatioRule = new DegradeRule("getUser")
    .setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType())
    .setCount(0.5)           // 异常比例阈值：50%
    .setTimeWindow(10)       // 熔断时长10秒
    .setMinRequestAmount(5)  // 最小请求数5
    .setStatIntervalMs(1000);// 统计时长1秒
```

#### 异常数

```java
// 异常数熔断规则
DegradeRule exceptionCountRule = new DegradeRule("getUser")
    .setGrade(CircuitBreakerStrategy.ERROR_COUNT.getType())
    .setCount(10)            // 异常数阈值：10次
    .setTimeWindow(60)       // 熔断时长60秒
    .setMinRequestAmount(5)  // 最小请求数5
    .setStatIntervalMs(60000);// 统计时长60秒
```

**三种熔断策略对比**：

| 策略 | 触发条件 | 适用场景 | 优势 |
|------|----------|----------|------|
| 慢调用比例 | 慢调用比例超阈值 | 对延迟敏感的服务 | 提前预警，不等报错 |
| 异常比例 | 异常比例超阈值 | 不稳定的依赖服务 | 识别部分故障 |

### 4.8 Sentinel 核心数据结构：LeapArray

LeapArray 是 Sentinel 实现滑动窗口的核心数据结构：

```java
// LeapArray 核心实现（简化版）
public class LeapArray<T> {
    // 窗口长度（毫秒）
    protected int windowLengthInMs;
    // 采样窗口数量
    protected int sampleCount;
    // 总时间窗口长度（毫秒）
    protected int intervalInMs;
    // 窗口数组（AtomicReferenceArray保证线程安全）
    protected final AtomicReferenceArray<WindowWrap<T>> array;

    /**
     * 根据当前时间获取对应的窗口
     * 核心算法：timeId = currentTimeMillis / windowLengthInMs
     *          idx = timeId % array.length()
     */
    public WindowWrap<T> currentWindow(long currentTimeMillis) {
        // 计算当前时间对应的窗口索引
        long timeId = currentTimeMillis / windowLengthInMs;
        int idx = (int) (timeId % array.length());

        // 计算窗口开始时间
        long windowStart = currentTimeMillis 
            - (currentTimeMillis % windowLengthInMs);

        while (true) {
            WindowWrap<T> old = array.get(idx);
            if (old == null) {
                // 创建新窗口
                WindowWrap<T> window = new WindowWrap<>(
                    windowLengthInMs, windowStart, newEmptyBucket());
                if (array.compareAndSet(idx, null, window)) {
                    return window;
                }
            } else if (windowStart == old.windowStart()) {
                // 当前窗口已存在且时间匹配
                return old;
            } else if (windowStart > old.windowStart()) {
                // 窗口已过期，重置
                if (updateLock.tryLock()) {
                    try {
                        return resetWindowTo(old, windowStart);
                    } finally {
                        updateLock.unlock();
                    }
                }
            }
        }
    }
}
```

**LeapArray 时间复杂度分析**：

| 操作 | 时间复杂度 | 说明 |
|------|-----------|------|
| 获取当前窗口 | O(1) | 取模计算索引 |
| 更新窗口数据 | O(1) | CAS更新 |
| 统计窗口数据 | O(n) | n=sampleCount，遍历窗口 |
| 重置过期窗口 | O(1) | 直接替换 |

### 7.3 多级降级实战

```java
@Service
public class ProductService {
    
    @Autowired
    private ProductClient productClient;
    @Autowired
    private RedisTemplate<String, Product> redisTemplate;
    @Autowired
    private LocalCacheManager localCache;
    
    /**
     * 三级降级策略：
     * 1. 正常调用远程服务
     * 2. 远程服务失败 → 从Redis缓存获取
     * 3. Redis缓存未命中 → 从本地缓存获取
     * 4. 本地缓存未命中 → 返回默认商品
     */
    @SentinelResource(
        value = "getProduct",
        blockHandler = "getProductBlockHandler",
        fallback = "getProductFallback"
    )
    public Product getProduct(Long productId) {
        // 第一级：远程调用
        Product product = productClient.getProduct(productId);
        // 成功后写入缓存
        redisTemplate.opsForValue().set(
            "product:" + productId, product, 30, TimeUnit.MINUTES);
        localCache.put("product:" + productId, product);
        return product;
    }
    
    // 熔断/限流时的降级处理
    public Product getProductBlockHandler(Long productId, BlockException ex) {
        log.warn("商品服务被限流/熔断，productId={}", productId);
        return getProductFromCache(productId);
    }
    
    // 异常时的降级处理
    public Product getProductFallback(Long productId, Throwable ex) {
        log.error("商品服务异常，productId={}", productId, ex);
        return getProductFromCache(productId);
    }
    
    private Product getProductFromCache(Long productId) {
        // 第二级：Redis缓存
        Product product = redisTemplate.opsForValue()
            .get("product:" + productId);
        if (product != null) {
            log.info("从Redis缓存获取商品，productId={}", productId);
            return product;
        }
        
        // 第三级：本地缓存
        product = localCache.get("product:" + productId, Product.class);
        if (product != null) {
            log.info("从本地缓存获取商品，productId={}", productId);
            return product;
        }
        
        // 第四级：默认值
        log.warn("所有降级策略失效，返回默认商品，productId={}", productId);
        return Product.defaultProduct(productId);
    }
}
```

## 九、面试高频问题

### Q1：熔断器的三种状态是什么？如何转换？

**答**：熔断器有三种状态——CLOSED、OPEN、HALF-OPEN。

- **CLOSED → OPEN**：当统计窗口内的失败率（或慢调用率）超过阈值，并且请求量达到最小请求量时，熔断器打开。
- **OPEN → HALF-OPEN**：经过 waitDurationInOpenState 后，熔断器自动转入半开状态。
- **HALF-OPEN → CLOSED**：半开状态允许少量请求通过，如果这些请求成功率达标，说明服务已恢复，切换回关闭状态。
- **HALF-OPEN → OPEN**：如果半开状态的请求仍然失败，说明服务未恢复，切换回打开状态。

### Q2：Hystrix 和 Sentinel 的主要区别？

**答**：
1. **功能维度**：Sentinel 功能更全面，除了熔断降级还有流量控制、系统保护、热点参数限流等；Hystrix 只有熔断降级和资源隔离。
2. **隔离策略**：Hystrix 默认线程池隔离，开销大；Sentinel 基于信号量计数器，轻量级。
3. **动态配置**：Sentinel 支持动态规则推送（Nacos/ZooKeeper/Apollo），Hystrix 需要重启。
4. **监控能力**：Sentinel Dashboard 功能更强大，支持实时监控和规则管理。
5. **维护状态**：Hystrix 已停止维护，Sentinel 仍活跃维护。

### Q3：什么是雪崩效应？如何防止？

**答**：雪崩效应是指一个服务的故障引发连锁反应，导致整个系统崩溃。

**防止方法**：
1. **熔断**：快速切断对故障服务的调用，防止故障扩散
2. **降级**：提供兜底方案，保证核心功能可用
3. **限流**：限制请求量，防止系统过载
4. **资源隔离**：线程池/信号量隔离，防止故障蔓延
5. **超时控制**：设置合理的超时时间，避免线程长时间阻塞
6. **重试机制**：合理重试（注意重试风暴）

### Q4：Sentinel 的热点参数限流是什么？适用场景？

**答**：热点参数限流是 Sentinel 独有的功能，可以根据请求参数的值维度做细粒度的限流。比如对热门商品ID、VIP用户ID等特殊参数值设置不同的限流阈值。

适用场景：
- 电商秒杀：热门商品更严格的限流
- 社交平台：热点用户做特殊处理
- 多租户系统：不同租户差异化限流

### Q5：如何设计多级降级策略？

**答**：多级降级策略从优先级高到低：
1. **远程调用** → 正常路径
2. **分布式缓存（Redis）** → 数据可能略有过期但可用
3. **本地缓存（Caffeine/Guava）** → 数据可能更旧但仍可用
4. **默认值** → 最后兜底，保证不报错

关键点：
- 核心链路不允许降级（如支付、下单），直接快速失败
- 非核心链路优先保证可用性（如推荐、评价），允许降级
- 缓存数据要设置合理的过期时间
- 降级策略要有监控和告警

### Q6：熔断器的时间窗口有哪些类型？各有什么特点？

**答**：
1. **基于计数的滑动窗口**：统计最近N次请求的指标。优点是内存高效（环形数组），缺点是对突发流量不敏感。
2. **基于时间的滑动窗口**：统计最近T时间内的指标。优点是对时间维度更精确，缺点是实现稍复杂。

Hystrix 使用基于时间的滑动窗口（10秒），Sentinel 使用 LeapArray 实现两种窗口，Resilience4j 也支持两种类型。

### Q7：Sentinel 的 Slot Chain 是什么？执行流程？

**答**：Slot Chain 是 Sentinel 的核心处理链，采用责任链模式，每个 Slot 负责一个功能：
1. **NodeSelectorSlot**：构建调用树
2. **ClusterBuilderSlot**：构建集群节点
3. **LogSlot**：记录日志
4. **StatisticSlot**：实时数据统计
5. **AuthoritySlot**：授权控制
6. **SystemSlot**：系统保护
7. **FlowSlot**：流量控制
8. **DegradeSlot**：熔断降级

请求依次通过每个 Slot，任何一个 Slot 拦截请求都会抛出 BlockException。

### Q8：Resilience4j 如何组合多个容错功能？

**答**：Resilience4j 提供 Decorators 工具类和函数式组合方式：

```java
Supplier<User> decorated = Decorators.ofSupplier(() -> client.call())
    .withCircuitBreaker(circuitBreaker)
    .withRateLimiter(rateLimiter)
    .withRetry(retry)
    .withBulkhead(bulkhead)
    .get();
```

也可以使用 Spring 注解组合：
```java
@CircuitBreaker(name = "service")
@Bulkhead(name = "service")
@RateLimiter(name = "service")
public User call() { ... }
```

注意：装饰器的顺序很重要，建议顺序为 RateLimiter → Bulkhead → CircuitBreaker → Retry → 实际调用。

## 十、总结

### 核心要点

1. **熔断器是状态机**：CLOSED → OPEN → HALF-OPEN 三态转换，是应对雪崩效应的第一道防线
2. **降级是兜底策略**：保证核心功能可用，非核心功能可以牺牲
3. **Hystrix 是历史**：已停止维护，但设计理念值得学习
4. **Sentinel 是全能王**：功能最全面，适合 Spring Cloud Alibaba 体系
5. **Resilience4j 是轻量之选**：函数式风格，Spring Cloud 默认推荐

### 最佳实践

1. 核心链路不降级，非核心链路多级降级
2. 熔断阈值要根据业务特点设置（核心服务更低容错率）
3. 监控和告警必不可少（熔断器OPEN必须P1告警）
4. 规则要支持动态配置（不重启生效）
5. 降级策略要有演练（定期验证降级是否有效）
6. 注意重试风暴（重试间隔要递增，重试次数要合理）

### 技术演进路线

```
Hystrix（Netflix，已停止维护）
    │
    ├──────► Resilience4j（官方推荐替代，轻量级）
    │
    └──────► Sentinel（阿里巴巴，功能最全）
```

选择建议：
- **新项目**：Spring Cloud Alibaba → Sentinel；Spring Cloud → Resilience4j
- **老项目**：Hystrix → 迁移到 Resilience4j
- **全功能需求**：Sentinel（限流+熔断+系统保护+热点参数）
