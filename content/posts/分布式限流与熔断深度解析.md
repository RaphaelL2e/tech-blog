## 一、引言

在高并发分布式系统中，限流和熔断是保护系统的两道核心防线。限流控制请求速率，熔断隔离故障服务，两者配合确保系统在流量洪峰和下游故障时仍能稳定运行。

1. 限流算法（固定窗口、滑动窗口、令牌桶、漏桶）

4. Sentinel 实战
5. 服务降级策略
6. 面试高频问题

## 二、限流算法

### 2.1 固定窗口计数器

**原理**：将时间划分为固定窗口，每个窗口内维护一个计数器。

```
时间轴：[0s, 1s) [1s, 2s) [2s, 3s) ...
限流：100 次/秒

窗口 1（0-1s）：计数 = 98  ✅ 允许
窗口 2（1-2s）：计数 = 105 ❌ 拒绝 5 个请求
窗口 3（2-3s）：计数 = 50  ✅ 允许
```

**临界问题（边界突发）**：

```
0.9s-1.0s：95 个请求（窗口 1）
1.0s-1.1s：95 个请求（窗口 2）
→ 0.2 秒内 190 个请求！远超 100/秒的限流

虽然每个窗口都不超过 100，但窗口边界处可能出现 2 倍流量
```

**代码实现**：

```java
public class FixedWindowRateLimiter {
    
    private final long windowSize;       // 窗口大小（毫秒）
    private final int maxRequests;       // 窗口内最大请求数
    private long windowStart;            // 当前窗口起始时间
    private int counter;                 // 当前窗口计数器
    
    public FixedWindowRateLimiter(long windowSizeMs, int maxRequests) {
        this.windowSize = windowSizeMs;
        this.maxRequests = maxRequests;
        this.windowStart = System.currentTimeMillis();
        this.counter = 0;
    }
    
    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        
        // 检查是否需要切换窗口
        if (now - windowStart >= windowSize) {
            windowStart = now;
            counter = 0;
        }
        
        // 判断是否超过限制
        if (counter < maxRequests) {
            counter++;
            return true;
        }
        return false;
    }
}
```

### 2.2 滑动窗口计数器

**原理**：将窗口细分为多个小窗口，滑动统计。

```
1 秒窗口，分为 10 个小窗口（每个 100ms）

0-100ms  100-200ms  ...  900-1000ms
  10        12      ...     8

当前 0.5s-1.5s 的请求数 = 后 5 个小窗口之和
= 小窗口[5] + [6] + [7] + [8] + [9]
```

**优点**：解决了固定窗口的边界突发问题

**代码实现**：

```java
public class SlidingWindowRateLimiter {
    
    private final long windowSize;       // 总窗口大小（毫秒）
    private final int subWindowCount;    // 子窗口数
    private final int maxRequests;       // 总窗口最大请求数
    private final long subWindowSize;    // 子窗口大小
    private final int[] counters;        // 每个子窗口的计数器
    private long windowStart;            // 窗口起始时间
    
    public SlidingWindowRateLimiter(long windowSizeMs, int subWindowCount, int maxRequests) {
        this.windowSize = windowSizeMs;
        this.subWindowCount = subWindowCount;
        this.maxRequests = maxRequests;
        this.subWindowSize = windowSizeMs / subWindowCount;
        this.counters = new int[subWindowCount];
        this.windowStart = System.currentTimeMillis();
    }
    
    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        
        // 计算当前子窗口索引
        int currentIndex = (int) ((now - windowStart) / subWindowSize) % subWindowCount;
        
        // 检查窗口是否滑动
        long elapsedWindows = (now - windowStart) / subWindowSize;
        if (elapsedWindows >= subWindowCount) {
            // 重置所有计数器
            for (int i = 0; i < subWindowCount; i++) {
                counters[i] = 0;
            }
            windowStart = now - (now - windowStart) % subWindowSize;
        }
        
        // 统计当前窗口内的总请求数
        int totalRequests = 0;
        for (int count : counters) {
            totalRequests += count;
        }
        
        // 判断是否超过限制
        if (totalRequests < maxRequests) {
            counters[currentIndex]++;
            return true;
        }
        return false;
    }
}
```

### 2.3 漏桶算法

**原理**：请求进入漏桶，以固定速率流出。

```
请求 →  ┌──────────┐  → 以固定速率处理
        │  漏桶     │
        │  ═══════  │
        │  ↓↓↓↓↓↓↓  │  → 溢出则丢弃
        └──────────┘

特点：
  - 固定处理速率（平滑输出）
  - 突发流量被削平
  - 溢出的请求直接丢弃

适用：保护下游服务（控制处理速率）
```

**代码实现**：

```java
public class LeakyBucketRateLimiter {
    
    private final int capacity;           // 桶容量
    private final double leakRate;        // 漏出速率（个/毫秒）
    private int water;                    // 当前水量
    private long lastLeakTime;            // 上次漏水时间
    
    public LeakyBucketRateLimiter(int capacity, double leakRatePerSecond) {
        this.capacity = capacity;
        this.leakRate = leakRatePerSecond / 1000.0;
        this.water = 0;
        this.lastLeakTime = System.currentTimeMillis();
    }
    
    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        
        // 计算漏出的水量
        int leaked = (int) ((now - lastLeakTime) * leakRate);
        if (leaked > 0) {
            water = Math.max(0, water - leaked);
            lastLeakTime = now;
        }
        
        // 判断桶是否已满
        if (water < capacity) {
            water++;
            return true;
        }
        return false;
    }
}
```

### 2.4 令牌桶算法

**原理**：以固定速率生成令牌放入桶中，请求需要获取令牌才能通过。

```
              令牌（固定速率生成）
                ↓ ↓ ↓
        ┌──────────────┐
请求 →  │  令牌桶       │ → 获取令牌 → 处理
        │  ○ ○ ○ ○ ○ ○ │
        │  (最多 N 个)  │ → 无令牌 → 拒绝/等待
        └──────────────┘

特点：
  - 允许突发流量（桶中有令牌时）
  - 长期平均速率受控
  - 最常用的限流算法

适用：API 网关限流（允许短时突发）
```

**代码实现**：

```java
public class TokenBucketRateLimiter {
    
    private final int capacity;            // 桶容量（最大令牌数）
    private final double refillRate;       // 令牌生成速率（个/毫秒）
    private int tokens;                    // 当前令牌数
    private long lastRefillTime;           // 上次补充时间
    
    public TokenBucketRateLimiter(int capacity, double refillRatePerSecond) {
        this.capacity = capacity;
        this.refillRate = refillRatePerSecond / 1000.0;
        this.tokens = capacity;            // 初始满桶
        this.lastRefillTime = System.currentTimeMillis();
    }
    
    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        
        // 补充令牌
        int newTokens = (int) ((now - lastRefillTime) * refillRate);
        if (newTokens > 0) {
            tokens = Math.min(capacity, tokens + newTokens);
            lastRefillTime = now;
        }
        
        // 尝试获取令牌
        if (tokens > 0) {
            tokens--;
            return true;
        }
        return false;
    }
    
    // 获取多个令牌（批量请求）
    public synchronized boolean tryAcquire(int permits) {
        long now = System.currentTimeMillis();
        
        int newTokens = (int) ((now - lastRefillTime) * refillRate);
        if (newTokens > 0) {
            tokens = Math.min(capacity, tokens + newTokens);
            lastRefillTime = now;
        }
        
        if (tokens >= permits) {
            tokens -= permits;
            return true;
        }
        return false;
    }
}
```

### 2.5 四种算法对比

| 算法 | 突发流量 | 平滑输出 | 实现复杂度 | 适用场景 |
|------|---------|---------|-----------|---------|
| **固定窗口** | 允许（边界突发2倍） | 否 | 低 | 简单限流 |
| **滑动窗口** | 限制 | 较平滑 | 中 | 精确限流 |
| **漏桶** | 削平 | 严格平滑 | 中 | 保护下游 |
| **令牌桶** | 允许（桶内令牌） | 长期平滑 | 中 | API 网关 |

## 三、分布式限流

### 3.1 单机限流的局限

```
问题：每个节点各自限流 100 QPS，3 个节点 = 300 QPS
→ 全局限流 100 QPS 的目标被突破

解决：分布式限流（全局计数器）
```

### 3.2 Redis + Lua 分布式限流

```lua
-- 令牌桶算法（Redis + Lua）
local key = KEYS[1]           -- 限流 Key
local capacity = tonumber(ARGV[1])  -- 桶容量
local rate = tonumber(ARGV[2])      -- 令牌生成速率（个/秒）
local now = tonumber(ARGV[3])       -- 当前时间（毫秒）
local requested = tonumber(ARGV[4]) -- 请求数

-- 获取当前令牌数和上次补充时间
local tokens = tonumber(redis.call('hget', key, 'tokens') or capacity)
local last_time = tonumber(redis.call('hget', key, 'last_time') or now)

-- 计算补充的令牌
local elapsed = (now - last_time) / 1000
local new_tokens = elapsed * rate
tokens = math.min(capacity, tokens + new_tokens)

-- 尝试获取令牌
if tokens >= requested then
    tokens = tokens - requested
    redis.call('hset', key, 'tokens', tokens)
    redis.call('hset', key, 'last_time', now)
    return 1  -- 允许
else
    redis.call('hset', key, 'tokens', tokens)
    redis.call('hset', key, 'last_time', now)
    return 0  -- 拒绝
end
```

## 四、熔断器模式

### 4.1 熔断器状态机

```
         ┌──────────────────────────────────────────┐
         │                                          │
         ↓   失败率超过阈值                          │ 成功率恢复
    ┌─────────┐           ┌──────────┐         ┌─────────┐
    │  CLOSED │ ────────→ │   OPEN   │ ──────→ │HALF-OPEN│
    │ (正常)   │           │ (熔断)    │         │ (半开)   │
    └─────────┘           └──────────┘         └─────────┘
         ↑                                          │
         │           探测成功                         │
         └──────────────────────────────────────────┘
                                          │ 探测失败
                                          ↓
                                    ┌──────────┐
                                    │   OPEN   │
                                    └──────────┘
```

**三个状态**：

| 状态 | 行为 | 转换条件 |
|------|------|---------|
| **CLOSED** | 正常放行请求 | 失败率超过阈值 → OPEN |
| **OPEN** | 直接拒绝/降级 | 超时后 → HALF-OPEN |
| **HALF-OPEN** | 放行少量探测请求 | 成功 → CLOSED，失败 → OPEN |

### 4.2 熔断器关键参数

| 参数 | 描述 | 典型值 |
|------|------|--------|
| **failureRateThreshold** | 失败率阈值 | 50% |
| **slowCallRateThreshold** | 慢调用率阈值 | 80% |
| **waitDurationInOpenState** | OPEN 状态持续时间 | 30s |
| **permittedNumberOfCallsInHalfOpenState** | HALF-OPEN 放行数 | 10 |
| **slidingWindowSize** | 滑动窗口大小 | 100 次调用 |
| **slidingWindowType** | 窗口类型 | COUNT_BASED / TIME_BASED |

### 4.3 熔断器代码实现

```java
public class CircuitBreaker {
    
    public enum State { CLOSED, OPEN, HALF_OPEN }
    
    private State state = State.CLOSED;
    private final double failureRateThreshold;
    private final long openTimeoutMs;
    
    private int totalCount = 0;
    private int failureCount = 0;
    private long openTime;
    
    public CircuitBreaker(double failureRateThreshold, long openTimeoutMs) {
        this.failureRateThreshold = failureRateThreshold;
        this.openTimeoutMs = openTimeoutMs;
    }
    
    public synchronized boolean allowRequest() {
        switch (state) {
            case CLOSED:
                return true;
            case OPEN:
                if (System.currentTimeMillis() - openTime >= openTimeoutMs) {
                    state = State.HALF_OPEN;
                    return true;  // 放行探测请求
                }
                return false;
            case HALF_OPEN:
                return true;  // 放行少量请求
            default:
                return false;
        }
    }
    
    public synchronized void recordSuccess() {
        if (state == State.HALF_OPEN) {
            state = State.CLOSED;
            reset();
        }
        totalCount++;
    }
    
    public synchronized void recordFailure() {
        failureCount++;
        totalCount++;
        
        if (state == State.HALF_OPEN) {
            state = State.OPEN;
            openTime = System.currentTimeMillis();
            return;
        }
        
        if (totalCount >= 10 && (double) failureCount / totalCount >= failureRateThreshold) {
            state = State.OPEN;
            openTime = System.currentTimeMillis();
        }
    }
    
    private void reset() {
        totalCount = 0;
        failureCount = 0;
    }
}
```

## 五、Sentinel 实战

### 5.1 Sentinel 概述

**Sentinel** 是阿里开源的流量防护组件，提供限流、熔断、系统保护等功能。

**核心概念**：

| 概念 | 描述 |
|------|------|
| **Resource** | 受保护的资源（方法、接口） |
| **Rule** | 规则（限流、熔断、系统保护） |
| **Slot Chain** | 责任链（统计、限流、熔断、降级） |

### 5.2 Sentinel 限流

```java
// 代码方式
try (Entry entry = SphU.entry("orderService")) {
    // 受保护的业务逻辑
    doBusiness();
} catch (BlockException e) {
    // 被限流/熔断
    handleBlock(e);
}

// 注解方式
@SentinelResource(value = "orderService", blockHandler = "handleBlock")
public Order getOrder(Long orderId) {
    return orderMapper.selectById(orderId);
}

// 限流规则
FlowRule rule = new FlowRule("orderService")
    .setCount(100)                   // QPS 阈值
    .setGrade(RuleConstant.FLOW_GRADE_QPS)  // QPS 模式
    .setControlBehavior(RuleConstant.CONTROL_BEHAVIOR_DEFAULT);  // 直接拒绝
FlowRuleManager.loadRules(Collections.singletonList(rule));
```

**流控模式**：

| 模式 | 描述 |
|------|------|
| **直接** | 资源自身限流 |
| **关联** | 关联资源超阈值时限流自身 |
| **链路** | 只统计从入口进来的流量 |

**流控效果**：

| 效果 | 描述 |
|------|------|
| **快速失败** | 直接抛出异常 |
| **Warm Up** | 预热，从阈值/3 逐渐升到阈值 |
| **排队等待** | 请求排队，匀速通过（漏桶） |

### 5.3 Sentinel 熔断

```java
// 熔断规则（慢调用比例）
DegradeRule rule = new DegradeRule("orderService")
    .setGrade(CircuitBreakerStrategy.SLOW_REQUEST_RATIO.getType())  // 慢调用比例
    .setCount(0.5)       // 慢调用比例阈值 50%
    .setSlowRatioThreshold(0.5)
    .setTimeWindow(30)   // 熔断持续时间 30s
    .setMinRequestAmount(10)  // 最小请求数
    .setStatIntervalMs(60000); // 统计时长 60s

// 熔断规则（异常比例）
DegradeRule rule2 = new DegradeRule("orderService")
    .setGrade(CircuitBreakerStrategy.ERROR_RATIO.getType())  // 异常比例
    .setCount(0.5)       // 异常比例阈值 50%
    .setTimeWindow(30);

// 熔断规则（异常数）
DegradeRule rule3 = new DegradeRule("orderService")
    .setGrade(CircuitBreakerStrategy.ERROR_COUNT.getType())  // 异常数
    .setCount(10)        // 异常数阈值
    .setTimeWindow(60);

DegradeRuleManager.loadRules(Arrays.asList(rule, rule2, rule3));
```

## 六、服务降级策略

### 6.1 降级方式

| 降级方式 | 描述 | 示例 |
|---------|------|------|
| **返回默认值** | 返回预设值 | 推荐系统降级 → 返回热门列表 |
| **返回缓存值** | 返回上次成功的结果 | 商品详情降级 → 返回缓存 |
| **简化逻辑** | 减少非核心逻辑 | 搜索降级 → 不算推荐分 |
| **写降级** | 同步写 → 异步写 | 日志降级 → 写本地文件 |
| **读降级** | 读数据库 → 读缓存 | 用户信息降级 → 读 Redis |

### 6.2 降级代码实现

```java
// 使用 Sentinel + 降级
@SentinelResource(value = "getRecommendations",
    blockHandler = "getRecommendationsBlock",
    fallback = "getRecommendationsFallback")
public List<Product> getRecommendations(Long userId) {
    return recommendService.getPersonalized(userId);
}

// 限流降级（BlockException）
public List<Product> getRecommendationsBlock(Long userId, BlockException e) {
    return getHotProducts();  // 返回热门商品
}

// 异常降级（业务异常）
public List<Product> getRecommendationsFallback(Long userId, Throwable t) {
    return getCachedRecommendations(userId);  // 返回缓存
}
```

### 6.3 多级降级

```
正常：推荐服务（个性化推荐）
   ↓ 限流/熔断
降级1：热门列表（通用推荐）
   ↓ 热门服务也挂
降级2：本地缓存（上次成功的结果）
   ↓ 缓存也过期
降级3：静态数据（配置文件预设）
```

## 七、限流与熔断的配合

```
请求 → 网关限流（全局限流）
         ↓ 通过
       服务限流（单机限流）
         ↓ 通过
       熔断器检查
         ↓ CLOSED
       调用下游服务
         ↓ 成功/失败
       更新熔断器状态

任何一环被拒绝 → 执行降级策略
```

**配置建议**：

```
网关限流：粗粒度（按服务/API 限流）
服务限流：细粒度（按方法限流）
熔断器：保护下游（失败率/慢调用）
降级策略：兜底方案（返回默认值/缓存）
```

## 八、面试高频问题

### Q1: 令牌桶和漏桶的区别？

**答案要点**：
- 漏桶：固定处理速率，削平突发，严格平滑
- 令牌桶：允许突发（桶内有令牌时），长期平均受控
- 令牌桶更适合 API 网关（允许短时突发）
- 漏桶更适合保护下游（严格控制速率）

### Q2: 熔断器的三个状态？

**答案要点**：
- CLOSED：正常放行，统计失败率
- OPEN：直接拒绝，超时后进入 HALF-OPEN
- HALF-OPEN：放行探测请求，成功→CLOSED，失败→OPEN

### Q3: 如何实现分布式限流？

**答案要点**：
- Redis + Lua 脚本（原子操作）
- 网关层限流（Nginx / Spring Cloud Gateway）
- 中心化限流服务（Sentinel Dashboard）

### Q4: Sentinel 的限流模式有哪些？

**答案要点**：
- 直接：资源自身限流
- 关联：关联资源超阈值时限流自身
- 链路：只统计从入口进来的流量
- 效果：快速失败 / Warm Up / 排队等待

### Q5: 限流和熔断的区别？

**答案要点**：
- 限流：控制请求速率（主动保护）
- 熔断：隔离故障服务（被动保护）
- 限流是入口控制，熔断是出口保护
- 两者配合：限流防过载，熔断防级联故障
