在面试中，评估大语言模型（LLM）的能力和设计有效的提示词，是两个出现频率极高的话题。前者考察你对模型能力的系统性认知，后者考察你对模型交互方式的工程理解。掌握这两块内容，不仅能回答面试问题，更能直接提升你在实际项目中使用大模型的效率。

本文从**大模型评估体系**和**提示工程实战**两个维度展开，既覆盖主流评测基准的理论背景，也给出可直接落地的提示词工程技巧。

## 一、大模型评估的挑战与分层体系

### 1.1 为什么评估 LLM 如此困难

传统软件评测有明确的正确输出，模型测试有精确的准确率指标，但 LLM 的评估面临三个根本性挑战：

**开放式生成的本质**：LLM 的输出是自然语言，同一个问题可以有多种合理答案。例如让模型解释量子纠缠，"玻尔比喻"和"爱因斯坦的思考"都是正确答案。传统的精确匹配评测方式完全失效。

**能力边界的不透明性**：我们无法直接观察模型的"思考过程"。一个模型可能在数学推导上表现优异，却在简单常识推理上犯低级错误。能力边界的不均匀分布使得单维度评测无法反映真实水平。

**任务多样性与主观性**：代码生成要求语法正确，创意写作要求文采优美，客户服务要求语气得体。不同任务对"好"的标准截然不同，同一个模型在不同任务上的表现可能天差地别。

正是这些挑战催生了多种评估范式的并存。

### 1.2 评估任务的三层分类

按照评测目的，LLM 评估可以分为三个层次：

**第一层：通用能力评测（General Benchmark）**

关注模型的综合认知能力，涵盖语言理解、知识推理、数学计算、代码生成等维度。目标是评估模型在通用任务上的整体水平。MMLU、Big-Bench Hard、CMMLU 等基准属于这一层。

**第二层：垂直任务评测（Task-specific Benchmark）**

针对特定应用场景设计评测集，考察模型在专业领域的能力。例如：
- **代码场景**：HumanEval、MBPP、LiveCodeBench
- **数学场景**：GSM8K、MATH
- **医疗场景**：MedQA、PubMedQA
- **法律场景**：JEC-QA

**第三层：人类偏好评测（Human Preference）**

有些能力无法通过自动化指标衡量——风格的优雅性、回答的实用性、与用户意图的对齐程度。这些需要人类评估员对模型输出进行主观打分。Chatbot Arena（后文详述）就是这一类评估的代表。

理解这三层分类是回答面试题"如何评估一个大模型"的基础框架。

## 二、主流评测基准详解

### 2.1 MMLU：大规模多任务语言理解

**MMLU**（Massive Multitask Language Understanding）由研究者于 2020 年提出，是当前最具影响力的通用评测基准之一。

**数据集构成**：涵盖 57 个学科领域，从基础数学（AMC 10/12）、大学物理，到历史、法律、伦理，形成了对模型知识广度的全面考察。

**评测格式**：所有题目均为四选一选择题。这种格式的优势在于可以自动化评分，无需人工介入或复杂的文本生成评估。

**评分标准**：衡量模型正确率，57 个领域取加权平均得到最终分数。

**代表性数据**（供参考）：
- GPT-3（2020）：约 43%
- GPT-3.5（2022）：约 70%
- GPT-4（2023）：约 86%
- Claude 3 Opus（2024）：约 88%

**面试关注点**：MMLU 反映的是模型的**知识召回能力**——模型需要从训练数据中准确提取并应用知识。高 MMLU 分数意味着模型在大量知识密集型任务上表现可靠，但并不代表它在推理深度、创造性或对话体验上同样出色。

### 2.2 Big-Bench Hard：拒绝捷径推理

**Big-Bench Hard**（BBH）是 Big-Bench 数据集中最难子集，专门筛选出**标准提示（standard prompting）下大模型表现低于人类水平**的任务，共 23 个任务。

**BBH 与 Big-Bench 的区别**：Big-Bench 整体包含 150+ 任务，其中许多任务大模型已经能做得很好。BBH 刻意选出了那些模型仍然困难的子集，聚焦于模型能力的上限。

**典型任务示例**：
- **目标物跟踪**（Tracking Shuffled Objects）：模型需要追踪打乱顺序后的物体位置
- **语义解析**（Dyck Languages）：判断括号字符串是否合法
- **多步推理**（Multi-step Arithmetic）：多步数值计算

**BBH 的面试价值**：BBH 是检验模型**真正推理能力**而非记忆能力的重要标尺。如果候选人在项目中声称使用了" GPT-4 级别"的模型，BBH 可以作为验证其实际能力的参考。

### 2.3 HumanEval 与 LiveCodeBench：代码能力评测

代码生成能力是当前 LLM 评测中最具实际价值的维度之一。

**HumanEval** 由 OpenAI 在 2021 年提出，包含 164 道由人工编写的编程题，每道题附带单元测试。评测指标为 **Pass@1**，即模型一次生成的代码能否通过测试。

**MBPP**（Mostly Basic Python Problems）补充了 HumanEval，包含 974 道题目，覆盖更基础的 Python 编程任务。

**LiveCodeBench**（2024）是更严格的代码评测集，做了两项关键改进：
- **无污染设计**：题目来自 GitHub LeetCode 等平台近两年的真实题目，模型训练数据不太可能覆盖
- **时间跨度**：从 2023 年 8 月至今持续更新，避免数据泄露导致的虚假高分

**面试关注点**：当你评估一个代码生成场景的模型选型时，LiveCodeBench 比 HumanEval 更有参考价值。同时要注意，代码评测考察的是**可运行的代码**生成能力，不是代码解释或文档生成。

### 2.4 GSM8K 与 MATH：数学推理能力

数学推理是检验 LLM 逻辑链完整性的最佳标尺。

**GSM8K**（Grade School Math 8K）：包含 8,500 道小学数学应用题，难度大约在小学高年级水平。题目要求多步推理，但不需要复杂的数学知识。主流大模型的 Pass@1 已在 90% 以上。

**MATH**：来自 AMC 10/12、AIME 等竞赛的 12,500 道数学题，难度更高，需要更复杂的推理链。顶级模型在此的准确率仍在 50-70% 区间，远低于饱和水平。

**两者关系**：GSM8K 接近"会做"的水平，MATH 接近"做得好"的水平。如果候选人说模型"数学能力很强"，追问" MATH 多少分"可以迅速定位其真实水平。

### 2.5 Chatbot Arena：人类偏好评估

**Chatbot Arena**（lmarena.ai）由 UC Berkeley LMSYS 组织发起，采用**众包人类盲评**的方式：对两个模型的回答进行匿名对比，人类评估员选择更优的一个。

**Elo 评分系统**：类似国际象棋elo_rating的系统，模型之间的对战结果不断更新评分，最终得到一个全局的排名分数。这个分数比任何单一基准都更接近"用户体验"层面的模型质量。

**Arena 的局限性**：
- 人类评估员的专业背景参差不齐
- 评估成本高、更新频率有限
- 不同语言、文化背景的评估员对"好"的理解存在差异

**面试应用**：当你被问到" GPT-4 和 Claude 哪个更好"时，Chatbot Arena 的最新排名是一个有说服力的参考维度，但也要说明其局限性。

## 三、提示工程：从技巧到系统方法论

### 3.1 为什么提示工程如此重要

面试中常听到这样的说法："模型都差不多，关键看 Prompt。"这句话有失偏颇，但也在一定程度上反映了现实：**不同提示词技巧可以让同一模型在同一任务上的表现差距达到 30 个百分点以上**。

举一个经典例子：让 GPT-3.5 计算"把 298 乘以 47"，直接提问的准确率只有约 15%，但加上 Chain-of-Thought（CoT）引导后，准确率可以提升到超过 70%。这种巨大的差异正是提示工程价值的体现。

### 3.2 核心提示技巧详解

**技巧一：Few-Shot Prompting（少样本提示）**

当零样本（zero-shot）无法达到期望效果时，给出 1-5 个示例通常是最有效的改进手段。

```
示例：
用户问：把以下句子改写成正式邮件：
"你好，我明天有事，不能来了"

好的提示词设计：
把以下句子改写成正式商务邮件，语气得体、礼貌。

示例1：
输入："嘿老王，帮我把那个文件发过来"
输出："尊敬的王先生，烦请将相关文件发送至我的邮箱，谢谢。"

示例2：
输入："这事儿不行啊"
输出："很遗憾，经评估，该方案暂不符合我们的要求。"

现在请改写：
输入："你好，我明天有事，不能来了"
输出：
```

**Few-Shot 的关键细节**：
- 示例数量通常 2-5 个效果最好，过多反而可能稀释核心模式
- 示例应覆盖常见的边缘情况
- 示例的格式和风格要与目标任务一致

**技巧二：Chain-of-Thought Prompting（思维链提示）**

通过引导模型"先思考再回答"，显著提升复杂推理任务的准确率。

**零样本 CoT**（不需要示例）：在问题后加上" Let's think step by step."，模型自动生成推理过程。

**少样本 CoT**（需要示例）：示例中展示完整的推理步骤。

```
问题：小明有 5 个苹果，小红给了他 3 个，然后小明吃掉了 2 个。
小明现在有几个苹果？

推理过程：
1. 小明开始有 5 个苹果
2. 小红给了他 3 个 → 5 + 3 = 8 个
3. 小明吃掉了 2 个 → 8 - 2 = 6 个

最终答案：6 个
```

**面试追问**：CoT 对哪些任务效果最明显？答案是一切需要**多步推理**的任务——数学计算、逻辑推理、代码调试、多跳问答。简单的事实问答几乎看不到提升。

**技巧三：角色扮演提示（Role Prompting）**

明确赋予模型一个专业角色，可以有效提升回答的相关性和专业深度。

```
不推荐：介绍一下 Spring IoC 容器
推荐：你是一位有 10 年经验的 Java 架构师，请从设计模式和源码层面深入介绍 Spring IoC 容器的工作原理，包括 BeanFactory 与 ApplicationContext 的区别、依赖注入的三种方式以及循环依赖的处理机制。
```

**注意**：角色提示的效果取决于任务类型。对于通用知识问答，角色设定帮助不大；对于专业领域咨询（法律、医疗、技术架构），角色设定能显著提升回答质量。

**技巧四：结构化输出（Structured Output）**

当需要模型输出机器可读的格式时，直接要求 JSON 或特定结构比后期解析更可靠。

```json
{
  "instruction": "你是一个情感分析模型。请分析以下评论的情感倾向，以JSON格式返回结果。",
  "input": "这家餐厅的菜品味道一般，但服务态度很好，价格也公道。",
  "output_format": "请严格按以下JSON格式返回：\n{\"sentiment\": \"positive|neutral|negative\", \"score\": <-1到1之间的浮点数>, \"reasons\": [\"原因1\", \"原因2\"]}",
  "response_format": "JSON"
}
```

**技术细节**：部分模型（如 GPT-4、Claude）原生支持 `response_format: { type: "json_object" }` 参数，可以严格约束输出格式，减少格式错误率。

### 3.3 提示工程中的常见误区

**误区一：提示词越长越好**

实践中，过长的系统提示会稀释关键指令，模型的注意力机制对"中间部分"的关注度相对较低。最有效的提示词往往简洁有力，核心指令不超过 200 个词。

**误区二：一次性写好提示词**

提示工程是一个迭代过程。推荐的工作流：
1. 写下初始提示词
2. 在几个典型用例上测试
3. 分析失败案例
4. 针对失败案例添加约束或示例
5. 重复步骤 2-4

**误区三：忽略模型的上下文窗口限制**

GPT-4o 的上下文窗口为 128K tokens，看起来很大，但当提示词、系统提示、few-shot 示例加起来超过窗口的 80% 时，模型在长序列末尾的"遗忘"问题会显著影响输出质量。合理的做法是将上下文控制在窗口的 50-60% 以内。

## 四、企业级评测与提示管理的工程实践

### 4.1 模型评测的工程化流程

在企业中，模型评测不是一次性活动，而是一个持续的质量保障环节。一个典型的工程化评测流程包含以下步骤：

**第一步：确定评测目标矩阵**

不同业务场景对模型能力的要求权重不同。以客服 AI 为例：

| 能力维度 | 权重 | 评测方法 |
|----------|------|----------|
| 回答准确性 | 40% | 人工抽检 |
| 语气友好度 | 20% | 人类偏好评估 |
| 响应速度 | 20% | 自动化计时 |
| 安全合规性 | 20% | 红队测试 + 自动化检测 |

**第二步：建立评测数据集**

评测集的质量直接决定评测结果的可信度。好的评测集应满足：
- **覆盖性**：涵盖业务场景的主要 case 类型
- **代表性**：case 分布与真实用户请求分布一致
- **无泄露**：评测集不应出现在训练数据中

**第三步：设计 A/B 对比实验**

当你在多个候选模型中做选择时，A/B 对比是最可靠的方式。在同一批测试用例上运行不同模型，控制随机性，收集客观指标和主观评分。

### 4.2 提示词的版本管理与回滚

在生产环境中，提示词也是代码，需要版本化管理：

```
prompts/
├── v1.0/
│   ├── system_prompt.md
│   └── user_template.md
├── v1.1/
│   ├── system_prompt.md  # 修复了角色定义模糊的问题
│   └── user_template.md
└── production.yaml       # 当前生产环境使用的版本
```

提示词变更需要和代码变更一样经过 Code Review。同时，每次变更应附带评测结果对比：修改后的提示词在核心指标上是否有提升？

### 4.3 提示词安全的边界控制

面试中常被问到的一个实际场景："用户输入恶意 Prompt，试图诱导模型泄露系统指令，怎么处理？"

**三层防护体系**：
1. **输入层**：检测用户输入中是否包含对系统 Prompt 的引用或注入尝试（如" Ignore previous instructions..."）
2. **输出层**：对模型输出进行合规性检查，敏感信息过滤
3. **行为层**：监控异常使用模式，如短时间内大量 API 调用

## 五、面试高频问题与标准回答

**Q1：如何全面评估一个大语言模型？**

评估 LLM 需要从三个维度入手：通用能力（用 MMLU、BBH 等基准）、垂直任务能力（用对应领域的 Task Benchmark）以及人类偏好（用 Chatbot Arena 等）。此外，还要结合具体业务场景设计定制化评测集。单一基准无法反映模型全貌。

**Q2：MMLU 和 MATH 的区别是什么？**

MMLU 衡量知识广度，题目以选择题为主；MATH 衡量推理深度，题目来自数学竞赛，需要完整的多步推导。一个 GPT-4 级别的模型在 MMLU 上可达 86% 以上，但在 MATH 上通常不超过 70%，反映了两类能力的本质差异。

**Q3：Chain-of-Thought 在什么场景下效果最明显？**

CoT 对需要多步推理的任务效果最显著，包括数学计算、逻辑推理、代码调试、多跳问答等。对简单的事实问答或情感分类等任务，CoT 带来的提升非常有限，甚至可能增加不必要的 token 消耗。

**Q4：提示工程和模型微调应该如何选择？**

当任务模式复杂、用例多样，且可以通过提示词引导时，优先尝试提示工程（成本低、见效快）。当任务需要模型内化特定领域知识或行为模式，且有充足的标注数据时，可以考虑微调。现实中两者往往配合使用：先通过提示工程快速验证任务可行性，再考虑微调做长期优化。
