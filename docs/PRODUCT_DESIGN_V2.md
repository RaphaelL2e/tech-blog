# Tech Blog V2 产品与优化设计

> 状态：P0 已实现并通过首发验收
>
> 设计对象：`tech-blog`（Hugo + GitHub Pages）
>
> 更新时间：2026-07-20

### P0 实现验证（2026-07-20）

- `hugo --gc --minify` 通过：254 篇文章、296 个 HTML 页面，公开产物 19.74 MiB。
- `python3 scripts/normalize_content.py --check` 与 `python3 scripts/validate_build.py` 全部通过。
- 360 px 真实浏览器全量巡检 254/254 通过：页面均为 200、单 H1、无全局横向滚动、无运行时错误或资源 404。
- Lighthouse 移动端：首页 Performance / Accessibility / Best Practices / SEO 均为 100；代表长文分别为 99 / 100 / 100 / 100。
- 公开产物无求职简历 PDF、手机号或明文密码；财富页保持 `noindex,nofollow` 且不在主导航。

## 1. 结论先行

V2 不应只是一次视觉换肤，而应把当前“按时间堆叠文章的静态博客”升级为：

**个人技术品牌首页 + 系列化知识库 + 面向工程师的高质量阅读器。**

核心价值从“我写过哪些文章”转变为：

1. 新访客能在 10 秒内知道作者擅长什么、这里能解决什么问题。
2. 有明确目标的读者能通过搜索、主题和系列快速找到内容。
3. 长文读者能顺畅阅读、定位章节，并自然进入同系列下一篇。
4. 招聘方、同行和潜在合作方能看到作者的工程能力、项目和持续输出。
5. 站点继续保持 Hugo 静态部署的低成本、可维护和高可靠特性。

品牌命名已经确认：

- 英文名：**Raphael Lab**
- 中文名：**飞哥 AI 实验室**
- 一句话定位：**记录后端工程、分布式系统与 AI 工程化的真实实践。**

品牌使用规则：

- 导航 Logo、英文渠道和代码仓库统一使用 `Raphael Lab`。
- 首页中文品牌说明、中文内容渠道和对外传播统一使用“飞哥 AI 实验室”。
- 浏览器标题使用“Raphael Lab｜飞哥 AI 实验室”。
- “记录后端工程、分布式系统与 AI 工程化的真实实践”作为定位文案，不作为品牌名称的一部分。
- 旧名称“Raphael Tech Blog”和“飞哥的 AI 折腾日记”不再作为站点主品牌，避免多套名称并存。

## 2. 当前产品审计

### 2.1 已有资产

- 254 篇已发布文章，时间跨度为 2018—2026 年。
- 2026 年新增内容集中在 Java、Spring、分布式系统、微服务、数据库、操作系统、网络与 AI。
- 已经形成多个 5—30 篇的系列，具备建设学习路径的基础。
- 有独立关于页、简历、GitHub 入口和一个加密财富页面。
- Hugo 静态生成、GitHub Actions 自动部署，发布成本低。

### 2.2 核心问题与证据

| 维度 | 当前情况 | 影响 | 优先级 |
| --- | --- | --- | --- |
| 首页 | 一次渲染全部 254 篇文章，页面约 5.2 万 px 高 | 首屏没有定位，浏览成本极高 | P0 |
| 移动端 | 文章页在 390 px 宽度出现横向溢出，长英文标题被截断 | 核心阅读场景不可用 | P0 |
| 阅读页 | 无目录、阅读进度、系列上下篇、相关文章 | 1.7 万 px 长文难以导航，读完即离站 | P0 |
| 内容结构 | 29 个分类、644 个标签，存在 Java/JAVA/java 等重复 | 分类失去导航意义，标签成为噪音 | P0 |
| 构建产物 | `public/` 约 296 MB，其中标签页约 276 MB | 部署、抓取和维护成本异常 | P0 |
| 模板路由 | 每个标签详情 URL 实际渲染完整标签云 | 615 个标签目录大量复制约 460 KB 页面 | P0 |
| 元数据 | 242/254 篇缺少 description，250/254 篇缺少 keywords | 搜索摘要和分享卡片质量差 | P0 |
| 标题语义 | 197 篇正文以 H1 开头，134 篇与模板标题完全重复 | 页面出现两个主标题，SEO 与阅读层级混乱 | P0 |
| SEO | 缺少 description、canonical、Open Graph、JSON-LD、robots 和 404 | 搜索收录、分享和错误体验不完整 | P0 |
| 品牌 | 站名、作者名、AI 日记、Java 面试内容并存 | 用户无法形成清晰记忆 | P0 |
| 关于页 | 作为独立静态 HTML 维护，样式与主站重复；隐私字段在源码可见 | 维护分叉且有隐私风险 | P0 |
| 导航 | 文章、分类、标签并列，财富页公开出现在主导航 | 信息架构由技术实现而非用户任务驱动 | P0 |
| 视觉 | 全站单一深色卡片、所有内容同等权重 | 层级弱，像模板站，长文阅读疲劳 | P1 |
| 可发现性 | 无站内搜索、筛选、系列入口 | 254 篇内容无法形成资产复利 | P0 |

### 2.3 根因判断

目前的问题不是文章少，而是**内容规模已经超过“简单博客模板”的承载上限**：

- 时间流适合十几篇文章，不适合 254 篇知识内容。
- 标签能辅助检索，但 644 个标签无法承担导航。
- 长文依赖目录和上下文，单列正文只能完成“展示”。
- 视觉变化无法解决定位、查找和内容关系缺失。

## 3. 产品定位

### 3.1 目标用户

#### A. 有具体问题的工程师

- 任务：快速找到一个技术概念、故障分析或实现方案。
- 入口：搜索引擎、站内搜索、主题页。
- 成功标准：30 秒内找到目标章节或相关系列。

#### B. 准备面试或系统学习的开发者

- 任务：按顺序学习 Java、Spring、分布式系统等完整主题。
- 入口：系列页、学习路径。
- 成功标准：清楚知道总篇数、当前位置、前后顺序和推荐下一篇。

#### C. 评估作者能力的人

- 任务：了解作者的专业方向、工程经验、项目与输出质量。
- 入口：首页、项目页、关于页。
- 成功标准：两次点击内看到代表作、项目案例和联系入口。

#### D. 回访读者

- 任务：查看最近更新或继续未读系列。
- 入口：首页最近更新、RSS、系列页。
- 成功标准：无需翻长列表即可看到新增内容。

### 3.2 核心产品指标

V2 首版不追求复杂增长系统，先关注可用性和内容消费：

- 首页到文章/系列的点击率。
- 站内搜索使用率与无结果率。
- 文章目录点击率。
- 同系列下一篇点击率。
- 文章阅读深度（25% / 50% / 90%）。
- 项目页与关于页访问率。
- 移动端横向溢出、404 和构建错误数量。

## 4. 信息架构

### 4.1 主导航

桌面端：

`首页 / 专题 / 文章库 / 项目 / 关于 / 搜索 / 主题切换`

移动端：

`Logo / 搜索 / 菜单`

具体调整：

- “分类”升级为“专题”，只保留少量稳定领域。
- “标签”退出主导航，仅在文章页和搜索筛选中使用。
- “财富”退出公开主导航，作为独立私有产品处理。
- “项目”成为一级入口，用案例证明工程能力。
- 搜索成为全局高频操作，不隐藏在文章列表内。

### 4.2 站点地图

```text
/
├── /series/                  专题与学习路径
│   ├── /series/java/
│   ├── /series/distributed-systems/

│   └── /projects/:slug/
```

### 4.3 顶层主题建议

分类控制在 5—7 个，建议首版使用：

1. Java 与 Spring
2. 分布式与微服务
3. 数据库与中间件
4. 系统设计与工程效能
5. AI 工程化
6. 计算机基础
7. 历史归档

标签仅用于细粒度筛选，每篇控制在 3—5 个；统一大小写和别名。

## 5. 页面产品设计

## 5.1 首页

首页不再是文章全集，而是品牌与内容分发页。

### 页面结构

```text
┌──────────────────────────────────────────────────────┐
│ Raphael Lab        专题  文章库  项目  关于   搜索 ◐ │
├──────────────────────────────────────────────────────┤
│ 飞哥 AI 实验室                                       │
│ 后端工程、分布式系统与 AI 工程化的实践笔记           │
│ 不只解释“是什么”，更关注真实系统里“为什么这样做”。   │
│ [搜索 254 篇文章…]        [浏览专题] [了解作者]       │
│ 254 篇文章 · 7 个主题 · 持续更新                      │
├──────────────────────────────────────────────────────┤
│ 推荐学习路径                                           │
│ [Java/Spring] [分布式系统] [数据库] [AI 工程化]       │
├──────────────────────────────────────────────────────┤
│ 精选文章                         最近更新              │
│ 3 篇编辑精选                      6 篇时间流            │
├──────────────────────────────────────────────────────┤
│ 工程项目 / 案例                                       │
│ [Claude-Go] [平台后端] [AI 工程化实验]                │
├──────────────────────────────────────────────────────┤
│ 关于作者 + GitHub / RSS / 联系                        │
└──────────────────────────────────────────────────────┘
```

### 规则

- 最近更新最多展示 6 篇，精选最多展示 3 篇。
- 学习路径展示文章数、难度、状态和一句话收益。
- 首页标题只出现一次，浏览器标题为“Raphael Lab｜飞哥 AI 实验室”。
- 首屏必须在 10 秒内回答“谁写的、写什么、我能做什么”。

## 5.2 专题与系列页

系列是 V2 的核心内容产品，不只是分类列表。

每个系列页包含：

- 系列简介与适合人群。
- 学习前置要求。
- 总篇数、预计阅读时间、难度。
- 分章节目录，而非简单按日期排序。
- 每篇状态：已发布 / 即将发布 / 已更新。
- 一篇“先从这里开始”的推荐入口。
- 同主题相关系列。

系列顺序通过明确字段维护，不能依赖发布日期或文件名猜测。

## 5.3 文章库

目标是“查找”，不是“从上往下翻”。

### 桌面布局

- 顶部搜索框。
- 左侧或顶部筛选：主题、系列、难度、年份。
- 结果卡片：标题、描述、主题、系列位置、更新时间、预计阅读时长。
- 默认按最近更新排序。
- 分页每页 12—18 篇，URL 可分享。

### 移动布局

- 搜索固定在顶部内容区。
- 筛选使用抽屉或横向 chips。
- 每页 10—12 篇。
- 不使用无限滚动，保留可定位、可返回的分页。

## 5.4 文章阅读页

### 桌面结构

```text
┌───────────────┬───────────────────────────┬──────────┐
│ 系列位置/返回  │ 标题、摘要、元数据         │ 本文目录 │
│                │ 正文（约 720px）          │ 阅读进度 │
│                │ 代码、表格、引用           │          │
│                │ 上一篇 / 下一篇            │          │
│                │ 相关文章 / 作者卡           │          │
└───────────────┴───────────────────────────┴──────────┘
```

### 必备功能

- 只由模板输出一个 H1，正文从 H2 开始。
- 面包屑：专题 / 系列 / 当前文章。
- 摘要、发布日期、最后更新、阅读时长。
- 桌面端粘性目录；移动端折叠目录。
- 顶部细线阅读进度。
- 标题锚点与“复制章节链接”。
- 代码块语言标签、复制按钮、横向滚动。
- 宽表格使用局部横向滚动，不撑破页面。
- 系列上一篇/下一篇。
- 2—4 篇规则驱动的相关文章。
- 文末反馈：内容是否有帮助、发现错误可去 GitHub 提交 issue。

### 内容可信度

旧技术文章显示状态：

- `持续维护`
- `最后验证：YYYY-MM`
- `历史归档，部分内容可能过时`

这样既保留搜索价值，也避免把 2019 年内容当成当前最佳实践。

## 5.5 项目页

项目页不是仓库链接列表，而是简短案例库。

每个项目包含：

- 问题背景。
- 我的角色与约束。
- 关键架构或技术决策。
- 最难的问题与解决方式。
- 结果与可量化改进。
- 代码仓库、演示或相关文章。

首版只选 3—5 个公开且最能代表能力的项目，不追求数量。

## 5.6 关于页

关于页改为 Hugo 内容页，复用主站布局和组件。

建议结构：

- 80—120 字个人定位。
- 核心能力与当前关注方向。
- 代表项目和精选文章。
- 精简经历，详细版本通过简历提供。
- GitHub、邮箱、RSS 等联系入口。

隐私要求：

- 手机号等敏感字段不得以明文写入公开 HTML。
- 下载密码不得直接显示在页面源码中。
- 公开简历与求职简历分离，公开版先做脱敏。

## 5.7 搜索

搜索必须支持中文标题与正文，保持静态站点架构。

首版需求：

- 搜索标题、描述、正文小标题和正文。
- 结果高亮关键词并显示命中片段。
- 支持键盘快捷键 `/` 或 `⌘K`。
- 支持按主题与系列二次筛选。
- 记录无结果关键词，用于补内容或增加别名。
- 私有页面、标签噪音页和归档附件不进入索引。

具体搜索实现库在开发阶段通过中文分词效果、索引体积和 GitHub Pages 兼容性再定，不在产品阶段锁死。

## 6. 内容模型

### 6.1 新文章 Front Matter

```yaml
---
title: "文章标题"
slug: "stable-english-slug"
date: 2026-07-20T10:00:00+08:00
updated: 2026-07-20T10:00:00+08:00
description: "80—120 字、可独立理解的文章摘要"
topic: "distributed-systems"
series: "distributed-interview"
series_order: 7
level: "intermediate"
status: "maintained"
featured: false
tags: ["调度", "工作流", "幂等"]
draft: false
---
```

### 6.2 字段规则

- `title`、`date`、`description`、`topic` 必填。
- `description` 必须可作为搜索摘要和分享描述直接使用。
- `topic` 只能从固定集合选择。
- `series` 可空，但有值时必须同时提供 `series_order`。
- `tags` 最多 5 个，使用统一词表。
- 正文不得重复写文章 H1。
- 对公开 URL 使用稳定 slug，新文章不再生成超长中文编码 URL。
- 已被搜索引擎收录的旧 URL 不批量更换；如调整必须保留重定向映射。

### 6.3 存量内容治理

分三批处理：

1. **结构修复**：移除重复 H1、补 description、归一分类和标签。
2. **系列编排**：为面试系列、设计模式和系统基础建立顺序。
3. **可信度治理**：对旧文标记归档或复核状态，优先更新仍有流量价值的内容。

建议建立内容校验脚本，在构建前检查必填字段、重复 slug、非法主题、标签数量和重复 H1。

## 7. 视觉与交互方向

### 7.1 已确认的视觉基准

以 2026-07-20 提供的“一周热点”界面为 V2 视觉基准，采用**深色编辑台 + 信息仪表盘**方向：

- 深黑蓝背景，顶部带非常克制的青绿色环境光，不使用大面积纯黑。
- 首屏使用“小状态点 + 辅助信息 + 大标题 + 简介”的编辑式层级。
- 标题关键词使用薄荷青到天蓝的渐变，形成品牌记忆点。
- 筛选器使用低对比胶囊按钮，选中态只增加边框与轻微底色。
- 搜索框与结果数量放在同一控制条，保持工具感而非营销感。
- 内容库在宽屏使用三列卡片；卡片内部固定为“主题标签 / 标题 / 摘要 / 元信息”。
- 卡片使用蓝灰色表面、细边框和极弱阴影，只在 hover 时轻微提亮，不做强发光。
- 青蓝色是主品牌色；粉色只用于少量异常、热度或提醒状态，不进入日常正文链接。

这套风格主要用于首页、文章库、专题页和项目页。文章详情页沿用相同色彩与导航，但减少卡片感、加宽留白，以保证长文耐读。

### 7.2 设计关键词

**深色、清晰、克制、工程感、可信、耐读。**

保留参考图的深色科技感，但避免“每块内容都加发光边框”的模板化处理。工程感来自信息结构、排版和细节，不来自大量霓虹效果。

### 7.3 色彩

- **默认深色主题**，与确认的参考界面保持一致。
- 页面背景使用带蓝绿色偏色的深黑，内容表面使用深蓝灰，不使用纯黑叠纯灰。
- 主色为薄荷青与理性蓝，用于标题渐变、链接、焦点与主要动作。
- 粉色仅用于热度或异常状态；成功状态使用薄荷绿。
- 浅色主题作为 P1 可选增强，不阻塞 V2 首发。
- 正文对比度满足 WCAG AA。

建议设计令牌：

| 用途 | 建议值 |
| --- | --- |
| 页面背景 | `#080B11` |
| 顶部环境光 | `rgba(35, 186, 151, 0.10)` |
| 卡片表面 | `#121722` |
| 卡片悬停 | `#161D2A` |
| 细边框 | `rgba(132, 151, 183, 0.16)` |
| 主文字 | `#F1F4F8` |
| 次文字 | `#98A2B3` |
| 薄荷青 | `#63D8C2` |
| 天蓝 | `#6FA8FF` |
| 提醒粉 | `#F062B7` |

最终实现时再通过对比度测试微调具体数值，视觉关系以本表为准。

### 7.4 排版

- 中文正文优先系统中文字体，避免依赖无法稳定加载的远程字体。
- 英文与数字可使用本地或自托管的现代无衬线字体。
- 代码使用等宽字体。
- 正文字号：桌面 17—18 px，移动端 16—17 px。
- 正文行高：1.75—1.9。
- 正文行宽：约 680—760 px。
- 页面最大宽度：约 1180—1240 px，用于容纳正文与目录。

### 7.5 组件风格

- 首页和文章库使用参考图式卡片；文章正文减少全包围容器，更多使用留白与细分隔线。
- 卡片只用于真正可点击的独立对象，整张卡片均可点击。
- 圆角控制在 8—14 px，不让所有元素胶囊化。
- 标题最多显示两行，摘要最多显示三行；卡片底部元信息始终对齐。
- 默认不位移卡片，hover 仅改变边框、背景和极弱阴影，避免页面漂浮感。
- 动效 150—220 ms，尊重 `prefers-reduced-motion`。
- 键盘焦点必须清晰可见。

### 7.6 响应式断点

- `< 640px`：单列、折叠菜单、折叠目录。
- `640—1023px`：文章卡片两列；正文单列，目录改为正文顶部。
- `>= 1024px`：文章卡片三列；正文 + 右侧粘性目录。

最低验收宽度为 360 px，任何页面不得产生全局横向滚动。

## 8. SEO、性能、安全与可访问性

### 8.1 SEO

每个公开页面必须包含：

- 唯一且不重复的 `<title>`。
- meta description。
- canonical URL。
- Open Graph 与分享图。
- Article / Person / WebSite JSON-LD。
- sitemap、RSS、robots.txt、404 页面。
- 正确的 H1—H3 层级。
- 旧 URL 的兼容或重定向策略。

标签索引策略：

- 低质量、单篇和重复标签页不进入索引。
- 搜索结果页与私有页面 `noindex`。
- 优先让专题页和高质量文章成为搜索入口。

### 8.2 性能预算

- 首页只渲染精选与最近文章，不超过 12 个文章卡片。
- CSS 从 887 行内联模板拆成可缓存静态资源。
- JavaScript 按功能加载，非核心功能延迟执行。
- 图片必须有尺寸、懒加载与现代格式。
- 公开构建产物从约 296 MB 降至 30 MB 以内。
- 标签产物不再出现数百个约 460 KB 的重复页面。
- 移动端 Lighthouse 目标：Performance ≥ 90，Accessibility / SEO ≥ 95。
- Web Vitals 目标：LCP ≤ 2.5s，CLS ≤ 0.1，INP ≤ 200ms。

### 8.3 安全与隐私

- 财富页移出主导航并设置 `noindex, nofollow`。
- 评估是否需要继续部署财富页；仅靠“URL 不公开”不构成权限控制。
- 继续保证解密只在本地完成，不在日志或分析工具中记录敏感值。
- 关于页与简历移除公开源码中的敏感字段和明文密码。
- 外部链接使用安全的 `rel` 属性。
- 建立基本 Content Security Policy 与 Referrer Policy。

### 8.4 可访问性

- 跳到正文链接。
- 语义化 header/nav/main/article/footer。
- 所有交互支持键盘。
- 弹窗具备焦点管理、Esc 关闭和可读名称；更推荐直接使用可链接页面而非标签弹窗。
- 颜色不是唯一状态表达。
- 代码复制后有屏幕阅读器可感知的反馈。

## 12. 关键取舍

1. **先做好查找与阅读，再做社交功能。** 评论、收藏、账号系统不是当前主要瓶颈。
2. **系列优先于标签。** 系列能够建立顺序和学习承诺，标签只能辅助检索。
3. **首页是品牌页，不是数据库导出页。** 全量内容交给文章库和搜索。
4. **保留静态架构。** 当前规模不需要引入后端和数据库。
5. **旧内容不删除，但要标记可信度。** 归档比假装永不过时更专业。
6. **公开产品与私有产品分离。** 财富功能不应决定技术博客的主导航和公开信息架构。
